# A1. Big-O Complexity

**학습 목표**: *"시간 복잡도가 뭔가요?"* 와 *"이 코드 복잡도는요?"* 에 다이어그램 그리며 5분 답할 수 있다.

## 1. 비유 — 전화번호부에서 이름 찾기
- **O(1)**: 펼치자마자 첫 장에 이름이 있음 (해시맵 인덱스 직행).
- **O(log n)**: 중간을 펴고 앞/뒤 절반씩 버려가며 찾음 (이진 탐색).
- **O(n)**: 첫 장부터 한 장씩 넘기며 찾음 (순차 탐색).
- **O(n log n)**: 전화번호부 전체를 "정렬"하는 효율적 방법 (병합/퀵 정렬).
- **O(n²)**: 모든 사람과 모든 사람을 한 번씩 악수시키기 (이중 루프).

## 2. 개념 정의 (1줄)
> **시간 복잡도** = 입력 크기 `n`이 커질 때 **연산 횟수가 늘어나는 추세**를 함수로 표현한 것. 실제 초 단위 시간이 아니라 **증가율**에 관심.
> **공간 복잡도** = 입력 크기 `n`에 따라 **추가로 필요한 메모리**의 증가율. (입력 자체를 저장하는 공간은 보통 제외)

## 3. 다이어그램 — 증가율 비교
```
연산 횟수
  ▲
  │                                    ● O(n²)
  │                              ●
  │                        ●
  │                  ●              ● O(n log n)
  │            ●
  │        ●                    ● O(n)
  │     ●
  │  ●●●●●●●●●●●●●●●●●●●●●●●●●●● O(log n)
  │──────────────────────────────  O(1)
  └───────────────────────────────► 입력 크기 n
```
n이 커질수록 그래프 간 격차가 폭발적으로 벌어진다 — 이것이 Big-O가 상수를 무시하고도 실무에서 유효한 이유.

## 4. 대표 복잡도 표
| 복잡도 | 이름 | 예시 | n=1,000,000일 때 체감 |
|--------|------|------|------------------------|
| O(1) | 상수 | 배열 인덱스 접근, 해시맵 조회 | 즉시 |
| O(log n) | 로그 | 이진 탐색, 균형 트리 탐색 | ~20번 연산 |
| O(n) | 선형 | 배열 순회, 선형 탐색 | 100만 번 |
| O(n log n) | 선형로그 | 병합/힙/퀵(평균) 정렬 | ~2000만 번 |
| O(n²) | 제곱 | 버블/삽입 정렬, 이중 루프 | 1조 번 — 사실상 못 씀 |

## 5. Best / Average / Worst 구분
```java
int linearSearch(int[] arr, int target) {
    for (int i = 0; i < arr.length; i++) {
        if (arr[i] == target) return i;   // best: 첫 원소면 O(1)
    }
    return -1;                             // worst: 없으면 O(n)
}
```
- **Best case**: 가장 운 좋은 입력 (면접에서 언급 빈도 낮음, 보통 의미 없음).
- **Average case**: 일반적 분포에서 기대값 (퀵 정렬이 "평균 O(n log n)"이라 불리는 이유).
- **Worst case**: 가장 나쁜 입력 (면접/실무에서 **가장 중요** — SLA·타임아웃은 최악을 기준으로 설계).

## 6. 왜 상수를 무시하나
> `3n + 100` 도, `n` 도 결국 **O(n)**. n이 충분히 커지면 상수배 차이(3배)나 상수항(100)은 **증가율 앞에서 무의미**해지기 때문.
- Big-O는 "정확한 실행 시간"이 아니라 "**n이 두 배가 되면 얼마나 느려지는가**"를 예측하는 도구.
- 단, 실무에서 n이 작으면 상수항이 실제 성능을 좌우할 수 있음 (예: 삽입 정렬이 작은 배열에서 퀵 정렬보다 빠름 → Java `Arrays.sort`가 작은 구간에 삽입 정렬 혼용하는 이유, A5 참고).

## 7. 핵심 포인트 (자주 하는 실수)
- 🔴 "Big-O = 실제 실행 시간" ❌ — **증가율**이지 초 단위 시간이 아님. O(n)이 O(log n)보다 항상 느리다는 보장도 (작은 n에서는) 없음.
- 🔴 "반복문 2개면 무조건 O(n²)" ❌ — 중첩 루프라도 안쪽 루프 범위가 줄어들면 O(n²)이 아닐 수 있음 (예: 투 포인터).
- 🟡 재귀 복잡도는 **점화식**으로 분석 (예: 병합 정렬 `T(n) = 2T(n/2) + O(n)` → O(n log n)).
- 🟡 공간 복잡도 계산 시 **재귀 호출 스택**도 포함해야 함 (재귀 깊이 n이면 O(n) 공간).

## 8. 예상 면접 질문 + 답변 골격
**Q1. "시간 복잡도가 뭔가요?"**
> 입력 크기 n에 따라 연산 횟수가 늘어나는 **증가율**. 실제 초 단위 시간이 아니라 n이 커질 때의 추세를 봄. 상수배·상수항은 n이 충분히 크면 무의미해지므로 무시.

**Q2. "왜 상수를 무시하나요?"**
> `3n+100`도 결국 O(n). n이 커질수록 상수배 차이는 증가율 앞에서 미미해짐. 다만 n이 작을 때는 상수항이 실제 성능을 좌우할 수 있어 실무에선 하이브리드 알고리즘(팀소트 등)을 쓰기도 함.

**Q3. "퀵 정렬이 O(n log n)인데 최악은 왜 O(n²)인가요?"**
> 평균은 피벗이 데이터를 균등 분할해 O(n log n), 최악은 이미 정렬된 배열에서 매번 가장 작은/큰 값을 피벗으로 뽑으면 분할이 1:(n-1)로 치우쳐 O(n²) (A5 참고).

**Q4. "O(1) 공간복잡도가 뭔가요?"**
> 입력 크기와 무관하게 **일정한 추가 메모리**만 사용. 예: 두 변수만 써서 배열 최댓값을 찾는 순회. 반대로 입력 크기만큼 새 배열/재귀 스택을 쓰면 O(n).

# A2. Array and Linked List

**학습 목표**: *"배열과 연결 리스트 차이는요?"* 와 *"언제 뭘 쓰나요?"* 에 5분 답할 수 있다.

## 1. 비유 — 극장 좌석 vs 보물찾기 쪽지
- **배열** = 번호 붙은 극장 좌석. 몇 번 좌석인지 알면 **바로 찾아감(O(1))**. 대신 중간에 좌석을 끼워 넣으려면 뒷사람 전부 한 칸씩 밀어야 함.
- **연결 리스트** = 보물찾기 쪽지 릴레이. "다음 힌트는 여기"라며 다음 위치(포인터)만 앎. 중간에 쪽지 하나 추가/제거는 앞뒤만 연결 바꾸면 끝. 대신 5번째 쪽지를 보려면 1→2→3→4→5 **순서대로 따라가야만(O(n))** 함.

## 2. 개념 정의
> **Array**: 메모리에 **연속된 블록**으로 할당, 인덱스로 주소를 즉시 계산(`base + i*size`).
> **Linked List**: 각 노드가 **데이터 + 다음 노드 주소(포인터)** 를 가짐, 메모리 여기저기 흩어져 있음.

## 3. 다이어그램 — 메모리 배치
```
Array (연속 메모리)
 주소: 100   104   108   112   116
       ┌────┬────┬────┬────┬────┐
       │ 10 │ 20 │ 30 │ 40 │ 50 │
       └────┴────┴────┴────┴────┘
 arr[2] → base(100) + 2*4 = 108번지로 즉시 점프 (O(1))

Linked List (흩어진 메모리, 포인터로 연결)
 주소 240          주소 88            주소 512
 ┌────┬─────┐    ┌────┬─────┐    ┌────┬──────┐
 │ 10 │ →88 │───►│ 20 │→512 │───►│ 30 │ →null │
 └────┴─────┘    └────┴─────┘    └────┴──────┘
 3번째 원소 보려면 head부터 포인터 2번 따라가야 함 (O(n))
```

## 4. 접근/삽입/삭제 비교표
| 연산 | Array | Linked List (단일) |
|------|-------|---------------------|
| 인덱스 접근 (`get(i)`) | **O(1)** | O(n) |
| 맨 앞 삽입/삭제 | O(n) (전부 밀림) | **O(1)** |
| 맨 뒤 삽입/삭제 | O(1) (amortized, 공간 있을 때) | O(1) (tail 포인터 있으면) / O(n)(없으면) |
| 중간 삽입/삭제 | O(n) (밀기) | O(n) (탐색) + O(1) (연결 변경) |
| 메모리 오버헤드 | 낮음 (데이터만) | 높음 (포인터 추가 저장) |
| 캐시 지역성 | **좋음** | 나쁨 |

## 5. 코드 — Java에서의 대응
```java
List<Integer> arrList = new ArrayList<>();   // 내부: Object[] 배열
List<Integer> linkedList = new LinkedList<>(); // 내부: 이중 연결 리스트(Node)

arrList.get(500);      // O(1) - 배열 인덱스 계산
linkedList.get(500);   // O(n) - head부터 500번 순회 (자주 하는 실수 지점!)

arrList.add(0, 1);     // O(n) - 뒤 원소 전부 시프트
linkedList.add(0, 1);  // O(1) - head 포인터만 갱신
```

## 6. 캐시 지역성(Cache Locality) — 실제 성능 차이의 핵심 ⭐
> **cross-link**: CPU는 메모리를 읽을 때 한 워드가 아니라 **캐시 라인 단위(보통 64B)**로 통째로 가져온다.
- **배열**: 연속 메모리라 `arr[i]`를 읽으면 `arr[i+1]~arr[i+15]`(int 기준)까지 **캐시에 같이 적재**됨 → 순회 시 캐시 히트율 매우 높음.
- **연결 리스트**: 노드가 메모리 곳곳에 흩어져 있어 다음 노드를 읽을 때마다 **캐시 미스** 가능성이 높고, 메인 메모리 접근(캐시 대비 수십~수백 배 느림)이 잦음.
- 그 결과 **이론상 둘 다 O(n) 순회**여도, 실제로는 배열 순회가 연결 리스트보다 **훨씬 빠른 경우가 많음** — Big-O만으로 성능을 100% 예측할 수 없는 대표 사례.

## 7. 핵심 포인트 (자주 하는 실수)
- 🔴 `LinkedList.get(i)`를 배열처럼 O(1)이라 착각 — 내부적으로 head/tail부터 순회하는 **O(n)**.
- 🔴 "ArrayList 삽입은 항상 O(1)" ❌ — 맨 뒤 추가는 amortized O(1)이지만, 배열이 꽉 차면 **새 배열로 전체 복사(O(n))** 발생. 중간 삽입은 O(n).
- 🟡 순회가 잦으면 ArrayList, **빈번한 앞쪽 삽입/삭제**(큐처럼 사용)면 LinkedList나 `ArrayDeque` 고려.
- 🟡 실무에서는 캐시 지역성 때문에 **연결 리스트보다 배열/ArrayList가 실제로 더 빠른 경우가 많음** — 이론적 O(n)이 같아도 상수 계수 차이가 큼.

## 8. 패턴 정리표 — 언제 뭘 쓰나
| 상황 | 선택 |
|------|------|
| 인덱스로 랜덤 접근 잦음 | Array / ArrayList |
| 맨 앞/뒤 삽입·삭제 잦음 (큐/스택) | `ArrayDeque` (Java에서 LinkedList보다 권장) |
| 크기가 자주 바뀌고 중간 삽입 많음 | LinkedList (단, 실제 성능은 프로파일링 필요) |
| 캐시 성능이 중요한 대량 순회 | Array / ArrayList |

## 9. 예상 면접 질문 + 답변 골격
**Q1. "배열과 연결 리스트 차이는?"**
> 배열은 연속 메모리라 인덱스 접근 O(1)이지만 중간 삽입/삭제가 O(n). 연결 리스트는 포인터로 연결돼 삽입/삭제는 O(1)(위치를 안다면)이지만 접근이 O(n). 캐시 지역성은 배열이 압도적으로 유리.

**Q2. "ArrayList와 LinkedList 중 뭘 써야 하나요?"**
> 대부분의 경우 접근이 잦고 캐시 지역성이 좋은 ArrayList가 실무에서 더 빠름. 앞쪽 삽입/삭제가 매우 잦은 큐 용도라면 `ArrayDeque`를 우선 고려 (Java 공식 문서도 Stack/LinkedList보다 ArrayDeque 권장).

**Q3. "캐시 지역성이 뭐고 왜 중요한가요?"**
> CPU가 캐시 라인 단위로 메모리를 읽어오는 특성. 배열은 연속 메모리라 한 번 읽을 때 인접 데이터도 캐시에 올라와 히트율이 높음. 연결 리스트는 노드가 흩어져 있어 캐시 미스가 잦아, 같은 O(n)이어도 실제 체감 속도 차이가 큼.

# A3. Hash Table

**학습 목표**: *"해시 충돌은 어떻게 해결하나요?"* 와 *"Java HashMap 내부 구조는요?"* 에 다이어그램 그리며 5분 답할 수 있다.

## 1. 비유 — 사물함 배정 시스템
학생 이름을 받아 **특정 규칙(해시 함수)**으로 사물함 번호를 계산해 바로 배정. "김철수" → 계산 → 17번 사물함. 번호를 알면 바로 찾아감(O(1)). 문제는 서로 다른 두 학생이 같은 규칙으로 **같은 번호**를 받을 때(충돌) — 이때 한 사물함에 두 명을 어떻게 넣을지가 관건.

## 2. 개념 정의
> **Hash Table** = 키를 **해시 함수**로 정수(버킷 인덱스)로 변환해, 배열 기반 저장소에 O(1) 평균으로 접근하는 자료구조.
> **해시 함수**: 임의 크기 입력을 고정 크기 정수(해시코드)로 매핑. 좋은 해시 함수는 **결과가 버킷 전체에 고르게 분포**해야 함(충돌 최소화).

## 3. 다이어그램 — 체이닝 vs 개방 주소법
```
체이닝 (Separate Chaining)
 bucket[0] → null
 bucket[1] → ("apple",1) → ("grape",5) → null   ← 충돌 시 연결리스트로 이어붙임
 bucket[2] → ("banana",2) → null
 bucket[3] → null

개방 주소법 (Open Addressing, Linear Probing)
 bucket[0] │ empty
 bucket[1] │ ("apple",1)
 bucket[2] │ ("banana",2)
 bucket[3] │ ("grape",5)  ← 원래 1번을 원했지만 충돌 → 다음 빈 칸(2,3...) 탐색해 저장
```

## 4. 충돌 해결 비교표
| 방식 | 원리 | 장점 | 단점 |
|------|------|------|------|
| **체이닝 (Chaining)** | 같은 버킷을 연결리스트/트리로 연결 | 구현 간단, 로드팩터 1 넘어도 동작 | 포인터 오버헤드, 캐시 지역성 낮음 |
| **개방 주소법 (Open Addressing)** | 충돌 시 다른 빈 슬롯을 탐사(linear/quadratic/double hashing) | 캐시 지역성 좋음, 포인터 불필요 | 로드팩터가 높아지면 성능 급락, 삭제가 까다로움(tombstone 필요) |

## 5. 로드팩터 (Load Factor)
> `load factor = 저장된 요소 수 / 버킷(배열) 크기`
- 로드팩터가 높을수록 충돌 확률 증가 → 성능 저하.
- **Java HashMap 기본 로드팩터 = 0.75**. 이를 넘으면 **버킷 배열을 2배로 늘리고 전체 재해싱(resize)**.
- 로드팩터를 낮게 잡으면 충돌은 줄지만 메모리 낭비 ↑ (공간-시간 트레이드오프).

## 6. 시간 복잡도
| 연산 | 평균 | 최악 |
|------|------|------|
| 조회/삽입/삭제 | **O(1)** | O(n) |

- **최악 O(n)**: 모든 키가 같은 버킷으로 몰리는 **해시 충돌 폭탄**(악의적 입력 또는 나쁜 해시 함수) 발생 시, 체이닝된 연결리스트를 처음부터 끝까지 훑어야 함.

## 7. Java HashMap 내부 구조 — cross-link ⭐
> Java `HashMap`은 **체이닝 기반**. 각 버킷이 기본적으로 연결리스트지만:
- `hashCode()` 값을 **추가 해싱(perturbation)** 처리해 분포를 고르게 함.
- 한 버킷의 체인 길이가 **8 이상 & 전체 버킷 수 64 이상**이면 그 버킷을 연결리스트 → **레드-블랙 트리로 트리화(Treeification)** → 최악 시간이 O(n)에서 O(log n)으로 개선 (A4의 균형 트리 개념과 연결).
- `equals()`와 `hashCode()`를 **함께 재정의**해야 함 — hashCode만 같고 equals가 다르면 잘못된 곳에서 못 찾고, equals만 재정의하면 다른 버킷에 들어가 애초에 비교 자체가 안 일어남.

## 8. 핵심 포인트 (자주 하는 실수)
- 🔴 "HashMap은 항상 O(1)" ❌ — **평균**은 O(1), 최악은 O(n)(트리화 이후엔 O(log n)으로 완화).
- 🔴 `hashCode()`만 재정의하고 `equals()` 안 하기 (또는 반대) — 계약(contract) 위반으로 `Map`/`Set`에서 중복 제거나 조회가 깨짐.
- 🟡 **가변 객체를 키로 사용** 금지 — 저장 후 필드를 바꾸면 `hashCode()`가 달라져 다시 못 찾음.
- 🟡 초기 용량을 예상 크기에 맞춰 지정하면 **resize(재해싱) 비용**을 줄일 수 있음 (`new HashMap<>(expectedSize)`).

## 9. 패턴 정리표 — 체이닝 vs 개방 주소법 선택
| 상황 | 선택 |
|------|------|
| 삭제가 잦음 | 체이닝 (개방 주소법은 삭제 시 tombstone 관리 필요) |
| 메모리보다 속도(캐시 지역성) 중요 | 개방 주소법 |
| 로드팩터를 낮게 유지 어려움 | 체이닝 (로드팩터 1 넘어도 동작 가능) |
| Java 실무 기본값 | HashMap = 체이닝 + 트리화 |

## 10. 예상 면접 질문 + 답변 골격
**Q1. "해시 충돌이 뭐고 어떻게 해결하나요?"**
> 서로 다른 키가 같은 버킷 인덱스로 매핑되는 현상. 체이닝(같은 버킷을 연결리스트로 연결)과 개방 주소법(충돌 시 다른 빈 슬롯 탐사) 두 가지 접근이 있고, Java HashMap은 체이닝 기반.

**Q2. "Java HashMap 내부 구조는요?"**
> 배열 + 체이닝. `hashCode()`를 추가 해싱해 버킷 인덱스를 정함. 로드팩터 0.75 초과 시 버킷을 2배로 늘려 재해싱. 한 버킷 체인이 8개 이상이고 전체 버킷 64개 이상이면 레드-블랙 트리로 트리화해 최악 O(n)을 O(log n)으로 완화.

**Q3. "로드팩터가 뭐고 왜 0.75인가요?"**
> 저장된 요소 수/버킷 크기. 너무 높으면 충돌 잦아 성능↓, 너무 낮으면 메모리 낭비. 0.75는 시간-공간 트레이드오프의 실증적 절충값.

**Q4. "hashCode와 equals를 왜 같이 재정의해야 하나요?"**
> HashMap은 먼저 hashCode로 버킷을 찾고, 그 버킷 안에서 equals로 실제 키를 비교함. 한쪽만 재정의하면 "논리적으로 같은 객체인데 다른 버킷에 들어가거나(hashCode만 다름)", "같은 버킷인데 equals가 틀려 못 찾는" 문제가 생김.

# A4. Tree and BST

**학습 목표**: *"이진탐색트리가 뭐고 왜 불균형이 문제인가요?"* 와 *"AVL/Red-Black이 뭘 해결하나요?"* 에 5분 답할 수 있다.

## 1. 비유 — 족보(계보도) vs 한 줄로 늘어선 가훈
- **이진탐색트리(BST)**: 도서관에서 책을 "왼쪽엔 더 작은 제목, 오른쪽엔 더 큰 제목"으로 계속 나눠 정리 → 찾을 때마다 절반씩 후보를 버림.
- **불균형 트리**: 하필 책이 가나다 순으로 하나씩 꽂혀서 트리가 **오른쪽으로만 죽 늘어진 상태** → 결국 그냥 일렬로 늘어선 리스트와 다를 게 없음.
- **균형 트리(AVL/Red-Black)**: 사서가 책을 꽂을 때마다 **계속 서가를 다시 정돈**해서 어느 쪽으로도 너무 길게 치우치지 않도록 유지.

## 2. 개념 정의
> **BST(Binary Search Tree)**: 각 노드에 대해 **왼쪽 서브트리 < 노드 값 < 오른쪽 서브트리**가 성립하는 이진 트리. 탐색 시 매번 한쪽 서브트리를 통째로 배제.
> **균형 트리**: 삽입/삭제 시 **트리 높이를 O(log n)으로 유지**하도록 자동으로 재조정(rotation)하는 BST의 개선판.

## 3. 다이어그램 — 균형 vs 불균형
```
균형 BST (높이 ≈ log n)              불균형 BST (사실상 리스트, 높이 = n)
          8                          1
        /   \                          \
       4     12                         2
      / \    / \                          \
     2   6  10  14                         3
                                              \
    탐색: O(log n)                             4
                                                  \
                                                  ... (높이 n)
                                        탐색: O(n) — BST의 이점 상실!
```

## 4. 불균형 문제 — 왜 발생하나
- 삽입 순서가 **이미 정렬된 순서**(1,2,3,4,5...)이면 BST는 매번 한쪽으로만 자식을 붙여 **연결 리스트 모양**이 됨.
- 결과: 탐색/삽입/삭제가 이론상 O(log n)이어야 하는데 **실제로는 O(n)** 으로 퇴화 — Big-O 분석 시 "평균"과 "최악"을 구분해야 하는 이유(A1).

## 5. 균형 트리 해법
| 트리 | 균형 유지 방법 | 특징 |
|------|----------------|------|
| **AVL Tree** | 왼쪽/오른쪽 서브트리 높이 차이가 **항상 1 이하**가 되도록 회전(rotation) | 균형이 매우 엄격 → 탐색 빠름, 삽입/삭제 시 회전 잦아 쓰기 비용 ↑ |
| **Red-Black Tree** | 노드에 red/black 색을 칠하고 5가지 규칙으로 **"높이가 최대 2배 이상 차이 안 남"** 정도만 보장 (느슨한 균형) | 회전 횟수 적음 → 삽입/삭제 빠름, 탐색은 AVL보다 살짝 느릴 수 있음 |

## 6. 시간 복잡도 비교표
| 연산 | 불균형 BST (최악) | 균형 BST (AVL/RB) |
|------|---------------------|----------------------|
| 탐색 | O(n) | **O(log n)** |
| 삽입 | O(n) | **O(log n)** |
| 삭제 | O(n) | **O(log n)** |

## 7. 코드 — BST 탐색 기본형
```java
boolean search(Node root, int target) {
    if (root == null) return false;
    if (target == root.value) return true;
    return target < root.value
        ? search(root.left, target)
        : search(root.right, target);
}
```
> 균형이 깨지면 이 재귀가 실질적으로 **한쪽 방향으로만 계속 타고 내려가는 O(n) 순회**가 됨.

## 8. cross-link — Java TreeMap과 DB 인덱스 ⭐
- **Java `TreeMap` / `TreeSet`**: 내부적으로 **Red-Black Tree**로 구현. 키 정렬 순회, `floorKey`/`ceilingKey` 등 범위 탐색이 O(log n). (A3의 HashMap 트리화도 Red-Black Tree를 재사용)
- **DB 인덱스 (B-Tree / B+Tree)**: 개념적으로 BST의 확장. 다만 노드 하나가 자식을 2개가 아니라 **수백 개**를 가져 트리 높이를 극단적으로 낮춤 → 디스크 I/O(가장 비싼 연산) 횟수를 최소화. "왜 인덱스 조회가 빠른가"의 근거가 이 균형 트리 구조.

## 9. 핵심 포인트 (자주 하는 실수)
- 🔴 "BST면 탐색이 O(log n)" ❌ — **균형이 깨지면 O(n)**. 정렬된 순서로 삽입하면 최악의 케이스.
- 🔴 "AVL과 Red-Black이 같다" ❌ — AVL은 엄격 균형(읽기 최적), RB는 느슨한 균형(쓰기 최적). 트레이드오프가 다름.
- 🟡 실무에서 **TreeMap/TreeSet은 자동으로 균형을 유지**하므로 사용자가 직접 회전을 신경 쓸 필요 없음.
- 🟡 DB 인덱스가 이진 트리가 아니라 **B/B+Tree(다진 트리)** 인 이유: 디스크 블록 단위 I/O를 최소화하려고 분기 수를 늘림.

## 10. 패턴 정리표 — 언제 뭘 쓰나
| 상황 | 선택 |
|------|------|
| 정렬된 순서로 순회/범위 탐색 필요 | TreeMap/TreeSet (Red-Black) |
| 순서 무관, 최고 속도의 조회만 필요 | HashMap/HashSet (A3) |
| 읽기가 압도적으로 많고 쓰기 적음 | AVL (더 엄격한 균형 → 탐색 유리) |
| 삽입/삭제가 빈번 | Red-Black (회전 적어 쓰기 유리) |
| 디스크 기반 대용량 데이터(DB) | B-Tree/B+Tree |

## 11. 예상 면접 질문 + 답변 골격
**Q1. "이진탐색트리가 뭔가요?"**
> 왼쪽 서브트리는 노드 값보다 작고 오른쪽은 큰 이진 트리. 탐색 시 매번 절반씩 후보를 배제해 균형 잡혀 있으면 O(log n).

**Q2. "BST의 불균형 문제가 뭔가요?"**
> 삽입 순서에 따라(예: 정렬된 순서로 삽입) 트리가 한쪽으로 치우쳐 연결 리스트처럼 퇴화, 탐색이 O(log n)이 아니라 O(n)이 됨.

**Q3. "AVL과 Red-Black 차이는요?"**
> AVL은 좌우 높이차 1 이하로 엄격히 유지해 탐색이 빠르지만 회전이 잦아 쓰기 비용이 큼. Red-Black은 느슨한 균형(색 규칙)으로 회전을 줄여 삽입/삭제가 빠름. Java TreeMap은 Red-Black 채택.

**Q4. "DB 인덱스는 BST와 뭐가 다른가요?"**
> 개념은 같지만 B-Tree/B+Tree는 노드당 자식 수가 훨씬 많은 다진 트리. 디스크 I/O가 메모리 접근보다 훨씬 비싸므로 트리 높이를 낮춰 I/O 횟수를 줄이는 게 목적.

# A5. Sorting

**학습 목표**: *"정렬 알고리즘들을 비교해주세요"* 와 *"퀵 정렬 최악이 왜 O(n²)인가요?"* 에 5분 답할 수 있다.

## 1. 비유 — 카드 정리 방식
- **버블 정렬**: 인접한 두 카드를 계속 비교해 큰 걸 뒤로 밀어냄 — 가장 굼뜬 방식.
- **삽입 정렬**: 카드 한 장씩 뽑아 이미 정렬된 손패의 알맞은 위치에 끼워 넣음 — 손패가 적을 땐 빠름.
- **병합 정렬**: 카드 더미를 반씩 계속 쪼갠 뒤(전문가 여럿에게 분배), 각자 정렬한 걸 **합치며 병합**.
- **퀵 정렬**: 기준 카드(피벗) 하나를 정하고 그보다 작은/큰 카드로 좌우를 나눈 뒤 재귀.
- **힙 정렬**: 카드를 "가장 큰 카드가 항상 맨 위로 오는" 특수 더미(힙)로 만들고 하나씩 꺼냄.

## 2. 대표 정렬 비교표 ⭐
| 알고리즘 | 평균 복잡도 | 최악 복잡도 | 안정성(Stable) | In-place |
|----------|-------------|-------------|:---:|:---:|
| 버블 정렬 (Bubble) | O(n²) | O(n²) | ✅ | ✅ |
| 삽입 정렬 (Insertion) | O(n²) | O(n²) | ✅ | ✅ |
| 병합 정렬 (Merge) | O(n log n) | **O(n log n)** | ✅ | ❌ (추가 배열 필요) |
| 퀵 정렬 (Quick) | O(n log n) | **O(n²)** | ❌ | ✅ |
| 힙 정렬 (Heap) | O(n log n) | O(n log n) | ❌ | ✅ |

> **안정성(Stable)**: 값이 같은 원소들의 **원래 상대 순서가 정렬 후에도 유지**되는지. (예: "이름순 정렬 후 나이순 정렬"을 두 번 하면, 이름순이 유지되려면 나이순 정렬이 안정적이어야 함)

## 3. 다이어그램 — 병합 정렬 vs 퀵 정렬 분할 방식
```
병합 정렬 (분할은 항상 절반, 병합 단계에서 비교)
[5 3 8 1] → [5 3] [8 1] → [5][3] [8][1] → [3 5] [1 8] → [1 3 5 8]
             절반씩 쪼갬                     정렬하며 병합(추가 공간 필요)

퀵 정렬 (피벗 기준 분할, 분할 단계에서 비교)
[5 3 8 1], pivot=5
 → [3 1] 5 [8]   (5보다 작은 것 | 피벗 | 큰 것)
 → 재귀적으로 좌우를 각각 정렬 (제자리에서, in-place)
```

## 4. 퀵 정렬 최악 O(n²) — 왜 발생하나
```java
// 항상 마지막 원소를 피벗으로 고르는 구현
int partition(int[] arr, int lo, int hi) {
    int pivot = arr[hi];
    ...
}
```
- 이미 **정렬된 배열**(또는 역순 배열)에 대해 항상 마지막(혹은 첫) 원소를 피벗으로 고르면, 매번 **1개 : n-1개**로 극단적으로 치우친 분할이 발생.
- 분할이 균등하지 않으면 재귀 깊이가 O(log n)이 아니라 **O(n)**이 되고, 각 깊이마다 O(n) 비교 → 총 **O(n²)**.
- **완화책**: 피벗을 랜덤 선택하거나, "3-median(처음/중간/끝 중 중앙값)" 방식으로 선택 → 최악 케이스 발생 확률을 극히 낮춤 (여전히 이론적 최악은 존재).

## 5. Java `Arrays.sort` — cross-link ⭐
> Java의 `Arrays.sort()`는 데이터 타입에 따라 **내부 구현이 다르다**:
- **기본형(primitive: int, long, double...) 배열**: **듀얼 피벗 퀵 정렬(Dual-Pivot Quicksort)** 사용. 기본형은 `equals`/동일성 비교가 의미 없어 **안정성이 필요 없으므로** 빠르고 in-place인 퀵 정렬 계열을 채택.
- **객체(Object) 배열, `List`**: **팀소트(TimSort)** 사용 (병합 정렬 + 삽입 정렬의 하이브리드). 객체는 정렬 기준 외의 필드가 다를 수 있어(예: 이름순 정렬 시 나이 필드) **안정성이 중요**하므로 stable한 팀소트 채택.
```java
int[] primArr = {5, 3, 8, 1};
Arrays.sort(primArr);              // Dual-Pivot Quicksort (unstable, 하지만 int라 상관없음)

Integer[] objArr = {5, 3, 8, 1};
Arrays.sort(objArr);               // TimSort (stable)

List<Person> people = ...;
Collections.sort(people, ...);     // TimSort (stable) - Comparator 체이닝 시 필수
```

## 6. 핵심 포인트 (자주 하는 실수)
- 🔴 "퀵 정렬은 항상 O(n log n)" ❌ — **평균**만 그렇고, 나쁜 피벗 선택 시 **최악 O(n²)**.
- 🔴 "정렬 알고리즘은 다 안정적" ❌ — 퀵/힙 정렬은 **불안정(unstable)**. 다중 조건 정렬(1차 이름순 유지하며 2차 나이순 정렬)에는 안정 정렬(병합/삽입/TimSort)이 필요.
- 🟡 Java가 기본형엔 퀵 정렬 계열, 객체엔 TimSort를 쓰는 이유를 **"안정성이 필요한가"** 기준으로 설명할 수 있어야 함.
- 🟡 TimSort는 작은 구간(보통 32~64개 미만)에서는 **삽입 정렬로 전환** — 작은 n에서는 상수항이 유리하기 때문(A1의 "왜 상수를 무시하나"의 실무 반례).

## 7. 패턴 정리표 — 언제 뭘 쓰나
| 상황 | 선택 |
|------|------|
| 안정성 필요 (다중 키 정렬) | 병합 정렬 / TimSort |
| 추가 메모리 여유 없음(in-place 필수) | 퀵 정렬 / 힙 정렬 |
| 최악 케이스도 보장돼야 함(실시간 시스템) | 병합 정렬 / 힙 정렬 (퀵 정렬 최악 O(n²) 회피) |
| 거의 정렬된 데이터, 소규모 | 삽입 정렬 |
| Java 기본형 배열 정렬 | Arrays.sort → 자동으로 Dual-Pivot Quicksort |
| Java 객체/컬렉션 정렬 | Arrays.sort/Collections.sort → 자동으로 TimSort |

## 8. 예상 면접 질문 + 답변 골격
**Q1. "정렬 알고리즘들을 비교해주세요."**
> O(n²) 계열(버블/삽입, 단순하지만 작은 데이터에 유리)과 O(n log n) 계열(병합/퀵/힙)로 나뉨. 병합은 안정적이지만 추가 공간 필요, 퀵은 in-place에 평균 빠르지만 불안정+최악 O(n²), 힙은 in-place에 최악도 O(n log n)이지만 불안정.

**Q2. "퀵 정렬 최악이 왜 O(n²)인가요?"**
> 피벗이 매번 최소/최대값으로만 뽑히면(예: 정렬된 배열 + 고정 위치 피벗) 분할이 1:(n-1)로 치우쳐 재귀 깊이가 n이 됨. 각 깊이 O(n) 비교로 총 O(n²). 랜덤/median-of-three 피벗 선택으로 확률적으로 회피.

**Q3. "안정 정렬이 왜 중요한가요?"**
> 여러 키로 순차 정렬할 때(1차 이름순 정렬 후 2차 나이순 정렬) 안정 정렬이어야 나이가 같은 사람들 사이에서 이름순이 유지됨. 불안정 정렬(퀵/힙)은 이 순서를 보장하지 않음.

**Q4. "Java Arrays.sort는 어떤 알고리즘을 쓰나요?"**
> 기본형 배열은 Dual-Pivot Quicksort(안정성 불필요, 빠른 in-place), 객체 배열/List는 TimSort(병합+삽입 하이브리드, stable). 데이터 타입에 따라 요구사항이 달라 구현을 분리한 것.
