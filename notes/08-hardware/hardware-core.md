# H1. CPU and Pipeline

**학습 목표**: *"CPU가 명령어를 어떻게 실행하나요?"* 와 *"파이프라이닝이 뭐고 왜 빠른가요?"* 에 5분 답할 수 있다.

## 1. 비유 — 세탁소 컨베이어 벨트
빨래 한 벌을 "세탁 → 건조 → 개기" 3단계로 처리한다고 하자. 한 벌 끝난 뒤 다음 벌을 시작하면 굼뜨다. 대신 1번 옷이 건조 중일 때 2번 옷을 세탁 시작하면, 세 사람이 동시에 서로 다른 옷을 처리하며 벨트가 쉬지 않는다. CPU의 **파이프라이닝**이 이 컨베이어 벨트다 — 명령어 하나가 끝나길 기다리지 않고 여러 명령어를 단계별로 겹쳐 처리한다.

## 2. 개념 정의
> **CPU 실행 사이클** = **Fetch(가져오기) → Decode(해독) → Execute(실행)** (세분화하면 +Memory Access +Write Back). 이 사이클이 **클럭(clock)** 신호에 맞춰 반복된다.
> **클럭 속도(GHz)** = 1초에 몇 번 사이클을 도는가. 클럭이 높다고 무조건 빠른 건 아님(사이클당 처리량·코어 수·아키텍처가 함께 작용).
> **파이프라이닝** = 여러 명령어의 FDE 단계를 겹쳐(overlap) 처리해, 사이클마다 명령어 하나씩 완료되는 것처럼 만드는 기법(처리량↑, 개별 명령어 지연시간 자체는 그대로).

## 3. 다이어그램 — 파이프라인 없음 vs 있음
```
[파이프라이닝 없음] 명령어 하나 다 끝나야 다음 시작
 I1: F D E
 I2:       F D E
 I3:             F D E
 → 3개 명령어 = 9 사이클

[파이프라이닝] 단계를 겹쳐 처리 (5단계 예시: F D E M W)
 사이클:   1  2  3  4  5  6  7
 I1:       F  D  E  M  W
 I2:          F  D  E  M  W
 I3:             F  D  E  M  W
 → 사이클마다 명령어 1개씩 완료(steady state) → 처리량(throughput) 대폭 상승
```

## 4. 파이프라인 해저드 (간단히)
겹쳐 실행하다 보니 충돌이 생길 수 있다. 면접에서 "이름만 알면" 충분한 수준.
| 해저드 종류 | 원인 | 예시 |
|-------------|------|------|
| **구조적(Structural)** | 하드웨어 자원 충돌 | 같은 메모리 포트를 두 단계가 동시에 씀 |
| **데이터(Data)** | 이전 명령어 결과를 다음 명령어가 아직 못 받음 | `a=b+c` 다음 바로 `d=a+1` |
| **제어(Control)** | 분기(`if`) 방향이 확정 전이라 다음 fetch 대상이 불확실 | `if` 문 이후 명령어 |
> 해결책은 스톨(stall/버블 삽입), forwarding(우회 전달), 분기 예측(branch prediction) 등 — 상세 구현은 인터뷰에서 깊게 안 물어보는 편이니 "이런 문제가 있고 이렇게 완화한다" 정도만 알면 충분.

## 5. 핵심 포인트 (자주 하는 실수)
- 🔴 "클럭이 높으면 무조건 빠르다" ❌ — IPC(사이클당 명령어 수), 코어 수, 캐시 구조가 함께 성능을 결정.
- 🔴 "파이프라이닝하면 명령어 1개 지연시간이 줄어든다" ❌ — 줄어드는 건 **처리량**(throughput). 개별 명령어가 끝나는 데 걸리는 총 단계 수(latency)는 그대로거나 오히려 늘 수 있음.
- 🟡 분기(`if`/`for`) 예측이 틀리면 파이프라인을 비우고 다시 채워야 함(분기 예측 실패 = 성능 손실의 흔한 원인) — 알고리즘 문제 풀 때 예측 가능한 분기(정렬된 배열 순회 등)가 왜 더 빠른지의 배경.

## 6. 정리표 — CPU 실행 흐름
| 단계 | 하는 일 |
|------|---------|
| Fetch | 메모리(캐시)에서 다음 명령어를 가져옴 |
| Decode | 명령어 해독, 필요한 레지스터/제어 신호 준비 |
| Execute | ALU 연산 수행 |
| Memory Access | 필요 시 메모리 read/write |
| Write Back | 결과를 레지스터에 기록 |

## 7. 예상 면접 질문 + 답변 골격
**Q1. "CPU가 명령어를 실행하는 과정을 설명해주세요."**
> Fetch(명령어 가져오기) → Decode(해독) → Execute(연산) → (Memory Access) → (Write Back). 클럭 신호에 맞춰 이 사이클을 반복.

**Q2. "파이프라이닝이 뭐고 왜 빠른가요?"**
> 여러 명령어의 단계를 겹쳐 처리해, 개별 명령어 지연시간은 그대로여도 사이클마다 명령어가 하나씩 완료되도록 만들어 전체 처리량을 높이는 기법. 컨베이어 벨트처럼 단계별로 동시 작업.

**꼬리 Q2-1. "파이프라인 해저드 때문에 항상 이상적으로 빨라지진 않죠? 대표 예를 들어보세요."**
> 네. ① 데이터 해저드 — 앞 명령 결과를 다음 명령이 바로 써야 하는데 아직 안 나옴(포워딩/스톨로 대응). ② 컨트롤 해저드 — 분기 결과가 정해지기 전에 다음 명령을 어디서 가져올지 몰라 파이프라인이 비어버림. ③ 구조 해저드 — 같은 하드웨어 자원을 동시에 요구. 이들 때문에 실제 처리량은 이상값보다 낮습니다.

**꼬리 Q2-2. "컨트롤 해저드를 줄이려고 CPU가 쓰는 기법은?"**
> 분기 예측(branch prediction)입니다. 과거 분기 패턴을 기록해 "이 분기는 taken일 것"이라 예측하고 미리 그 경로 명령을 파이프라인에 채웁니다. 맞으면 손해가 없고, 틀리면 채운 걸 버리고(flush) 올바른 경로로 다시 채우는 페널티가 있죠. 그래서 예측 가능한 분기가 많은 코드가 더 빠릅니다(정렬된 배열 조건문이 빠른 유명한 예).

**Q3. "파이프라인 해저드가 뭔가요?"**
> 겹쳐 실행하는 과정에서 자원 충돌(구조적), 이전 결과 미확정(데이터), 분기 방향 불확실(제어) 때문에 다음 단계 진행이 막히는 상황. 스톨/forwarding/분기 예측으로 완화.

## 8. 심화 — 해저드 상세와 분기 예측(branch prediction) ⭐

### 8-1. 데이터 해저드 — forwarding으로 스톨을 줄이는 원리
```
I1: ADD R1, R2, R3   ; R1 = R2 + R3   (E 단계에서 R1 결과가 나옴)
I2: SUB R4, R1, R5   ; R4 = R1 - R5   ; R1이 필요한데 아직 Write Back 전!

스톨(버블) 없이 그냥 실행하면:
 I1:  F  D  E  M  W
 I2:     F  D  E  M  W     ← D 단계에서 R1을 읽으려 하지만 I1의 W는 아직 안 끝남 → 잘못된 값 위험

forwarding(우회 전달) 적용:
 I1:  F  D  E  M  W
 I2:     F  D  E  M  W     ← I1의 E 단계 결과를 레지스터 파일 대신 "직접" I2의 E 단계 입력으로 배선(bypass)
                              → 스톨 없이 바로 다음 사이클에 사용 가능
```
- forwarding(=bypassing)으로 해결 안 되는 경우(대표: load-use hazard — 바로 앞 명령어가 메모리에서 읽어온 값을 곧바로 쓰는 경우)는 **1사이클 스톨**이 불가피. 컴파일러가 명령어 순서를 재배치(instruction scheduling)해 그 사이에 무관한 명령어를 끼워 넣어 스톨을 감추기도 함.

### 8-2. 제어 해저드 — 분기 예측(branch prediction)과 실패 비용
분기(`if`, `for`, `while`)는 조건이 Execute 단계 즈음에야 확정되는데, Fetch는 그보다 훨씬 먼저 "다음 명령어"를 가져와야 한다. 그래서 CPU는 **분기 방향을 미리 예측**하고 그 예측대로 fetch를 계속 진행한다(speculative execution).

| 항목 | 내용 |
|------|------|
| 정적 예측(static) | "분기는 항상 미실행(not-taken)"처럼 고정 규칙 |
| 동적 예측(dynamic) | 과거 실행 이력(branch history table)을 참고해 예측 — 최근 반복 패턴 학습 |
| 예측 성공 | 파이프라인이 끊기지 않고 그대로 흘러감 — 페널티 없음 |
| **예측 실패(misprediction)** | 잘못 fetch한 명령어들을 전부 **버리고(flush)**, 정확한 target부터 파이프라인을 **다시 채워야** 함 → 파이프라인 깊이만큼 사이클 손실 |

```
분기 예측 실패 시:
 I1(if): F  D  E  M  W
 I2(잘못 예측된 다음 명령어): F  D  E  X  X   ← E에서 분기 결과 확정, 틀렸음을 발견 → 버림(flush)
 I3(진짜 다음 명령어):                F  D  E  M  W   ← 여기서부터 다시 시작 (파이프라인 재충전)
 → 버려진 단계 수만큼 "사이클 손실" = 분기 예측 실패 비용(misprediction penalty)
```
- 현대 CPU는 파이프라인이 깊어서(10~20단계) 예측 실패 페널티가 크다 — 그래서 분기 예측기 정확도가 성능에 직결.
- 알고리즘 문제에서 "정렬된 배열을 순회하며 조건 분기"가 "정렬 안 된 배열"보다 빠른 경우가 있는 이유 = 분기 패턴이 반복/예측 가능해지면 분기 예측 성공률↑ → 파이프라인 flush가 줄어듦(branch prediction-friendly access pattern).

### 8-3. 핵심 포인트 (자주 하는 실수)
- 🔴 "forwarding이 있으면 데이터 해저드로 인한 스톨은 전혀 없다" ❌ — load-use hazard처럼 forwarding만으로 못 막는 경우 1사이클 스톨은 여전히 필요.
- 🔴 "분기 예측 실패는 결과가 틀리게 나온다" ❌ — 결과는 항상 정확함(틀린 speculative 실행분은 커밋 전에 버림). 손해는 **성능**(사이클 손실)뿐.
- 🟡 파이프라인이 깊을수록(=단계가 많을수록) 처리량 이득은 커지지만, 분기 예측 실패 시 버려야 할 단계도 많아져 misprediction penalty도 커짐 — 트레이드오프.

### 꼬리 질문(follow-up)
- Q. "분기 예측 실패 비용을 줄이기 위해 소프트웨어 레벨에서 할 수 있는 게 있나요?" → (답변 골격) 분기를 줄이거나(branchless code), 조건이 실행마다 잘 안 바뀌게 패턴을 고정하거나(예측기가 학습하기 쉬운 형태), 정렬을 통해 분기 방향을 일정하게 만드는 방법 등이 있다.
- Q. "load-use hazard가 forwarding으로 안 풀리는 이유는?" → (답변 골격) load 명령어의 결과값 자체가 Memory Access 단계에서야 나오는데, forwarding은 Execute 단계 결과를 다음 Execute로 넘기는 배선이라 그 시점에 값이 아직 준비 안 됐기 때문 — 그래서 1사이클 스톨이 구조적으로 필요.

# H2. Memory Hierarchy and Cache

**학습 목표**: *"메모리 계층 구조를 설명해주세요"* 와 *"캐시가 왜 필요한가요?"* 에 5분 답할 수 있다.

## 1. 비유 — 책상 위, 서랍, 창고, 지하 보관소
자주 쓰는 펜은 **책상 위**(레지스터)에 둔다. 오늘 쓸 서류는 **서랍**(캐시)에. 이번 달 서류는 **사무실 캐비닛**(RAM). 작년 서류는 **회사 창고**(SSD). 10년 전 서류는 **외부 문서 보관소**(HDD) — 가지러 가는데 오래 걸리지만 저렴하게 대량 보관 가능. 자주 쓸수록 가까이, 덜 쓸수록 멀리 두는 게 전체 비용 대비 최선이다.

## 2. 개념 정의
> **메모리 계층 구조(Memory Hierarchy)** = 속도·용량·비용이 서로 트레이드오프 관계인 저장 장치를 계층으로 쌓아, "빠르지만 비싼 소량"과 "느리지만 싼 대량"을 조합해 평균적으로 빠르면서 저렴한 시스템을 만드는 설계.
> **캐시(Cache)**가 필요한 이유 = CPU 속도 대비 메모리(RAM) 속도가 훨씬 느려서(메모리 병목, "memory wall") — 자주/최근 쓴 데이터를 CPU 가까운 빠른 저장소에 미리 둬 평균 접근 시간을 줄인다.

## 3. 다이어그램 — 계층 구조
```
     빠름 ▲                       ▲ 작음/비쌈
         │  Register (CPU 내부)    │
         │  L1 Cache               │
         │  L2 Cache               │
         │  L3 Cache (코어 간 공유)  │
         │  RAM (Main Memory)       │
         │  SSD                     │
         ▼  HDD                     ▼
     느림                       큼/저렴
```

## 4. 속도·용량·비용 비교표 ⭐
| 계층 | 대략적 접근 속도 | 대략적 용량 | 비트당 비용 | 특징 |
|------|----------------|------------|------------|------|
| Register | < 1ns | 수백 byte~KB | 최고가 | CPU 내부, 코드로 직접 제어 불가 |
| L1 Cache | ~1ns | 수십 KB | 매우 비쌈 | 코어 전용 |
| L2 Cache | ~수ns | 수백 KB~MB | 비쌈 | 코어 전용(경우에 따라 공유) |
| L3 Cache | ~수십ns | 수~수십 MB | 상대적으로 저렴 | 코어 간 공유 |
| RAM | ~100ns | GB 단위 | 저렴 | 휘발성, 프로세스 종료 시 소실 |
| SSD | ~수십µs | 수백GB~TB | 더 저렴 | 비휘발성, 플래시 |
| HDD | ~수ms | TB 단위 | 최저가 | 비휘발성, 기계적 회전 |

> 계층을 한 단계 내려갈수록 **속도는 자릿수 단위(10배~10만 배)로 느려지고, 용량은 커지고, 비트당 비용은 싸진다.**

## 5. 다른 계층에서도 반복되는 같은 발상 — cross-link ⭐
"빠른 저장소에 자주 쓰는 것을 둔다"는 아이디어가 스택 전 영역에서 반복된다.
| 계층 | 빠른 저장소 | 느린 저장소 | 판단 기준 |
|------|------------|------------|----------|
| **하드웨어** | L1/L2/L3 Cache | RAM | 최근/자주 접근한 메모리 주소(locality) |
| **JVM (JIT)** | Code Cache (컴파일된 기계어) | 인터프리터 재해석 | hot spot(자주 실행되는 바이트코드) — `jvm-memory-gc.md`의 T3 참고 |
| **OS (가상 메모리)** | 페이지 캐시 / TLB(주소 변환 캐시) | 실제 물리 메모리 재탐색 / 디스크 스왑 | 최근 접근한 페이지·주소 변환 결과 |

> 면접에서 "캐시가 왜 이렇게 다양한 곳에 있나요?"라고 물으면: **속도 차이가 큰 두 저장소 사이에서, 자주/최근 쓰는 데이터를 빠른 쪽에 미리 갖다 놓아 평균 비용을 낮추는 원리는 하드웨어 캐시나 JIT code cache나 OS TLB나 동일하다** — 라고 답하면 큰 그림을 이해하고 있음을 보여줄 수 있음.

## 6. 핵심 포인트 (자주 하는 실수)
- 🔴 "캐시가 크면 무조건 좋다" ❌ — 캐시가 커지면 탐색(indexing) 비용도 늘어 접근 속도 자체가 느려질 수 있음. 그래서 L1(작고 빠름)~L3(크고 느림)으로 단계를 나눔.
- 🔴 "RAM도 SSD처럼 재부팅해도 남는다" ❌ — RAM은 **휘발성**(전원 꺼지면 소실), SSD/HDD는 비휘발성.
- 🟡 레지스터는 코드로 직접 할당 지시 불가(컴파일러/CPU가 결정) — Java에서 "이 변수를 레지스터에 둬라" 같은 제어는 불가능.
- 🟡 계층이 존재하는 근본 이유 = **속도와 용량/비용은 함께 가질 수 없다**는 물리적 한계 때문(SRAM 빠르지만 비쌈 vs DRAM/플래시/자기디스크로 갈수록 저렴하지만 느림).

## 7. 예상 면접 질문 + 답변 골격
**Q1. "메모리 계층 구조를 설명해주세요."**
> Register(가장 빠름·최소 용량) → L1/L2/L3 Cache → RAM → SSD → HDD(가장 느림·최대 용량) 순으로, 아래로 갈수록 느려지지만 용량이 커지고 저렴해짐. 빠른 저장소는 비싸서 소량만 둘 수 있으니, 계층을 쌓아 평균적으로 빠르면서 저렴한 시스템을 구성.

**Q2. "캐시가 왜 필요한가요?"**
> CPU 속도에 비해 RAM 접근이 훨씬 느려(메모리 병목) CPU가 항상 대기하게 됨. 최근/자주 쓰는 데이터를 CPU에 더 가까운 빠른 저장소(캐시)에 두면 평균 접근 시간이 크게 줄어듦.

**Q3. "이런 캐시 개념이 소프트웨어에도 적용되나요?"**
> 네. JVM JIT의 code cache(자주 실행되는 바이트코드를 컴파일된 기계어로 저장), OS의 페이지 캐시/TLB(자주 쓰는 페이지·주소 변환 결과 저장)가 모두 같은 원리 — 빠른 저장소에 자주 쓰는 것을 두어 평균 비용을 낮춤.

## 8. 심화 — NUMA(Non-Uniform Memory Access, 비균일 메모리 접근) ⭐

### 8-1. 비유 — 2인 사무실의 각자 서랍
멀티코어 서버를 "책상 두 개가 있는 사무실"이라고 생각하자. 각 책상(CPU 소켓)마다 **자기 서랍(로컬 RAM)**이 있다. 내 서랍에서 서류를 꺼내는 건 빠르지만, 옆 책상 서랍에서 서류를 꺼내려면 일어나서 걸어가야 한다(다른 소켓의 메모리 컨트롤러를 거쳐야 함) — 더 오래 걸린다.

### 8-2. 개념 정의
> **NUMA** = 멀티 소켓(멀티 CPU) 서버에서, 각 CPU가 **자신에게 물리적으로 가까운 로컬 메모리**와 **다른 CPU에 붙은 원격 메모리**를 모두 접근할 수 있지만, 로컬 접근이 원격 접근보다 빠른 구조. (반대 개념: 모든 메모리 접근 속도가 동일한 **UMA**, Uniform Memory Access — 단일 소켓/저사양 서버에서 흔함.)

```
[CPU 0] ── 로컬 RAM 0 (빠름)          [CPU 1] ── 로컬 RAM 1 (빠름)
   │                                      │
   └──────── 인터커넥트(QPI/Infinity Fabric 등) ────────┘
   CPU 0이 RAM 1 접근 = "원격 접근" → 인터커넥트를 거쳐야 해서 로컬보다 느림
```

### 8-3. 왜 백엔드 개발자가 알아야 하나
- 대규모 서버(멀티 소켓)에서 애플리케이션 스레드가 CPU 0에서 실행되는데, 할당받은 메모리는 CPU 1의 로컬 RAM이면 매 접근마다 원격 접근 페널티가 붙음.
- JVM 힙이 크고 GC가 스레드/객체를 어느 소켓 메모리에 두는지에 따라 대규모 서버의 GC 성능·지연시간이 달라질 수 있음(실무에서 `-XX:+UseNUMA` 같은 JVM 옵션이나 OS의 `numactl`로 제어).
- 컨테이너/쿠버네티스 환경에서도 NUMA-aware 스케줄링이 지연시간에 영향을 줄 수 있다는 정도만 알아도 충분("이런 것도 있다" 수준).

### 8-4. 핵심 포인트 (자주 하는 실수)
- 🔴 "메모리는 다 똑같은 속도로 접근된다" ❌ — 멀티 소켓 서버에서는 로컬/원격 접근 속도 차이가 실재(NUMA).
- 🟡 NUMA는 **일반 개발 업무에서 직접 튜닝할 일은 드물지만**, 대규모 서버·고성능 튜닝·JVM GC 옵션 문맥에서 등장하는 개념 — "이름과 왜 생기는지" 정도 답하면 충분한 수준.

### 꼬리 질문(follow-up)
- Q. "NUMA와 캐시 일관성(coherence)이 관계가 있나요?" → (답변 골격) 있음 — 멀티 소켓이라도 코어들은 캐시 일관성 프로토콜(MESI 등)로 데이터 일관성을 유지해야 하고, 원격 소켓의 캐시까지 확인해야 하는 경우 그 비용도 NUMA 접근 지연에 포함된다.

## 9. 심화 — 메모리 정렬(Alignment)과 구조체 패딩 ⭐

### 9-1. 비유 — 한 칸에 한 사람씩 앉는 버스 좌석
버스 좌석이 "1인용 칸"으로 나뉘어 있으면, 몸집이 작은 사람도 한 칸을 통째로 차지한다. CPU도 메모리를 **워드(word) 단위**(32bit CPU는 4byte, 64bit CPU는 8byte)로 묶어서 읽는 게 가장 효율적이라, 데이터 타입 크기가 그 경계에 안 맞아도 어차피 정렬된 경계에 맞춰 자리를 배정한다 — 그 결과 빈 공간(패딩)이 생긴다.

### 9-2. 개념 정의
> **정렬(Alignment)** = 데이터가 메모리에서 자신의 타입 크기의 배수인 주소에서 시작하도록 배치하는 규칙(예: 4byte `int`는 4의 배수 주소에서 시작). CPU가 워드 단위로 메모리를 읽기 때문에, 정렬되지 않은 데이터는 **두 번의 메모리 접근**이 필요하거나(성능 저하), 일부 아키텍처에서는 아예 접근 오류가 날 수 있음.
> **구조체 패딩(struct padding)** = 컴파일러가 구조체의 각 필드를 정렬 규칙에 맞추기 위해 필드 사이/끝에 사용하지 않는 **빈 byte**를 끼워 넣는 것.

```
struct Example {
    byte  a;   // 1 byte
    // 패딩 3 byte (int는 4의 배수 주소에서 시작해야 하므로)
    int   b;   // 4 byte
    byte  c;   // 1 byte
    // 패딩 3 byte (구조체 전체 크기도 가장 큰 필드의 배수로 맞춤)
};
// 실제 크기: 1 + 3(패딩) + 4 + 1 + 3(패딩) = 12 byte (필드만 더하면 6 byte인데 2배 차이)
```

### 9-3. 핵심 포인트 (자주 하는 실수)
- 🔴 "구조체 크기 = 필드 크기의 합" ❌ — 정렬 때문에 패딩이 끼어들어 실제 크기가 더 큰 경우가 흔함.
- 🟡 필드 선언 **순서**를 크기 큰 것부터(혹은 정렬 맞춰) 배치하면 패딩을 줄여 구조체 크기를 절약할 수 있음(C/C++, 메모리 최적화가 중요한 시스템 프로그래밍에서 자주 나오는 팁). Java는 객체 헤더·필드 배치를 JVM이 자동 관리하지만, 근본 원리(정렬)는 동일하게 적용됨.
- 🟡 정렬은 캐시 라인 개념과도 연결 — 정렬이 잘못되면 하나의 데이터가 두 캐시 라인에 걸쳐(cache line split) 저장돼 접근 시 캐시 미스가 두 번 날 수도 있음.

### 꼬리 질문(follow-up)
- Q. "왜 CPU는 임의 주소를 그냥 못 읽고 정렬을 요구하나요?" → (답변 골격) 메모리 버스와 캐시가 워드/캐시 라인 단위로 데이터를 옮기도록 설계돼 있어서, 정렬되지 않은 접근은 두 워드에 걸쳐 걸쳐있는 데이터를 합치는 추가 작업이 필요하거나(성능 손실), 일부 아키텍처는 이를 아예 지원하지 않아 하드웨어 예외가 발생하기 때문.
- Q. "구조체 필드 순서를 바꾸면 실제로 메모리를 얼마나 아낄 수 있나요?" → (답변 골격) 위 예시처럼 필드 순서에 따라 같은 필드 조합도 패딩 양이 달라져 크기가 2배 가까이 차이 날 수 있다 — 대규모 배열/컬렉션에서는 이 차이가 누적되어 캐시 효율과 메모리 사용량 모두에 영향을 준다.

# H3. Cache Locality and Performance

**학습 목표**: *"캐시 히트/미스가 뭔가요?"* 와 *"배열이 왜 연결리스트보다 순회가 빠른가요?"* 에 5분 답할 수 있다.

## 1. 비유 — 도서관에서 책 찾기
방금 꺼내 본 책(시간 지역성)은 다시 찾을 확률이 높으니 책상 위에 그대로 둔다. 그 책 옆 칸에 꽂힌 책들(공간 지역성)도 같은 주제라 곧 필요할 확률이 높으니, 사서가 책 한 권 가지러 갈 때 **옆 칸 몇 권을 통째로** 미리 챙겨온다. 이게 바로 캐시가 메모리를 "라인(line)" 단위로 통째로 가져오는 이유다.

## 2. 개념 정의
> **시간 지역성(Temporal Locality)** = 최근에 접근한 데이터는 곧 다시 접근될 가능성이 높다(예: 루프 변수, 반복 호출 함수).
> **공간 지역성(Spatial Locality)** = 접근한 주소 근처의 데이터도 곧 접근될 가능성이 높다(예: 배열 순차 순회).
> 캐시는 데이터를 **캐시 라인(보통 64byte)** 단위로 통째로 가져오는데, 이는 공간 지역성을 활용해 미래의 접근을 미리 대비하는 것.
> **캐시 히트(Hit)** = 요청한 데이터가 캐시에 있음(빠름). **캐시 미스(Miss)** = 없어서 더 느린 하위 계층(RAM 등)까지 내려가야 함.

## 3. 다이어그램 — 배열 순회 vs 연결리스트 순회
```
배열 (연속 메모리) — 공간 지역성 좋음
 [10][20][30][40][50] ... 캐시 라인 하나로 여러 원소가 한 번에 적재됨
  ▲
  한 번 미스 나면 이후 인접 원소는 캐시 히트 (같은 라인에 이미 있음)

연결리스트 (흩어진 메모리, 각 노드가 heap 임의 위치) — 공간 지역성 나쁨
 [10]→ (heap 저 멀리) [20]→ (heap 또 다른 곳) [30]→ ...
  ▲
  노드 하나 이동할 때마다 사실상 매번 캐시 미스 가능성 높음
```

## 4. 실무 영향 — 왜 배열 순회가 더 빠른가
- 배열은 원소가 메모리에 **연속**으로 저장 → 캐시 라인 하나에 여러 원소가 함께 적재됨 → 순차 접근 시 미스가 드묾.
- 연결리스트는 노드가 heap 여기저기 흩어짐(포인터로 연결) → 다음 노드로 이동할 때마다 새 캐시 라인을 가져와야 할 확률이 높음 → 미스 빈발.
- 알고리즘 문제에서 "Big-O가 같아도 배열 기반이 실제로 더 빠른" 흔한 이유 = 이 캐시 친화성(cache-friendliness) 차이.
- Java에서 `ArrayList` vs `LinkedList` 순회 성능 차이도 같은 맥락(단, `LinkedList`는 각 노드가 별도 객체라 참조를 따라가야 함).

## 5. False Sharing (간단히)
멀티스레드 환경에서 서로 다른 스레드가 **각자 다른 변수**를 쓰지만, 그 변수들이 **같은 캐시 라인**에 우연히 같이 있으면 한 스레드가 쓸 때마다 다른 코어의 캐시 라인이 무효화(invalidate)되어 성능이 떨어지는 현상. 해결책: 변수 사이에 padding을 넣어 캐시 라인을 분리(예: Java의 `@Contended` 애너테이션, `LongAdder`의 내부 구현이 이 문제를 완화).

## 6. 핵심 포인트 (자주 하는 실수)
- 🔴 "Big-O만 같으면 성능도 같다" ❌ — 캐시 지역성 차이로 실제 실행 시간은 몇 배씩 차이 날 수 있음(배열 vs 연결리스트가 대표 예시).
- 🔴 "캐시 미스는 드물게 일어난다" ❌ — 랜덤 접근 패턴(해시맵 순회, 연결리스트, 포인터 체이싱)에서는 미스가 매우 빈번.
- 🟡 시간 지역성 = "같은 것 다시 씀", 공간 지역성 = "가까운 것도 곧 씀" — 이 둘을 헷갈리지 말 것.
- 🟡 false sharing은 **버그가 아니라 성능 문제**(결과는 맞지만 느림) — 동시성 정확성 이슈(T4의 가시성/원자성)와는 다른 층위.

## 7. 정리표 — 지역성 종류
| 지역성 | 정의 | 활용 예 |
|--------|------|---------|
| 시간 지역성 | 최근 접근한 데이터 재접근 가능성 높음 | 루프 변수, 반복 호출 |
| 공간 지역성 | 인접 주소 데이터 접근 가능성 높음 | 배열 순차 순회, 캐시 라인 단위 적재 |

## 8. 예상 면접 질문 + 답변 골격
**Q1. "캐시 히트/미스가 뭔가요?"**
> 요청한 데이터가 캐시에 있으면 히트(빠름), 없어서 하위 계층까지 내려가야 하면 미스(느림). 캐시 설계 목표는 히트율을 최대화하는 것.

**Q2. "배열 순회가 연결리스트 순회보다 빠른 이유는?"**
> 배열은 메모리에 연속 저장돼 공간 지역성이 좋아 캐시 라인 하나로 여러 원소가 함께 적재됨. 연결리스트는 노드가 heap에 흩어져 있어 이동할 때마다 캐시 미스 가능성이 높음.

**꼬리 Q2-1. "캐시 라인 하나에 여러 원소가 실린다는 걸 성능 관점에서 더 설명해보세요."**
> 캐시는 바이트 단위가 아니라 보통 64바이트 라인 단위로 메모리를 당겨옵니다. int 배열이면 한 번의 미스로 16개(64/4)가 함께 올라와 다음 15번 접근은 미스 없이 처리됩니다(공간 지역성). 그래서 연속 순회는 미스가 1/16로 줄지만, 포인터로 흩어진 노드나 큰 stride 접근은 매번 새 라인을 당겨 이점이 사라집니다.

**꼬리 Q2-2. "그럼 2차원 배열을 행 우선으로 도느냐 열 우선으로 도느냐가 왜 성능에 영향을 주나요?"**
> 대부분 언어(C·자바)는 2차원 배열을 행 우선(row-major)으로 연속 저장합니다. 행 방향 순회는 인접 원소가 같은 캐시 라인에 있어 빠르지만, 열 방향 순회는 매 접근이 한 행 길이만큼 떨어진 주소라 캐시 라인을 계속 새로 당겨 미스가 폭증합니다. 접근 순서를 메모리 배치에 맞추는 게 핵심입니다.

**Q3. "false sharing이 뭔가요?"**
> 서로 다른 스레드가 각자 다른 변수를 쓰는데, 그 변수들이 같은 캐시 라인에 있어서 한쪽이 쓸 때마다 다른 코어의 캐시 라인이 무효화되며 성능이 떨어지는 현상. padding으로 변수를 다른 캐시 라인에 분리해 완화.

## 9. 심화 — 캐시 일관성(Cache Coherence)과 MESI 프로토콜 ⭐

### 9-1. 비유 — 여러 명이 같이 보는 화이트보드 사본
멀티코어는 각 코어가 자기 L1/L2 캐시(화이트보드 사본)를 따로 들고 있다. 그런데 원본(메모리 값)을 여러 사람이 각자 사본에 베껴 적어두고 있으면, 한 명이 자기 사본을 고치는 순간 다른 사람들의 사본은 "낡은 정보"가 된다. **캐시 일관성 프로토콜**은 "누가 언제 자기 사본을 지우거나 갱신해야 하는지"를 코어끼리 실시간으로 알려주는 규칙이다.

### 9-2. 개념 정의
> **캐시 일관성(Cache Coherence)** = 멀티코어 환경에서 같은 메모리 주소를 여러 코어의 캐시가 각자 들고 있을 때, 한 코어의 값 변경이 다른 코어에도 올바르게 반영되도록(낡은 값을 읽지 않도록) 보장하는 메커니즘.
> **MESI** = 캐시 라인이 가질 수 있는 4가지 상태로 일관성을 관리하는 대표적 프로토콜.

| 상태 | 의미 |
|------|------|
| **M**odified | 이 코어만 값을 가지고 있고, 메모리보다 최신(dirty) — 아직 메모리에 안 써짐 |
| **E**xclusive | 이 코어만 값을 가지고 있고, 메모리와 값이 같음(clean, 나만 독점) |
| **S**hared | 여러 코어가 같은 값을 읽기 전용으로 공유 중 |
| **I**nvalid | 이 캐시 라인은 무효 — 다시 읽어와야 함 |

```
코어 A가 캐시 라인을 Shared 상태로 읽음        코어 A가 그 라인을 write하려 함
 A: [S]  B: [S]                       →        A: [M]  B: [I]  ← B에게 "무효화(invalidate)" 신호 전파
 (둘 다 같은 값을 읽기전용 캐시)                (A는 독점+수정, B의 사본은 낡았으니 무효 처리)

이후 B가 같은 주소를 다시 읽으면:
 B: [I] → 캐시 미스 발생 → A의 캐시(or 메모리)에서 최신 값 가져와 다시 [S]/[M]으로 갱신
```
- 이 무효화(invalidate) 트래픽이 바로 **false sharing이 느려지는 원인의 하드웨어적 실체**다 — 서로 다른 변수라도 같은 캐시 라인에 있으면, 한쪽 write마다 다른 코어의 캐시 라인이 [S]→[I]로 계속 무효화되고, 무효화된 코어는 매번 캐시 미스로 재적재해야 한다.

### 9-3. 멀티코어에서 false sharing의 실제 성능 영향
- 캐시 라인이 무효화될 때마다 코어 간 **코히런시 트래픽**(invalidate 메시지 + 캐시 라인 재전송)이 인터커넥트(버스/링/메시)를 타고 오가야 함 → 로컬 캐시 히트보다 수십~수백 배 느려질 수 있음.
- 코어 수가 많을수록(예: 8코어, 16코어) 같은 캐시 라인을 두고 경쟁하는 코어가 늘어나 무효화 트래픽이 기하급수적으로 증가 — 스레드 수를 늘렸는데 오히려 처리량이 떨어지는(scalability collapse) 현상의 흔한 원인 중 하나.
- 실무 예: 여러 스레드가 각자 다른 카운터를 증가시키는데 그 카운터들이 배열로 인접 선언돼 한 캐시 라인에 몰리면, 논리적으로는 서로 무관한 연산인데도 물리적으로 캐시 라인을 두고 서로 방해(ping-pong)하며 급격히 느려짐. Java의 `LongAdder`/`@Contended`, C++의 `alignas(64)` 등이 이 문제를 padding으로 회피.

### 9-4. 핵심 포인트 (자주 하는 실수)
- 🔴 "캐시 일관성 프로토콜은 소프트웨어(OS/JVM)가 구현한다" ❌ — MESI는 **하드웨어(CPU)** 수준에서 자동으로 동작. 개발자가 직접 호출/제어하는 API가 아님(대신 그 존재를 알고 설계에 반영하는 것이 개발자 역할).
- 🔴 "false sharing은 데이터를 공유해서 생기는 문제다" ❌ — 오히려 **변수는 완전히 독립적인데 캐시 라인만 우연히 겹쳐서** 생기는 문제. 논리적 공유가 아니라 물리적(메모리 레이아웃) 근접이 원인.
- 🟡 MESI 상세 상태 전이는 인터뷰에서 깊게 묻는 경우는 드묾 — "여러 코어 캐시 간 최신성을 맞추는 프로토콜이 있고, 이게 false sharing의 근본 원인"이라는 인과관계를 설명할 수 있으면 충분.

### 꼬리 질문(follow-up)
- Q. "false sharing은 어떻게 탐지/디버깅하나요?" → (답변 골격) 프로파일러(예: Java의 JFR, perf의 cache-misses/coherence 이벤트)로 특정 코드 라인에서 캐시 미스나 코히런시 트래픽이 비정상적으로 높은지 확인하고, 의심되는 필드들이 같은 캐시 라인(보통 64byte)에 몰려 있는지 메모리 레이아웃을 점검한다.
- Q. "코어가 많아질수록 왜 오히려 성능이 떨어질 수 있나요(scalability collapse)?" → (답변 골격) 공유 자원(캐시 라인, 락)에 대한 코히런시 트래픽/경쟁이 코어 수에 비례해 늘어나기 때문 — 병렬성 이득보다 무효화·동기화 오버헤드가 더 커지는 지점이 생김.

# H4. Storage HDD and SSD

**학습 목표**: *"HDD와 SSD 차이가 뭔가요?"* 와 *"이게 DB 성능에 왜 중요한가요?"* 에 5분 답할 수 있다.

## 1. 비유 — 레코드판 vs USB
**HDD**는 레코드판(LP)과 바늘 같다 — 원하는 트랙(위치)까지 바늘(헤드)이 물리적으로 움직이고 판이 회전해야 소리(데이터)가 나온다. 옆 트랙은 금방 넘어가지만, 반대쪽 끝 트랙으로 가려면 바늘이 한참 이동해야 한다(탐색 시간). **SSD**는 USB 메모리처럼 전기 신호로 어느 위치든 거의 동일한 시간에 접근한다 — 움직이는 부품이 없다.

## 2. 개념 정의
> **HDD(Hard Disk Drive)** = 회전하는 자기 디스크 플래터 + 움직이는 헤드로 데이터를 읽고 쓰는 기계식 저장장치. 데이터 위치에 따라 접근 시간이 달라짐(헤드 이동 = seek time, 플래터 회전 대기 = rotational latency).
> **SSD(Solid State Drive)** = 플래시 메모리 칩에 전기적으로 데이터를 저장하는 저장장치. 움직이는 부품이 없어 어느 주소든 접근 시간이 거의 균일.

## 3. 다이어그램 — 구조 차이
```
HDD                                    SSD
┌─────────────────────┐               ┌─────────────────────┐
│   회전하는 플래터      │               │   플래시 메모리 칩     │
│      ⟲ ⟲ ⟲          │               │   (전기 신호로 read/   │
│   ▲                   │               │    write, 기계 동작    │
│   │ 헤드가 seek 이동    │               │    없음)              │
│   └─ 트랙 위치까지 이동 │               └─────────────────────┘
└─────────────────────┘
   seek time + rotational latency          거의 균일한 접근 시간
   → 순차 접근 ≫ 랜덤 접근 (속도 차 큼)      → 순차·랜덤 접근 속도 차 작음(상대적)
```

## 4. 순차 접근 vs 랜덤 접근 비교표 ⭐
| | HDD | SSD |
|--|-----|-----|
| 구조 | 회전 플래터 + 기계식 헤드 | 플래시 메모리(전기적) |
| 순차 접근 | 상대적으로 빠름(헤드 이동 적음) | 빠름 |
| 랜덤 접근 | **느림**(매번 seek + 회전 대기) | 순차 대비 느려질 수 있으나 HDD보다 훨씬 빠름 |
| 순차 vs 랜덤 속도 차 | 매우 큼 | 상대적으로 작음 |
| 내구성/수명 | 기계 마모 | 쓰기 횟수 제한(wear leveling으로 완화) |
| 비용/용량 | 저렴, 대용량 | 더 비쌈(가격 격차는 계속 축소 중) |

## 5. DB/애플리케이션 성능 함의
- HDD 환경에서는 **랜덤 I/O가 매우 비싸므로**, DB가 인덱스·데이터를 최대한 순차적으로 읽도록 설계하는 게 중요했음(B-Tree의 순차 스캔 친화적 구조, WAL의 append-only 쓰기 등).
- SSD는 랜덤 I/O 페널티가 훨씬 작아 랜덤 읽기가 많은 워크로드(인덱스 탐색, OLTP)에서 큰 성능 이득.
- 그래도 **순차 접근이 랜덤보다 유리한 경향 자체는 SSD에서도 남아있음**(내부적으로 페이지/블록 단위 read, 컨트롤러 캐시 활용 등) — "SSD니까 접근 패턴 신경 안 써도 된다"는 과장된 결론.
- 애플리케이션 관점: 파일을 순차적으로 append하는 로그, 시퀀셜 스캔이 유리한 배치 작업 등은 여전히 스토리지 유형과 무관하게 성능에 유리.

## 6. 핵심 포인트 (자주 하는 실수)
- 🔴 "SSD는 랜덤/순차 접근 속도가 완전히 같다" ❌ — HDD보다 차이가 훨씬 작을 뿐, 여전히 순차가 유리한 경향 존재.
- 🔴 "HDD가 느린 이유는 용량이 커서" ❌ — 느린 근본 원인은 **기계적 이동(seek + rotation)**, 용량과는 별개.
- 🟡 SSD도 쓰기 횟수 제한이 있어 **wear leveling**(마모 균등화)으로 수명 관리 — HDD엔 없는 개념.
- 🟡 DB 인덱스 자료구조(B-Tree 등)가 왜 "디스크 친화적"으로 설계됐는지 물으면, HDD 시대의 순차 접근 최적화가 배경이라는 답이 자연스러운 연결.

## 7. 예상 면접 질문 + 답변 골격
**Q1. "HDD와 SSD의 차이는?"**
> HDD는 회전하는 자기 디스크 + 기계식 헤드로 seek time·rotational latency가 존재해 랜덤 접근이 느림. SSD는 플래시 메모리에 전기적으로 접근해 움직이는 부품이 없고, 접근 시간이 위치에 관계없이 훨씬 균일함.

**Q2. "순차 접근과 랜덤 접근 속도 차이가 왜 중요한가요?"**
> HDD는 랜덤 접근마다 헤드 이동·회전 대기가 발생해 순차 대비 훨씬 느림. 그래서 DB나 로그 시스템이 순차 쓰기(append-only)·순차 스캔에 최적화된 구조(WAL, B-Tree)를 채택해온 배경이 됨. SSD는 이 격차가 작아졌지만 완전히 사라지진 않음.

**Q3. "SSD로 옮기면 DB 성능이 무조건 좋아지나요?"**
> 랜덤 I/O가 많은 워크로드(인덱스 탐색 등)에서는 크게 좋아짐. 다만 순차 접근이 여전히 상대적으로 유리하고, SSD 고유의 쓰기 제한(wear leveling)도 고려 대상이라 무조건이라 단정하긴 어려움.

# H5. 캐시 일관성 (MESI)

**학습 목표**: *"멀티코어에서 각 코어가 캐시를 따로 갖는데 어떻게 같은 값을 보나요?"* / *"MESI가 뭐고 왜 필요한가요?"* / *"false sharing이 왜 하드웨어적으로 느린가요?"* / *"volatile/메모리 배리어와는 무슨 관계인가요?"* 에 상태 전이를 그리며 5분 답할 수 있다.
> H3 §9에서 false sharing을 다뤘다면, 여기서는 그 밑바탕인 **캐시 일관성 프로토콜 자체**를 상태 전이 중심으로 정리한다.

## 1. 비유 — 여러 명이 각자 복사한 문서

한 개의 원본 문서(메인 메모리)를 4명(4코어)이 각자 복사본(캐시)을 떠서 자기 책상에서 본다. A가 자기 복사본을 고치면(write), 나머지 사람의 복사본은 **낡은 값(stale)** 이 된다. 이때 규칙이 없으면 각자 다른 값을 진실로 믿게 된다. **MESI = "누가 최신본을 갖고 있고, 누구 복사본이 무효인지"를 서로에게 방송(snoop)해 맞추는 규칙**이다.

## 2. 개념 정의 (1줄)
> **캐시 일관성(cache coherence)** = 멀티코어에서 각 코어의 프라이빗 캐시에 같은 메모리 블록의 복사본이 여러 개 있어도, 어느 코어가 읽어도 **최신 값을 보도록 보장**하는 성질.
> **MESI** = 각 캐시 라인을 **Modified / Exclusive / Shared / Invalid** 4상태로 관리하며, 코어 간 버스 스누핑으로 상태를 전이시켜 일관성을 유지하는 대표 프로토콜.

## 3. 4가지 상태 — 캐시 라인 단위

| 상태 | 의미 | 메모리와 일치? | 다른 코어에도 있나 |
|------|------|----------------|--------------------|
| **M** (Modified) | 내가 고쳐서 나만 최신본을 가짐(dirty) | ❌ 불일치(내 캐시가 진실) | 없음(나만 가짐) |
| **E** (Exclusive) | 나만 가지고 있고, 아직 안 고침(clean) | ✅ 일치 | 없음(나만 가짐) |
| **S** (Shared) | 여러 코어가 같은 값을 읽기용으로 공유 | ✅ 일치 | 있을 수 있음 |
| **I** (Invalid) | 내 복사본은 무효(누가 고쳤음) → 재적재 필요 | — | — |

## 4. 다이어그램 — 상태 전이 (핵심 흐름)

```
코어0이 x를 처음 읽음 → 아무도 안 갖고 있음 → [E] (나만, 깨끗)
코어1도 x를 읽음        → 둘이 공유          → 둘 다 [S]
코어0이 x에 write       → 코어0: [S]→[M]     → 코어1에게 "무효화(invalidate)" 방송
                          코어1: [S]→[I]      → 코어1 복사본 폐기
코어1이 다시 x를 읽음   → 캐시 미스([I])     → 코어0의 [M] 값을 받아옴(write-back)
                          코어0: [M]→[S], 코어1: [I]→[S]
```

핵심 규칙: **한 라인이 [M] 또는 [E]이면 그 코어만 소유**. write를 하려면 반드시 다른 모든 코어의 복사본을 [I]로 무효화해서 "나만 [M]" 상태를 만들어야 한다 → 이 **무효화 방송(bus traffic)** 이 비용의 원천.

## 5. false sharing과의 연결 (왜 느린가)

- 캐시 일관성은 **변수 단위가 아니라 캐시 라인 단위(보통 64byte)** 로 동작한다.
- 서로 완전히 독립적인 변수 A, B가 **우연히 같은 캐시 라인**에 있으면: 코어0이 A를 write할 때마다 라인 전체가 [M]이 되어 코어1의 B 복사본까지 [I]로 무효화된다. 코어1은 B만 쓰는데도 매번 캐시 미스로 재적재 → **논리적으로 안 겹치는데 물리적으로 겹쳐 느려지는 false sharing**.
- 회피: 핫한 필드를 **패딩으로 서로 다른 캐시 라인에 분리**(Java 8 `@Contended`, C의 `alignas(64)` 등). → H3 §9 참고.

## 6. volatile / 메모리 배리어와의 관계 (자주 혼동)

- **MESI(하드웨어)** 는 "여러 캐시 복사본의 최신성"을 맞춘다 — **가시성(visibility)의 물리적 토대**.
- 하지만 코어는 성능을 위해 **store buffer / 명령어 재정렬**을 하므로, MESI만으로는 "write가 언제 다른 코어에 보이는지 순서"까지 보장되지 않는다.
- **메모리 배리어(memory barrier)** 는 이 재정렬·버퍼링을 막아 "이 지점 이전 write는 여기서 반드시 반영/가시화"를 강제하는 명령. Java의 `volatile`은 read/write에 이 배리어를 삽입해 **가시성 + 순서**를 보장한다(JMM).
- 즉: **volatile(언어/JMM) → 메모리 배리어(CPU 명령) → MESI(캐시 일관성)** 로 층이 내려간다. MESI는 토대, volatile은 그 위에서 순서까지 보장하는 상위 계약.

## 7. 핵심 포인트 (자주 하는 실수)

- 🔴 "캐시 일관성은 소프트웨어(락)로 맞춘다" ❌ — MESI는 **하드웨어(캐시 컨트롤러 + 버스 스누핑)** 가 자동 처리. 락은 그 위에서 원자성·순서를 다루는 상위 계층.
- 🔴 "MESI가 있으니 volatile 없이도 멀티스레드 가시성이 보장된다" ❌ — MESI는 캐시 최신성만, **재정렬/store buffer로 인한 순서 문제는 배리어(volatile)** 가 있어야 해결.
- 🟡 일관성 단위는 변수가 아니라 **캐시 라인(64byte)** — false sharing의 근본 원인이자 성능 튜닝 포인트.
- 🟡 MESI 상세 전이는 면접에서 깊게 묻는 경우가 드묾 — "코어별 캐시 복사본의 최신성을 버스 스누핑으로 맞추는 프로토콜이고, false sharing과 volatile 가시성의 하드웨어적 토대"라는 **인과 연결**을 설명하면 충분.

## 8. 예상 면접 질문 + 답변 골격

**Q1. "멀티코어에서 캐시 일관성이 왜 필요한가요?"**
> ① 각 코어가 같은 메모리 블록의 복사본을 프라이빗 캐시에 따로 가짐 → ② 한 코어가 고치면 나머지 복사본이 낡은 값이 됨 → ③ MESI 같은 프로토콜이 "누가 최신본을 갖고 누구 게 무효인지"를 버스 스누핑으로 맞춰 어느 코어가 읽어도 최신 값을 보게 함.

**Q2. "MESI 4상태를 간단히 설명해주세요."**
> ① M=나만 고친 최신본(메모리와 불일치) → ② E=나만 가진 깨끗한 값 → ③ S=여러 코어가 읽기용 공유 → ④ I=무효(누가 고쳐서 폐기). write하려면 다른 코어를 모두 I로 무효화해 M을 확보.

**Q3. "false sharing이 하드웨어적으로 왜 느린가요?"**
> ① 일관성은 변수가 아니라 캐시 라인(64byte) 단위 → ② 독립 변수라도 같은 라인에 있으면 한쪽 write가 다른 코어의 라인을 통째로 무효화 → ③ 상대는 매번 캐시 미스로 재적재 → 논리적 무관인데 물리적 근접으로 느려짐. 패딩으로 라인 분리해 해결.

**Q4. "MESI가 있으면 volatile은 왜 필요한가요?"**
> ① MESI는 캐시 복사본의 최신성(가시성 토대)만 보장 → ② store buffer·명령어 재정렬로 write가 보이는 "순서"는 별개 문제 → ③ volatile이 메모리 배리어를 삽입해 재정렬을 막고 가시성+순서를 함께 보장(JMM). MESI는 토대, volatile은 상위 계약.

# 핵심 질문 (Quiz)

> 답변을 먼저 떠올린 뒤 펼쳐서 확인하세요.

<details>
<summary>Q1. CPU가 명령어를 실행하는 과정을 설명해주세요.</summary>

- Fetch(가져오기) → Decode(해독) → Execute(연산) → (Memory Access) → (Write Back)
- 이 사이클이 클럭(clock) 신호에 맞춰 반복됨
- 클럭 속도(GHz)가 높다고 무조건 빠른 건 아님 — IPC, 코어 수, 캐시 구조가 함께 성능 결정

</details>

<details>
<summary>Q2. 파이프라이닝이 뭐고 왜 빠른가요?</summary>

- 여러 명령어의 FDE 단계를 겹쳐(overlap) 처리하는 기법
- 컨베이어 벨트 비유 — 한 명령어 끝나길 기다리지 않고 단계별로 동시 처리
- 개별 명령어의 지연시간(latency)은 그대로거나 오히려 늘 수 있지만, 사이클마다 명령어 하나씩 완료 → 전체 처리량(throughput)은 대폭 상승

</details>

<details>
<summary>Q3. 분기 예측(branch prediction) 실패는 왜 성능에 큰 손해를 주나요?</summary>

- 분기 결과는 Execute 단계 즈음에야 확정되는데 Fetch는 그보다 먼저 다음 명령어를 가져와야 함 → CPU가 분기 방향을 미리 예측(speculative execution)
- 예측 실패 시 잘못 fetch한 명령어들을 전부 버리고(flush), 정확한 target부터 파이프라인을 다시 채워야 함 → 파이프라인 깊이만큼 사이클 손실
- 결과 자체는 항상 정확함(틀린 speculative 실행분은 커밋 전에 버림) — 손해는 성능(사이클 손실)뿐
- 파이프라인이 깊을수록(10~20단계) 처리량 이득도 크지만 misprediction penalty도 커지는 트레이드오프

</details>

<details>
<summary>Q4. 메모리 계층 구조(Memory Hierarchy)를 설명해주세요.</summary>

- Register(가장 빠름·최소 용량) → L1/L2/L3 Cache → RAM → SSD → HDD(가장 느림·최대 용량)
- 아래로 갈수록 느려지지만 용량이 커지고 비트당 비용은 저렴해짐
- 빠른 저장소는 비싸서 소량만 둘 수 있으니, 계층을 쌓아 평균적으로 빠르면서 저렴한 시스템을 구성
- 계층이 존재하는 근본 이유 = 속도와 용량/비용을 동시에 가질 수 없다는 물리적 한계

</details>

<details>
<summary>Q5. 캐시가 왜 필요한가요?</summary>

- CPU 속도에 비해 RAM 접근이 훨씬 느림(메모리 병목, memory wall)
- 최근/자주 쓰는 데이터를 CPU에 더 가까운 빠른 저장소(캐시)에 미리 둬 평균 접근 시간을 줄임
- 같은 발상이 소프트웨어에도 반복됨: JVM JIT의 code cache(hot spot 컴파일 저장), OS의 페이지 캐시/TLB(최근 페이지·주소 변환 결과 저장)

</details>

<details>
<summary>Q6. 배열 순회가 연결리스트 순회보다 빠른 이유는?</summary>

- 배열: 메모리에 연속 저장 → 공간 지역성(spatial locality) 좋음 → 캐시 라인(보통 64byte) 하나로 여러 원소가 함께 적재 → 순차 접근 시 미스가 드묾
- 연결리스트: 노드가 heap 여기저기 흩어짐(포인터로 연결) → 노드 이동마다 새 캐시 라인을 가져와야 할 확률 높음 → 미스 빈발
- Big-O가 같아도 캐시 친화성(cache-friendliness) 차이로 실제 성능은 몇 배씩 차이 날 수 있음

</details>

<details>
<summary>Q7. false sharing이 뭔가요?</summary>

- 서로 다른 스레드가 각자 다른(독립적인) 변수를 쓰는데, 그 변수들이 우연히 같은 캐시 라인에 있어서 발생
- 한 스레드가 쓸 때마다 다른 코어의 캐시 라인이 무효화(invalidate)되어 성능 저하
- 논리적 공유가 아니라 물리적(메모리 레이아웃) 근접이 원인 — 결과는 맞지만 느려지는 "성능 문제"이지 동시성 정확성(버그) 문제가 아님
- 해결책: 변수 사이에 padding을 넣어 캐시 라인을 분리 (Java `@Contended`, `LongAdder` 내부 구현 등)

</details>

<details>
<summary>Q8. HDD와 SSD의 차이는 무엇이고, 왜 DB 성능에서 중요한가요?</summary>

| 항목 | HDD | SSD |
|------|-----|-----|
| 구조 | 회전 플래터 + 기계식 헤드 | 플래시 메모리(전기적) |
| 랜덤 접근 | 느림(매번 seek + 회전 대기) | 순차 대비 느려질 수 있으나 HDD보다 훨씬 빠름 |
| 순차 vs 랜덤 속도 차 | 매우 큼 | 상대적으로 작음 |

- HDD는 랜덤 접근마다 seek time·rotational latency가 커서, DB가 순차 쓰기(append-only, WAL)·순차 스캔 친화적 구조(B-Tree)를 채택해온 배경이 됨
- SSD는 이 격차가 작아졌지만 완전히 사라지진 않음 — "SSD니까 접근 패턴 신경 안 써도 된다"는 과장된 결론

</details>
