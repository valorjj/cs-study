# O1. Process and Thread

**학습 목표**: *"프로세스와 스레드 차이는?"* / *"컨텍스트 스위칭이 뭐죠?"* 에 5분 답할 수 있다.

## 1. 비유 — 회사와 직원
- **프로세스** = 독립된 **회사 하나**. 자체 사무실(주소 공간)·금고(메모리)·비품을 가짐. 다른 회사와 자원 공유 안 함.
- **스레드** = 회사 안의 **직원**. 같은 사무실(주소 공간)·금고를 **공유**하며 각자 자기 책상(스택)에서 일함.

핵심 직관: **프로세스 간은 격리(독립 메모리), 스레드 간은 공유(같은 프로세스 메모리)**. 그래서 스레드는 통신이 빠르지만 동기화 문제가 생긴다.

## 2. 개념 정의
> **프로세스** = 실행 중인 프로그램. OS가 자원을 할당하는 단위. PCB(Process Control Block)로 관리.
> **스레드** = 프로세스 내 실행 흐름의 단위. CPU 스케줄링의 단위.

## 3. 다이어그램 — 메모리 공유 경계
```
┌──────────── Process A ────────────┐   ┌──── Process B ────┐
│  Code | Data | Heap  (공유)        │   │  (완전히 독립)     │
│  ┌─────────┬─────────┬─────────┐   │   │                    │
│  │Thread 1 │Thread 2 │Thread 3 │   │   │   Thread 1         │
│  │ Stack   │ Stack   │ Stack   │   │   │   Stack            │
│  │ 레지스터 │ 레지스터 │ 레지스터 │   │   │                    │
│  └─────────┴─────────┴─────────┘   │   │                    │
└────────────────────────────────────┘   └────────────────────┘
    스레드끼리 Code/Data/Heap 공유          프로세스는 서로 격리
    → Stack·레지스터·PC만 스레드별
```
> Java의 "Heap은 공유, Stack은 스레드별"(JVM T1)이 바로 이 그림. JVM 스레드 = OS 스레드에 매핑.

## 4. 비교표
| 구분 | 프로세스 | 스레드 |
|------|----------|--------|
| 메모리 | 독립(격리) | Code/Data/Heap 공유, Stack은 개별 |
| 생성 비용 | 큼(자원 복제) | 작음 |
| 통신 | IPC 필요(파이프, 소켓, 공유메모리) | 공유 메모리로 즉시 |
| 한 개 죽으면 | 다른 프로세스 영향 없음 | 같은 프로세스 전체 위험 |
| 컨텍스트 스위치 | 무거움(주소 공간 교체) | 가벼움(주소 공간 유지) |

## 5. 컨텍스트 스위칭 (Context Switching)
- CPU가 실행 중이던 작업의 상태(레지스터, PC 등)를 **PCB/TCB에 저장**하고, 다음 작업의 상태를 **복원**하는 것.
- 비용: 이때 CPU는 실제 일을 안 함(오버헤드). 프로세스 스위치는 주소 공간(페이지 테이블·TLB flush)까지 바꿔 더 비쌈.
- 스레드 스위치는 같은 주소 공간이라 TLB flush 없어 상대적으로 저렴.

## 6. User Mode vs Kernel Mode, 시스템 콜

### 비유 — 은행 창구 뒤 금고
고객(응용 프로그램)은 창구 앞(**User Mode**)에서만 일할 수 있고, 금고(하드웨어 자원)는 직원(**Kernel**)만 직접 만짐. 고객이 돈을 인출하려면 창구 직원에게 **요청서(System Call)** 를 제출해야 함 — 직접 금고에 들어갈 수 없다.

- **User Mode**: 응용 프로그램이 실행되는 제한된 모드. 하드웨어 직접 접근(디스크, 메모리 관리, I/O 포트 등) 불가.
- **Kernel Mode**: OS 커널이 실행되는 특권 모드. 모든 명령어·자원에 접근 가능.
- **시스템 콜(System Call)**: User Mode 프로그램이 파일 열기, 네트워크 전송, 메모리 할당 등 커널 기능이 필요할 때 커널에 요청하는 인터페이스. `read()`, `write()`, `fork()`, `malloc()`(내부적으로 `brk`/`mmap`) 등.
  - 흐름: 사용자 프로그램 → 시스템 콜 호출(트랩/소프트웨어 인터럽트) → **모드 전환(User→Kernel)** → 커널이 요청 처리 → **모드 전환(Kernel→User)** → 실행 재개.
  - Java에서 `System.out.println()`, `new FileInputStream()`, 소켓 I/O 모두 내부적으로 시스템 콜을 호출한다.

### 컨텍스트 스위칭 비용과의 관계
- 시스템 콜 자체도 **모드 전환** 비용이 든다(스레드/프로세스 컨텍스트 스위치와는 별개지만 둘 다 "지금 하던 일 잠깐 멈추고 다른 모드/작업으로 전환"이라는 공통점).
- 시스템 콜이 잦으면(예: 파일을 1바이트씩 읽기) 모드 전환 오버헤드가 누적됨 → **버퍼링**(`BufferedReader` 등)으로 시스템 콜 횟수를 줄이는 이유가 여기 있다.
- 컨텍스트 스위치 비용 요소: ① 레지스터/PC 저장·복원 ② (프로세스면) 페이지 테이블 교체 + TLB flush ③ 캐시 미스 증가(다른 프로세스 데이터로 캐시 재적재) ④ 스케줄러 실행 자체의 오버헤드.

## 7. 프로세스 상태 다이어그램

```
        생성됨              스케줄러가 선택
  ┌──▶ [ New ] ──────▶ [ Ready ] ──────────▶ [ Running ]
  │                        ▲                    │  │
  │                        │   타임슬라이스 만료/  │  │
  │                        └── 선점(preempt) ────┘  │
  │                                                  │ I/O·이벤트 대기
  │                                                  ▼
  │                                            [ Waiting ]
  │                                                  │
  │              I/O 완료 등                          │
  │              ◄──────────────────────────────────┘
  │                        │
  │                        ▼
  └──────────────────  [ Terminated ] ◄── Running에서 종료
```
- **New**: 프로세스 생성 중(PCB 할당). **Ready**: CPU만 배정되면 바로 실행 가능(준비 큐 대기). **Running**: CPU 점유 중. **Waiting(Blocked)**: I/O 등 이벤트 대기 중(CPU 못 씀). **Terminated**: 실행 종료, 자원 회수 대기.
- 헷갈리는 포인트: **Ready→Running**은 스케줄러의 선택, **Running→Ready**는 선점(타임아웃 등), **Running→Waiting**은 자발적(I/O 요청) — 이 방향은 스케줄러가 아니라 프로세스 자신이 유발.

## 8. IPC(Inter-Process Communication) 종류 비교

프로세스는 메모리가 격리돼 있어 서로 데이터를 주고받으려면 OS가 제공하는 IPC 메커니즘이 필요하다.

| 방식 | 개념 | 특징 | 예 |
|------|------|------|-----|
| **파이프(Pipe)** | 커널 버퍼를 통한 단방향(named pipe는 양방향 가능) 스트림 | 보통 **부모-자식 관계** 프로세스 간, 관련 없는 프로세스는 named pipe(FIFO) 필요 | 셸의 `\|` |
| **공유 메모리(Shared Memory)** | 같은 물리 메모리 영역을 여러 프로세스 주소 공간에 매핑 | **가장 빠름**(커널 개입 없이 직접 읽기/쓰기), 대신 동기화(세마포어 등)를 직접 구현해야 함 | `shmget`, Java의 `MappedByteBuffer` |
| **메시지 큐(Message Queue)** | 커널이 관리하는 큐에 메시지 단위로 송수신 | 프로세스 간 관계 없어도 됨, 메시지 경계 유지, 우선순위 지정 가능 | POSIX MQ, Kafka(응용 계층 버전) |
| **소켓(Socket)** | 네트워크 프로토콜 기반 양방향 통신 | **같은 머신이든 다른 머신이든** 동작(유일하게 네트워크 확장 가능), 오버헤드 가장 큼 | TCP/UDP 소켓, HTTP |
| **시그널(Signal)** | 비동기 이벤트 통지(데이터 전달 아님) | 매우 가벼움, 정해진 신호 종류만 | `SIGKILL`, `SIGTERM` |

> 성능 순서(대략): 공유 메모리 > 파이프 ≈ 메시지 큐 > 소켓. 하지만 공유 메모리는 동기화를 직접 짜야 해서 구현 난이도가 가장 높다.

## 9. 핵심 포인트 (자주 하는 실수)
- 🔴 "스레드는 메모리를 전부 공유" — ❌ **Stack·레지스터·PC는 스레드별**. Heap/Data/Code만 공유.
- 🔴 "멀티프로세스가 항상 느리다" — 격리로 **안정성**↑(한 프로세스 죽어도 나머지 생존). Chrome 탭 분리가 예.
- 🟡 멀티스레드는 공유 자원 → **동기화 문제**(O4로 연결).
- 🔴 "시스템 콜은 그냥 함수 호출" — ❌ 일반 함수 호출과 달리 **모드 전환(트랩)** 이 일어나 비용이 더 크다.
- 🟡 "Waiting에서 Ready로 못 가고 바로 Running" — ❌ I/O 완료해도 **Ready 큐로 돌아가 스케줄러 선택을 기다림**, 곧바로 CPU를 얻지 않음.

## 10. 예상 면접 질문 + 답변 골격
**Q1. "프로세스와 스레드 차이?"**
> 프로세스는 독립 메모리를 가진 실행 단위, 스레드는 그 안에서 Code/Data/Heap을 공유하며 각자 Stack만 갖는 실행 흐름. 스레드는 통신이 빠르나 동기화 이슈, 프로세스는 격리로 안정적이나 컨텍스트 스위치가 무겁습니다.

**Q2. "컨텍스트 스위칭이 뭐고 왜 비싼가요?"**
> CPU가 현재 작업 상태를 저장하고 다음 작업 상태를 복원하는 것. 그동안 실제 연산을 못 해 오버헤드이고, 프로세스 전환은 주소 공간·TLB까지 교체해 스레드 전환보다 비쌉니다.

**Q3. "시스템 콜이 뭔가요? 왜 필요하죠?"**
> 사용자 프로그램이 하드웨어·커널 자원(파일, 네트워크, 메모리 등)에 접근하려 할 때 커널에 요청하는 인터페이스입니다. User Mode는 직접 접근 권한이 없어서 트랩을 통해 Kernel Mode로 전환해 처리하고 다시 돌아옵니다. 이 전환 자체가 오버헤드라 잦은 시스템 콜은 성능에 영향을 줍니다.

**Q4. "공유 메모리와 소켓 중 IPC로 뭘 쓸지 어떻게 고르나요?"**
> 같은 머신에서 최대 성능이 필요하면 공유 메모리(동기화는 직접 구현), 프로세스 관계와 무관하게 구조화된 메시지를 주고받으려면 메시지 큐, 다른 머신과도 통신해야 하면 소켓을 씁니다.

# O2. CPU Scheduling

**학습 목표**: *"스케줄링 알고리즘 종류와 선점/비선점 차이는?"* 에 답할 수 있다.

## 1. 비유 — 은행 창구
직원(CPU) 하나, 손님(프로세스) 여럿. 누구를 먼저 처리할지 규칙이 **스케줄링**.
- 번호표 순서대로 = FCFS
- 볼일 짧은 손님 먼저 = SJF
- 정해진 시간만큼 돌아가며 = Round Robin

## 2. 선점 vs 비선점
- **비선점(Non-preemptive)**: 실행 중인 작업이 **스스로 CPU를 놓을 때까지** 안 뺏음. (FCFS, SJF)
- **선점(Preemptive)**: OS가 **강제로 CPU를 회수** 가능(타임 슬라이스 만료, 더 급한 작업 도착). (Round Robin, Priority-preemptive)

## 3. 알고리즘 비교표
| 알고리즘 | 방식 | 장점 | 단점 |
|----------|------|------|------|
| **FCFS** | 먼저 온 순 | 단순·공정 | Convoy 효과(긴 작업이 뒤 막음) |
| **SJF** | 짧은 작업 먼저 | 평균 대기시간 최소 | 실행시간 예측 어려움, starvation |
| **Priority** | 우선순위 높은 순 | 중요 작업 우선 | 낮은 우선순위 starvation → **Aging**으로 완화 |
| **Round Robin** | 타임 퀀텀 순환 | 응답성 좋음(대화형) | 퀀텀 너무 작으면 스위치 오버헤드 |
| **MLFQ** | 다단계 큐+피드백 | 실무형(짧은 작업 우대+적응) | 복잡 |

## 4. 평가 기준
- **처리량(Throughput)**: 단위시간 완료 작업 수
- **응답시간(Response)**: 요청 후 첫 반응까지 (대화형 중요)
- **대기시간(Waiting)**: 준비 큐에서 기다린 시간
- **반환시간(Turnaround)**: 제출~완료 전체

## 5. 핵심 포인트
- 🟡 **starvation**(기아): 낮은 우선순위가 계속 밀림 → **Aging**(대기할수록 우선순위↑)으로 해결.
- 🟡 Round Robin의 퀀텀: 크면 FCFS화, 작으면 스위치 오버헤드 → 균형이 중요.

## 6. 예상 면접 질문
**Q1. "선점형과 비선점형 스케줄링 차이?"**
> 비선점은 실행 중 작업이 스스로 CPU를 반납할 때까지 유지(FCFS/SJF), 선점은 OS가 강제 회수(Round Robin). 선점형이 응답성은 좋지만 컨텍스트 스위치 비용과 동기화 고려가 늘어납니다.

**Q2. "MLFQ가 실무에서 선호되는 이유는?"**
> 짧은 작업은 높은 우선순위 큐에서 빨리 처리하고(응답성), 오래 실행되는 작업은 점차 낮은 큐로 내려 CPU 집약 작업과 대화형 작업을 자동으로 구분합니다. Aging도 결합해 starvation을 막을 수 있어 리눅스 CFS 등 실제 스케줄러의 기반 아이디어가 됩니다.

# O3. Virtual Memory and Paging

**학습 목표**: *"가상 메모리가 뭐고 왜 쓰나요?"* / *"페이지 폴트는?"* 에 답할 수 있다. (JVM GC·메모리와 직접 연결)

## 1. 비유 — 도서관 열람실
책(실제 물리 메모리, RAM)은 한정. 하지만 이용자에겐 "원하는 책 다 있는 것처럼" 보이게 함. 실제론 대부분 서고(디스크)에 있고, 필요할 때만 열람실(RAM)로 가져옴 = **가상 메모리**.

## 2. 개념 정의
> **가상 메모리** = 각 프로세스에 **실제 RAM보다 큰 연속된 주소 공간**이 있는 것처럼 보이게 하는 기법. 물리 메모리 한계를 넘어 실행 가능 + 프로세스 간 격리.
> **페이징** = 가상 주소 공간을 고정 크기 **페이지**로, 물리 메모리를 **프레임**으로 나눠 매핑.

## 3. 다이어그램 — 주소 변환
```
 가상 주소 (프로세스가 보는 주소)
      │  [ 페이지 번호 | 오프셋 ]
      ▼
 ┌─────────────┐   TLB(캐시) 히트 시 초고속
 │ Page Table  │◄──── TLB (최근 변환 캐시)
 │ (페이지→프레임)│
 └──────┬──────┘
        ▼
 물리 주소 [ 프레임 번호 | 오프셋 ] → RAM
        │
        └─ 페이지가 RAM에 없음 → Page Fault → 디스크에서 로드
```

## 4. 핵심 메커니즘
- **페이지 테이블**: 가상 페이지 → 물리 프레임 매핑표 (프로세스별).
- **TLB (Translation Lookaside Buffer)**: 최근 주소 변환을 캐싱하는 하드웨어. 히트하면 페이지 테이블 안 거침. **JIT의 code cache나 CPU 캐시와 같은 "자주 쓰는 걸 빠르게" 발상**.
- **페이지 폴트(Page Fault)**: 접근한 페이지가 RAM에 없음 → OS가 디스크에서 프레임으로 로드(느림). **Demand Paging**(필요할 때만 로드).
- **페이지 교체(Page Replacement)**: RAM 꽉 참 → 어떤 페이지를 내보낼지. **LRU**(가장 오래 안 쓴 것), FIFO, Optimal.

## 5. JVM GC와의 연결 ⭐ (cross-link)
| OS 메모리 관리 | JVM GC |
|----------------|--------|
| 안 쓰는 페이지를 디스크로 내보냄(교체) | 도달 불가 객체를 회수 |
| LRU로 교체 대상 선정 | 세대 가설로 Young 우선 수거 |
| Page Fault = RAM에 없어 디스크 접근 | Major GC = Old 정리(느림) |
> 둘 다 "한정된 빠른 메모리를 효율적으로 재활용"하는 문제. 면접에서 GC 얘기하다 "OS도 비슷하게 페이지 교체를..."로 확장하면 좋다.

## 6. 페이지 교체 알고리즘 상세

목표: RAM(프레임)이 꽉 찼을 때 **어떤 페이지를 내보낼지** 결정. 잘못 고르면 곧바로 다시 폴트(→ 성능 저하).

### LRU (Least Recently Used)
> "가장 오래전에 사용된 페이지를 내보낸다." — 과거를 보고 미래를 예측(최근에 안 썼으면 앞으로도 안 쓸 확률↑)하는 **지역성(locality)** 가정.

구현 방법:
- **연결 리스트 + 이동**: 페이지 접근 시마다 해당 노드를 리스트 맨 앞으로 이동. 교체 시 맨 뒤(꼬리) 제거. → 접근마다 O(1) 재배치 가능(연결 리스트라서), 하지만 하드웨어 지원 없이 매 메모리 접근마다 하기엔 비쌈.
- **카운터/타임스탬프**: 각 페이지에 마지막 접근 시각 기록, 교체 시 전체 스캔해 최솟값 탐색 → 스캔 비용 O(n).
- 실제 하드웨어는 완벽한 LRU를 구현하기엔 비용이 커서, 근사 알고리즘인 **Clock**을 대신 쓴다.
- (Java 개발자용 비유: `LinkedHashMap(accessOrder=true)` + `removeEldestEntry`로 LRU 캐시를 구현하는 것과 원리가 동일 — Java T3 자료구조에서 이미 다룬 패턴.)

### Clock 알고리즘 (Second-Chance)
LRU를 정확히 구현하는 대신, 저렴하게 근사하는 방법.
```
        ┌───┐
     ┌─▶│ P1│ref=1
     │  └───┘
     │   ▼
   ┌───┐   ┌───┐
   │P4 │   │P2 │ref=0  ◄── 시계 바늘(hand)이 여기 있으면:
   │ref│   │ref│         ref=0 → 이 페이지를 교체(victim)
   │=0 │   │=0 │         ref=1 → 0으로 낮추고 다음 칸으로(한 번의 기회 부여)
   └───┘   └───┘
     ▲   ┌───┐
     └───│ P3│ref=1
         └───┘
```
- 각 페이지에 **참조 비트(reference bit)** 1개. 접근하면 OS/하드웨어가 1로 세팅.
- 교체 필요 시 시계 바늘이 프레임을 순회: `ref=0`이면 즉시 교체, `ref=1`이면 **0으로 내리고(2nd chance) 다음으로 넘어감**.
- 전체 스캔·정밀한 순서 추적 없이 "최근에 안 쓰인 것 같은" 페이지를 찾아 LRU를 값싸게 근사 → 대부분 실제 OS(Linux 등)가 이 계열 사용.

### Belady's Anomaly (벨레이디의 변칙)
> 직관: "프레임(캐시)을 늘리면 페이지 폴트가 줄어야 한다." — **FIFO에서는 항상 성립하지 않는다.**
- FIFO 교체 알고리즘은 프레임 수를 늘렸는데 오히려 폴트 횟수가 **늘어나는** 경우가 존재(반례로 유명한 참조열: `1 2 3 4 1 2 5 1 2 3 4 5`).
- 원인: FIFO는 "얼마나 오래 있었나"만 보고 "얼마나 자주/최근에 쓰였나"는 무시 → 자주 쓰이는 페이지도 그냥 오래됐다는 이유로 쫓아냄.
- **LRU와 Optimal(OPT, 미래를 안다고 가정하는 이상적 알고리즘)은 이 변칙이 발생하지 않음**(스택 알고리즘 성질을 만족하기 때문). 그래서 실무/이론 모두 FIFO보다 LRU 계열을 선호.

## 7. 핵심 포인트 (자주 하는 실수)
- 🔴 "가상 메모리 = 디스크 스왑" — ❌ 스왑은 가상 메모리의 **일부 구현**. 가상 메모리의 본질은 **주소 추상화 + 격리**.
- 🔴 "페이징에 외부 단편화 있다" — ❌ **페이징은 외부 단편화 없음**(고정 크기). 대신 **내부 단편화**(마지막 페이지 낭비). 세그먼테이션이 외부 단편화.
- 🟡 **Thrashing**: 페이지 폴트가 과도해 CPU가 스와핑만 하는 상태 → 성능 급락.
- 🔴 "프레임을 늘리면 무조건 폴트가 준다" — ❌ **FIFO는 Belady's Anomaly로 예외 발생**. LRU/OPT는 그런 예외가 없다.
- 🟡 완벽한 LRU는 비용이 커서 실제 OS는 **Clock(2nd chance)** 같은 근사 알고리즘을 쓴다.

## 8. 예상 면접 질문
**Q1. "가상 메모리를 왜 쓰나요?"**
> ① 물리 RAM보다 큰 프로그램 실행 가능 ② 프로세스 간 주소 공간 격리(보안·안정) ③ 각 프로세스가 연속 주소를 보게 해 개발 단순화.

**Q2. "페이지 폴트 처리 과정은?"**
> 접근 페이지가 RAM에 없으면 트랩 발생 → OS가 디스크에서 해당 페이지를 빈 프레임(없으면 교체)으로 로드 → 페이지 테이블 갱신 → 명령 재실행. 잦으면 Thrashing.

**Q3. "페이징과 세그먼테이션 차이?"**
> 페이징은 고정 크기 페이지(외부 단편화 없음, 내부 단편화 있음), 세그먼테이션은 논리 단위 가변 크기(외부 단편화 있음). 현대 OS는 주로 페이징 기반.

**Q4. "LRU를 실제로 어떻게 구현하나요?"**
> 이론적으로는 연결 리스트를 접근 순서대로 재배치하거나 타임스탬프를 비교해 최솟값을 찾지만, 매 메모리 접근마다 하기엔 비용이 큽니다. 그래서 실제 하드웨어/OS는 참조 비트 하나로 근사하는 **Clock(2nd chance) 알고리즘**을 주로 사용합니다.

**Q5. "Belady's Anomaly가 뭔가요?"**
> 일반적으로 프레임(캐시 크기)을 늘리면 페이지 폴트가 줄어드는데, FIFO 교체 알고리즘에서는 프레임을 늘렸는데 오히려 폴트가 늘어나는 경우가 존재합니다. LRU나 Optimal 알고리즘은 이 현상이 발생하지 않아 FIFO보다 선호됩니다.

# O4. Synchronization and Deadlock

**학습 목표**: *"임계구역/뮤텍스/세마포어?"* / *"데드락 조건과 해결?"* 에 답할 수 있다. (Java 동시성과 직접 연결)

## 1. 비유 — 화장실 열쇠 하나
칸(공유 자원) 하나, 열쇠(락) 하나. 열쇠 가진 사람만 들어감. 두 명이 동시에 들어가려 하면 문제(race condition) → 열쇠로 **상호 배제**.

## 2. 핵심 개념
- **Race Condition**: 둘 이상이 공유 자원에 동시 접근·수정 → 순서에 따라 결과가 달라짐. (Java `count++`와 동일 문제 — JMM T4)
- **Critical Section(임계구역)**: 공유 자원에 접근하는 코드 영역. 한 번에 하나만 실행돼야 함.
- **상호 배제(Mutual Exclusion)**: 임계구역에 한 스레드만 진입.

## 3. 동기화 도구 비교
| 도구 | 개념 | 특징 |
|------|------|------|
| **Mutex** | 열쇠 1개(binary), 소유 개념 있음 | 잠근 스레드만 해제 가능 |
| **Semaphore** | 카운터 N개 자원 허용 | counting(N개)/binary. 소유 개념 없음 |
| **Monitor** | 언어 차원 상호배제 캡슐화 | Java `synchronized`가 이것 |
| **Spinlock** | 락 풀릴 때까지 busy-wait | 짧은 대기에 유리, CPU 낭비 위험 |

> Java `synchronized`/`ReentrantLock`은 OS의 뮤텍스/모니터를 언어로 감싼 것. `volatile`의 가시성 문제도 결국 CPU 캐시 vs 메인메모리(하드웨어) 얘기.

## 4. 데드락(Deadlock) — 4가지 필요조건 ⭐
네 개가 **동시에** 성립해야 데드락 발생 (하나만 깨도 예방):
```
1. 상호 배제 (Mutual Exclusion)  — 자원을 한 번에 하나만
2. 점유와 대기 (Hold and Wait)   — 자원 쥔 채 다른 자원 대기
3. 비선점 (No Preemption)        — 강제로 뺏을 수 없음
4. 환형 대기 (Circular Wait)     — A→B→C→A 원형으로 대기
```
비유: 사거리에서 네 차가 서로 앞차 꼬리를 막고 아무도 못 감.

## 5. 데드락 처리 전략
| 전략 | 방법 |
|------|------|
| **예방(Prevention)** | 4조건 중 하나를 원천 차단 (예: 자원 순서 정해 환형 대기 제거) |
| **회피(Avoidance)** | 안전 상태만 허용 — **은행원 알고리즘(Banker's)** |
| **탐지·회복(Detection)** | 주기적 탐지 후 프로세스 강제 종료/자원 회수 |
| **무시(Ostrich)** | 드물면 그냥 무시 (실제 많은 OS) |

## 6. 세마포어로 생산자-소비자 문제 풀기

### 비유 — 빵집 진열대
생산자(제빵사)는 빵을 만들어 진열대(버퍼)에 놓고, 소비자(손님)는 진열대에서 빵을 꺼내 감. 진열대 **자리 수가 제한**돼 있고, **빵이 없으면 손님은 기다려야** 함 — 이 두 제약을 세마포어 두 개로 표현한다.

```
세마포어 empty = N   // 버퍼의 "빈 자리" 개수 (처음엔 N개 다 빔)
세마포어 full  = 0   // 버퍼의 "채워진 자리" 개수 (처음엔 0개)
뮤텍스   mutex = 1   // 버퍼 자체를 건드리는 동안의 상호배제

Producer:                          Consumer:
loop {                             loop {
  item = produce()                   wait(full)    // 꺼낼 빵 있을 때까지 대기
  wait(empty)   // 빈 자리 있을 때까지 대기          wait(mutex)
  wait(mutex)                         item = buffer에서 꺼냄
  buffer에 item 삽입                   signal(mutex)
  signal(mutex)                       signal(empty) // 빈 자리 하나 늘어남
  signal(full)  // 채워진 자리 하나 늘어남           consume(item)
}                                 }
```
- `empty`/`full`은 **카운팅 세마포어**로 "자원 개수"(빈 자리/찬 자리)를 표현 — 세마포어의 전형적 용도.
- `mutex`는 버퍼 자료구조 자체를 동시에 두 스레드가 건드리지 않게 하는 **이진 세마포어**(뮤텍스 역할).
- 순서가 중요: `wait(empty)` → `wait(mutex)` 순으로 잡아야 함. 반대로 하면 mutex를 쥔 채 empty를 기다리다 소비자도 mutex를 못 얻어 **데드락**.

## 7. 뮤텍스와 세마포어 차이 (심화)
| 구분 | Mutex | Semaphore |
|------|-------|-----------|
| 값의 범위 | 0/1 (binary) | 0~N (counting) 또는 binary |
| **소유(ownership)** | 있음 — **잠근 스레드만 해제** 가능 | 없음 — **다른 스레드가 signal 가능**(A가 wait, B가 signal해도 됨) |
| 용도 | **상호 배제**(임계구역 보호) | **자원 개수 제한 + 스레드 간 신호(signaling)** |
| 우선순위 역전 | 소유자가 명확해 **우선순위 상속(priority inheritance)** 적용 가능 | 소유자가 없어 적용 어려움 |
| 예 | Java `synchronized`, `ReentrantLock` | Java `Semaphore`, 생산자-소비자의 empty/full |

> 핵심 차이 한 줄: **뮤텍스는 "누가 잠갔는지"가 있고 그 사람만 풀 수 있는 자물쇠, 세마포어는 소유 없이 "카운트"만 관리하는 신호기**. 그래서 세마포어는 "다른 스레드에게 완료를 알리는" 용도(생산자→소비자 신호)로도 쓰이지만, 뮤텍스는 그 용도로 쓰면 안 된다(자기가 안 잠근 락을 풀 수 없으므로).

## 8. 데드락 회피 — 은행원 알고리즘(Banker's Algorithm) 예시

> 아이디어: 은행이 대출해줄 때 "이 대출 이후에도 모든 고객이 순서대로 최대 필요량을 받아 완제할 수 있는가"(= **안전 상태, Safe State**)를 미리 계산 후에만 승인.

예시: 자원 A 총 12개, 프로세스 3개.

| 프로세스 | Allocation(현재 보유) | Max(최대 필요) | Need(추가 필요 = Max-Allocation) |
|----------|------------------------|-----------------|-----------------------------------|
| P0 | 5 | 10 | 5 |
| P1 | 2 | 4 | 2 |
| P2 | 2 | 9 | 7 |

- 현재 할당 합 = 5+2+2 = 9 → **가용(Available) = 12 - 9 = 3**
- 안전 순서 탐색: Available=3으로 **Need를 만족시킬 수 있는 프로세스**부터 실행 후 자원 반환.
  1. P1의 Need=2 ≤ Available(3) → P1 실행 가능 → 완료 후 자원 반환: Available = 3+2 = 5
  2. P0의 Need=5 ≤ Available(5) → P0 실행 가능 → 완료 후 반환: Available = 5+5 = 10
  3. P2의 Need=7 ≤ Available(10) → P2 실행 가능 → 완료
- 안전 순서 `P1 → P0 → P2` 존재 → **Safe State** → 이 상태로의 자원 할당은 허용.
- 만약 어떤 순서로도 모든 프로세스의 Need를 채울 수 없으면 **Unsafe State**로 판단해 해당 요청을 아예 거절(데드락이 아니라 "데드락 가능성이 있는 상태"를 사전 차단).

## 9. 핵심 포인트
- 🔴 "데드락은 조건 하나만 있어도 발생" — ❌ **4조건 모두** 성립해야. 예방 = 하나 깨기.
- 🟡 실무 예방: **락 획득 순서 통일**(환형 대기 차단)이 가장 흔한 방법.
- 🟡 Java 데드락도 동일 — 두 스레드가 두 락을 반대 순서로 잡으면 발생.
- 🔴 "뮤텍스와 세마포어(binary)는 완전히 같다" — ❌ 값만 보면 비슷해도 **소유 개념 유무**가 다름. 세마포어는 남이 대신 풀어줄 수 있음(신호 용도), 뮤텍스는 안 됨.
- 🟡 은행원 알고리즘은 **회피(Avoidance)** 전략 — 예방(Prevention)처럼 조건 자체를 없애는 게 아니라, 매 요청마다 "이래도 안전한가"를 계산해 위험한 요청만 거절.
- 🔴 "Safe State가 아니면 곧바로 데드락" — ❌ Unsafe State는 데드락이 **될 수도 있는** 상태일 뿐, 반드시 데드락은 아님. 은행원 알고리즘은 안전을 위해 보수적으로 거절.

## 10. 예상 면접 질문
**Q1. "뮤텍스와 세마포어 차이?"**
> 뮤텍스는 소유 개념이 있는 이진 락(잠근 스레드만 해제), 세마포어는 N개 자원을 허용하는 카운터(소유 개념 없음). 임계구역 상호배제엔 뮤텍스, 자원 개수 제한이나 스레드 간 신호에는 세마포어를 씁니다.

**Q2. "데드락 발생 조건과 예방법?"**
> 상호배제·점유와대기·비선점·환형대기 4가지가 동시 성립 시 발생. 하나만 깨면 예방되며, 실무에선 락 획득 순서를 통일해 환형 대기를 제거하는 방법을 주로 씁니다.

**Q3. "OS 동기화와 Java 동시성의 관계?"**
> Java `synchronized`는 OS 모니터/뮤텍스를 언어로 감싼 것이고, `volatile`의 가시성 문제는 CPU 캐시와 메인메모리 사이 문제입니다. 결국 같은 상호배제·가시성 문제를 다른 층위에서 다룹니다.

**Q4. "세마포어로 생산자-소비자 문제를 어떻게 푸나요?"**
> 빈 슬롯 수를 세는 empty 세마포어와 채워진 슬롯 수를 세는 full 세마포어, 그리고 버퍼 접근 자체를 보호하는 이진 세마포어(mutex) 세 개를 씁니다. 생산자는 empty를 기다렸다가 넣고 full을 늘리고, 소비자는 full을 기다렸다가 꺼내고 empty를 늘립니다. wait(empty)를 wait(mutex)보다 먼저 호출해야 데드락을 피할 수 있습니다.

**Q5. "은행원 알고리즘은 예방(Prevention)과 뭐가 다른가요?"**
> 예방은 데드락 4조건 중 하나를 아예 성립 못 하게 구조적으로 막는 것이고, 회피(은행원 알고리즘)는 조건은 남겨두되 매 자원 요청마다 시뮬레이션해서 안전 상태를 유지할 수 있는 요청만 허용하는 방식입니다. 더 유연하지만 각 프로세스의 최대 필요량을 미리 알아야 하고 계산 비용이 듭니다.

# O5. I/O Models

**학습 목표**: *"블로킹/논블로킹, 동기/비동기 차이?"* 에 답할 수 있다. (JS 이벤트 루프·Netty와 연결)

## 1. 비유 — 라면 끓이기
- **블로킹 동기**: 냄비 앞에 서서 물 끓을 때까지 **아무것도 안 하고 기다림**.
- **논블로킹 동기**: 물 끓나 **계속 들여다보며**(폴링) 다른 일도 조금씩.
- **비동기**: 타이머 맞춰두고 **딴 일 하다가**, 다 되면 알림 받음(콜백).

## 2. 두 축으로 구분
- **블로킹 vs 논블로킹**: 호출이 **즉시 리턴하느냐**(제어권). 블로킹=결과 날 때까지 멈춤, 논블로킹=바로 리턴(아직이면 "아직" 응답).
- **동기 vs 비동기**: 결과를 **누가 신경 쓰느냐**(완료 통지). 동기=호출자가 결과 확인, 비동기=완료되면 콜백/이벤트로 통지.

## 3. 4가지 조합표
| 모델 | 설명 | 예 |
|------|------|-----|
| 블로킹+동기 | 호출 후 결과까지 멈춤 | 전통 `read()` |
| 논블로킹+동기 | 바로 리턴, 호출자가 반복 폴링 | `O_NONBLOCK` + 폴링 |
| I/O 멀티플렉싱 | 여러 fd를 한 스레드가 감시 | `select`/`poll`/**`epoll`** |
| 비동기(AIO) | 커널이 완료 후 통지 | `io_uring`, 콜백 |

## 4. JS 이벤트 루프·Netty와의 연결 ⭐ (cross-link)
- **JS 이벤트 루프**: 싱글 스레드가 **논블로킹 I/O + 이벤트 큐**로 동시성 확보. I/O를 커널/스레드풀에 맡기고 완료되면 콜백을 큐에 넣어 처리 → **블로킹 안 하니 한 스레드로 수천 연결**.
- **epoll/멀티플렉싱**: 한 스레드가 다수 소켓을 감시 → Node.js·Netty·Nginx의 고성능 비결.
- 정리: "왜 Node는 싱글 스레드인데 빠른가?" = **논블로킹 I/O + 이벤트 루프**로 CPU를 놀리지 않기 때문.

## 5. 핵심 포인트
- 🔴 "논블로킹 = 비동기" — ❌ 다른 축. 논블로킹+동기(폴링)도 있음.
- 🟡 스레드-per-connection(블로킹)은 연결 많으면 스레드 폭발 → **이벤트 기반(epoll)** 이 C10K 문제 해법.

## 6. 예상 면접 질문
**Q1. "블로킹/논블로킹과 동기/비동기 차이?"**
> 블로킹/논블로킹은 호출이 즉시 리턴하는지(제어권), 동기/비동기는 결과 완료를 누가 통지하는지(관심). 논블로킹이면서 동기(폴링)도 가능해 별개 축입니다.

**Q2. "Node.js가 싱글 스레드인데 어떻게 동시 처리를 하나요?"**
> 논블로킹 I/O로 요청을 커널/스레드풀에 맡기고, 이벤트 루프가 완료 콜백을 큐에서 꺼내 처리합니다. CPU가 I/O 대기로 놀지 않아 한 스레드로 많은 연결을 감당합니다. 단, CPU 집약 작업엔 부적합.

**Q3. "select/poll과 epoll의 차이는?"**
> select/poll은 감시할 fd 목록을 매번 커널에 통째로 넘기고 커널이 전체를 순회(O(n))하는 반면, epoll은 커널에 관심 있는 fd를 미리 등록해두고 준비된 fd만 알려줘서(이벤트 기반) 연결 수가 많아져도 성능이 크게 떨어지지 않습니다. 그래서 C10K 문제의 표준 해법으로 자리잡았습니다.

# 핵심 질문 (Quiz)

> 답변을 먼저 떠올린 뒤 펼쳐서 확인하세요.

<details>
<summary>Q1. 프로세스와 스레드의 차이는?</summary>

- 프로세스: 독립된 메모리(주소 공간)를 가진 실행 단위. 다른 프로세스와 자원 공유 없음.
- 스레드: 프로세스 안에서 Code/Data/Heap을 **공유**하고 Stack·레지스터·PC만 개별로 갖는 실행 흐름.
- 스레드는 통신이 빠르지만 공유 자원 때문에 동기화 문제가 생김.
- 프로세스는 격리 덕분에 하나가 죽어도 다른 프로세스는 안전, 대신 컨텍스트 스위치가 무거움(주소 공간·TLB flush 포함).

</details>

<details>
<summary>Q2. 시스템 콜이 무엇이고 왜 필요한가?</summary>

- User Mode 프로그램은 하드웨어(디스크, 메모리 관리, I/O 포트 등)에 직접 접근 불가.
- 파일 열기, 네트워크 전송, 메모리 할당 등이 필요하면 커널에 요청하는 인터페이스 = 시스템 콜.
- 흐름: 시스템 콜 호출(트랩) → User→Kernel 모드 전환 → 커널 처리 → Kernel→User 전환 → 재개.
- 모드 전환 자체가 오버헤드 → 잦은 시스템 콜(1바이트씩 read 등)은 성능 저하 → 버퍼링(`BufferedReader`)으로 횟수 감소.

</details>

<details>
<summary>Q3. 선점형(Preemptive)과 비선점형(Non-preemptive) 스케줄링의 차이는?</summary>

- 비선점: 실행 중인 작업이 스스로 CPU를 반납할 때까지 유지 (예: FCFS, SJF).
- 선점: OS가 강제로 CPU를 회수 가능 (타임 슬라이스 만료, 더 급한 작업 도착 — 예: Round Robin, Priority-preemptive).
- 선점형이 응답성은 좋지만 컨텍스트 스위치 비용과 동기화 고려가 늘어남.

</details>

<details>
<summary>Q4. 페이지 폴트(Page Fault) 처리 과정은?</summary>

- 접근한 가상 페이지가 RAM에 없음 → 트랩(page fault) 발생.
- OS가 디스크에서 해당 페이지를 빈 프레임(없으면 페이지 교체 알고리즘으로 victim 선정)에 로드.
- 페이지 테이블 갱신 후 명령 재실행.
- 페이지 폴트가 과도하면 **Thrashing**(CPU가 스와핑만 하며 성능 급락).

</details>

<details>
<summary>Q5. Belady's Anomaly란 무엇이며 어떤 알고리즘에서 발생하는가?</summary>

- 일반 직관: 프레임(캐시) 수를 늘리면 페이지 폴트가 줄어야 함.
- **FIFO 교체 알고리즘**은 프레임을 늘려도 오히려 폴트가 **늘어나는** 경우가 존재(반례: `1 2 3 4 1 2 5 1 2 3 4 5`).
- 원인: FIFO는 "얼마나 오래 있었나"만 보고 "얼마나 자주/최근에 쓰였나"는 무시.
- LRU와 Optimal(OPT)은 스택 알고리즘 성질을 만족해 이 변칙이 발생하지 않음 → 실무는 FIFO보다 LRU 계열 선호.

</details>

<details>
<summary>Q6. 뮤텍스(Mutex)와 세마포어(Semaphore)의 차이는?</summary>

| 구분 | Mutex | Semaphore |
|------|-------|-----------|
| 값 범위 | 0/1 (binary) | 0~N (counting) 또는 binary |
| 소유(ownership) | 있음 — 잠근 스레드만 해제 가능 | 없음 — 다른 스레드가 signal 가능 |
| 용도 | 상호 배제(임계구역 보호) | 자원 개수 제한 + 스레드 간 신호(signaling) |
| 우선순위 역전 | 소유자 명확 → 우선순위 상속 적용 가능 | 소유자 없어 적용 어려움 |

- 핵심 한 줄: 뮤텍스는 "누가 잠갔는지"가 있어 그 사람만 풀 수 있는 자물쇠, 세마포어는 소유 없이 카운트만 관리하는 신호기.

</details>

<details>
<summary>Q7. 데드락(Deadlock) 발생 조건과 예방법은?</summary>

- 4가지 필요조건이 **동시에** 성립해야 발생(하나만 깨도 예방):
  1. 상호 배제(Mutual Exclusion)
  2. 점유와 대기(Hold and Wait)
  3. 비선점(No Preemption)
  4. 환형 대기(Circular Wait)
- 처리 전략: 예방(조건 자체 차단) / 회피(은행원 알고리즘 — 안전 상태만 허용) / 탐지·회복(주기적 탐지 후 강제 종료) / 무시(Ostrich).
- 실무에서 가장 흔한 예방법: **락 획득 순서 통일**(환형 대기 제거).

</details>

<details>
<summary>Q8. 블로킹/논블로킹과 동기/비동기의 차이는?</summary>

- 블로킹 vs 논블로킹: 호출이 **즉시 리턴하느냐**(제어권 관점). 블로킹=결과 날 때까지 멈춤, 논블로킹=바로 리턴.
- 동기 vs 비동기: 결과를 **누가 신경 쓰느냐**(완료 통지 관점). 동기=호출자가 직접 확인, 비동기=콜백/이벤트로 통지.
- 두 축은 독립적 → 논블로킹+동기(폴링)도 존재.
- 예: 블로킹+동기(전통 `read()`), 논블로킹+동기(`O_NONBLOCK` 폴링), 멀티플렉싱(`select`/`poll`/`epoll`), 비동기(`io_uring`).

</details>
