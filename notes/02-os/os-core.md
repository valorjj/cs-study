# O1. Process and Thread

**학습 목표**: *"프로세스와 스레드 차이는?"* / *"컨텍스트 스위칭이 뭐죠?"* 에 5분 답할 수 있다.

## 1. 비유 — 회사와 직원
- **프로세스** = 독립된 **회사 하나**. 자체 사무실(주소 공간)·금고(메모리)·비품을 가짐. 다른 회사와 자원 공유 안 함.
- **스레드** = 회사 안의 **직원**. 같은 사무실(주소 공간)·금고를 **공유**하며 각자 자기 책상(스택)에서 일함.

핵심 직관: **프로세스 간은 격리(독립 메모리), 스레드 간은 공유(같은 프로세스 메모리)**. 그래서 스레드는 통신이 빠르지만 동기화 문제가 생긴다.

## 2. 개념 정의
> **프로세스** = 실행 중인 프로그램. OS가 자원을 할당하는 단위. PCB(Process Control Block)로 관리.
> **스레드** = 프로세스 내 실행 흐름의 단위. CPU 스케줄링의 단위.

## 3. 다이어그램 — 메모리 공유 경계
```
┌──────────── Process A ────────────┐   ┌──── Process B ────┐
│  Code | Data | Heap  (공유)        │   │  (완전히 독립)     │
│  ┌─────────┬─────────┬─────────┐   │   │                    │
│  │Thread 1 │Thread 2 │Thread 3 │   │   │   Thread 1         │
│  │ Stack   │ Stack   │ Stack   │   │   │   Stack            │
│  │ 레지스터 │ 레지스터 │ 레지스터 │   │   │                    │
│  └─────────┴─────────┴─────────┘   │   │                    │
└────────────────────────────────────┘   └────────────────────┘
    스레드끼리 Code/Data/Heap 공유          프로세스는 서로 격리
    → Stack·레지스터·PC만 스레드별
```
> Java의 "Heap은 공유, Stack은 스레드별"(JVM T1)이 바로 이 그림. JVM 스레드 = OS 스레드에 매핑.

## 4. 비교표
| 구분 | 프로세스 | 스레드 |
|------|----------|--------|
| 메모리 | 독립(격리) | Code/Data/Heap 공유, Stack은 개별 |
| 생성 비용 | 큼(자원 복제) | 작음 |
| 통신 | IPC 필요(파이프, 소켓, 공유메모리) | 공유 메모리로 즉시 |
| 한 개 죽으면 | 다른 프로세스 영향 없음 | 같은 프로세스 전체 위험 |
| 컨텍스트 스위치 | 무거움(주소 공간 교체) | 가벼움(주소 공간 유지) |

## 5. 컨텍스트 스위칭 (Context Switching)
- CPU가 실행 중이던 작업의 상태(레지스터, PC 등)를 **PCB/TCB에 저장**하고, 다음 작업의 상태를 **복원**하는 것.
- 비용: 이때 CPU는 실제 일을 안 함(오버헤드). 프로세스 스위치는 주소 공간(페이지 테이블·TLB flush)까지 바꿔 더 비쌈.
- 스레드 스위치는 같은 주소 공간이라 TLB flush 없어 상대적으로 저렴.

## 6. User Mode vs Kernel Mode, 시스템 콜

### 비유 — 은행 창구 뒤 금고
고객(응용 프로그램)은 창구 앞(**User Mode**)에서만 일할 수 있고, 금고(하드웨어 자원)는 직원(**Kernel**)만 직접 만짐. 고객이 돈을 인출하려면 창구 직원에게 **요청서(System Call)** 를 제출해야 함 — 직접 금고에 들어갈 수 없다.

- **User Mode**: 응용 프로그램이 실행되는 제한된 모드. 하드웨어 직접 접근(디스크, 메모리 관리, I/O 포트 등) 불가.
- **Kernel Mode**: OS 커널이 실행되는 특권 모드. 모든 명령어·자원에 접근 가능.
- **시스템 콜(System Call)**: User Mode 프로그램이 파일 열기, 네트워크 전송, 메모리 할당 등 커널 기능이 필요할 때 커널에 요청하는 인터페이스. `read()`, `write()`, `fork()`, `malloc()`(내부적으로 `brk`/`mmap`) 등.
  - 흐름: 사용자 프로그램 → 시스템 콜 호출(트랩/소프트웨어 인터럽트) → **모드 전환(User→Kernel)** → 커널이 요청 처리 → **모드 전환(Kernel→User)** → 실행 재개.
  - Java에서 `System.out.println()`, `new FileInputStream()`, 소켓 I/O 모두 내부적으로 시스템 콜을 호출한다.

### 컨텍스트 스위칭 비용과의 관계
- 시스템 콜 자체도 **모드 전환** 비용이 든다(스레드/프로세스 컨텍스트 스위치와는 별개지만 둘 다 "지금 하던 일 잠깐 멈추고 다른 모드/작업으로 전환"이라는 공통점).
- 시스템 콜이 잦으면(예: 파일을 1바이트씩 읽기) 모드 전환 오버헤드가 누적됨 → **버퍼링**(`BufferedReader` 등)으로 시스템 콜 횟수를 줄이는 이유가 여기 있다.
- 컨텍스트 스위치 비용 요소: ① 레지스터/PC 저장·복원 ② (프로세스면) 페이지 테이블 교체 + TLB flush ③ 캐시 미스 증가(다른 프로세스 데이터로 캐시 재적재) ④ 스케줄러 실행 자체의 오버헤드.

## 7. 프로세스 상태 다이어그램

```
        생성됨              스케줄러가 선택
  ┌──▶ [ New ] ──────▶ [ Ready ] ──────────▶ [ Running ]
  │                        ▲                    │  │
  │                        │   타임슬라이스 만료/  │  │
  │                        └── 선점(preempt) ────┘  │
  │                                                  │ I/O·이벤트 대기
  │                                                  ▼
  │                                            [ Waiting ]
  │                                                  │
  │              I/O 완료 등                          │
  │              ◄──────────────────────────────────┘
  │                        │
  │                        ▼
  └──────────────────  [ Terminated ] ◄── Running에서 종료
```
- **New**: 프로세스 생성 중(PCB 할당). **Ready**: CPU만 배정되면 바로 실행 가능(준비 큐 대기). **Running**: CPU 점유 중. **Waiting(Blocked)**: I/O 등 이벤트 대기 중(CPU 못 씀). **Terminated**: 실행 종료, 자원 회수 대기.
- 헷갈리는 포인트: **Ready→Running**은 스케줄러의 선택, **Running→Ready**는 선점(타임아웃 등), **Running→Waiting**은 자발적(I/O 요청) — 이 방향은 스케줄러가 아니라 프로세스 자신이 유발.

## 8. IPC(Inter-Process Communication) 종류 비교

프로세스는 메모리가 격리돼 있어 서로 데이터를 주고받으려면 OS가 제공하는 IPC 메커니즘이 필요하다.

| 방식 | 개념 | 특징 | 예 |
|------|------|------|-----|
| **파이프(Pipe)** | 커널 버퍼를 통한 단방향(named pipe는 양방향 가능) 스트림 | 보통 **부모-자식 관계** 프로세스 간, 관련 없는 프로세스는 named pipe(FIFO) 필요 | 셸의 `\|` |
| **공유 메모리(Shared Memory)** | 같은 물리 메모리 영역을 여러 프로세스 주소 공간에 매핑 | **가장 빠름**(커널 개입 없이 직접 읽기/쓰기), 대신 동기화(세마포어 등)를 직접 구현해야 함 | `shmget`, Java의 `MappedByteBuffer` |
| **메시지 큐(Message Queue)** | 커널이 관리하는 큐에 메시지 단위로 송수신 | 프로세스 간 관계 없어도 됨, 메시지 경계 유지, 우선순위 지정 가능 | POSIX MQ, Kafka(응용 계층 버전) |
| **소켓(Socket)** | 네트워크 프로토콜 기반 양방향 통신 | **같은 머신이든 다른 머신이든** 동작(유일하게 네트워크 확장 가능), 오버헤드 가장 큼 | TCP/UDP 소켓, HTTP |
| **시그널(Signal)** | 비동기 이벤트 통지(데이터 전달 아님) | 매우 가벼움, 정해진 신호 종류만 | `SIGKILL`, `SIGTERM` |

> 성능 순서(대략): 공유 메모리 > 파이프 ≈ 메시지 큐 > 소켓. 하지만 공유 메모리는 동기화를 직접 짜야 해서 구현 난이도가 가장 높다.

## 9. 핵심 포인트 (자주 하는 실수)
- 🔴 "스레드는 메모리를 전부 공유" — ❌ **Stack·레지스터·PC는 스레드별**. Heap/Data/Code만 공유.
- 🔴 "멀티프로세스가 항상 느리다" — 격리로 **안정성**↑(한 프로세스 죽어도 나머지 생존). Chrome 탭 분리가 예.
- 🟡 멀티스레드는 공유 자원 → **동기화 문제**(O4로 연결).
- 🔴 "시스템 콜은 그냥 함수 호출" — ❌ 일반 함수 호출과 달리 **모드 전환(트랩)** 이 일어나 비용이 더 크다.
- 🟡 "Waiting에서 Ready로 못 가고 바로 Running" — ❌ I/O 완료해도 **Ready 큐로 돌아가 스케줄러 선택을 기다림**, 곧바로 CPU를 얻지 않음.

## 10. 예상 면접 질문 + 답변 골격
**Q1. "프로세스와 스레드 차이?"**
> 프로세스는 독립 메모리를 가진 실행 단위, 스레드는 그 안에서 Code/Data/Heap을 공유하며 각자 Stack만 갖는 실행 흐름입니다. 스레드는 통신이 빠르나 공유 자원 동기화 이슈가 있고, 프로세스는 격리로 안정적이나 컨텍스트 스위치가 무겁습니다.

**꼬리 Q1-1. "그럼 멀티스레드 대신 멀티프로세스를 일부러 택하는 경우는?"**
> 격리가 안정성·보안으로 직결될 때입니다. 크롬이 탭마다 프로세스를 분리하는 이유는 한 탭이 크래시해도 브라우저 전체가 안 죽고, 사이트 간 메모리 침범을 막기 위해서입니다. 또 Python처럼 GIL로 스레드 병렬 CPU 연산이 막힌 언어는 `multiprocessing`으로 코어를 실제로 나눠 씁니다. Java 백엔드에서는 보통 스레드로 충분하지만, 배치 잡을 별도 JVM 프로세스로 띄워 장애 격리와 힙 튜닝을 독립적으로 하는 선택도 이 맥락입니다.

**꼬리 Q1-2. "JVM 스레드와 OS 스레드는 어떤 관계인가요?"**
> HotSpot은 Java 스레드 하나를 OS 커널 스레드 하나에 1:1로 매핑합니다(1:1 모델). 그래서 스레드 생성·스위치 비용이 OS에 그대로 실리고, 수만 개를 만들면 메모리(스레드당 스택 ~1MB)와 스위치 비용이 커집니다. Java 21의 Virtual Thread(Project Loom)는 이 한계를 풀려고 다수의 가상 스레드를 소수의 OS 캐리어 스레드에 M:N으로 태워, I/O 대기 시 OS 스레드를 붙잡지 않고 반납하도록 만든 것입니다.

**Q2. "컨텍스트 스위칭이 뭐고 왜 비싼가요?"**
> CPU가 현재 작업의 레지스터·PC를 PCB/TCB에 저장하고 다음 작업의 상태를 복원하는 것입니다. 그동안 실제 연산을 못 해 순수 오버헤드이고, 프로세스 전환은 주소 공간·페이지 테이블·TLB까지 교체하며, 여기에 더해 캐시가 새 데이터로 재적재되며 발생하는 캐시 미스가 실제로는 더 큰 숨은 비용입니다.

**꼬리 Q2-1. "스위치 비용을 직접 측정하거나 줄여본 경험이 있나요?"**
> 스레드 풀 크기를 코어 수 대비 과도하게 크게 잡았을 때 `vmstat`의 `cs`(context switch) 수치가 치솟고 CPU는 높은데 처리량이 안 오르는 패턴을 본 적이 있습니다. CPU 바운드 작업은 풀 크기를 코어 수 근처로 낮춰 불필요한 스위치를 줄였고, 반대로 I/O 바운드는 대기 시간을 메우려 코어 수보다 크게 잡되 `cs`와 응답시간을 보며 조정했습니다. 핵심은 "스레드를 많이 만들수록 빨라진다"가 아니라는 점입니다.

**Q3. "시스템 콜이 뭔가요? 왜 필요하죠?"**
> 사용자 프로그램이 하드웨어·커널 자원(파일, 네트워크, 메모리)에 접근하려 할 때 커널에 요청하는 인터페이스입니다. User Mode는 직접 접근 권한이 없어 트랩으로 Kernel Mode로 전환해 처리하고 돌아옵니다. 이 모드 전환 자체가 오버헤드라 잦은 시스템 콜은 성능에 영향을 줍니다.

**꼬리 Q3-1. "그 오버헤드를 실제 코드에서 줄인 예를 들어보세요."**
> 파일을 1바이트씩 `read`하면 매 바이트마다 시스템 콜이 나가 모드 전환이 누적됩니다. `BufferedReader`/`BufferedInputStream`은 커널에서 한 번에 8KB씩 당겨와 유저 버퍼에 채워두고 그 안에서 잘라주므로 시스템 콜 횟수를 수천 분의 1로 줄입니다. 로그를 라인마다 flush하다 처리량이 안 나와, 버퍼링 후 주기적 flush로 바꿔 시스템 콜·디스크 I/O를 줄인 경험이 이 원리와 같습니다.

**Q4. "공유 메모리와 소켓 중 IPC로 뭘 쓸지 어떻게 고르나요?"**
> 같은 머신에서 대용량·저지연이 최우선이면 공유 메모리(대신 세마포어 등 동기화를 직접 구현), 프로세스 관계와 무관하게 구조화된 메시지를 비동기로 주고받으려면 메시지 큐, 다른 머신으로 확장될 여지가 있으면 처음부터 소켓을 씁니다. 실무에서는 성능이 압도적으로 부족한 게 아니라면, 확장성과 유지보수가 좋은 소켓/메시지 큐를 기본값으로 두고 공유 메모리는 프로파일링으로 병목이 확인됐을 때만 도입하는 편이 안전합니다.

# O2. CPU Scheduling

**학습 목표**: *"스케줄링 알고리즘 종류와 선점/비선점 차이는?"* 에 답할 수 있다.

## 1. 비유 — 은행 창구
직원(CPU) 하나, 손님(프로세스) 여럿. 누구를 먼저 처리할지 규칙이 **스케줄링**.
- 번호표 순서대로 = FCFS
- 볼일 짧은 손님 먼저 = SJF
- 정해진 시간만큼 돌아가며 = Round Robin

## 2. 선점 vs 비선점
- **비선점(Non-preemptive)**: 실행 중인 작업이 **스스로 CPU를 놓을 때까지** 안 뺏음. (FCFS, SJF)
- **선점(Preemptive)**: OS가 **강제로 CPU를 회수** 가능(타임 슬라이스 만료, 더 급한 작업 도착). (Round Robin, Priority-preemptive)

## 3. 알고리즘 비교표
| 알고리즘 | 방식 | 장점 | 단점 |
|----------|------|------|------|
| **FCFS** | 먼저 온 순 | 단순·공정 | Convoy 효과(긴 작업이 뒤 막음) |
| **SJF** | 짧은 작업 먼저 | 평균 대기시간 최소 | 실행시간 예측 어려움, starvation |
| **Priority** | 우선순위 높은 순 | 중요 작업 우선 | 낮은 우선순위 starvation → **Aging**으로 완화 |
| **Round Robin** | 타임 퀀텀 순환 | 응답성 좋음(대화형) | 퀀텀 너무 작으면 스위치 오버헤드 |
| **MLFQ** | 다단계 큐+피드백 | 실무형(짧은 작업 우대+적응) | 복잡 |

## 4. 평가 기준
- **처리량(Throughput)**: 단위시간 완료 작업 수
- **응답시간(Response)**: 요청 후 첫 반응까지 (대화형 중요)
- **대기시간(Waiting)**: 준비 큐에서 기다린 시간
- **반환시간(Turnaround)**: 제출~완료 전체

## 5. 핵심 포인트
- 🟡 **starvation**(기아): 낮은 우선순위가 계속 밀림 → **Aging**(대기할수록 우선순위↑)으로 해결.
- 🟡 Round Robin의 퀀텀: 크면 FCFS화, 작으면 스위치 오버헤드 → 균형이 중요.

## 6. 예상 면접 질문
**Q1. "선점형과 비선점형 스케줄링 차이?"**
> 비선점은 실행 중 작업이 스스로 CPU를 반납할 때까지 유지(FCFS/SJF), 선점은 OS가 타임 슬라이스 만료나 더 급한 작업 도착 시 강제 회수합니다(Round Robin). 선점형이 응답성과 공정성은 좋지만 컨텍스트 스위치 비용이 늘고, 임계구역 도중 선점될 수 있어 동기화를 더 신경 써야 합니다.

**Q2. "MLFQ가 실무에서 선호되는 이유는?"**
> 실행 시간을 미리 몰라도 되기 때문입니다. 짧은 작업은 높은 우선순위 큐에서 빨리 끝내 응답성을 확보하고(대화형 우대), CPU를 오래 쓰는 작업은 점차 낮은 큐로 내려 자동으로 CPU 집약/대화형을 구분합니다. Aging을 결합해 starvation도 막습니다. SJF의 장점(짧은 작업 우대)을 실행 시간 예측 없이 근사한다는 점이 실무적입니다.

**꼬리 Q2-1. "리눅스 CFS는 어떻게 공정성을 달성하나요?"**
> CFS는 우선순위 큐 대신 각 태스크의 누적 실행 시간(vruntime)을 레드-블랙 트리로 관리해, 항상 vruntime이 가장 작은(= 가장 덜 실행된) 태스크를 다음에 실행합니다. 이렇게 하면 모든 태스크가 CPU를 "공평하게 나눠 받은 것처럼" 수렴합니다. nice 값은 vruntime이 쌓이는 가중치를 조절해 우선순위를 반영합니다. 즉 고정된 타임 퀀텀이 아니라 상대적 실행량 기반이라는 게 전통 RR과의 차이입니다.

**Q3(TRADE-OFF). "Round Robin 타임 퀀텀을 크게 vs 작게 잡을 때의 트레이드오프는?"**
> 퀀텀이 너무 크면 사실상 FCFS가 되어 대화형 응답성이 떨어지고, 너무 작으면 컨텍스트 스위치가 잦아 스위치 오버헤드가 유효 처리량을 갉아먹습니다. 그래서 스위치 비용보다 충분히 크되(보통 수~수십 ms) 대화형 반응이 체감되지 않을 만큼 작게 잡는 균형점을 찾습니다. 이건 스레드 풀 사이징이나 배치 청크 크기를 정할 때의 "오버헤드 vs 반응성" 트레이드오프와 같은 결의 문제입니다.

# O3. Virtual Memory and Paging

**학습 목표**: *"가상 메모리가 뭐고 왜 쓰나요?"* / *"페이지 폴트는?"* 에 답할 수 있다. (JVM GC·메모리와 직접 연결)

## 1. 비유 — 도서관 열람실
책(실제 물리 메모리, RAM)은 한정. 하지만 이용자에겐 "원하는 책 다 있는 것처럼" 보이게 함. 실제론 대부분 서고(디스크)에 있고, 필요할 때만 열람실(RAM)로 가져옴 = **가상 메모리**.

## 2. 개념 정의
> **가상 메모리** = 각 프로세스에 **실제 RAM보다 큰 연속된 주소 공간**이 있는 것처럼 보이게 하는 기법. 물리 메모리 한계를 넘어 실행 가능 + 프로세스 간 격리.
> **페이징** = 가상 주소 공간을 고정 크기 **페이지**로, 물리 메모리를 **프레임**으로 나눠 매핑.

## 3. 다이어그램 — 주소 변환
```
 가상 주소 (프로세스가 보는 주소)
      │  [ 페이지 번호 | 오프셋 ]
      ▼
 ┌─────────────┐   TLB(캐시) 히트 시 초고속
 │ Page Table  │◄──── TLB (최근 변환 캐시)
 │ (페이지→프레임)│
 └──────┬──────┘
        ▼
 물리 주소 [ 프레임 번호 | 오프셋 ] → RAM
        │
        └─ 페이지가 RAM에 없음 → Page Fault → 디스크에서 로드
```

## 4. 핵심 메커니즘
- **페이지 테이블**: 가상 페이지 → 물리 프레임 매핑표 (프로세스별).
- **TLB (Translation Lookaside Buffer)**: 최근 주소 변환을 캐싱하는 하드웨어. 히트하면 페이지 테이블 안 거침. **JIT의 code cache나 CPU 캐시와 같은 "자주 쓰는 걸 빠르게" 발상**.
- **페이지 폴트(Page Fault)**: 접근한 페이지가 RAM에 없음 → OS가 디스크에서 프레임으로 로드(느림). **Demand Paging**(필요할 때만 로드).
- **페이지 교체(Page Replacement)**: RAM 꽉 참 → 어떤 페이지를 내보낼지. **LRU**(가장 오래 안 쓴 것), FIFO, Optimal.

## 5. JVM GC와의 연결 ⭐ (cross-link)
| OS 메모리 관리 | JVM GC |
|----------------|--------|
| 안 쓰는 페이지를 디스크로 내보냄(교체) | 도달 불가 객체를 회수 |
| LRU로 교체 대상 선정 | 세대 가설로 Young 우선 수거 |
| Page Fault = RAM에 없어 디스크 접근 | Major GC = Old 정리(느림) |
> 둘 다 "한정된 빠른 메모리를 효율적으로 재활용"하는 문제. 면접에서 GC 얘기하다 "OS도 비슷하게 페이지 교체를..."로 확장하면 좋다.

## 6. 페이지 교체 알고리즘 상세

목표: RAM(프레임)이 꽉 찼을 때 **어떤 페이지를 내보낼지** 결정. 잘못 고르면 곧바로 다시 폴트(→ 성능 저하).

### LRU (Least Recently Used)
> "가장 오래전에 사용된 페이지를 내보낸다." — 과거를 보고 미래를 예측(최근에 안 썼으면 앞으로도 안 쓸 확률↑)하는 **지역성(locality)** 가정.

구현 방법:
- **연결 리스트 + 이동**: 페이지 접근 시마다 해당 노드를 리스트 맨 앞으로 이동. 교체 시 맨 뒤(꼬리) 제거. → 접근마다 O(1) 재배치 가능(연결 리스트라서), 하지만 하드웨어 지원 없이 매 메모리 접근마다 하기엔 비쌈.
- **카운터/타임스탬프**: 각 페이지에 마지막 접근 시각 기록, 교체 시 전체 스캔해 최솟값 탐색 → 스캔 비용 O(n).
- 실제 하드웨어는 완벽한 LRU를 구현하기엔 비용이 커서, 근사 알고리즘인 **Clock**을 대신 쓴다.
- (Java 개발자용 비유: `LinkedHashMap(accessOrder=true)` + `removeEldestEntry`로 LRU 캐시를 구현하는 것과 원리가 동일 — Java T3 자료구조에서 이미 다룬 패턴.)

### Clock 알고리즘 (Second-Chance)
LRU를 정확히 구현하는 대신, 저렴하게 근사하는 방법.
```
        ┌───┐
     ┌─▶│ P1│ref=1
     │  └───┘
     │   ▼
   ┌───┐   ┌───┐
   │P4 │   │P2 │ref=0  ◄── 시계 바늘(hand)이 여기 있으면:
   │ref│   │ref│         ref=0 → 이 페이지를 교체(victim)
   │=0 │   │=0 │         ref=1 → 0으로 낮추고 다음 칸으로(한 번의 기회 부여)
   └───┘   └───┘
     ▲   ┌───┐
     └───│ P3│ref=1
         └───┘
```
- 각 페이지에 **참조 비트(reference bit)** 1개. 접근하면 OS/하드웨어가 1로 세팅.
- 교체 필요 시 시계 바늘이 프레임을 순회: `ref=0`이면 즉시 교체, `ref=1`이면 **0으로 내리고(2nd chance) 다음으로 넘어감**.
- 전체 스캔·정밀한 순서 추적 없이 "최근에 안 쓰인 것 같은" 페이지를 찾아 LRU를 값싸게 근사 → 대부분 실제 OS(Linux 등)가 이 계열 사용.

### Belady's Anomaly (벨레이디의 변칙)
> 직관: "프레임(캐시)을 늘리면 페이지 폴트가 줄어야 한다." — **FIFO에서는 항상 성립하지 않는다.**
- FIFO 교체 알고리즘은 프레임 수를 늘렸는데 오히려 폴트 횟수가 **늘어나는** 경우가 존재(반례로 유명한 참조열: `1 2 3 4 1 2 5 1 2 3 4 5`).
- 원인: FIFO는 "얼마나 오래 있었나"만 보고 "얼마나 자주/최근에 쓰였나"는 무시 → 자주 쓰이는 페이지도 그냥 오래됐다는 이유로 쫓아냄.
- **LRU와 Optimal(OPT, 미래를 안다고 가정하는 이상적 알고리즘)은 이 변칙이 발생하지 않음**(스택 알고리즘 성질을 만족하기 때문). 그래서 실무/이론 모두 FIFO보다 LRU 계열을 선호.

## 7. 핵심 포인트 (자주 하는 실수)
- 🔴 "가상 메모리 = 디스크 스왑" — ❌ 스왑은 가상 메모리의 **일부 구현**. 가상 메모리의 본질은 **주소 추상화 + 격리**.
- 🔴 "페이징에 외부 단편화 있다" — ❌ **페이징은 외부 단편화 없음**(고정 크기). 대신 **내부 단편화**(마지막 페이지 낭비). 세그먼테이션이 외부 단편화.
- 🟡 **Thrashing**: 페이지 폴트가 과도해 CPU가 스와핑만 하는 상태 → 성능 급락.
- 🔴 "프레임을 늘리면 무조건 폴트가 준다" — ❌ **FIFO는 Belady's Anomaly로 예외 발생**. LRU/OPT는 그런 예외가 없다.
- 🟡 완벽한 LRU는 비용이 커서 실제 OS는 **Clock(2nd chance)** 같은 근사 알고리즘을 쓴다.

## 8. 예상 면접 질문
**Q1. "가상 메모리를 왜 쓰나요?"**
> ① 물리 RAM보다 큰 프로그램 실행(디맨드 페이징) ② 프로세스 간 주소 공간 격리로 보안·안정 확보 ③ 각 프로세스가 0번지부터 연속 주소를 보게 해 컴파일·링킹·재배치를 단순화. 요약하면 "주소 추상화 + 격리"가 본질이고, 스왑(디스크 사용)은 그 위에 얹힌 한 구현일 뿐입니다.

**Q2. "페이지 폴트 처리 과정은?"**
> 접근한 페이지의 PTE가 유효하지 않으면 CPU가 트랩을 걸어 커널로 진입 → OS가 정당한 접근인지 확인 → 디스크(스왑/파일)에서 해당 페이지를 빈 프레임(없으면 교체 알고리즘으로 victim 선정)에 로드 → 페이지 테이블·TLB 갱신 → 폴트를 일으킨 명령을 재실행합니다. 디스크 I/O가 끼어 수 마이크로~밀리초가 들고, 폴트가 과도하면 Thrashing으로 성능이 급락합니다.

**꼬리 Q2-1. "모든 페이지 폴트가 디스크를 읽나요?"**
> 아닙니다. 디스크를 실제로 읽는 major fault와, 페이지가 이미 메모리 어딘가(페이지 캐시, COW 공유 등)에 있어 매핑만 연결하면 되는 minor fault로 나뉩니다. minor fault는 매우 저렴합니다. `ps`나 `/proc`에서 majflt/minflt를 보면 앱이 실제로 스왑을 때리는지 판단할 수 있어, "폴트가 많다 = 무조건 느리다"로 단정하지 않는 게 중요합니다.

**Q3. "페이징과 세그먼테이션 차이?"**
> 페이징은 고정 크기 페이지로 나눠 외부 단편화가 없고(마지막 페이지에 내부 단편화만), 세그먼테이션은 코드/스택/힙 같은 논리 단위를 가변 크기로 나눠 프로그래머 관점에 자연스럽지만 외부 단편화가 생깁니다. 현대 OS는 페이징을 기반으로 하고 세그먼트 개념은 보호·논리 구분 정도로만 남겨 둔 하이브리드입니다.

**Q4. "LRU를 실제로 어떻게 구현하나요?"**
> 이론적으로는 연결 리스트를 접근 순서로 재배치하거나 타임스탬프 최솟값을 찾지만, 매 메모리 접근마다 하기엔 비용이 큽니다. 그래서 실제 OS는 참조 비트 하나로 근사하는 Clock(2nd chance) 계열을 씁니다. 반대로 애플리케이션 레벨 캐시라면 접근 빈도가 낮아 비용을 감당할 수 있어, 자바에서 `LinkedHashMap(accessOrder=true)`이나 Caffeine으로 정밀한 LRU/LFU를 직접 구현합니다. "어느 계층이냐"에 따라 정밀도와 비용의 타협점이 다릅니다.

**Q5. "Belady's Anomaly가 뭔가요?"**
> 보통 프레임(캐시)을 늘리면 폴트가 주는데, FIFO에서는 늘렸는데 오히려 폴트가 느는 경우가 존재합니다(반례: `1 2 3 4 1 2 5 1 2 3 4 5`). FIFO가 "얼마나 오래 있었나"만 보고 사용 빈도를 무시하기 때문입니다. LRU·OPT는 스택 알고리즘 성질(프레임 k개가 항상 k+1개의 부분집합)을 만족해 이 변칙이 없어, 이론·실무 모두 FIFO보다 LRU 계열을 선호합니다.

# O4. Synchronization and Deadlock

**학습 목표**: *"임계구역/뮤텍스/세마포어?"* / *"데드락 조건과 해결?"* 에 답할 수 있다. (Java 동시성과 직접 연결)

## 1. 비유 — 화장실 열쇠 하나
칸(공유 자원) 하나, 열쇠(락) 하나. 열쇠 가진 사람만 들어감. 두 명이 동시에 들어가려 하면 문제(race condition) → 열쇠로 **상호 배제**.

## 2. 핵심 개념
- **Race Condition**: 둘 이상이 공유 자원에 동시 접근·수정 → 순서에 따라 결과가 달라짐. (Java `count++`와 동일 문제 — JMM T4)
- **Critical Section(임계구역)**: 공유 자원에 접근하는 코드 영역. 한 번에 하나만 실행돼야 함.
- **상호 배제(Mutual Exclusion)**: 임계구역에 한 스레드만 진입.

## 3. 동기화 도구 비교
| 도구 | 개념 | 특징 |
|------|------|------|
| **Mutex** | 열쇠 1개(binary), 소유 개념 있음 | 잠근 스레드만 해제 가능 |
| **Semaphore** | 카운터 N개 자원 허용 | counting(N개)/binary. 소유 개념 없음 |
| **Monitor** | 언어 차원 상호배제 캡슐화 | Java `synchronized`가 이것 |
| **Spinlock** | 락 풀릴 때까지 busy-wait | 짧은 대기에 유리, CPU 낭비 위험 |

> Java `synchronized`/`ReentrantLock`은 OS의 뮤텍스/모니터를 언어로 감싼 것. `volatile`의 가시성 문제도 결국 CPU 캐시 vs 메인메모리(하드웨어) 얘기.

## 4. 데드락(Deadlock) — 4가지 필요조건 ⭐
네 개가 **동시에** 성립해야 데드락 발생 (하나만 깨도 예방):
```
1. 상호 배제 (Mutual Exclusion)  — 자원을 한 번에 하나만
2. 점유와 대기 (Hold and Wait)   — 자원 쥔 채 다른 자원 대기
3. 비선점 (No Preemption)        — 강제로 뺏을 수 없음
4. 환형 대기 (Circular Wait)     — A→B→C→A 원형으로 대기
```
비유: 사거리에서 네 차가 서로 앞차 꼬리를 막고 아무도 못 감.

## 5. 데드락 처리 전략
| 전략 | 방법 |
|------|------|
| **예방(Prevention)** | 4조건 중 하나를 원천 차단 (예: 자원 순서 정해 환형 대기 제거) |
| **회피(Avoidance)** | 안전 상태만 허용 — **은행원 알고리즘(Banker's)** |
| **탐지·회복(Detection)** | 주기적 탐지 후 프로세스 강제 종료/자원 회수 |
| **무시(Ostrich)** | 드물면 그냥 무시 (실제 많은 OS) |

## 6. 세마포어로 생산자-소비자 문제 풀기

### 비유 — 빵집 진열대
생산자(제빵사)는 빵을 만들어 진열대(버퍼)에 놓고, 소비자(손님)는 진열대에서 빵을 꺼내 감. 진열대 **자리 수가 제한**돼 있고, **빵이 없으면 손님은 기다려야** 함 — 이 두 제약을 세마포어 두 개로 표현한다.

```
세마포어 empty = N   // 버퍼의 "빈 자리" 개수 (처음엔 N개 다 빔)
세마포어 full  = 0   // 버퍼의 "채워진 자리" 개수 (처음엔 0개)
뮤텍스   mutex = 1   // 버퍼 자체를 건드리는 동안의 상호배제

Producer:                          Consumer:
loop {                             loop {
  item = produce()                   wait(full)    // 꺼낼 빵 있을 때까지 대기
  wait(empty)   // 빈 자리 있을 때까지 대기          wait(mutex)
  wait(mutex)                         item = buffer에서 꺼냄
  buffer에 item 삽입                   signal(mutex)
  signal(mutex)                       signal(empty) // 빈 자리 하나 늘어남
  signal(full)  // 채워진 자리 하나 늘어남           consume(item)
}                                 }
```
- `empty`/`full`은 **카운팅 세마포어**로 "자원 개수"(빈 자리/찬 자리)를 표현 — 세마포어의 전형적 용도.
- `mutex`는 버퍼 자료구조 자체를 동시에 두 스레드가 건드리지 않게 하는 **이진 세마포어**(뮤텍스 역할).
- 순서가 중요: `wait(empty)` → `wait(mutex)` 순으로 잡아야 함. 반대로 하면 mutex를 쥔 채 empty를 기다리다 소비자도 mutex를 못 얻어 **데드락**.

## 7. 뮤텍스와 세마포어 차이 (심화)
| 구분 | Mutex | Semaphore |
|------|-------|-----------|
| 값의 범위 | 0/1 (binary) | 0~N (counting) 또는 binary |
| **소유(ownership)** | 있음 — **잠근 스레드만 해제** 가능 | 없음 — **다른 스레드가 signal 가능**(A가 wait, B가 signal해도 됨) |
| 용도 | **상호 배제**(임계구역 보호) | **자원 개수 제한 + 스레드 간 신호(signaling)** |
| 우선순위 역전 | 소유자가 명확해 **우선순위 상속(priority inheritance)** 적용 가능 | 소유자가 없어 적용 어려움 |
| 예 | Java `synchronized`, `ReentrantLock` | Java `Semaphore`, 생산자-소비자의 empty/full |

> 핵심 차이 한 줄: **뮤텍스는 "누가 잠갔는지"가 있고 그 사람만 풀 수 있는 자물쇠, 세마포어는 소유 없이 "카운트"만 관리하는 신호기**. 그래서 세마포어는 "다른 스레드에게 완료를 알리는" 용도(생산자→소비자 신호)로도 쓰이지만, 뮤텍스는 그 용도로 쓰면 안 된다(자기가 안 잠근 락을 풀 수 없으므로).

## 8. 데드락 회피 — 은행원 알고리즘(Banker's Algorithm) 예시

> 아이디어: 은행이 대출해줄 때 "이 대출 이후에도 모든 고객이 순서대로 최대 필요량을 받아 완제할 수 있는가"(= **안전 상태, Safe State**)를 미리 계산 후에만 승인.

예시: 자원 A 총 12개, 프로세스 3개.

| 프로세스 | Allocation(현재 보유) | Max(최대 필요) | Need(추가 필요 = Max-Allocation) |
|----------|------------------------|-----------------|-----------------------------------|
| P0 | 5 | 10 | 5 |
| P1 | 2 | 4 | 2 |
| P2 | 2 | 9 | 7 |

- 현재 할당 합 = 5+2+2 = 9 → **가용(Available) = 12 - 9 = 3**
- 안전 순서 탐색: Available=3으로 **Need를 만족시킬 수 있는 프로세스**부터 실행 후 자원 반환.
  1. P1의 Need=2 ≤ Available(3) → P1 실행 가능 → 완료 후 자원 반환: Available = 3+2 = 5
  2. P0의 Need=5 ≤ Available(5) → P0 실행 가능 → 완료 후 반환: Available = 5+5 = 10
  3. P2의 Need=7 ≤ Available(10) → P2 실행 가능 → 완료
- 안전 순서 `P1 → P0 → P2` 존재 → **Safe State** → 이 상태로의 자원 할당은 허용.
- 만약 어떤 순서로도 모든 프로세스의 Need를 채울 수 없으면 **Unsafe State**로 판단해 해당 요청을 아예 거절(데드락이 아니라 "데드락 가능성이 있는 상태"를 사전 차단).

## 9. 핵심 포인트
- 🔴 "데드락은 조건 하나만 있어도 발생" — ❌ **4조건 모두** 성립해야. 예방 = 하나 깨기.
- 🟡 실무 예방: **락 획득 순서 통일**(환형 대기 차단)이 가장 흔한 방법.
- 🟡 Java 데드락도 동일 — 두 스레드가 두 락을 반대 순서로 잡으면 발생.
- 🔴 "뮤텍스와 세마포어(binary)는 완전히 같다" — ❌ 값만 보면 비슷해도 **소유 개념 유무**가 다름. 세마포어는 남이 대신 풀어줄 수 있음(신호 용도), 뮤텍스는 안 됨.
- 🟡 은행원 알고리즘은 **회피(Avoidance)** 전략 — 예방(Prevention)처럼 조건 자체를 없애는 게 아니라, 매 요청마다 "이래도 안전한가"를 계산해 위험한 요청만 거절.
- 🔴 "Safe State가 아니면 곧바로 데드락" — ❌ Unsafe State는 데드락이 **될 수도 있는** 상태일 뿐, 반드시 데드락은 아님. 은행원 알고리즘은 안전을 위해 보수적으로 거절.

## 10. 예상 면접 질문
**Q1. "뮤텍스와 세마포어 차이?"**
> 뮤텍스는 소유 개념이 있는 이진 락으로 잠근 스레드만 해제할 수 있고, 세마포어는 N개 자원을 허용하는 카운터로 소유 개념이 없어 다른 스레드가 signal할 수 있습니다. 그래서 임계구역 상호배제엔 뮤텍스, 자원 개수 제한이나 스레드 간 완료 신호(signaling)엔 세마포어를 씁니다. (더 깊은 스핀락·모니터 비교는 O7에서 다룹니다.)

**Q2. "데드락 발생 조건과 예방법?"**
> 상호배제·점유와대기·비선점·환형대기 4가지가 동시에 성립할 때 발생하고, 하나만 깨면 예방됩니다. 실무에서 가장 현실적인 건 락 획득 순서를 전역으로 통일해 환형 대기를 없애는 방법입니다. (운영 중 진단·회복 전략은 O6에서 이어집니다.)

**Q3. "OS 동기화와 Java 동시성의 관계?"**
> Java `synchronized`는 OS 모니터/뮤텍스를 언어로 감싼 것이고, `volatile`의 가시성 문제는 CPU 코어별 캐시와 메인메모리 사이의 문제입니다. 결국 상호배제(원자성)와 가시성이라는 같은 문제를 하드웨어·OS·언어의 다른 층위에서 다루는 것이고, JMM은 이 층위를 프로그래머에게 계약으로 노출한 것입니다.

**Q4. "세마포어로 생산자-소비자 문제를 어떻게 푸나요?"**
> 빈 슬롯 수를 세는 empty, 채워진 슬롯 수를 세는 full, 버퍼 접근 자체를 보호하는 이진 세마포어(mutex) 셋을 씁니다. 생산자는 empty를 기다렸다 넣고 full을 늘리고, 소비자는 full을 기다렸다 꺼내고 empty를 늘립니다. 반드시 `wait(empty)`를 `wait(mutex)`보다 먼저 호출해야 합니다. 순서를 뒤집으면 mutex를 쥔 채 empty를 기다리고, 소비자는 그 mutex를 못 얻어 서로 막히는 데드락이 납니다.

**Q5. "은행원 알고리즘은 예방(Prevention)과 뭐가 다른가요?"**
> 예방은 4조건 중 하나를 구조적으로 성립 못 하게 막는 것이고, 회피(은행원)는 조건은 남겨두되 매 요청마다 "이걸 승인해도 모든 프로세스가 완료 가능한 안전 순서가 존재하는가"를 시뮬레이션해 위험한 요청만 거절하는 것입니다. 더 유연하지만 최대 필요량을 미리 알아야 하고 매 요청마다 O(n²) 계산이 들어, 프로세스·자원이 유동적인 범용 OS에서는 거의 안 쓰이고 이론·특수 임베디드에서만 씁니다.

**Q6(SCENARIO). "실무에서 데드락을 실제로 만난 적 있나요? 어떻게 대응했나요?"**
> DB 쪽이 가장 흔했습니다. 두 트랜잭션이 여러 행을 서로 반대 순서로 잠가 MySQL이 `Deadlock found; try restarting transaction`을 뱉은 적이 있습니다. `SHOW ENGINE INNODB STATUS`의 LATEST DETECTED DEADLOCK 섹션으로 어느 두 트랜잭션이 어떤 인덱스 락을 물고 있었는지 확인했고, 근본 해결은 애플리케이션에서 항상 같은 키 순서(예: 작은 id부터)로 갱신하도록 락 순서를 통일한 것이었습니다. 즉시 완화책으로는 재시도 로직과 락 타임아웃을 두어 데드락 희생자만 롤백 후 재시도되게 했습니다. (진단 흐름은 O6에서 상세히 정리합니다.)

# O5. I/O Models

**학습 목표**: *"블로킹/논블로킹, 동기/비동기 차이?"* 에 답할 수 있다. (JS 이벤트 루프·Netty와 연결)

## 1. 비유 — 라면 끓이기
- **블로킹 동기**: 냄비 앞에 서서 물 끓을 때까지 **아무것도 안 하고 기다림**.
- **논블로킹 동기**: 물 끓나 **계속 들여다보며**(폴링) 다른 일도 조금씩.
- **비동기**: 타이머 맞춰두고 **딴 일 하다가**, 다 되면 알림 받음(콜백).

## 2. 두 축으로 구분
- **블로킹 vs 논블로킹**: 호출이 **즉시 리턴하느냐**(제어권). 블로킹=결과 날 때까지 멈춤, 논블로킹=바로 리턴(아직이면 "아직" 응답).
- **동기 vs 비동기**: 결과를 **누가 신경 쓰느냐**(완료 통지). 동기=호출자가 결과 확인, 비동기=완료되면 콜백/이벤트로 통지.

## 3. 4가지 조합표
| 모델 | 설명 | 예 |
|------|------|-----|
| 블로킹+동기 | 호출 후 결과까지 멈춤 | 전통 `read()` |
| 논블로킹+동기 | 바로 리턴, 호출자가 반복 폴링 | `O_NONBLOCK` + 폴링 |
| I/O 멀티플렉싱 | 여러 fd를 한 스레드가 감시 | `select`/`poll`/**`epoll`** |
| 비동기(AIO) | 커널이 완료 후 통지 | `io_uring`, 콜백 |

## 4. JS 이벤트 루프·Netty와의 연결 ⭐ (cross-link)
- **JS 이벤트 루프**: 싱글 스레드가 **논블로킹 I/O + 이벤트 큐**로 동시성 확보. I/O를 커널/스레드풀에 맡기고 완료되면 콜백을 큐에 넣어 처리 → **블로킹 안 하니 한 스레드로 수천 연결**.
- **epoll/멀티플렉싱**: 한 스레드가 다수 소켓을 감시 → Node.js·Netty·Nginx의 고성능 비결.
- 정리: "왜 Node는 싱글 스레드인데 빠른가?" = **논블로킹 I/O + 이벤트 루프**로 CPU를 놀리지 않기 때문.

## 5. 핵심 포인트
- 🔴 "논블로킹 = 비동기" — ❌ 다른 축. 논블로킹+동기(폴링)도 있음.
- 🟡 스레드-per-connection(블로킹)은 연결 많으면 스레드 폭발 → **이벤트 기반(epoll)** 이 C10K 문제 해법.

## 6. 예상 면접 질문
**Q1. "블로킹/논블로킹과 동기/비동기 차이?"**
> 블로킹/논블로킹은 호출이 즉시 리턴하는지(제어권 관점), 동기/비동기는 결과 완료를 누가 통지하는지(완료 통지 관점)로 서로 독립된 축입니다. 논블로킹이면서 동기(바로 리턴하지만 호출자가 반복 폴링)인 조합도 있어서 둘을 같은 것으로 묶으면 안 됩니다.

**Q2. "Node.js가 싱글 스레드인데 어떻게 동시 처리를 하나요?"**
> 논블로킹 I/O로 요청을 커널(epoll)이나 내부 스레드풀(libuv)에 맡기고, 이벤트 루프가 완료 콜백을 큐에서 꺼내 처리합니다. CPU가 I/O 대기로 놀지 않아 한 스레드로 수천 연결을 감당합니다. 단, 한 콜백이 CPU를 오래 잡으면 이벤트 루프 전체가 막히므로 CPU 집약 작업엔 부적합하고, 그럴 땐 워커 스레드나 별도 프로세스로 뺍니다.

**Q3. "select/poll과 epoll의 차이는?"**
> select/poll은 감시할 fd 목록을 매 호출마다 커널에 통째로 넘기고 커널이 전체를 순회(O(n))합니다. epoll은 관심 fd를 `epoll_ctl`로 한 번 등록해두고 준비된 fd만 `epoll_wait`로 돌려줘(O(준비된 개수)) 연결 수가 늘어도 성능이 유지됩니다. 그래서 C10K의 표준 해법이 됐고, Netty·Nginx·Redis가 이 위에 서 있습니다.

**Q4(SCENARIO/TRADE-OFF). "Spring MVC(블로킹)와 WebFlux(논블로킹) 중 어떤 기준으로 고르나요?"**
> 스레드-per-요청 블로킹 모델은 요청당 스레드를 잡아 대기시키므로, 외부 API·DB를 오래 기다리는 I/O 바운드 트래픽이 많고 동시 연결이 크면 스레드 풀이 고갈됩니다. 이때 WebFlux(Reactor+Netty의 논블로킹)가 적은 스레드로 버팁니다. 반대로 트래픽이 크지 않거나 JDBC처럼 블로킹 드라이버에 묶여 있고 팀이 명령형 코드에 익숙하다면, 리액티브의 디버깅·학습 비용이 이득을 넘어서 MVC가 낫습니다. 저라면 기본은 MVC로 두고, 스레드 풀 고갈이 실측된 게이트웨이성 서비스에만 WebFlux(또는 Java 21 Virtual Thread로 블로킹 코드를 유지하며 확장)를 검토하겠습니다.

# O6. 데드락 (Deadlock)

**학습 목표**: *"운영 중 데드락이 의심될 때 어떻게 진단하고 회복하나요?"* 에 실무 사례로 5분 답할 수 있다. (O4의 이론을 "진단·운영" 관점으로 확장)

## 1. 비유 — 좁은 골목의 마주 오는 두 트럭
편도 골목에서 두 트럭이 마주 봤다. 각자 "상대가 후진해야 내가 간다"며 버티면 둘 다 영원히 못 감. 아무도 자기 자리를 양보(비선점 깨기)하지 않고, 서로 상대의 자리를 기다리며(환형 대기), 골목은 한 대만 지날 수 있고(상호 배제), 각자 이미 골목 절반을 점유(점유와 대기)한 상태 — **데드락 4조건이 그대로 재현**된다.

## 2. 개념 정의
> **데드락(Deadlock)** = 둘 이상의 프로세스/스레드가 서로가 쥔 자원을 무한히 기다려 **아무도 진행하지 못하는** 상태. O4의 4조건(상호배제·점유와대기·비선점·환형대기)이 동시에 성립할 때 발생.
> **혼동 주의 — Livelock**: 서로 양보하느라 상태만 계속 바뀌고 실질 진행이 없는 상태(두 사람이 복도에서 계속 같은 쪽으로 비켜 못 지나감). **Starvation**: 데드락은 아니지만 특정 스레드만 자원을 못 얻어 무한 대기(우선순위·스케줄링 편향).

## 3. 데드락 vs Livelock vs Starvation 비교표
| 구분 | 진행(progress) | 원인 | 대표 해법 |
|------|----------------|------|-----------|
| **Deadlock** | 완전 정지(상태 변화 없음) | 4조건 동시 성립, 환형 대기 | 락 순서 통일, 타임아웃, 탐지·롤백 |
| **Livelock** | 상태는 변하나 실질 진행 0 | 서로 반복 양보/재시도 | 랜덤 backoff, 재시도 상한 |
| **Starvation** | 일부만 진행 못 함 | 우선순위·스케줄 편향 | Aging, 공정(fair) 락 |

## 4. 운영 중 진단 흐름 (JVM & DB)
```
증상: 특정 API 무응답 / 스레드 풀 고갈 / TPS 급락, CPU는 낮음
   │
   ▼  ① JVM 스레드 데드락 의심
jstack <pid>  또는  jcmd <pid> Thread.print
   └─▶ "Found one Java-level deadlock:" 섹션 확인
       - 어떤 스레드가 어떤 모니터/락을 hold하고 무엇을 wait하는지 원형 확인
       - BLOCKED 상태 스레드가 서로의 락을 물고 있으면 데드락 확정
   │
   ▼  ② DB 데드락 의심 (스레드는 대기 중, 커넥션 풀 고갈)
MySQL:  SHOW ENGINE INNODB STATUS → LATEST DETECTED DEADLOCK
PostgreSQL: 로그의 "deadlock detected" + pg_locks / pg_stat_activity
   └─▶ 어느 두 트랜잭션이 어떤 인덱스/행 락을 반대 순서로 잡았는지 확인
   │
   ▼  ③ 회복 & 근본 대책
즉시: 데드락 희생자 롤백/재시도, 락 타임아웃, 문제 스레드 덤프 확보
근본: 락(또는 갱신 대상) 획득 순서 전역 통일 → 환형 대기 제거
```

## 5. 데드락 처리 4전략 요약 (O4 심화)
| 전략 | 실무 위치 | 특징 |
|------|-----------|------|
| **예방(Prevention)** | 애플리케이션 설계 | 락/행 획득 순서 통일(환형대기↓)이 가장 현실적. `Hold and Wait`를 없애려 필요한 락을 한 번에(원자적으로) 획득하기도 |
| **회피(Avoidance)** | 이론/특수 임베디드 | 은행원 알고리즘. 범용 OS·서버에선 최대 필요량 사전 파악·계산비용 때문에 거의 안 씀 |
| **탐지·회복(Detection)** | **DB 엔진**(InnoDB 등) | 대기 그래프(wait-for graph)의 사이클을 탐지 → 비용 낮은 트랜잭션을 victim으로 롤백 |
| **무시(Ostrich)** | 범용 OS | 발생이 드물고 예방 비용이 크면 그냥 무시(재부팅). Linux 커널의 유저 락 상당수가 이 태도 |

## 6. 핵심 포인트 (자주 하는 실수)
- 🔴 "데드락이면 CPU가 100%다" — ❌ 데드락 스레드는 **BLOCKED로 대기만** 하므로 CPU는 오히려 낮고 처리량만 0에 수렴. CPU 100%는 오히려 livelock/무한루프 의심.
- 🔴 "타임아웃을 걸면 데드락이 해결된다" — ❌ 타임아웃은 **회복(완화)**일 뿐 근본 원인(환형 대기)은 남음. 근본 대책은 락 순서 통일.
- 🟡 DB 데드락은 애플리케이션 락과 별개 계층 — `synchronized`를 아무리 잘 짜도 트랜잭션이 행을 반대 순서로 잠그면 발생.
- 🟡 `tryLock(timeout)`으로 락을 못 얻으면 **가진 락을 모두 풀고 재시도**(전부 아니면 전무) 하면 점유와 대기를 깰 수 있음.
- 🔴 "데드락과 스타베이션은 같은 것" — ❌ 데드락은 상호 순환 대기로 **전원 정지**, 스타베이션은 특정 스레드만 계속 밀림(전체는 진행).

## 7. 예상 면접 질문
**Q1(SCENARIO). "운영 중 특정 API가 먹통이고 스레드 풀이 꽉 찼습니다. 데드락인지 어떻게 진단하나요?"**
> 먼저 CPU 지표를 봅니다. CPU가 낮은데 무응답이면 스레드가 어딘가 BLOCKED된 것이라 데드락/락 경합을 의심합니다. `jstack`이나 `jcmd Thread.print`로 스레드 덤프를 뜨면 JVM이 "Found one Java-level deadlock"으로 순환 대기를 직접 짚어줍니다. 락 경합이 아니라 커넥션 풀만 고갈됐다면 DB를 봐야 하고, MySQL이면 `SHOW ENGINE INNODB STATUS`의 데드락 섹션으로 어느 트랜잭션이 어떤 락을 반대로 잡았는지 확인합니다. 진단 후에는 희생자 롤백·재시도로 즉시 회복시키고, 근본적으로는 락/행 획득 순서를 통일합니다.

**꼬리 Q1-1. "스레드 덤프에서 구체적으로 무엇을 보고 데드락이라 확신하나요?"**
> 두 스레드 이상이 서로 `BLOCKED (on object monitor)` 상태이고, 각자 "waiting to lock <A>"인데 상대가 "locked <A>"를 이미 hold하며 반대로 "waiting to lock <B>"를 물고 있는 원형 대기가 보이면 확정입니다. JVM은 이걸 자동 탐지해 데드락 블록으로 요약해 주지만, ReentrantLock 같은 java.util.concurrent 락은 탐지가 제한적이라 스레드 스택의 `park`/`lock` 지점을 직접 따라가 봐야 할 때도 있습니다.

**Q2(TRADE-OFF). "데드락 예방·회피·탐지 중 서버 애플리케이션에서 뭘 택하겠습니까?"**
> 예방(락 순서 통일)을 1순위로 둡니다. 코드 규율만으로 되고 런타임 비용이 없기 때문입니다. 회피(은행원)는 최대 자원 요구량을 미리 알아야 하고 요청마다 계산이 들어 동적인 웹 서버엔 안 맞습니다. 탐지·회복은 제가 직접 구현하기보다 DB 엔진에 맡기고, 애플리케이션은 데드락 예외를 잡아 재시도하는 쪽으로 갑니다. 즉 "설계로 예방 + DB의 탐지에 위임 + 재시도로 회복"의 조합이 현실적입니다.

**Q3. "락 획득 순서 통일이 왜 환형 대기를 없애나요?"**
> 모든 스레드가 자원을 전역적으로 정해진 동일 순서(예: 자원 번호 오름차순)로만 획득하면, "A를 쥐고 B를 기다리는 스레드"와 "B를 쥐고 A를 기다리는 스레드"가 동시에 존재할 수 없습니다. 대기 방향이 항상 한쪽으로만 흘러 사이클이 생기지 않기 때문입니다. 사이클이 없으면 환형 대기 조건이 깨지고, 4조건 중 하나가 무너지므로 데드락이 원천적으로 불가능해집니다.

# O7. 뮤텍스 · 세마포어 · 모니터

**학습 목표**: *"스핀락 vs 뮤텍스 언제 뭘 쓰나요?"* / *"synchronized와 ReentrantLock 차이는?"* 에 근거를 대며 답할 수 있다. (O4의 도구 비교를 "선택 기준"으로 확장)

## 1. 비유 — 회의실 잠금 장치들
- **뮤텍스**: 회의실 열쇠 1개. 빌린 사람만 반납 가능(소유). 없으면 **자리에 돌아가 앉아 호출을 기다림(sleep, 블로킹)**.
- **스핀락**: 열쇠 없이 문 앞에서 **손잡이를 계속 흔들며 기다림(busy-wait)**. 곧 열릴 거면 자리 왕복(스위치)보다 빠르지만, 오래 걸리면 그 시간 내내 CPU를 태움.
- **세마포어**: 회의실이 N개. 입장권 N장을 나눠주고 다 나가면 대기. 표를 **누가 냈든 아무나 반납 가능**(소유 없음, 신호용).
- **모니터**: 회의실에 **자동 잠금+안내데스크**가 내장. 들어가면 자동으로 잠기고, "자료 준비되면 부를게요"(조건 변수)로 대기·통지까지 관리. 언어가 이 데스크를 대신 운영(Java `synchronized`).

## 2. 개념 정의
> **모니터(Monitor)** = 공유 자원과 그것을 다루는 연산을 하나로 캡슐화하고, **상호배제 + 조건 동기화(wait/notify)** 를 언어/런타임이 자동 보장하는 고수준 동기화 구조. 개발자가 락을 명시적으로 잠그고 푸는 실수를 줄여준다.
> **조건 변수(Condition Variable)** = "특정 조건이 될 때까지 락을 놓고 잠들었다가, 다른 스레드가 깨워주면 다시 락을 잡고 재개"하는 대기 큐. Java의 `wait()/notify()`, `Condition.await()/signal()`.

## 3. 스핀락 vs 뮤텍스 판단 기준
```
        임계구역 대기 예상 시간이 짧다? (락 보유 시간 < 컨텍스트 스위치 비용)
                    │
          ┌─────────┴──────────┐
         YES                    NO
          │                      │
   멀티코어에서 대기?         뮤텍스(블로킹)
          │                  - 대기 중 CPU 반납 → 다른 일 수행
   스핀락 유리                - 스위치 비용은 있지만 CPU 낭비 없음
   - 스위치 없이 즉시 진입     - 긴 임계구역/단일코어의 정답
   - 커널 진입 회피
   ※ 단일코어에서 스핀락은
     최악(락 가진 스레드가
     선점됐는데 CPU를 계속 태움)
```

## 4. 비교표 — 4가지 동기화 도구 (선택 기준 중심)
| 도구 | 대기 방식 | 소유 | 언제 쓰나 |
|------|-----------|------|-----------|
| **Spinlock** | busy-wait(안 잠) | 있음 | 임계구역이 매우 짧고 멀티코어일 때. 커널 내부(인터럽트 컨텍스트)에서 자주 |
| **Mutex** | blocking(잠듦) | 있음 | 임계구역이 길거나 대기 시간이 불확실할 때. 일반 애플리케이션의 기본 |
| **Semaphore** | blocking | 없음 | 자원 N개 제한(커넥션 풀, 동시 실행 수 제한) + 스레드 간 신호 |
| **Monitor** | blocking + 조건변수 | 있음(암묵) | 상호배제와 "조건 대기"를 함께 다룰 때. 고수준 언어의 기본(Java `synchronized`) |

## 5. Java에서의 대응 & synchronized vs ReentrantLock
| 구분 | `synchronized`(모니터) | `ReentrantLock` |
|------|------------------------|-----------------|
| 획득/해제 | JVM이 자동(블록 벗어나면 해제) | 수동 `lock()`/`unlock()`(finally 필수) |
| 공정성(fairness) | 불공정만 | `new ReentrantLock(true)`로 공정 모드 선택 가능 |
| 대기 중단 | 불가 | `lockInterruptibly()`, `tryLock(timeout)` 지원 |
| 조건 변수 | 단일(`wait/notify`) | 여러 개(`newCondition()`)로 생산자/소비자 분리 대기 |
| 성능 | JIT 최적화(락 생략·바이어스드 등)로 대부분 충분 | 세밀한 제어가 필요할 때 |

> 실무 기본은 `synchronized`(해제 누락 실수가 없어 안전). **타임아웃·인터럽트·tryLock(데드락 회피)·공정성·다중 조건**이 필요할 때만 `ReentrantLock`을 꺼낸다.

## 6. 핵심 포인트 (자주 하는 실수)
- 🔴 "스핀락이 뮤텍스보다 항상 빠르다" — ❌ **짧은 대기 + 멀티코어**에서만. 대기가 길면 그 시간 내내 CPU를 태워 오히려 손해. 단일코어에선 거의 항상 나쁨.
- 🔴 "세마포어(binary)는 뮤텍스와 같다" — ❌ **소유 개념**이 다름. 세마포어는 A가 wait, B가 signal 가능(신호용). 뮤텍스는 잠근 스레드만 해제 → 신호용으로 쓰면 안 됨.
- 🟡 **우선순위 역전(Priority Inversion)**: 낮은 우선순위 스레드가 락을 쥔 채 밀려, 그 락을 기다리는 높은 우선순위 스레드가 못 감. 뮤텍스는 소유자가 명확해 **우선순위 상속(priority inheritance)** 으로 완화 가능(세마포어는 소유자 없어 어려움). 화성 탐사선 Pathfinder 리셋 사고가 유명한 사례.
- 🔴 "`ReentrantLock`을 쓰면 `unlock`을 안 해도 GC가 정리" — ❌ 반드시 `finally`에서 `unlock`. 안 풀면 영구 락 → 데드락.
- 🟡 재진입(reentrant): 같은 스레드가 이미 쥔 락을 다시 요청해도 통과. `synchronized`·`ReentrantLock` 모두 재진입 지원(재귀 호출·같은 객체 메서드 상호 호출 시 자기 데드락 방지).

## 7. 예상 면접 질문
**Q1(TRADE-OFF). "스핀락과 뮤텍스, 언제 무엇을 쓰나요?"**
> 판단 기준은 "임계구역 보유 시간 vs 컨텍스트 스위치 비용"입니다. 임계구역이 아주 짧아 대기가 스위치 비용보다 짧게 끝날 것 같고 멀티코어라면, 잠들었다 깨는 스위치 없이 즉시 진입하는 스핀락이 유리합니다. 커널 내부의 짧은 크리티컬 섹션이 대표적입니다. 반대로 임계구역이 길거나 대기 시간이 불확실하면, 대기 중 CPU를 반납하는 뮤텍스가 맞습니다. 특히 단일코어에서 스핀락은 락을 쥔 스레드가 선점당했는데 대기 스레드가 CPU를 계속 태워 최악이므로 피합니다.

**꼬리 Q1-1. "그럼 자바 애플리케이션 개발자가 스핀락을 직접 쓸 일이 있나요?"**
> 순수 스핀락을 직접 짜는 일은 드뭅니다. 다만 JVM의 락은 내부적으로 **적응형 스피닝(adaptive spinning)** 을 합니다. 블로킹 전에 잠깐 스핀하며 곧 풀릴 것 같으면 스위치를 아끼는 방식입니다. 또 `java.util.concurrent`의 CAS 기반 논블로킹 자료구조(`AtomicInteger`, `ConcurrentLinkedQueue`)가 실패 시 재시도하는 루프가 사실상 스핀의 성격을 갖습니다. 그래서 개념 자체는 락프리 프로그래밍을 이해할 때 중요합니다.

**Q2. "모니터가 뮤텍스+세마포어보다 나은 점은?"**
> 모니터는 상호배제와 조건 동기화(wait/notify)를 언어 차원에서 캡슐화해, 개발자가 락을 명시적으로 잠그고 푸는 과정을 자동화합니다. `synchronized` 블록을 벗어나면 예외가 나든 정상이든 락이 자동 해제돼 해제 누락으로 인한 데드락을 구조적으로 막습니다. 세마포어/뮤텍스를 직접 다루면 이 해제를 사람이 책임져야 해 실수 여지가 큽니다. 대신 세밀한 제어(타임아웃, 다중 조건)는 포기하는 트레이드오프가 있습니다.

**Q3(SCENARIO). "동시에 최대 10개만 외부 API를 호출하도록 제한하려면 뭘 쓰겠습니까? 왜죠?"**
> 카운팅 세마포어입니다. `Semaphore(10)`을 두고 호출 전 `acquire`, 완료 후 `release`하면 자연스럽게 동시 실행이 10으로 제한됩니다. 뮤텍스는 1개만 허용하니 부적합하고, 세마포어는 "N개 자원 허용 + 소유 없이 다른 흐름이 반납" 이라는 성질이 이 요구에 정확히 맞습니다. 자바라면 `Semaphore`나, 스레드 풀 크기 자체를 10으로 제한하는 방식(`Executors`의 고정 풀)으로도 같은 효과를 냅니다.

**꼬리 Q3-1. "우선순위 역전이 뭔지, 세마포어 대신 뮤텍스가 유리한 경우와 엮어 설명해 보세요."**
> 낮은 우선순위 스레드 L이 락을 쥔 상태에서 중간 우선순위 스레드 M이 CPU를 계속 차지하면, 그 락을 기다리는 높은 우선순위 스레드 H가 L보다도 늦게 실행되는 역전이 생깁니다. 뮤텍스는 "누가 락을 쥐었는지"라는 소유 정보가 있어, H가 기다리는 동안 락 소유자 L의 우선순위를 H만큼 일시적으로 올려주는 우선순위 상속으로 이를 완화할 수 있습니다. 세마포어는 소유자 개념이 없어 이 기법을 적용하기 어렵습니다. 실시간성이 중요한 시스템에서 상호배제엔 세마포어보다 뮤텍스를 권하는 이유가 이것입니다.

# O8. 페이징과 페이지 교체

**학습 목표**: *"주소 변환이 실제로 어떻게 일어나고, TLB·다단계 페이지 테이블이 왜 필요한가요?"* / *"Thrashing을 어떻게 판단·해결하나요?"* 에 답할 수 있다. (O3의 개념을 "변환 하드웨어·워킹셋" 관점으로 심화)

## 1. 비유 — 대형 도서관의 청구기호 시스템
이용자는 "가상 청구기호"(가상 주소)만 안다. 실제 책의 물리적 서가 위치는 **색인 카드(페이지 테이블)** 가 알려준다. 그런데 매번 색인실까지 가면 느리니, 사서 책상에 **최근 조회 기록 쪽지(TLB)** 를 붙여둔다. 책 종류가 폭증하면 색인 카드 자체가 방 하나를 차지 → 색인을 **여러 단계로 쪼개(다단계 페이지 테이블)** 실제 쓰는 부분만 만든다.

## 2. 개념 정의
> **다단계 페이지 테이블(Multi-level Page Table)** = 단일 페이지 테이블이 너무 커서(64비트 주소 공간이면 천문학적) 주소를 여러 조각으로 나눠, 상위 테이블이 하위 테이블을 가리키는 트리 구조로 만든 것. **실제 사용하는 영역의 하위 테이블만 만들어** 메모리를 절약.
> **TLB(Translation Lookaside Buffer)** = 최근 사용한 (가상페이지→물리프레임) 변환을 담는 CPU 내 초고속 캐시. 히트하면 페이지 테이블을 안 거치고 즉시 변환.
> **워킹셋(Working Set)** = 프로세스가 특정 시점에 활발히 참조하는 페이지 집합. 이 집합이 물리 프레임에 다 못 들어가면 폴트가 폭증(Thrashing).

## 3. 다이어그램 — 2단계 주소 변환 + TLB
```
 가상 주소:  [ P1(상위) | P2(하위) | offset ]
                 │          │         │
   ┌─────────────┘          │         └────────────┐
   ▼                        ▼                       │
 ① TLB 조회 (가상페이지 전체로) ─ 히트! ──▶ 프레임 번호 즉시 획득 ─┐
   │ 미스                                                        │
   ▼                                                            │
 ② CR3(페이지 테이블 base)                                       │
   → P1로 1단계 테이블 인덱싱 → 2단계 테이블 주소                 │
   → P2로 2단계 테이블 인덱싱 → 프레임 번호  (메모리 접근 2번!)   │
   │  └─ 유효 비트 0 이면 → Page Fault → 디스크 로드/교체         │
   │                                                            ▼
   └─▶ 얻은 변환을 TLB에 저장 ──────────────▶  물리 주소 [ 프레임 | offset ] → RAM
```
> 다단계는 메모리를 아끼는 대신 변환마다 메모리를 여러 번 읽는다(4레벨이면 최대 4번). 그래서 **TLB 히트율이 성능의 핵심** — 히트하면 이 다단계 walk를 통째로 건너뛴다.

## 4. 비교표 — 페이징 관련 핵심 개념
| 개념 | 해결하는 문제 | 대가/주의 |
|------|---------------|-----------|
| **단일 페이지 테이블** | 주소 변환 | 64비트에선 테이블이 비현실적으로 큼 |
| **다단계 페이지 테이블** | 테이블 크기 폭발 | 변환 시 메모리 접근 횟수 증가 |
| **TLB** | 다단계 walk 느림 | 컨텍스트 스위치 시 flush(프로세스 전환 비쌈) |
| **디맨드 페이징** | 안 쓰는 페이지까지 로드 낭비 | 첫 접근 시 폴트 지연 |
| **워킹셋 관리** | Thrashing 방지 | 워킹셋 추정 오버헤드 |

## 5. Thrashing (스래싱) — 판단과 해결
```
 다중 프로그래밍 정도(동시 프로세스 수) ↑
   │
   ▼
 각 프로세스 몫의 프레임 < 워킹셋
   │
   ▼
 페이지 폴트 폭증 → 디스크 I/O 대기 → CPU는 놀고 스와핑만 반복
   │
   ▼
 CPU 이용률 급락 (그래프가 어느 지점에서 급격히 꺾임)
```
- **증상**: CPU 사용률은 낮은데 디스크 I/O(스왑) 사용률이 극도로 높음. `vmstat`의 `si`/`so`(swap-in/out)가 지속적으로 큼.
- **원인**: 물리 메모리 대비 동시에 돌리는 프로세스/작업의 워킹셋 총합이 초과.
- **해결**: ① 다중 프로그래밍 정도를 낮춤(일부 프로세스를 swap-out/일시 중단) ② 물리 메모리 증설 ③ 워킹셋 모델·PFF(Page-Fault Frequency)로 각 프로세스에 필요한 최소 프레임을 보장.

## 6. 핵심 포인트 (자주 하는 실수)
- 🔴 "페이지 테이블은 프로세스마다 하나의 큰 배열" — ❌ 현대는 **다단계(또는 역페이지 테이블)** 로, 실제 쓰는 영역의 하위 테이블만 존재.
- 🔴 "TLB는 소프트웨어 캐시" — ❌ **CPU 하드웨어** 안의 캐시. 그래서 극도로 빠르지만 크기가 작아 히트율 관리가 중요.
- 🟡 **컨텍스트 스위치가 비싼 숨은 이유가 TLB flush** — 프로세스가 바뀌면 주소 매핑이 달라져 TLB를 비워야 하고, 이후 한동안 TLB 미스가 쏟아짐(ASID/PCID로 완화하는 CPU도 있음).
- 🔴 "Thrashing은 메모리를 늘리면 무조건 해결" — 🟡 대개 맞지만, 근본은 워킹셋 대비 프레임 부족이라 **동시 실행 수를 줄이는 것**이 더 직접적일 때도 있음.
- 🟡 큰 힙을 쓰는 JVM에서 **Huge Page(대형 페이지)** 를 켜면 TLB 엔트리 하나가 더 큰 영역을 커버해 TLB 미스가 줄어 성능이 오르기도 함.

## 7. 예상 면접 질문
**Q1. "가상 주소가 물리 주소로 변환되는 과정을 단계별로 설명해 보세요."**
> CPU가 가상 주소를 내면 먼저 그 페이지 번호로 TLB를 조회합니다. 히트하면 프레임 번호를 즉시 얻어 오프셋과 합쳐 물리 주소가 나옵니다. 미스면 페이지 테이블을 walk하는데, 현대는 다단계라 상위 인덱스로 하위 테이블을 찾아 내려가며 여러 번 메모리를 읽습니다. 최종 PTE의 유효 비트가 0이면 페이지 폴트가 발생해 OS가 디스크에서 로드하고, 유효하면 프레임 번호를 얻어 TLB에 캐싱한 뒤 물리 주소를 완성합니다.

**꼬리 Q1-1. "그럼 64비트 시스템에서 페이지 테이블을 왜 다단계로 만드나요?"**
> 64비트 주소 공간을 4KB 페이지로 단일 테이블에 담으면 엔트리 수가 천문학적이라 프로세스마다 그런 표를 물리 메모리에 통째로 둘 수 없습니다. 다단계로 쪼개면 실제로 매핑된 영역에 해당하는 하위 테이블만 생성하면 되므로(대부분의 주소 공간은 비어 있음) 페이지 테이블 자체의 메모리를 크게 절약합니다. 대가로 변환 시 테이블을 여러 단계 타고 내려가야 해서, TLB가 그만큼 더 중요해집니다.

**Q2(SCENARIO). "배포 후 서버 응답이 느려졌는데 CPU는 낮고 디스크 I/O가 치솟습니다. 무엇을 의심하나요?"**
> 메모리 부족으로 인한 스와핑, 즉 Thrashing을 우선 의심합니다. `vmstat`으로 `si`/`so`(swap-in/out)가 지속적으로 큰지, `free`로 여유 메모리와 스왑 사용량을 봅니다. 원인이 맞으면 동시에 돌리는 프로세스 수나 각 프로세스의 힙/워킹셋이 물리 메모리를 초과한 것이라, 즉시로는 동시 실행 수를 줄이거나 메모리를 증설하고, 애플리케이션 쪽으론 JVM 힙을 물리 메모리에 맞게 재설정합니다. JVM이라면 GC 로그도 함께 봐서 GC 폭주와 스와핑을 구분합니다.

**꼬리 Q2-1. "컨텍스트 스위치가 잦을 때 TLB가 왜 성능에 영향을 주나요?"**
> 프로세스가 바뀌면 가상→물리 매핑이 완전히 달라지므로 이전 프로세스의 TLB 엔트리는 무효가 되어 flush됩니다. 스위치 직후엔 TLB가 비어 있어 한동안 미스가 쏟아지고, 그때마다 다단계 페이지 테이블 walk 비용이 붙어 실효 성능이 떨어집니다. 이것이 프로세스 컨텍스트 스위치가 스레드 스위치보다 비싼 숨은 이유 중 하나입니다. 최신 CPU는 PCID/ASID로 프로세스별 TLB 엔트리에 태그를 붙여 flush를 줄이기도 합니다.

**Q3. "페이지 크기를 크게 하면 어떤 트레이드오프가 있나요?"**
> 페이지가 크면 같은 메모리를 더 적은 페이지로 매핑하니 페이지 테이블이 작아지고, TLB 엔트리 하나가 더 넓은 영역을 커버해 TLB 미스가 줄어 큰 메모리를 쓰는 워크로드(대형 힙 JVM, DB)에 유리합니다. 반대로 실제 필요보다 많이 할당되는 내부 단편화가 커지고, 디맨드 페이징 시 한 번에 읽어 들이는 양이 많아 첫 접근 지연이 늘 수 있습니다. 그래서 Huge Page는 큰 연속 메모리를 다루는 워크로드에 선택적으로 켭니다.

# 핵심 질문 (Quiz)

> 답변을 먼저 떠올린 뒤 펼쳐서 확인하세요.

<details>
<summary>Q1. 프로세스와 스레드의 차이는?</summary>

- 프로세스: 독립된 메모리(주소 공간)를 가진 실행 단위. 다른 프로세스와 자원 공유 없음.
- 스레드: 프로세스 안에서 Code/Data/Heap을 **공유**하고 Stack·레지스터·PC만 개별로 갖는 실행 흐름.
- 스레드는 통신이 빠르지만 공유 자원 때문에 동기화 문제가 생김.
- 프로세스는 격리 덕분에 하나가 죽어도 다른 프로세스는 안전, 대신 컨텍스트 스위치가 무거움(주소 공간·TLB flush 포함).

</details>

<details>
<summary>Q2. 시스템 콜이 무엇이고 왜 필요한가?</summary>

- User Mode 프로그램은 하드웨어(디스크, 메모리 관리, I/O 포트 등)에 직접 접근 불가.
- 파일 열기, 네트워크 전송, 메모리 할당 등이 필요하면 커널에 요청하는 인터페이스 = 시스템 콜.
- 흐름: 시스템 콜 호출(트랩) → User→Kernel 모드 전환 → 커널 처리 → Kernel→User 전환 → 재개.
- 모드 전환 자체가 오버헤드 → 잦은 시스템 콜(1바이트씩 read 등)은 성능 저하 → 버퍼링(`BufferedReader`)으로 횟수 감소.

</details>

<details>
<summary>Q3. 선점형(Preemptive)과 비선점형(Non-preemptive) 스케줄링의 차이는?</summary>

- 비선점: 실행 중인 작업이 스스로 CPU를 반납할 때까지 유지 (예: FCFS, SJF).
- 선점: OS가 강제로 CPU를 회수 가능 (타임 슬라이스 만료, 더 급한 작업 도착 — 예: Round Robin, Priority-preemptive).
- 선점형이 응답성은 좋지만 컨텍스트 스위치 비용과 동기화 고려가 늘어남.

</details>

<details>
<summary>Q4. 페이지 폴트(Page Fault) 처리 과정은?</summary>

- 접근한 가상 페이지가 RAM에 없음 → 트랩(page fault) 발생.
- OS가 디스크에서 해당 페이지를 빈 프레임(없으면 페이지 교체 알고리즘으로 victim 선정)에 로드.
- 페이지 테이블 갱신 후 명령 재실행.
- 페이지 폴트가 과도하면 **Thrashing**(CPU가 스와핑만 하며 성능 급락).

</details>

<details>
<summary>Q5. Belady's Anomaly란 무엇이며 어떤 알고리즘에서 발생하는가?</summary>

- 일반 직관: 프레임(캐시) 수를 늘리면 페이지 폴트가 줄어야 함.
- **FIFO 교체 알고리즘**은 프레임을 늘려도 오히려 폴트가 **늘어나는** 경우가 존재(반례: `1 2 3 4 1 2 5 1 2 3 4 5`).
- 원인: FIFO는 "얼마나 오래 있었나"만 보고 "얼마나 자주/최근에 쓰였나"는 무시.
- LRU와 Optimal(OPT)은 스택 알고리즘 성질을 만족해 이 변칙이 발생하지 않음 → 실무는 FIFO보다 LRU 계열 선호.

</details>

<details>
<summary>Q6. 뮤텍스(Mutex)와 세마포어(Semaphore)의 차이는?</summary>

| 구분 | Mutex | Semaphore |
|------|-------|-----------|
| 값 범위 | 0/1 (binary) | 0~N (counting) 또는 binary |
| 소유(ownership) | 있음 — 잠근 스레드만 해제 가능 | 없음 — 다른 스레드가 signal 가능 |
| 용도 | 상호 배제(임계구역 보호) | 자원 개수 제한 + 스레드 간 신호(signaling) |
| 우선순위 역전 | 소유자 명확 → 우선순위 상속 적용 가능 | 소유자 없어 적용 어려움 |

- 핵심 한 줄: 뮤텍스는 "누가 잠갔는지"가 있어 그 사람만 풀 수 있는 자물쇠, 세마포어는 소유 없이 카운트만 관리하는 신호기.

</details>

<details>
<summary>Q7. 데드락(Deadlock) 발생 조건과 예방법은?</summary>

- 4가지 필요조건이 **동시에** 성립해야 발생(하나만 깨도 예방):
  1. 상호 배제(Mutual Exclusion)
  2. 점유와 대기(Hold and Wait)
  3. 비선점(No Preemption)
  4. 환형 대기(Circular Wait)
- 처리 전략: 예방(조건 자체 차단) / 회피(은행원 알고리즘 — 안전 상태만 허용) / 탐지·회복(주기적 탐지 후 강제 종료) / 무시(Ostrich).
- 실무에서 가장 흔한 예방법: **락 획득 순서 통일**(환형 대기 제거).

</details>

<details>
<summary>Q8. 블로킹/논블로킹과 동기/비동기의 차이는?</summary>

- 블로킹 vs 논블로킹: 호출이 **즉시 리턴하느냐**(제어권 관점). 블로킹=결과 날 때까지 멈춤, 논블로킹=바로 리턴.
- 동기 vs 비동기: 결과를 **누가 신경 쓰느냐**(완료 통지 관점). 동기=호출자가 직접 확인, 비동기=콜백/이벤트로 통지.
- 두 축은 독립적 → 논블로킹+동기(폴링)도 존재.
- 예: 블로킹+동기(전통 `read()`), 논블로킹+동기(`O_NONBLOCK` 폴링), 멀티플렉싱(`select`/`poll`/`epoll`), 비동기(`io_uring`).

</details>

<details>
<summary>Q9. 운영 중 데드락이 의심될 때 진단·회복 절차는?</summary>

- 먼저 CPU를 확인: 데드락 스레드는 BLOCKED로 대기만 하므로 **CPU는 낮고 처리량만 0**(CPU 100%면 오히려 무한루프/livelock 의심).
- JVM: `jstack <pid>` 또는 `jcmd <pid> Thread.print` → "Found one Java-level deadlock" 섹션에서 순환 대기 확인.
- DB: MySQL `SHOW ENGINE INNODB STATUS`의 LATEST DETECTED DEADLOCK / PostgreSQL 로그의 "deadlock detected".
- 즉시 회복: 희생자 롤백·재시도, 락 타임아웃. 근본 대책: **락/행 획득 순서 전역 통일**로 환형 대기 제거.
- 타임아웃은 완화일 뿐 근본 해결이 아님을 구분.

</details>

<details>
<summary>Q10. Deadlock, Livelock, Starvation의 차이는?</summary>

- **Deadlock**: 서로가 쥔 자원을 순환 대기 → 상태 변화 없이 전원 정지. (4조건 동시 성립)
- **Livelock**: 서로 계속 양보/재시도해 상태는 바뀌지만 실질 진행 0 → 랜덤 backoff로 완화.
- **Starvation**: 특정 스레드만 우선순위·스케줄 편향으로 계속 밀림(전체는 진행) → Aging, 공정(fair) 락으로 해결.

</details>

<details>
<summary>Q11. 스핀락과 뮤텍스는 각각 언제 쓰는가?</summary>

- 판단 기준: **임계구역 보유 시간 vs 컨텍스트 스위치 비용**.
- **스핀락**: 임계구역이 매우 짧고(대기 < 스위치 비용) **멀티코어**일 때. 잠들지 않고 busy-wait해 스위치·커널 진입을 아낌. 단일코어에선 락 소유자가 선점되면 CPU만 태워 최악.
- **뮤텍스**: 임계구역이 길거나 대기 시간이 불확실할 때. 대기 중 CPU를 반납(블로킹)해 다른 일 수행.
- 자바 앱은 순수 스핀락을 직접 쓰기보다 JVM의 적응형 스피닝·CAS 재시도 루프 형태로 접함.

</details>

<details>
<summary>Q12. synchronized와 ReentrantLock의 차이, 그리고 모니터의 이점은?</summary>

- `synchronized`(모니터): JVM이 락을 **자동 해제**(블록 이탈 시) → 해제 누락으로 인한 데드락을 구조적으로 방지. 실무 기본값.
- `ReentrantLock`: 수동 `lock/unlock`(finally 필수). 대신 **타임아웃(`tryLock`)·인터럽트·공정성·다중 조건 변수** 지원 → 세밀한 제어나 데드락 회피가 필요할 때.
- 둘 다 **재진입** 지원(같은 스레드가 쥔 락 재요청 통과).
- 모니터의 이점: 상호배제 + 조건 동기화(wait/notify)를 언어 차원에서 캡슐화해 락 관리 실수를 줄임.

</details>

<details>
<summary>Q13. 다단계 페이지 테이블과 TLB는 왜 필요한가?</summary>

- **다단계 페이지 테이블**: 64비트 주소 공간을 단일 테이블로 담으면 비현실적으로 큼 → 트리로 쪼개 **실제 매핑된 영역의 하위 테이블만** 생성해 메모리 절약. 대가로 변환 시 메모리 접근 횟수 증가.
- **TLB**: 최근 (가상페이지→프레임) 변환을 담는 **CPU 하드웨어 캐시**. 히트하면 다단계 walk를 통째로 건너뜀 → 히트율이 성능의 핵심.
- 컨텍스트 스위치 시 TLB flush가 프로세스 전환을 비싸게 만드는 숨은 요인(PCID/ASID로 완화 가능).

</details>

<details>
<summary>Q14. Thrashing은 어떻게 판단하고 해결하는가?</summary>

- **정의**: 워킹셋 총합이 물리 프레임을 초과해 페이지 폴트가 폭증, CPU는 놀고 스와핑만 반복하는 상태.
- **증상**: CPU 사용률은 낮은데 디스크 I/O(스왑)가 극도로 높음. `vmstat`의 `si`/`so`가 지속적으로 큼.
- **해결**: ① 다중 프로그래밍 정도(동시 프로세스 수)를 낮춤 ② 물리 메모리 증설 ③ 워킹셋 모델·PFF로 프로세스별 최소 프레임 보장.
- JVM이라면 힙을 물리 메모리에 맞게 재설정하고 GC 폭주와 스와핑을 구분.

</details>
