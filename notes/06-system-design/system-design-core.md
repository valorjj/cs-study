# SD1. Scalability and Load Balancing

**학습 목표**: *"트래픽이 늘면 어떻게 확장하나요?"* / *"로드밸런서는 뭘 하나요?"* / *"왜 서버를 stateless로 만드나요?"* 에 다이어그램 그리며 5분 답할 수 있다.

## 1. 비유 — 식당의 손님이 몰릴 때
손님(트래픽)이 몰리면 두 가지 선택지가 있다. ① 주방장 한 명을 더 유능하게 만든다(칼 좋은 걸로 바꾸고 화력 세게 — **scale-up**), ② 주방을 하나 더 열어 요리사를 늘린다(**scale-out**). 손님이 어느 주방으로 갈지는 **홀 매니저(로드밸런서)** 가 정한다. 이때 각 요리사가 "이 손님 정보는 내 머릿속에만 있어(session이 서버 로컬)" 라고 하면, 다음 방문 때 다른 요리사가 받으면 곤란해진다. 그래서 모든 요리사가 손님 정보를 공용 장부(외부 저장소)에서 조회하게 만들어야 아무 요리사에게나 보내도 된다 — 이게 **stateless**.

## 2. 개념 정의 (1줄)
> **수직 확장(Scale-up)** = 서버 1대의 스펙(CPU/RAM) 증강. **수평 확장(Scale-out)** = 서버 대수를 늘려 분산. **로드밸런서(LB)** = 여러 서버로 요청을 분배하는 트래픽 교통정리 계층.

## 3. 다이어그램 — 수평 확장 + LB
```
                    Client
                      │
              ┌───────▼────────┐
              │  Load Balancer  │  (L4: TCP/IP 레벨, L7: HTTP 헤더/URL 레벨)
              └───┬───┬───┬────┘
        ┌─────────┘   │   └─────────┐
        ▼             ▼             ▼
   ┌─────────┐   ┌─────────┐   ┌─────────┐
   │ Server A│   │ Server B│   │ Server C│   ← stateless (세션 로컬 저장 X)
   └────┬────┘   └────┬────┘   └────┬────┘
        └──────────────┼──────────────┘
                        ▼
              ┌───────────────────┐
              │ 공유 세션 저장소     │  (Redis 등) — HTTP stateless 참고
              │ 공유 DB             │
              └───────────────────┘
```
- 클라이언트의 각 HTTP 요청은 LB를 거쳐 A/B/C 아무 서버든 갈 수 있음(HTTP 요청 분산). 서버가 세션을 로컬 메모리에 들고 있으면 "다음 요청이 다른 서버로 가면 로그인 풀림" 문제 발생 → **HTTP 자체가 stateless 프로토콜**이란 특성과 맞물려, 서버도 stateless로 설계해야 아무 서버에 붙여도 동작.

## 4. 로드밸런싱 알고리즘
| 알고리즘 | 동작 | 적합 상황 |
|----------|------|-----------|
| **Round Robin** | 순서대로 균등 분배 | 서버 스펙 동일, 요청 처리 시간 비슷 |
| **Weighted Round Robin** | 서버 스펙에 비례한 가중치 | 서버 스펙이 다를 때 |
| **Least Connections** | 현재 연결(처리 중 요청) 적은 서버로 | 요청 처리 시간 편차 큼(장기 커넥션) |
| **IP Hash** | 클라이언트 IP 해시로 고정 서버 배정 | 캐시 지역성(sticky) 필요할 때 |
| **Health Check** | 위 알고리즘 + 죽은 서버는 목록에서 제외 | 장애 서버 자동 배제(필수 기능) |

## 5. L4 vs L7 로드밸런서
| | L4 (Transport) | L7 (Application) |
|--|----------------|-------------------|
| 판단 기준 | IP/Port(TCP) | URL/Header/Cookie(HTTP) |
| 속도 | 빠름(패킷만 봄) | 느림(헤더 파싱) |
| 기능 | 단순 분배 | URL 기반 라우팅, A/B 테스트, SSL termination |

## 6. 핵심 포인트 (🔴 자주 하는 실수)
- 🔴 "서버 늘리면 알아서 확장된다" — ❌ 서버가 **stateful**(세션 로컬 저장)이면 수평 확장 시 특정 사용자가 특정 서버에 고정(sticky session)되어야 해서 확장 이점이 반감되고, 그 서버가 죽으면 세션 유실.
- 🔴 "Round Robin이면 항상 공평하다" — ❌ 서버 스펙이 다르거나 요청 처리 시간 편차가 크면 특정 서버에 부하 쏠림 → Least Connections/Weighted가 더 적합.
- 🟡 Scale-up은 한계(물리적 최대 스펙)와 SPOF(단일 장애점) 문제가 있고, Scale-out은 무한 확장 가능하지만 stateless 설계 + 데이터 정합성 문제(SD3)가 뒤따름.
- 🟡 LB 자체도 SPOF가 될 수 있어 LB를 이중화(Active-Standby)하는 경우가 많음.

## 7. 패턴 정리표
| 구분 | Scale-up | Scale-out |
|------|----------|-----------|
| 방법 | 서버 1대 스펙 증강 | 서버 대수 증가 |
| 한계 | 물리적 최대치 존재 | 이론상 무제한 |
| 장애 대응 | 서버 죽으면 전체 다운(SPOF) | 한 대 죽어도 나머지가 처리 |
| 전제 조건 | 없음 | **stateless 서버** + LB 필요 |
| 비용 | 초반 저렴, 후반 급증 | 초반부터 인프라 복잡도↑ |

## 8. 예상 면접 질문 + 답변 골격
**Q1. "수직 확장과 수평 확장의 차이는?"**
> ① 수직=서버 1대 스펙 증강, 간단하지만 물리적 한계·SPOF 존재 → ② 수평=서버 대수 증가, LB로 분산, 무한 확장 가능하지만 서버가 stateless여야 함.

**Q2. "로드밸런서는 무슨 역할을 하나요?"**
> ① 여러 서버로 요청 분배(Round Robin/Least Connections 등) → ② Health Check로 죽은 서버 자동 배제 → ③ L7이면 URL/헤더 기반 라우팅까지 담당.

**Q3. "왜 서버를 stateless로 만들어야 하나요?"**
> ① HTTP 자체가 stateless라 요청마다 어느 서버든 처리 가능해야 수평 확장의 이점을 살림 → ② 세션을 서버 로컬에 두면 특정 서버에 고정(sticky)돼야 하고 그 서버 장애 시 세션 유실 → ③ 세션을 Redis 등 외부 공유 저장소로 빼면 어떤 서버가 받아도 동일하게 처리 가능.

# SD2. Caching

**학습 목표**: *"캐싱 전략을 설명해주세요"* / *"Cache Stampede가 뭐죠?"* 에 다이어그램 그리며 5분 답할 수 있다.

## 1. 비유 — 자주 찾는 책은 책상 위에
도서관(DB) 전체를 매번 뒤지면 느리다. 자주 빌리는 책은 사서가 **책상 위(캐시)** 에 미리 꺼내둔다. 손님은 책상부터 확인하고, 없으면 그제서야 서고(DB)까지 간다. 이 발상은 CPU가 자주 쓰는 데이터를 L1/L2 캐시에 두는 것과 완전히 같은 원리 — **"자주 쓰는 것을 가까이 둔다"**.

## 2. 개념 정의 (1줄)
> **캐싱** = 자주 조회되는 데이터를 원본(DB)보다 빠른 저장소에 사본으로 두어 응답 속도를 높이고 원본의 부하를 줄이는 기법.

## 3. 다이어그램 — 캐시 계층 (요청이 지나가는 순서)
```
Client
  │
  ▼
① 브라우저 캐시        (Cache-Control, ETag) — 요청 자체를 안 보냄
  │ (miss)
  ▼
② CDN                  (정적 리소스, 이미지/JS/CSS, 지리적으로 가까운 엣지)
  │ (miss)
  ▼
③ 애플리케이션 캐시      (Redis/Memcached — 세션, API 응답, 조회 결과)
  │ (miss)
  ▼
④ DB 캐시               (DB 자체 버퍼 풀/쿼리 캐시)
  │ (miss)
  ▼
Database (원본, 가장 느림)
```
- 계층이 앞쪽(브라우저)일수록 빠르지만 신선도(freshness)가 떨어지고, 뒤쪽(DB)일수록 느리지만 정확함. → 트레이드오프.

## 4. 캐싱 전략 — Look-Aside vs Write-Through
```
[Look-Aside (Lazy Loading)]           [Write-Through]
 읽기:                                 쓰기:
  App → Cache 조회                     App → Cache에 먼저 씀
   miss → DB 조회 → Cache에 채움          Cache → DB에도 즉시 씀 (동기)
  hit → 바로 반환                       (그 다음 App에 응답)
 쓰기:
  App → DB만 직접 씀 (캐시는 안 건드림, 다음 읽기 때 채워짐 또는 무효화)
```
| | Look-Aside (Cache-Aside) | Write-Through |
|--|--------------------------|----------------|
| 읽기 흐름 | 캐시 먼저 확인, miss시 DB 조회 후 캐시에 채움 | 캐시 자체가 최신 유지되어 있음 |
| 쓰기 흐름 | DB만 갱신(캐시는 무효화/방치) | 캐시와 DB에 동시(동기) 기록 |
| 장점 | 구현 단순, 캐시 장애에 강함(DB로 우회 가능) | 캐시-DB 데이터 항상 일치, 읽기 시 항상 hit |
| 단점 | 첫 요청은 항상 miss(느림), 캐시-DB 불일치 가능 구간 존재 | 쓰기 지연↑(이중 쓰기), 안 읽는 데이터도 캐싱되어 낭비 |
| 실무 사용 | **가장 흔함**(Redis + Look-Aside) | 쓰기보다 읽기 압도적으로 많고 최신성이 중요할 때 |

## 5. 캐시 무효화(Invalidation) & TTL
- **TTL(Time-To-Live)**: 일정 시간 후 자동 만료. 구현 쉽지만, 만료 전까지는 **stale(오래된) 데이터**를 반환할 수 있음.
- **명시적 무효화**: DB 갱신 시 캐시를 직접 삭제/갱신. 정합성은 좋지만 삭제 누락 시 영구 stale 위험.
- 원칙: "Cache invalidation is one of the two hard things in CS" — 정답은 없고 **정합성 vs 성능**의 트레이드오프를 상황에 맞게 선택.

## 6. Cache Stampede (Thundering Herd)
```
캐시 만료 순간 ──► 동시에 수천 요청 도착
                        │
              전부 "miss" 판정
                        │
              전부 DB로 동시에 쿼리 ──► DB 순간 과부하/다운
```
- **원인**: 인기 있는 키 하나가 만료되는 순간 대량 요청이 동시에 캐시 miss를 겪고 전부 DB를 때림.
- **해결책**:
  - **Mutex/Lock**: 첫 요청만 DB 조회 + 캐시 채움, 나머지는 대기 후 캐시에서 읽음.
  - **확률적 조기 갱신(Probabilistic Early Expiration)**: TTL 임박 시 일부 요청이 미리 갱신.
  - **논리적 만료**: 캐시엔 "만료됐다"는 플래그만 두고 실제 삭제는 안 함 → 백그라운드에서 갱신, 그동안은 stale 데이터라도 반환(가용성 우선).

## 7. 핵심 포인트 (🔴 자주 하는 실수)
- 🔴 "캐시에 다 넣으면 무조건 빠르다" — ❌ 캐시 용량은 한정적. 자주 안 쓰는 데이터까지 넣으면 **eviction(LRU 등)** 이 잦아져 오히려 hit율 저하.
- 🔴 "TTL만 걸면 무효화 걱정 끝" — ❌ TTL 동안은 stale 데이터가 나갈 수 있음. 가격/재고처럼 정확성이 중요한 데이터는 명시적 무효화 병행 필요.
- 🟡 Redis는 애플리케이션 캐시 계층의 대표 구현체. in-memory라 빠르지만 서버 자체가 다운되면 캐시 전체 유실(→ replication/AOF로 대비).
- 🟡 Look-Aside에서 "DB 쓰기 → 캐시 삭제" 순서가 중요. 캐시 먼저 지우고 DB 쓰다 실패하면 다음 읽기가 옛 DB값을 다시 캐싱하는 레이스 컨디션 발생 가능.

## 8. 패턴 정리표 — 캐시 계층 & 전략
| 계층/전략 | 예시 | 장점 | 단점 |
|-----------|------|------|------|
| 브라우저 캐시 | Cache-Control, ETag | 네트워크 요청 자체 생략 | 클라이언트 제어권 없음 |
| CDN | CloudFront, Cloudflare | 지리적으로 가까움, 정적 자원에 최적 | 동적 데이터엔 부적합 |
| 애플리케이션 캐시(Redis) | 세션, API 응답 | 빠름, 유연 | 정합성 관리 필요 |
| Look-Aside | 대부분의 실무 | 단순, 장애 격리 | 첫 요청 miss |
| Write-Through | 결제·재고 등 정합성 중요 | 항상 최신 | 쓰기 비용↑ |

## 9. 예상 면접 질문 + 답변 골격
**Q1. "캐싱 전략을 설명해주세요."**
> ① Look-Aside는 캐시 먼저 확인 후 miss시 DB 조회+캐시 채움, 대부분의 실무에서 사용 → ② Write-Through는 쓰기 시점에 캐시·DB를 동시 갱신해 항상 최신 유지하지만 쓰기 지연 증가.

**Q2. "Cache Stampede가 뭐고 어떻게 막나요?"**
> ① 인기 키 만료 순간 대량 요청이 동시에 miss → DB 몰림 → ② Mutex lock으로 첫 요청만 DB 조회, 나머지 대기 / 확률적 조기 갱신 / 논리적 만료(stale 허용) 등으로 완화.

**Q3. "캐시 무효화는 어떻게 하나요?"**
> ① TTL 기반 자동 만료(간단하지만 stale 구간 존재) → ② DB 갱신 시 캐시 명시적 삭제(정합성 우선) → ③ 데이터 성격(정확성 vs 성능 요구)에 따라 선택.

# SD3. Database Scaling

**학습 목표**: *"DB가 병목이면 어떻게 확장하나요?"* / *"Replication과 Sharding 차이는?"* 에 다이어그램 그리며 5분 답할 수 있다.

## 1. 비유 — 은행 창구
읽기(잔액 조회)만 몰리면 **창구 사본을 여러 개 만들어(복제) 조회 전담 창구**를 늘리면 된다(Replication). 하지만 쓰기(입출금)까지 몰리면, 한 은행 건물(서버 1대)로는 감당이 안 되니 **고객 이름 앞글자 A~M은 1지점, N~Z는 2지점**처럼 아예 **은행을 분할**해야 한다(Sharding). 지점 간엔 서로 다른 고객 데이터를 갖고 있어 조회가 지점을 넘나들면 복잡해진다.

## 2. 개념 정의 (1줄)
> **Replication** = 같은 데이터를 여러 DB 인스턴스에 복제해 **읽기**를 분산. **Sharding** = 데이터를 여러 DB 인스턴스에 나눠 담아 **쓰기(+저장용량)** 를 분산.

## 3. 다이어그램 — Replication (Master-Slave)
```
                Write
                  │
                  ▼
             ┌─────────┐
             │  Master  │  (쓰기 전담)
             └────┬────┘
        복제(비동기/동기)
        ┌─────────┼─────────┐
        ▼         ▼         ▼
   ┌────────┐┌────────┐┌────────┐
   │ Slave 1││ Slave 2││ Slave 3│   (읽기 전담, Read Replica)
   └────────┘└────────┘└────────┘
        ▲         ▲         ▲
        └────── Read 요청 분산 ──┘
```
- 읽기(SELECT)가 압도적으로 많은 서비스(조회 위주 커머스 등)에 적합. Master가 죽으면 Slave 승격(failover)으로 대응.
- **복제 지연(Replication Lag)**: Master에 쓰고 바로 Slave에서 읽으면 반영 전일 수 있음 → "내가 방금 쓴 글이 안 보이는" 문제.

## 4. 다이어그램 — Sharding (수평 분할)
```
        Client
          │
   ┌──────▼──────┐
   │ Shard Router │  (user_id % N 또는 해시 기반으로 어느 shard인지 결정)
   └──┬───┬───┬──┘
      ▼   ▼   ▼
  ┌─────┐┌─────┐┌─────┐
  │Shard│Shard│Shard│
  │  A  ││  B  ││  C  │   각자 다른 데이터(user 1~1000, 1001~2000, ...)
  └─────┘└─────┘└─────┘
```
- 각 shard는 전체 데이터의 일부만 가짐 → 쓰기·저장 용량이 shard 수만큼 분산.
- Shard Key 선택이 핵심(예: user_id). 잘못 고르면 특정 shard에만 쏠리는 **hot shard** 문제.

## 5. 파티셔닝(Partitioning) — Sharding과의 관계
| 구분 | 파티셔닝(Partitioning) | 샤딩(Sharding) |
|------|------------------------|-----------------|
| 범위 | 한 DB 인스턴스 **내부**에서 테이블을 나눔 | **여러 DB 인스턴스**로 나눔 |
| 목적 | 큰 테이블 관리 용이(파티션 단위 인덱스/삭제) | 서버 자체의 부하·용량 분산 |
| 예 | 날짜별 파티션(월별 로그 테이블) | user_id 기준으로 서버 3대에 분산 |
> Sharding = "물리적으로 다른 서버로의" 수평 분할(파티셔닝의 확장 개념).

## 6. 트레이드오프
- **Replication**: 읽기 확장엔 강하지만 쓰기는 여전히 Master 1대 병목. 복제 지연으로 인한 일관성 문제 상존.
- **Sharding**: 쓰기·용량 모두 확장되지만, **JOIN이 shard를 넘나들 수 없음**, 트랜잭션이 여러 shard에 걸치면 분산 트랜잭션 필요(복잡), 재샤딩(re-sharding) 비용 큼.
- DB 인덱스는 단일 서버 내 조회 속도를 높이는 것이고, Replication/Sharding은 그 서버 자체를 늘리는 것 — **레벨이 다른 최적화**. 트랜잭션(ACID)도 Sharding 환경에선 shard 경계를 넘으면 보장이 어려워짐.

## 7. 핵심 포인트 (🔴 자주 하는 실수)
- 🔴 "Replication이 쓰기도 분산해준다" — ❌ 쓰기는 Master 1대만 처리. 읽기만 분산됨.
- 🔴 "Sharding하면 인덱스가 필요 없다" — ❌ 각 shard 내부에서도 여전히 인덱스 설계는 그대로 필요. Sharding은 "어느 서버로 갈지"의 문제, 인덱스는 "그 서버 안에서 얼마나 빨리 찾을지"의 문제.
- 🟡 Shard Key를 잘못 고르면(예: 가입일 기준) 최근 가입자가 몰린 shard 하나만 뜨거워짐(hot shard) → 균등 분산되는 키(해시) 선택이 중요.
- 🟡 Replication Lag 때문에 "쓰고 바로 읽기(read-your-own-write)"가 필요한 요청은 Master에서 직접 읽도록 예외 처리하는 경우가 많음.

## 8. 패턴 정리표
| | Replication | Sharding |
|--|-------------|-----------|
| 분산 대상 | 읽기(Read) | 쓰기(Write) + 저장용량 |
| 데이터 | 모든 노드가 동일 데이터 | 노드마다 다른 데이터 |
| 확장 한계 | Master가 병목(쓰기) | Shard Key 설계, JOIN 제약 |
| 장애 대응 | Slave 승격(failover) | 특정 shard 장애 시 그 데이터만 영향 |
| 대표 문제 | Replication Lag | Hot shard, 분산 트랜잭션 |

## 9. 예상 면접 질문 + 답변 골격
**Q1. "DB 병목을 어떻게 해결하나요?"**
> ① 읽기가 많으면 Read Replica(Master-Slave Replication)로 읽기 분산 → ② 쓰기·용량 자체가 문제면 Sharding으로 데이터를 여러 서버에 수평 분할 → ③ 두 방법을 함께 쓰는 경우도 흔함(각 shard마다 replica).

**Q2. "Replication과 Sharding 차이는?"**
> Replication은 같은 데이터를 복제해 읽기 분산(쓰기는 Master 1대). Sharding은 데이터 자체를 쪼개 여러 서버에 나눠 담아 쓰기·용량을 분산. 목적이 다름.

**Q3. "Sharding의 단점은?"**
> ① shard 간 JOIN 불가/제약 → ② 여러 shard에 걸친 트랜잭션은 분산 트랜잭션 필요(복잡·느림) → ③ Shard Key를 잘못 고르면 hot shard 발생, 재샤딩 비용도 큼.

# SD4. Message Queue

**학습 목표**: *"메시지 큐를 왜 쓰나요?"* / *"Kafka와 RabbitMQ 차이는?"* / *"at-least-once가 뭐죠?"* 에 5분 답할 수 있다.

## 1. 비유 — 식당 주문 티켓
손님이 주문하면 요리사가 바로 요리해야만 주문을 받아줄 수 있다면, 요리사가 바쁠 때 손님은 홀에서 계속 기다려야 한다(직접 호출 = 강결합 + 부하 그대로 전달). 대신 **주문 티켓(메시지)** 을 주문 걸이(큐)에 걸어두면, 손님은 바로 돌아가고(비동기), 요리사는 자기 속도대로 티켓을 하나씩 처리한다(버퍼링). 홀 담당(주문받는 쪽)과 주방(처리하는 쪽)이 서로 몰라도 티켓 걸이만 공유하면 되니 **분리(decoupling)** 된다.

## 2. 개념 정의 (1줄)
> **메시지 큐(MQ)** = 송신자(Producer)와 수신자(Consumer) 사이에 메시지를 중간 저장해, 시스템을 **비동기**로 연결하고 **decoupling**하며 트래픽 급증 시 **버퍼** 역할을 하는 미들웨어.

## 3. 다이어그램
```
[동기 직접 호출 — 문제]                  [메시지 큐 — 해결]
 Client → ServiceA → ServiceB           Client → ServiceA → [Queue] → ServiceB
   (B 느리면 A도 느려짐,                        (A는 큐에 넣고 즉시 응답,
    B 다운되면 A도 실패)                          B는 자기 속도로 소비,
                                                B 다운돼도 메시지는 큐에 남아있음)

           Producer                     Queue                    Consumer
              │                    ┌─────────┐                      │
              └── publish ────────►│ msg msg │◄──── poll/consume ───┘
                                    │ msg ... │
                                    └─────────┘
                    급증한 요청도 큐에 쌓이며 Consumer는
                    자기 처리 속도만큼만 꺼내감 → 부하 완충(버퍼링)
```

## 4. Kafka vs RabbitMQ (간단 비교)
| | Kafka | RabbitMQ |
|--|-------|-----------|
| 모델 | 로그 기반(분산 커밋 로그), Consumer가 원하는 offset부터 재소비 가능 | 전통적 메시지 브로커(큐에서 꺼내면 사라짐) |
| 처리량 | 매우 높음(대용량 스트리밍에 강함) | 중간(복잡한 라우팅에 강함) |
| 메시지 순서 | 파티션 내에서 순서 보장 | 큐 단위로 순서 보장 |
| 라우팅 | 단순(topic 기반) | 복잡한 라우팅 규칙(exchange: direct/topic/fanout) |
| 대표 용도 | 로그 수집, 이벤트 스트리밍, 대용량 데이터 파이프라인 | 작업 큐(task queue), 마이크로서비스 간 메시징 |

## 5. 전달 보장(Delivery Semantics)
| 방식 | 의미 | 문제/특징 |
|------|------|-----------|
| **At-most-once** | 최대 한 번 전달(손실 가능) | 재시도 없음 → 유실 위험, 성능은 좋음 |
| **At-least-once** | 최소 한 번 전달(중복 가능) | ack 실패 시 재전송 → **중복 처리 가능**, Consumer가 멱등성(idempotency) 구현 필요 |
| **Exactly-once** | 정확히 한 번 전달 | 이상적이지만 구현 복잡·비용 큼(Kafka는 트랜잭션+멱등 Producer 조합으로 근사 지원) |

## 6. 핵심 포인트 (🔴 자주 하는 실수)
- 🔴 "메시지 큐 쓰면 무조건 exactly-once다" — ❌ 대부분 시스템의 기본값은 **at-least-once**. Exactly-once는 별도 설정(멱등 Producer + 트랜잭션)과 비용이 필요.
- 🔴 "MQ는 순서를 항상 보장한다" — ❌ Kafka는 **파티션 내에서만** 순서 보장. 여러 파티션에 걸치면 전역 순서 보장 안 됨.
- 🟡 At-least-once 환경에서는 Consumer 로직을 **멱등하게**(같은 메시지 여러 번 처리해도 결과가 같도록) 설계해야 함 — 예: "잔액 +1000" 대신 "주문ID로 이미 처리했는지 체크 후 처리".
- 🟡 MQ는 동기 호출의 실패 전파를 막아주지만, 대신 "메시지가 언제 처리될지 보장 안 됨(eventual)"이라는 결과적 일관성 트레이드오프를 받아들여야 함(SD5 CAP과 연결).

## 7. 패턴 정리표
| 목적 | 메시지 큐가 주는 것 |
|------|----------------------|
| 비동기 처리 | Producer는 응답 기다리지 않고 즉시 리턴 |
| 시스템 decoupling | Producer/Consumer가 서로의 존재·스펙을 몰라도 됨 |
| 부하 완충(버퍼링) | 트래픽 급증 시 큐에 쌓이고 Consumer는 자기 속도로 처리 |
| 장애 격리 | Consumer가 죽어도 메시지는 큐에 남아 복구 후 처리 가능 |

## 8. 예상 면접 질문 + 답변 골격
**Q1. "메시지 큐를 왜 쓰나요?"**
> ① 동기 직접 호출은 수신 서비스가 느리거나 다운되면 발신 서비스까지 영향받음 → ② 큐를 두면 비동기 처리로 즉시 응답, 시스템 간 decoupling, 트래픽 급증 시 버퍼 역할까지 가능.

**Q2. "Kafka와 RabbitMQ 차이는?"**
> Kafka는 로그 기반이라 대용량 스트리밍·재소비에 강하고, RabbitMQ는 전통적 브로커로 복잡한 라우팅 규칙에 강함. 대용량 이벤트 파이프라인엔 Kafka, 작업 큐엔 RabbitMQ가 흔한 선택.

**Q3. "at-least-once가 뭐고 왜 문제가 되나요?"**
> ① ack 실패 시 재전송하는 방식이라 메시지가 유실되진 않지만 **중복 전달**될 수 있음 → ② Consumer가 멱등하게 설계되어 있지 않으면 중복 처리(예: 중복 결제)로 이어질 수 있어 반드시 멱등성 보장 필요.

# SD5. CAP Theorem

**학습 목표**: *"CAP 이론이 뭔가요?"* / *"왜 셋 다 만족 못 하나요?"* / *"결과적 일관성이 뭐죠?"* 에 5분 답할 수 있다.

## 1. 비유 — 두 도시의 은행 지점
서울 지점과 부산 지점이 동시에 잔액 정보를 갖고 있다. 어느 날 두 지점을 잇는 통신선이 끊겼다(**Partition 발생**). 이때 선택은 둘 중 하나다. ① 통신이 복구될 때까지 **아예 거래를 중단**한다 → 모든 지점의 잔액이 항상 정확함은 보장되지만(Consistency), 손님은 거래를 못 한다(Availability 포기). ② 통신이 끊겨도 각 지점이 **자기가 가진 정보로 일단 거래를 계속 받는다** → 손님은 항상 거래 가능하지만(Availability), 두 지점의 잔액이 서로 달라질 수 있다(Consistency 포기).

## 2. 개념 정의 (1줄)
> **CAP 이론** = 분산 시스템은 **Consistency**(모든 노드가 항상 같은 데이터를 봄), **Availability**(모든 요청이 항상 응답을 받음), **Partition Tolerance**(노드 간 통신 장애가 있어도 시스템이 계속 동작)를 **동시에 세 가지 다 만족할 수 없고, 그 중 2개만 선택 가능**하다는 이론.

## 3. 다이어그램
```
                Consistency
                    ▲
                   ╱ ╲
                  ╱   ╲
                 ╱ CA  ╲     ← 현실 분산 시스템에서 사실상 선택 불가
                ╱ (이론상)╲      (네트워크 파티션은 언젠가 발생하므로)
               ╱───────────╲
              ╱             ╲
             ╱  CP     AP    ╲
            ╱ (정합성)  (가용성) ╲
           ▼───────────────────▼
     Partition                Availability
     Tolerance
     (분산 시스템은 이게 전제 조건 → 결국 C vs A 트레이드오프)
```
- 분산 시스템은 네트워크 장애(Partition)가 **언젠가는 반드시 발생**하므로, **P는 선택이 아니라 전제**. 실제 설계 고민은 "Partition 상황에서 C를 지킬지 A를 지킬지"의 **CP vs AP** 선택이다.

## 4. CP vs AP 예시
| | CP (Consistency + Partition Tolerance) | AP (Availability + Partition Tolerance) |
|--|------------------------------------------|--------------------------------------------|
| Partition 발생 시 | 정합성 깨질 위험 있으면 **응답을 거부**(에러/타임아웃) | 오래된 데이터라도 **응답은 반환** |
| 예시 | 은행 계좌 이체, 재고 차감(정확성이 중요) | SNS 좋아요 수, 조회수(약간 늦어도 무방) |
| 대표 시스템 | ZooKeeper, HBase, RDBMS(강한 일관성 모드) | Cassandra, DynamoDB, 대부분의 NoSQL 기본 설정 |

## 5. 결과적 일관성 (Eventual Consistency)
```
t0: Node A에 write(x=5)
t1: Node B는 아직 x=3 (전파 지연)
t2: 복제 전파 완료 → Node B도 x=5로 수렴
```
> **즉시 일관성을 포기하는 대신 가용성을 얻고, 시간이 지나면(전파가 끝나면) 결국 모든 노드가 같은 값으로 수렴**한다는 보장. AP 시스템(대부분 NoSQL)의 기본 동작 방식이며, **BASE(Basically Available, Soft state, Eventually consistent)** 모델이 CAP의 AP 선택을 구체화한 실무 원칙.

## 6. 핵심 포인트 (🔴 자주 하는 실수)
- 🔴 "CA를 선택하면 된다" — ❌ 분산 시스템에서 네트워크 파티션은 피할 수 없는 현실이라 **P는 선택지가 아니라 전제**. 사실상 CP vs AP 중 선택.
- 🔴 "NoSQL은 무조건 일관성이 없다" — ❌ 정확히는 **대부분 AP를 기본값으로 택해 결과적 일관성을 채택**한 것. 일부 NoSQL은 설정으로 강한 일관성 모드도 지원(예: Cassandra의 quorum read/write).
- 🟡 CAP은 **Partition이 실제로 발생했을 때만** 적용되는 트레이드오프. 평상시(파티션 없을 때)는 C와 A를 동시에 만족 가능.
- 🟡 어떤 데이터에 CP, 어떤 데이터에 AP를 쓸지는 **서비스 특성**에 달림 — 결제/재고는 CP, 좋아요 수/추천 피드는 AP가 흔한 선택.

## 7. 패턴 정리표
| 상황 | 선택 | 이유 |
|------|------|------|
| 계좌 이체, 재고 차감 | CP | 정합성이 깨지면 손실/오배송으로 직결 |
| SNS 좋아요, 조회수, 추천 | AP | 약간 늦게 반영돼도 서비스 지장 없음, 항상 응답이 더 중요 |
| 분산 락, 리더 선출(ZooKeeper) | CP | 여러 노드가 다른 리더를 보면 시스템 오동작 |
| 글로벌 서비스의 캐시/세션 | AP | 지역 간 지연보다 즉시 응답이 중요 |

## 8. 예상 면접 질문 + 답변 골격
**Q1. "CAP 이론이 뭔가요?"**
> ① 분산 시스템은 Consistency, Availability, Partition Tolerance 세 가지를 동시에 만족할 수 없고 2개만 선택 가능 → ② 네트워크 파티션은 필연적으로 발생하므로 P는 전제 조건 → ③ 실제로는 파티션 상황에서 C를 지킬지(CP) A를 지킬지(AP)의 문제.

**Q2. "왜 셋 다 만족할 수 없나요?"**
> 노드 간 통신이 끊긴 상황(Partition)에서, 정합성을 지키려면 통신 복구 전까지 응답을 거부해야 하고(Availability 포기), 항상 응답하려면 오래된 데이터를 줄 수밖에 없다(Consistency 포기). 둘 다 동시에는 불가능.

**Q3. "결과적 일관성이 뭔가요?"**
> 쓰기 직후엔 노드마다 값이 다를 수 있지만, 복제가 전파되면 결국 모든 노드가 같은 값에 수렴한다는 보장. AP를 택한 분산 시스템(대부분 NoSQL, BASE 모델)이 즉시 일관성 대신 항상 응답 가능함을 얻기 위해 채택하는 방식.

**Q4. "어떤 상황에 CP, 어떤 상황에 AP를 쓰나요?"**
> 정확성이 서비스 성패를 좌우하는 결제·재고는 CP, 약간 늦어도 무방하고 항상 응답이 중요한 좋아요 수·피드는 AP를 선택하는 게 일반적.
