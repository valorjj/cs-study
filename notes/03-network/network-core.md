# N1. OSI and TCP/IP Model

**학습 목표**: *"OSI 7계층 / TCP-IP 계층을 설명하고, 데이터가 어떻게 흐르나요?"* 에 답할 수 있다.

## 1. 비유 — 국제 택배
물건(데이터)을 보낼 때 여러 단계 포장을 거침: 물건 → 상자 → 운송장 부착 → 트럭 적재. 받는 쪽은 역순으로 개봉. 각 계층이 자기 "포장 정보(헤더)"를 붙였다 떼는 게 **캡슐화/역캡슐화**.

## 2. 계층 대응표
| OSI 7계층 | TCP/IP 4계층 | 역할 | 예 |
|-----------|--------------|------|-----|
| 7 Application | Application | 사용자 서비스 | HTTP, DNS, FTP |
| 6 Presentation | (Application) | 암호화·인코딩 | TLS, JPEG |
| 5 Session | (Application) | 연결 세션 관리 | - |
| 4 Transport | Transport | 종단 간 전달·신뢰성 | **TCP, UDP** |
| 3 Network | Internet | 라우팅·논리주소 | **IP**, ICMP |
| 2 Data Link | Network Access | 인접 노드 전달·물리주소 | Ethernet, MAC |
| 1 Physical | (Network Access) | 비트 전송 | 케이블, 전기신호 |

## 3. 다이어그램 — 캡슐화
```
송신 (내려가며 헤더 추가)          수신 (올라가며 헤더 제거)
 App:  [        Data        ]        [        Data        ]
 TCP:  [TCP hdr| Data ]  ← 세그먼트    ▲ 역캡슐화
 IP:   [IP hdr|TCP| Data]  ← 패킷      │
 Link: [Eth|IP|TCP|Data|]  ← 프레임 ───┘ 물리 전송
```

## 4. 핵심 포인트
- 🟡 각 계층은 **바로 위/아래 계층하고만** 인터페이스 → 계층 독립성(한 계층 바뀌어도 나머지 영향 최소).
- 🟡 "IP는 3계층(주소·라우팅), TCP/UDP는 4계층(포트·종단전달)" — 계층 헷갈리지 말 것.
- 🔴 OSI는 **참조 모델**(이론), 실제 인터넷은 **TCP/IP** 스택으로 동작.

## 5. 예상 면접 질문
**Q. "OSI 계층에서 TCP와 IP는 각각 몇 계층이고 왜 나뉘나요?"**
> IP는 3계층(네트워크)으로 논리 주소·라우팅 담당, TCP는 4계층(전송)으로 포트 기반 종단 간 신뢰성 전달 담당. 역할을 계층으로 분리해 독립적으로 교체·발전할 수 있게 한 것이 계층 모델의 핵심입니다.

> 🔎 꼬리 질문: *"OSI 7계층인데 왜 실제론 TCP/IP 4계층을 쓰나요?"* → OSI는 통신을 이해·설계하기 위한 참조 모델(이론)이고, TCP/IP는 실제 인터넷에서 구현·표준화된 프로토콜 스택이기 때문입니다. Presentation·Session 계층 역할은 실무에서 Application 계층 프로토콜(TLS, HTTP 등)이 흡수해 처리합니다.

# N2. TCP and UDP

**학습 목표**: *"TCP와 UDP 차이, 3-way handshake?"* 에 다이어그램 그리며 답할 수 있다.

## 1. 비유 — 등기우편 vs 엽서
- **TCP** = 등기우편. 받았는지 확인(ACK), 순서 보장, 유실 시 재전송. 느리지만 확실.
- **UDP** = 엽서. 그냥 던짐. 도착·순서 보장 없음. 대신 빠르고 가벼움.

## 2. 비교표
| 구분 | TCP | UDP |
|------|-----|-----|
| 연결 | 연결형(handshake) | 비연결형 |
| 신뢰성 | 보장(ACK·재전송·순서) | 없음 |
| 속도/오버헤드 | 느림·큼 | 빠름·작음 |
| 흐름/혼잡 제어 | 있음 | 없음 |
| 용도 | HTTP, 파일, 메일 | 스트리밍, 게임, DNS, VoIP |

## 3. 3-way Handshake (연결 수립)
```
Client                         Server
   │  ── SYN (seq=x) ──────────►│   "연결하자"
   │  ◄── SYN+ACK (seq=y,ack=x+1)│   "그래, 나도 준비"
   │  ── ACK (ack=y+1) ────────►│   "확인, 시작"
   │        [연결 established]    │
```
> 왜 3번? 양쪽 모두 **상대의 송신·수신 능력**을 확인하려면 최소 3번 필요.

## 4. 4-way Handshake (연결 종료)
```
   │ ── FIN ──►│   "나 끝"
   │ ◄── ACK ──│   "알겠어"
   │ ◄── FIN ──│   "나도 끝"   (남은 데이터 보내고)
   │ ── ACK ──►│   "확인"      → TIME_WAIT 후 close
```
> 종료가 4번인 이유: 서버가 보낼 데이터가 남았을 수 있어 **ACK와 FIN을 분리**해서 보냄.

## 5. 신뢰성·제어 (TCP)
- **흐름 제어(Flow Control)**: 수신자 버퍼 넘치지 않게 window로 속도 조절.
- **혼잡 제어(Congestion Control)**: 네트워크 혼잡 시 전송량 조절(slow start 등).

## 6. 흐름 제어 vs 혼잡 제어 심화 ⭐

### 6-1. 흐름 제어(Flow Control) — Sliding Window
비유: 물 따르기. 받는 사람 **컵(수신 버퍼)** 크기를 넘게 따르면 넘침 → 상대가 "내 컵 이만큼 남았어"를 매번 알려줘야 함.
- 수신자는 매 ACK에 **rwnd(receive window)** 값을 실어 보냄 = "지금 이만큼 더 받을 수 있어".
- 송신자는 ACK를 기다리지 않고도 rwnd 범위 안에서는 **여러 세그먼트를 미리 연속 전송**(sliding window) → 한 번 보내고 기다리는 stop-and-wait보다 훨씬 효율적.
- `rwnd = 0`(Zero Window) → 송신 중단, 수신자가 버퍼 비우면 **Window Update**로 재개.
```
송신자 window(rwnd만큼)                수신 버퍼
[1][2][3][4][5][6][7][8] ─────►  ACK(ack=3, rwnd=5) 오면 window가 오른쪽으로 슬라이드
        └── 이미 보낸/ACK 대기 ──┘
```
> 핵심: 흐름 제어는 **수신자(상대방) 버퍼 상태** 기준 — "너 못 따라오니 늦출게".

### 6-2. 혼잡 제어(Congestion Control) — Slow Start / AIMD
흐름 제어와 달리, 혼잡 제어는 **네트워크(중간 경로)의 처리 능력**을 추정해서 조절 — "회선 자체가 막힐 것 같으니 늦출게". 기준 변수는 **cwnd(congestion window)**이고, 실제 전송량은 `min(cwnd, rwnd)`.

1. **Slow Start**: cwnd를 작게(1~2 MSS) 시작해 **RTT마다 2배(지수적)** 증가 → `ssthresh`(임계값) 도달 또는 손실 발생 시 다음 단계로.
2. **Congestion Avoidance (AIMD)**: ssthresh 이후엔 **RTT마다 +1 MSS(선형 증가, Additive Increase)** — 조심스럽게 늘림.
3. **손실 발생 시 (Multiplicative Decrease)**:
   - **Timeout**(패킷 완전 유실 추정) → cwnd를 1로 리셋하고 slow start부터 재시작(강한 처벌).
   - **3중복 ACK(Fast Retransmit)** → 아직 네트워크가 살아있다는 신호이므로 cwnd를 **절반으로만** 줄이고 congestion avoidance 재개(Fast Recovery) — timeout보다 훨씬 덜 가혹.

```
cwnd
 │        /\slow start(지수)   AIMD(선형↑) 후 손실→반으로 뚝(톱니 모양)
 │       /  \___/\        /\
 │      /        \      _/  \___
 │     /  ssthresh \  _/        \_
 │    /              \/
 └──────────────────────────────► time (RTT)
```

### 6-3. 흐름 제어 vs 혼잡 제어 비교표
| 구분 | 흐름 제어 | 혼잡 제어 |
|------|-----------|-----------|
| 목적 | 수신자 버퍼 오버플로 방지 | 네트워크(경로) 혼잡 방지 |
| 기준 변수 | rwnd (수신 윈도우) | cwnd (혼잡 윈도우) |
| 판단 근거 | 상대방이 알려주는 버퍼 여유 | 손실·타임아웃으로 **추정** |
| 증가 방식 | 상대가 알려준 만큼 즉시 반영 | slow start(지수) → AIMD(선형) |
| 감소 트리거 | rwnd=0 | timeout(급감) / 3중복ACK(반으로) |

- 🔴 "혼잡 제어는 수신자가 알려준다" — ❌ 수신자는 흐름 제어(rwnd)만 알려주고, 혼잡 제어(cwnd)는 **송신자가 손실 신호로 스스로 추정**합니다.
- 🟡 실제 전송 가능량은 `min(rwnd, cwnd)` — 버퍼도 여유 있고 네트워크도 안 막혀야 최대 속도.
- 🔎 꼬리 질문: *"타임아웃과 3중복 ACK, 왜 cwnd 감소 폭이 다른가요?"* → 타임아웃은 패킷이 완전히 사라진 심각한 신호(1로 리셋), 3중복 ACK는 뒤 패킷은 도착 중이라는 뜻(네트워크가 살아있음)이라 절반만 줄여 회복을 빠르게 합니다.

## 7. 핵심 포인트 (자주 하는 실수)
- 🔴 "UDP는 항상 나쁘다" — ❌ 실시간성이 중요한 곳(게임·스트리밍·DNS)에선 재전송 지연이 오히려 해로워 UDP가 적합.
- 🟡 **TIME_WAIT**: 종료 후 잠시 대기(늦게 온 패킷 처리·재사용 방지). 서버에 많이 쌓이면 튜닝 대상.
- 🟡 소켓 통신은 결국 **OS의 I/O 모델**(블로킹/논블로킹, epoll — OS N5) 위에서 동작.

## 8. 예상 면접 질문
**Q1. "TCP와 UDP 차이?"**
> TCP는 연결형으로 handshake·ACK·재전송·순서보장·흐름/혼잡제어를 제공해 신뢰적이지만 무겁고, UDP는 비연결형으로 보장이 없는 대신 빠릅니다. 신뢰성이 필요하면 TCP, 실시간성이 중요하면 UDP.

**Q2. "3-way handshake를 설명해주세요."**
> SYN(연결 요청) → SYN+ACK(수락+역요청) → ACK(확인) 3단계로, 양쪽의 송수신 능력을 상호 확인해 연결을 수립합니다. 종료는 FIN/ACK를 각각 주고받아 4-way입니다.

**Q3. "흐름 제어와 혼잡 제어의 차이는 무엇인가요?"**
> 흐름 제어는 수신자의 버퍼(rwnd)를 기준으로 상대가 알려준 만큼 속도를 맞추는 것이고, 혼잡 제어는 네트워크 경로 자체의 혼잡을 손실·타임아웃으로 추정해 송신자가 cwnd를 slow start(지수 증가) → AIMD(선형 증가, 손실 시 감소)로 조절하는 것입니다. 실제 전송량은 둘 중 작은 값(min(rwnd, cwnd))으로 제한됩니다.

# N3. HTTP

**학습 목표**: *"HTTP 메서드·상태코드, HTTP/1.1 vs 2 vs 3, stateless?"* 에 답할 수 있다.

## 1. 비유 — 식당 주문
손님(클라이언트)이 주문서(요청)를 내면 주방(서버)이 음식(응답)을 줌. 주방은 손님을 **기억하지 않음**(stateless) → 매 주문서에 필요한 정보를 다 적어야(헤더·토큰).

## 2. 요청 구조 + 메서드
- 요청 = **메서드 + URL + 헤더 + (바디)**.
| 메서드 | 용도 | 멱등성 |
|--------|------|--------|
| GET | 조회 | O |
| POST | 생성/처리 | X |
| PUT | 전체 교체 | O |
| PATCH | 부분 수정 | X |
| DELETE | 삭제 | O |
> **멱등(idempotent)**: 여러 번 호출해도 결과 동일. 재시도 안전성과 직결.

## 3. 상태 코드
| 범위 | 의미 | 예 |
|------|------|-----|
| 2xx | 성공 | 200 OK, 201 Created, 204 No Content |
| 3xx | 리다이렉트 | 301 영구, 302 임시, 304 Not Modified(캐시) |
| 4xx | 클라이언트 오류 | 400, 401 인증, 403 권한, 404, 429 Too Many |
| 5xx | 서버 오류 | 500, 502 Bad Gateway, 503 Unavailable |

## 4. HTTP 버전 진화 ⭐
| 버전 | 핵심 | 해결한 문제 |
|------|------|-------------|
| 1.0 | 요청마다 연결 | 매번 TCP 연결 비용 |
| **1.1** | keep-alive(연결 재사용), 파이프라이닝 | 연결 재사용. 단 **HOL 블로킹** 잔존 |
| **2** | 멀티플렉싱(한 연결에 병렬 스트림), 헤더 압축, 서버 푸시 | 1.1의 HOL 블로킹(HTTP 레벨) |
| **3** | **QUIC(UDP 기반)** | TCP HOL 블로킹까지 제거, 연결 수립 빠름 |

## 5. Stateless와 상태 유지
- HTTP는 **stateless**(요청 간 상태 기억 안 함) → 확장성↑.
- 로그인 유지는 별도 수단: **쿠키/세션**(서버 저장) 또는 **토큰(JWT)**(클라이언트 보관, 서버 무상태 유지).

## 6. HTTP 캐시 심화 ⭐
비유: 편의점에서 매번 사는 삼각김밥 — 유통기한(신선도) 안 지났으면 그냥 먹고(**강한 캐시**), 애매하면 점원한테 "이거 아직 괜찮아요?"만 물어보고(**약한 캐시/재검증**) 새로 안 사도 됨.

- **`Cache-Control`** (HTTP/1.1, 현재 표준): 캐시 정책의 핵심 헤더.
  - `max-age=N`: N초 동안 신선(재요청 없이 그대로 사용).
  - `no-cache`: 저장은 하되 **매번 서버에 검증**(재검증 없이 그대로 쓰면 안 됨) — 이름과 달리 캐시를 아예 안 쓰는 게 아님.
  - `no-store`: 아예 저장 금지(민감 정보 응답 등).
  - `public` / `private`: 중간 캐시(CDN 등)까지 허용할지 / 브라우저(개인)만 허용할지.
- **`Expires`**: HTTP/1.0의 절대 만료 **시각**(`Wed, 21 Oct 2026 07:28:00 GMT`) 지정 방식. 서버-클라 시계 오차 문제가 있어 `Cache-Control: max-age`가 있으면 그게 우선.
- **강한 캐시 vs 약한 캐시(재검증)**:
  - **강한 캐시(Strong Cache)**: `max-age`/`Expires`가 안 지났으면 **서버에 요청 자체를 보내지 않고** 로컬 캐시를 그대로 사용(네트워크 왕복 0).
  - **약한 캐시(Weak/Conditional Cache)**: 신선도 기간이 지나면 서버에 "그동안 안 바뀌었어?"라고 **조건부 요청**을 보냄 — 안 바뀌었으면 바디 없이 `304 Not Modified`만 응답(전송량 절약), 바뀌었으면 `200`+새 바디.
- **`ETag` / `If-None-Match`**: `ETag`는 리소스의 해시/버전 값(내용 기반이라 정확). 클라이언트가 다음 요청 때 `If-None-Match: <ETag값>`을 보내면 서버가 비교해 같으면 304.
- (참고) `Last-Modified` / `If-Modified-Since`: 수정 **시각** 기반이라 초 단위 미만의 변경은 놓칠 수 있음 → 정확도는 ETag가 더 높음.

```
[강한 캐시]                         [약한 캐시(재검증)]
브라우저: max-age 안 지남             브라우저: max-age 지남
   └─ 서버 요청 없이 바로 사용          └─ GET + If-None-Match: "abc123"
                                            └─ 서버: 안 바뀜 → 304 (바디 없음)
                                               바뀜   → 200 + 새 ETag + 새 바디
```

| 구분 | 강한 캐시 | 약한 캐시(재검증) |
|------|-----------|--------------------|
| 헤더 | `Cache-Control: max-age`, `Expires` | `Cache-Control: no-cache` + `ETag`/`Last-Modified` |
| 서버 요청 | 안 함(기간 내) | 매번 보내되 바디는 안 받을 수 있음 |
| 응답 | 로컬 캐시 그대로 | `304`(재사용) 또는 `200`(갱신) |
| 장점 | 가장 빠름(왕복 0) | 최신성 보장하며 트래픽 절약 |

- 🔴 "`no-cache`는 캐시를 안 쓴다는 뜻" — ❌ 저장은 하되 **매번 검증**한다는 뜻. 아예 저장을 막는 건 `no-store`.
- 🔎 꼬리 질문: *"ETag와 Last-Modified 중 뭐가 더 정확한가요?"* → ETag는 내용 기반 해시라 1초 이내 여러 번 바뀌어도 감지하지만, Last-Modified는 초 단위 시각이라 그 안의 변경은 놓칠 수 있어 ETag가 더 정확합니다.

## 7. 쿠키 vs 세션 vs JWT 심화 비교 ⭐
비유: **쿠키** = 손목에 찍힌 놀이공원 도장(내가 들고 다님), **세션** = 클럽 명부에 내 정보를 적어두고 손목엔 번호표만 줌(진짜 정보는 클럽/서버가 보관), **JWT** = 위조 방지 서명이 된 신분증(어느 지점을 가든 그 자체로 검증 가능, 본사에 확인 전화 안 해도 됨).

| 구분 | 쿠키 | 세션 | JWT |
|------|------|------|-----|
| 저장 위치 | 클라이언트(브라우저) | 서버(메모리/Redis 등) — 클라엔 세션 ID만 쿠키로 | 클라이언트(로컬스토리지/쿠키) — 서버는 저장 안 함 |
| 서버 확장성 | 서버 상태 없음 | 여러 서버 간 세션 **공유 필요**(Redis, sticky session) → 확장 시 부담 | 서버가 상태를 안 들고 있어 **무상태·수평 확장 용이** |
| 보안 | 값이 그대로 노출 → XSS 위험, `HttpOnly`/`Secure`/`SameSite` 필수 | 클라엔 ID만 노출, 실제 데이터는 서버 안에 있어 비교적 안전 | 페이로드는 서명만 되고 **암호화는 아님**(디코딩하면 내용 보임) → 민감정보 넣지 말 것 |
| 무효화(로그아웃) | 클라에서 삭제 요청만 가능, 서버가 강제로 막기 어려움 | 서버에서 세션 삭제 시 **즉시** 무효화 가능 | 이미 발급된 토큰은 만료 전까지 **서버가 즉시 무효화하기 어려움** → 블랙리스트 or 짧은 만료+Refresh Token 필요 |
| 대표 용도 | 클라이언트 측 상태·설정 저장 | 전통적 모놀리식 웹의 로그인 유지 | MSA·모바일 앱 등 무상태 인증(OAuth/JWT 기반) |

- 🔴 "JWT는 서버에 저장 안 하니 더 안전하다" — ❌ 저장을 안 해서 **확장성**은 좋지만, 탈취당하면 만료 전까지 막을 방법이 마땅치 않다는 게 오히려 약점입니다. 그래서 access token은 짧게(예: 15분), refresh token으로 재발급하는 패턴을 씁니다.
- 🔎 꼬리 질문: *"로그아웃을 눌렀는데 JWT는 어떻게 무효화하나요?"* → JWT 자체엔 서버 측 상태가 없어 즉시 무효화가 어렵습니다. 실무에서는 (1) 만료시간을 짧게 두고 refresh token으로 재발급, (2) 탈취/로그아웃된 토큰만 서버에 블랙리스트로 잠깐 보관해 검증 시 걸러내는 방식을 함께 씁니다.

## 8. CORS (동일 출처 정책과 Preflight)
비유: 남의 집(다른 출처) 물건은 함부로 못 가져감. 집주인(서버)이 "얘(이 출처)는 가져가도 좋다"는 허락장(CORS 응답 헤더)을 줘야, 문지기(브라우저)가 통과시켜 줌.

- **동일 출처(Same-Origin)**: **프로토콜 + 호스트 + 포트**가 모두 같아야 동일 출처. 하나라도 다르면 교차 출처(cross-origin).
- **CORS(Cross-Origin Resource Sharing)**: 브라우저가 강제하는 보안 정책. 서버가 막는 게 아니라 **브라우저가 응답을 JS에 넘길지 말지**를 CORS 헤더(`Access-Control-Allow-Origin` 등) 기준으로 결정.
- **Preflight(사전 요청)**: 상태를 바꿀 수 있는 "위험할 수 있는" 요청(PUT/DELETE, 커스텀 헤더, `application/json` 바디 등)은 실제 요청 전에 브라우저가 먼저 `OPTIONS` 메서드로 "이 요청 보내도 돼?"라고 물어봄 → 서버가 허용 응답(`Access-Control-Allow-Methods` 등)해야 실제 요청 진행.

```
브라우저(다른 출처 요청)                     서버
   │ ── OPTIONS (Preflight) ──────────────►│
   │ ◄─ Access-Control-Allow-Origin: ... ──│  "이 출처/메서드/헤더 허용"
   │ ── 실제 GET/POST/... ─────────────────►│
   │ ◄──────── 응답 ────────────────────────│
```

- 🔴 "CORS 에러는 서버가 요청을 거부한 것" — ❌ 서버는 보통 정상 응답을 보냈지만, **브라우저가** 응답 헤더에 허가가 없어 JS에서 그 응답을 못 읽게 막은 것. (서버 대 서버, Postman/curl 요청은 CORS의 영향을 받지 않음 — CORS는 브라우저 전용 정책.)
- 🔎 꼬리 질문: *"모든 교차 출처 요청이 Preflight를 거치나요?"* → 아니요. GET/HEAD/POST + 단순 헤더(Content-Type이 `text/plain` 등)로 이뤄진 "단순 요청(simple request)"은 Preflight 없이 바로 나가고, 서버 응답의 CORS 헤더로만 브라우저가 허용 여부를 판단합니다.

## 9. REST API 설계 원칙 + REST vs RPC
비유: REST는 "명사(자원) 중심 주소 체계 + 표준 동사(HTTP 메서드)"로 말하는 방식(예: "1번 유저를 조회해줘" → `GET /users/1`), RPC는 "이 함수를 실행해줘"라고 직접 지시하는 방식(예: `getUser(1)`).

- **자원(Resource) 중심**: URI는 **명사**로(`/users/1`), 행위는 URI가 아니라 **메서드**로 표현.
  - 좋은 예: `GET /users/1`, `POST /users`, `DELETE /users/1`
  - 나쁜 예: `GET /getUser?id=1`, `POST /deleteUser` — 동사가 URI에 들어감(RPC 스타일 잔재).
- **메서드로 행위 표현**: GET(조회)/POST(생성)/PUT(전체수정)/PATCH(부분수정)/DELETE(삭제) — N3-2 표 참고.
- **상태 코드로 결과 표현**: 성공/실패를 응답 바디 메시지가 아니라 **HTTP 상태 코드**로도 일관되게 표현(200/201/400/404 등).
- **무상태(Stateless)**: 서버가 클라이언트 상태를 기억하지 않음(N3-5) — 서버 확장에 유리.
- (참고) **HATEOAS**(응답에 다음 가능한 액션 링크 포함)까지가 REST 원 정의지만, 실무에서는 자원·메서드·상태코드 원칙만 지키고 HATEOAS는 생략하는 경우가 많음("REST-like").

| 구분 | REST | RPC(gRPC 등) |
|------|------|---------------|
| 스타일 | 자원(명사) + HTTP 메서드 | 함수/절차 호출 그 자체(동사) |
| 프로토콜 | 보통 HTTP/JSON | HTTP/2 + Protobuf(gRPC) 등 |
| 성능 | 텍스트(JSON) 기반, 상대적으로 큼 | 바이너리(Protobuf) 기반, 빠르고 작음 |
| 브라우저 친화성 | 높음(표준 HTTP) | 낮음(gRPC는 브라우저 직접 호출 제약) |
| 주 용도 | 공개 API, 웹-백엔드 통신 | 내부 마이크로서비스 간 고성능 통신 |

- 🔴 "REST가 항상 정답" — ❌ 외부에 공개하는 API·웹 프론트-백엔드 통신엔 REST(범용성)가 좋지만, 내부 서비스 간 고성능이 중요하면 gRPC 같은 RPC 방식을 더 선호하기도 합니다.
- 🔎 꼬리 질문: *"URI에 동사를 쓰면 왜 안 좋은가요?"* → REST는 자원(명사)과 행위(메서드)를 분리하는 게 원칙인데, `/deleteUser`처럼 URI에 동사가 들어가면 메서드와 URI에 행위 정보가 중복·충돌될 수 있고, 자원 기반 캐싱·라우팅의 일관성이 깨집니다.

## 10. WebSocket vs HTTP Polling (실시간 통신)
비유: **Polling**은 "소식 있어?"를 몇 초마다 계속 전화해서 물어보는 것, **Long Polling**은 전화해서 소식 생길 때까지 끊지 않고 기다리는 것, **WebSocket**은 아예 전화선을 열어두고 소식이 생기는 즉시 양쪽 다 자유롭게 말할 수 있는 것.

| 방식 | 동작 | 장점 | 단점 |
|------|------|------|------|
| Polling | 클라가 주기적으로(예: 3초마다) 요청 반복 | 구현 단순 | 지연(주기만큼) + 불필요한 요청으로 서버 부하 |
| Long Polling | 서버가 이벤트 생길 때까지 응답을 들고 있다가 응답, 클라는 받자마자 재요청 | Polling보다 지연 적음 | 연결 유지 비용, 재연결 반복 |
| **WebSocket** | 최초 HTTP Upgrade 핸드셰이크 후 **하나의 TCP 연결**에서 서버-클라 양방향(풀 듀플렉스) 실시간 통신 | 지연 최소, 매 메시지마다 HTTP 헤더 오버헤드 없음 | 연결 상태 유지 필요(서버 자원), 로드밸런서/프록시 설정 신경 써야 함 |

```
Polling:      클라 ──GET──►서버   (반복, 매번 새 요청/응답)
Long Polling: 클라 ──GET──►서버 (이벤트 생길 때까지 대기) ──응답──►클라 (즉시 재요청)
WebSocket:    클라 ──HTTP Upgrade──►서버 ──101 Switching Protocols──►
              이후 같은 TCP 연결 위에서 양방향 자유롭게 메시지 송수신
```

- 🟡 WebSocket도 시작은 HTTP 요청(`Upgrade: websocket` 헤더)이며, 서버가 `101 Switching Protocols`로 응답하면 그 뒤부턴 HTTP가 아닌 WebSocket 프로토콜로 전환됩니다.
- 🔎 꼬리 질문: *"실시간 채팅엔 무조건 WebSocket을 써야 하나요?"* → 아닙니다. 알림처럼 빈도가 낮고 약간의 지연이 허용되면 Long Polling만으로 충분할 수 있고, 양방향·고빈도·저지연(채팅, 실시간 협업, 게임)이 중요할 때 WebSocket 도입 비용(연결 관리, 스케일아웃 시 세션 고정 등)을 감수할 가치가 있습니다.

## 11. 핵심 포인트 (자주 하는 실수)
- 🔴 "POST는 멱등" — ❌ POST는 **비멱등**(호출마다 리소스 생성). 재시도 시 중복 주의.
- 🟡 **HOL(Head-of-Line) 블로킹**: 앞 요청이 막으면 뒤가 대기. HTTP/2가 HTTP레벨, HTTP/3(QUIC)가 TCP레벨까지 해결.
- 🟡 캐시 검증: `304 Not Modified` + `ETag`/`If-None-Match`로 재다운로드 회피.

## 12. 예상 면접 질문
**Q1. "HTTP가 stateless라는 게 무슨 의미인가요?"**
> 서버가 요청 간 상태를 기억하지 않는다는 것. 덕분에 서버 확장(로드밸런싱)이 쉽지만, 로그인 상태 유지는 쿠키/세션이나 토큰 같은 별도 수단이 필요합니다.

**Q2. "HTTP/1.1과 2, 3의 차이?"**
> 1.1은 연결 재사용은 되나 HOL 블로킹이 있고, 2는 한 연결에서 멀티플렉싱으로 HTTP레벨 HOL을 해결, 3은 UDP 기반 QUIC로 TCP레벨 HOL까지 없애고 연결 수립을 빠르게 했습니다.

**Q3. "GET과 POST 차이?"**
> GET은 조회용·멱등·캐시 가능·바디 없음이 원칙, POST는 생성/처리용·비멱등. 재시도 안전성이 다르므로 결제 같은 처리는 POST 중복에 주의합니다.

**Q4. "쿠키, 세션, JWT의 차이와 각각의 트레이드오프는?"**
> 쿠키는 클라이언트에 값을 그대로 저장, 세션은 서버가 실제 데이터를 들고 클라엔 ID만 주는 방식이라 로그아웃 즉시 무효화는 쉽지만 서버 확장 시 세션 공유가 필요합니다. JWT는 서버가 상태를 안 들고 있어 무상태·수평 확장에 유리하지만, 발급된 토큰을 즉시 무효화하기 어려워 짧은 만료 + Refresh Token 전략을 함께 씁니다.

**Q5. "브라우저에서 CORS 에러가 나면 어떻게 해결하나요?"**
> 서버가 거부한 게 아니라 브라우저가 응답의 CORS 헤더(`Access-Control-Allow-Origin` 등)가 없어 JS에 응답을 넘기지 않은 것이므로, 서버 쪽에서 허용 출처·메서드·헤더를 CORS 응답 헤더로 명시해야 합니다. 상태를 바꾸는 요청은 브라우저가 먼저 Preflight(OPTIONS)로 확인한다는 점도 함께 설명합니다.

**Q6. "REST API를 설계할 때 지키는 원칙은?"**
> URI는 자원을 나타내는 명사로 설계하고 행위는 HTTP 메서드로 표현하며, 결과는 상태 코드로 일관되게 나타내고 서버는 무상태로 유지합니다. 내부 고성능 서비스 간 통신처럼 REST보다 RPC(gRPC)가 더 적합한 상황도 있다는 점을 함께 언급합니다.

# N4. HTTPS and TLS

**학습 목표**: *"HTTPS는 어떻게 안전한가요? 대칭/비대칭 키를 왜 둘 다?"* 에 답할 수 있다.

## 1. 비유 — 자물쇠 상자 교환
비대칭키 = **공개 자물쇠**(누구나 잠글 수 있지만 열쇠는 주인만). 먼저 이 공개 자물쇠로 "진짜 대화용 비밀(대칭키)"을 안전하게 전달한 뒤, 이후 대화는 빠른 대칭키로. **비대칭으로 키 교환, 대칭으로 실제 통신**.

## 2. 왜 둘을 섞나
| | 비대칭(RSA/ECC) | 대칭(AES) |
|--|----------------|-----------|
| 키 | 공개키/개인키 쌍 | 하나의 공유키 |
| 속도 | 느림 | 빠름 |
| 역할 | **키 교환·인증** | **실제 데이터 암호화** |
> 비대칭은 느려서 데이터 전체엔 부적합 → **핸드셰이크에서 대칭키만 안전 교환**하고 본 통신은 대칭키.

## 3. TLS 핸드셰이크 (개념)
```
Client                              Server
  │ ── ClientHello (지원 암호군) ───►│
  │ ◄── ServerHello + 인증서(공개키) │  인증서로 서버 신원 확인(CA 서명)
  │ ── (공개키로 pre-master 전달) ──►│  → 양쪽이 같은 세션키(대칭) 생성
  │ ◄──── Finished (암호화 시작) ────│
  │      [이후 대칭키로 암호화 통신]  │
```
- **인증서(CA 서명)**: 서버가 진짜인지 신뢰된 기관(CA)이 보증 → 중간자 공격 방지.

## 4. 핵심 포인트
- 🔴 "HTTPS는 비대칭키로 통신" — ❌ 비대칭은 **키 교환·인증**에만, 실제 데이터는 **대칭키**.
- 🟡 HTTPS가 막는 것: 도청(암호화), 위변조(무결성), 신원 위장(인증서). = 기밀성+무결성+인증.
- 🟡 TLS는 OSI 표현/세션 계층 근처 — HTTP 아래, TCP 위에서 동작(HTTPS = HTTP over TLS).

## 5. 예상 면접 질문
**Q. "HTTPS에서 대칭키와 비대칭키를 왜 둘 다 쓰나요?"**
> 비대칭키는 안전하지만 느려서 실제 데이터 암호화엔 부적합합니다. 그래서 핸드셰이크 때 비대칭키로 서버 인증과 대칭키(세션키) 교환만 하고, 이후 본 통신은 빠른 대칭키로 암호화합니다. 속도와 안전성을 모두 취한 방식입니다.

> 🔎 꼬리 질문: *"인증서(CA 서명)가 없으면 어떤 문제가 생기나요?"* → 클라이언트가 서버의 진짜 신원을 확인할 방법이 없어, 중간자(MITM)가 자기 공개키를 서버인 척 건네줘도 알아챌 수 없습니다. CA가 서명한 인증서 덕분에 브라우저가 "이 공개키는 신뢰된 기관이 보증한 진짜 서버 것"임을 검증할 수 있습니다.

# N5. DNS and the Web Request Journey

**학습 목표**: *"주소창에 google.com을 치면 무슨 일이 일어나나요?"* — 이 유명한 질문에 계층을 관통하며 답할 수 있다.

## 1. 비유 — 전화번호부
사람은 이름(도메인 `google.com`)을 기억, 통신은 번호(IP `142.250...`)가 필요. **DNS = 이름 → IP 전화번호부**.

## 2. DNS 조회 흐름
```
브라우저 캐시 → OS/hosts 캐시 → Local DNS(리졸버)
   → Root 서버(.) → TLD 서버(.com) → Authoritative 서버(google.com)
   → IP 응답 (TTL 동안 캐시)
```

## 3. 주소창 입력 후 전 과정 ⭐ (계층 총정리)
```
1. URL 파싱 (google.com)
2. DNS 조회 → IP 획득                         (N5, Application)
3. TCP 3-way handshake로 서버와 연결            (N2, Transport)
4. HTTPS면 TLS 핸드셰이크 (대칭키 교환)          (N4)
5. HTTP 요청 전송 (GET / …)                     (N3, Application)
   └ 도중에 로드밸런서가 서버로 분산            (↔ System Design)
6. 서버 응답 (HTML/JSON) + 상태코드
7. 브라우저 렌더링 (HTML 파싱 → DOM → 표시)
8. 연결 종료 (4-way) 또는 keep-alive 재사용
```
> 이 한 흐름에 N1~N4가 다 나옴. 면접에서 이걸로 "계층을 실제로 이해했다"를 보여줄 수 있다.

## 4. 핵심 포인트
- 🟡 DNS는 주로 **UDP**(빠름, 작은 응답). 응답 크거나 재전송 필요 시 TCP.
- 🟡 **TTL**: DNS 응답 캐시 유효시간. 짧으면 변경 빠르게 반영, 길면 조회 부하↓.
- 🟡 캐시 계층이 많아(브라우저→OS→리졸버) 대부분 Root까지 안 감.

## 5. 예상 면접 질문
**Q. "브라우저에 URL을 입력하면 어떤 일이 일어나나요?"**
> ① URL 파싱 → ② DNS로 도메인→IP 조회 → ③ TCP 3-way handshake 연결 → ④ (HTTPS면) TLS 핸드셰이크로 대칭키 교환 → ⑤ HTTP 요청(로드밸런서가 서버 분산) → ⑥ 서버 응답 → ⑦ 브라우저 렌더링 → ⑧ 연결 종료/재사용. DNS(응용)·TCP(전송)·IP(네트워크)가 모두 관여합니다.

> 🔎 꼬리 질문: *"이미 캐시된 페이지를 재방문하면 이 과정 중 뭐가 생략되나요?"* → DNS는 브라우저/OS 캐시 TTL 내면 조회 생략, HTTP 응답은 강한 캐시(Cache-Control/Expires)가 살아있으면 서버 요청 자체가 생략되고, keep-alive 연결이 살아있으면 TCP/TLS 핸드셰이크도 재사용되어 건너뜁니다.

# N6. TCP 핸드셰이크 (3-way · 4-way)

**학습 목표**: *"연결 수립·종료 시 상태가 어떻게 전이되고, TIME_WAIT는 왜 필요한가요?"* 에 소켓 상태를 짚으며 답할 수 있다.

## 1. 비유 — 전화 통화 시작과 끝
전화를 걸 때 "여보세요?(SYN)" → "네 들려요, 그쪽은요?(SYN+ACK)" → "저도 잘 들려요(ACK)" 세 마디로 **양쪽이 서로 말이 통하는지** 확인하고 대화를 시작. 끊을 때는 "저 이만 끊을게요(FIN)" → "네(ACK)" → (상대가 할 말 마저 하고) "저도 끊어요(FIN)" → "네(ACK)"로, 한쪽이 끊자고 해도 **상대는 하던 말을 마칠 수 있어** 네 마디가 된다.

## 2. 개념 정의 — 왜 3번, 왜 4번
- **3-way(수립)**: 각자 자신의 **초기 시퀀스 번호(ISN)** 를 상대에게 알리고, 상대가 그걸 받았다는 ACK를 확인해야 한다. "내 송신을 상대가 받는가 + 상대 송신을 내가 받는가"를 양방향 모두 확인하려면 최소 3번. 2번이면 서버가 클라의 수신 능력을 확인하지 못한다.
- **4-way(종료)**: TCP는 **전이중(full-duplex)** 이라 각 방향 스트림을 독립적으로 닫는다. 한쪽이 FIN을 보내도 반대 방향은 아직 열려 있어(**half-close**) 남은 데이터를 마저 보낼 수 있다. 그래서 상대의 ACK와 상대의 FIN이 **분리**되어 총 4번. (보낼 데이터가 없으면 ACK+FIN이 합쳐져 3번으로 줄기도 함.)

## 3. 다이어그램 — 상태 전이 (state machine)
```
[수립]  Client                         Server
        CLOSED                          LISTEN
          │ ── SYN (seq=x) ───────────►│
        SYN_SENT                        SYN_RCVD
          │ ◄── SYN+ACK(seq=y,ack=x+1)─│
          │ ── ACK (ack=y+1) ─────────►│
        ESTABLISHED ◄──────────────► ESTABLISHED

[종료]  (active close)                 (passive close)
        ESTABLISHED                     ESTABLISHED
          │ ── FIN ───────────────────►│
        FIN_WAIT_1                      CLOSE_WAIT   ← 앱이 close() 호출 대기
          │ ◄── ACK ───────────────────│
        FIN_WAIT_2                      │ (남은 데이터 전송)
          │ ◄── FIN ───────────────────│
        TIME_WAIT ── ACK ─────────────►│ LAST_ACK → CLOSED
          │ (2×MSL 대기 후) CLOSED       │
```

## 4. 비교표 — 핵심 상태
| 상태 | 어느 쪽 | 의미 |
|------|---------|------|
| `SYN_SENT` / `SYN_RCVD` | 클라 / 서버 | 핸드셰이크 진행 중 |
| `ESTABLISHED` | 양쪽 | 연결 수립 완료(정상 통신) |
| `CLOSE_WAIT` | 수동 종료측 | 상대 FIN 받고 ACK는 보냄, **아직 내 close() 안 함** |
| `FIN_WAIT_2` | 능동 종료측 | 내 FIN에 ACK 받음, 상대 FIN 기다리는 중 |
| `TIME_WAIT` | 능동 종료측 | 마지막 ACK 후 **2×MSL** 대기 |

## 5. 핵심 포인트 (자주 하는 실수)
- 🟡 **TIME_WAIT는 능동 종료(먼저 FIN 보낸 쪽)에만** 생긴다. 이유 2가지: ① 마지막 ACK가 유실되면 상대가 FIN을 재전송하는데, 이를 받아 다시 ACK해 주려면 잠시 살아 있어야 함. ② 지연 도착한 이전 연결의 패킷이 **같은 4-tuple의 새 연결에 섞여드는 것**을 방지. 대기 시간은 `2×MSL`(패킷 최대 생존시간의 왕복분).
- 🔴 **CLOSE_WAIT가 쌓이면 = 애플리케이션 버그.** 상대가 FIN을 보냈는데 우리 앱이 소켓을 `close()` 하지 않아 CLOSE_WAIT에 눌러앉은 상태(커넥션 풀·스트림 누수). OS가 아니라 코드가 원인이라 재시작해도 재발한다.
- 🟡 **SYN 플러딩**: 3-way 도중 서버의 SYN_RCVD 백로그 큐를 SYN만 계속 보내 채우는 DoS → **SYN cookie** 로 방어(백로그에 상태를 안 들고도 검증).
- 🟡 자세한 흐름/혼잡 제어는 N7, HTTP 레벨 연결 재사용은 N8 참고. (N2는 개요, N6은 상태 전이 심화.)

## 6. 예상 면접 질문

**Q1. "3-way handshake가 왜 2-way면 안 되나요?"**
> 연결은 양방향이라 "내 데이터를 상대가 받는가"와 "상대 데이터를 내가 받는가"를 둘 다 확인해야 합니다. SYN→SYN+ACK까지 2번이면 서버는 클라의 수신 능력을 아직 검증받지 못했고, 클라의 마지막 ACK가 있어야 서버도 자신의 SYN이 도달했음을 압니다. 또 양쪽이 서로의 초기 시퀀스 번호(ISN)를 교환·확인해야 이후 순서·중복을 판별할 수 있어 최소 3번이 필요합니다.

**Q2. "종료는 왜 4-way인가요? 항상 4번인가요?"**
> TCP는 전이중이라 두 방향 스트림을 독립적으로 닫습니다. 한쪽이 FIN을 보내도 반대 방향은 열려 있어(half-close) 남은 데이터를 마저 보낼 수 있고, 그래서 상대의 ACK와 FIN이 분리되어 4번이 됩니다. 다만 수동 종료측에 보낼 데이터가 없으면 ACK와 FIN이 한 세그먼트로 합쳐져 3번으로 줄 수도 있습니다.

**Q3. (시나리오) "서버에 TIME_WAIT 소켓이 수만 개 쌓였습니다. 무슨 문제이고 어떻게 대응하나요?"**
> TIME_WAIT은 능동 종료측에 생기므로, 서버에 잔뜩 쌓였다면 대개 **서버가 매 요청마다 커넥션을 먼저 끊고 있다**는 뜻입니다(예: keep-alive 미사용, 짧은 커넥션 반복). 각 소켓이 로컬 포트·메모리를 2×MSL 동안 점유해 **로컬 포트 고갈(ephemeral port exhaustion)** 이나 연결 실패로 이어질 수 있습니다. 근본 대응은 커넥션 재사용(keep-alive, 커넥션 풀)으로 종료 빈도 자체를 줄이는 것이고, 굳이 능동 종료가 필요하면 **클라이언트/프록시 쪽이 먼저 끊게** 설계를 바꿉니다. 커널 튜닝(`net.ipv4.tcp_tw_reuse`로 아웃바운드 재사용 허용, `ip_local_port_range` 확대)은 보조 수단이며, 예전에 쓰던 `tcp_tw_recycle`은 NAT 환경에서 오작동해 지금은 제거되었습니다.

**Q4. (시나리오) "`netstat`에 CLOSE_WAIT이 계속 늘어납니다. 원인은?"**
> CLOSE_WAIT은 상대가 FIN을 보냈는데 우리 쪽 애플리케이션이 아직 소켓을 `close()` 하지 않은 상태입니다. 즉 커널 문제가 아니라 **우리 코드가 연결(또는 스트림·커넥션 풀 객체)을 닫지 않는 리소스 누수 버그**입니다. HTTP 클라이언트·JDBC·소켓 스트림을 try-with-resources 등으로 확실히 닫는지, 예외 경로에서 close가 누락됐는지를 봐야 합니다. TIME_WAIT과 달리 시간이 지나도 저절로 사라지지 않는다는 점이 결정적 단서입니다.

# N7. 흐름 제어와 혼잡 제어

**학습 목표**: *"슬라이딩 윈도우, 느린 시작, AIMD를 그림으로 설명하고, 둘의 판단 근거 차이를 말할 수 있나요?"* 에 답할 수 있다. (N2-6 심화와 짝을 이룸)

## 1. 비유 — 수도관과 물탱크
- **흐름 제어** = 받는 쪽 **물탱크(수신 버퍼)** 가 넘치지 않게. 탱크 주인이 "지금 이만큼 더 받을 수 있어(rwnd)"라고 계속 알려준다 → 상대방 사정.
- **혼잡 제어** = 중간 **수도관(네트워크 경로)** 이 터지지 않게. 관이 막히는지는 아무도 안 알려주니 물을 조금씩 늘려보다 새면(손실) 줄인다 → 경로 사정을 **추정**.

## 2. 개념 정의
- **흐름 제어(Flow Control)**: 송신 속도를 **수신자 버퍼 여유(rwnd)** 에 맞춤. 수신자가 ACK마다 rwnd를 실어 통지, `rwnd=0`이면 송신 중단 후 Window Update로 재개.
- **혼잡 제어(Congestion Control)**: 송신 속도를 **네트워크 처리 능력(cwnd)** 에 맞춤. 아무도 안 알려주므로 손실·타임아웃으로 추정. 실제 전송량 = `min(rwnd, cwnd)`.
- **슬라이딩 윈도우(Sliding Window)**: ACK를 매번 기다리지 않고 윈도우 크기만큼 **미리 연속 전송**하는 방식. ACK가 오면 윈도우가 오른쪽으로 미끄러지며 새 세그먼트를 내보낸다. stop-and-wait 대비 링크를 꽉 채워 쓴다(대역폭×지연 곱만큼).

## 3. 다이어그램 — cwnd 톱니와 단계
```
cwnd
 │            손실(timeout)→cwnd=1로 급락
 │   Slow Start │        Congestion Avoidance(선형 +1 MSS/RTT)
 │   (지수 ×2)   │        ╱│      3dupACK→절반(Fast Recovery)
 │      ╱│       ▼      ╱  │     ╱│
 │     ╱ │ ssthresh ─ ╱    ▼   ╱  ▼
 │    ╱  └──────────╱      ╱  ╱
 │   ╱             ╱  ×2? 아니오 선형↗
 └──╱────────────────────────────────► time (RTT)
    ↑ cwnd=1 MSS에서 시작
```

## 4. 비교표
| 구분 | 흐름 제어 | 혼잡 제어 |
|------|-----------|-----------|
| 막는 대상 | 수신자 버퍼 오버플로 | 네트워크(경로) 혼잡 |
| 기준 변수 | rwnd (수신 윈도우) | cwnd (혼잡 윈도우) |
| 판단 근거 | 수신자가 **명시적으로 통지** | 손실·타임아웃으로 **추정** |
| 증가 | 상대가 알려준 만큼 즉시 | Slow Start(지수) → AIMD(선형) |
| 감소 | rwnd=0이면 중단 | timeout→cwnd=1 / 3dup ACK→절반 |

**혼잡 제어 단계**: ① **Slow Start** — cwnd 1 MSS에서 RTT마다 2배(지수). ② `ssthresh` 도달 시 **Congestion Avoidance(AIMD)** — RTT마다 +1 MSS(선형). ③ **손실 시** — timeout이면 cwnd=1로 리셋+slow start 재시작(강한 처벌), 3중복 ACK면 cwnd 절반+congestion avoidance 재개(Fast Retransmit/Recovery, 가벼운 처벌).

## 5. 핵심 포인트 (자주 하는 실수)
- 🔴 "혼잡 제어는 수신자가 rwnd로 알려준다" — ❌ rwnd는 **흐름 제어**용. 혼잡 제어(cwnd)는 통지받는 게 아니라 **송신자가 손실 신호로 스스로 추정**한다.
- 🟡 timeout과 3중복 ACK의 처벌이 다른 이유: timeout은 패킷이 완전히 사라진 심각한 신호(1로 리셋), 3중복 ACK는 뒤 패킷이 도착 중이라 **네트워크가 살아있다는 신호**(절반만).
- 🟡 "Slow Start인데 왜 지수 증가?" — 이름은 **시작 값이 작다**는 뜻이지 증가 속도가 느리다는 뜻이 아니다. 증가 자체는 RTT마다 2배로 빠르다.
- 🟡 **BDP(대역폭×지연)** 가 크면(장거리·고속 회선) 윈도우가 작을수록 링크를 못 채운다. Window Scaling 옵션으로 64KB 한계를 넘긴다.

## 6. 예상 면접 질문

**Q1. "흐름 제어와 혼잡 제어의 판단 근거가 어떻게 다른가요?"**
> 흐름 제어는 수신자가 ACK에 실어 보내는 rwnd로 "내가 이만큼 더 받을 수 있다"를 명시적으로 통지받아 조절합니다. 혼잡 제어는 네트워크 경로 상태를 아무도 알려주지 않으므로 송신자가 손실·타임아웃을 관찰해 cwnd를 스스로 추정·조절합니다. 실제 전송량은 둘 중 작은 값 min(rwnd, cwnd)입니다.

**Q2. "느린 시작(Slow Start)과 AIMD를 설명해주세요."**
> 연결 초기엔 경로 용량을 모르니 cwnd를 1 MSS에서 시작해 RTT마다 2배로 늘리는 게 Slow Start입니다. ssthresh에 도달하면 조심스럽게 RTT마다 +1 MSS씩 선형 증가(Additive Increase)하다가, 손실이 나면 확 줄입니다(Multiplicative Decrease). 이 선형 증가·곱셈 감소의 반복이 AIMD이고, 그래서 cwnd 그래프가 톱니 모양이 됩니다.

**Q3. (트레이드오프) "왜 손실 종류에 따라 cwnd를 다르게 줄이나요?"**
> 타임아웃은 ACK가 아예 안 온 것이라 경로가 심각하게 막혔거나 끊겼다는 신호이므로 cwnd를 1로 리셋해 크게 물러섭니다. 반면 3중복 ACK는 그 뒤 세그먼트들은 잘 도착하고 있다는 뜻이라 네트워크가 살아있다고 보고 절반만 줄여 빠르게 회복합니다(Fast Recovery). 회복 속도와 혼잡 회피 사이의 균형을 손실의 심각도로 판단하는 것입니다.

**Q4. (시나리오) "대역폭은 충분한데 대용량 전송 처리량이 안 나옵니다. 무엇을 의심하나요?"**
> 장거리 고RTT 링크에서 흔한 BDP(대역폭×지연) 대비 윈도우 부족을 의심합니다. 윈도우가 BDP보다 작으면 ACK를 기다리느라 링크를 다 못 채웁니다. TCP Window Scaling이 켜져 있는지, 수신 버퍼(rwnd) 상한이 작지 않은지, 초기 cwnd·혼잡 알고리즘(CUBIC/BBR 등)이 적절한지를 봅니다. 손실이 잦다면 혼잡 제어가 계속 cwnd를 깎아 처리량이 눌리므로 패킷 손실률도 함께 확인합니다.

# N8. HTTP 버전 발전 (1.1 · 2 · 3)

**학습 목표**: *"HTTP/2를 쓰는데도 왜 느릴 수 있고, HTTP/3는 무엇을 더 해결했나요?"* 에 HOL 블로킹을 계층별로 나눠 답할 수 있다. (N3-4 심화)

## 1. 비유 — 계산대와 줄
- **HTTP/1.1** = 계산대 하나에 줄 하나. 앞사람이 오래 걸리면 뒤가 다 기다린다(**HOL 블로킹**). 급하면 계산대(TCP 연결)를 여러 개 연다(브라우저 도메인당 6개).
- **HTTP/2** = 계산대 하나인데 **여러 주문을 잘게 쪼개 번갈아 처리**(멀티플렉싱). 애플리케이션 레벨에선 앞사람이 안 막는다.
- **HTTP/3** = 계산대 자체를 바꿈. 한 물건이 바닥에 떨어져도(패킷 손실) **그 주문만 멈추고 나머지 주문은 계속**(QUIC).

## 2. 개념 정의 — 세 가지 HOL 블로킹을 구분하라
- **HTTP/1.1 HOL**: 파이프라이닝을 해도 응답은 **요청 순서대로** 와야 해서 앞 응답이 느리면 뒤가 막힘 → 실무에선 파이프라이닝을 거의 안 쓰고 커넥션을 여러 개 연다.
- **HTTP/2가 해결한 것 (애플리케이션 레벨 HOL)**: 하나의 TCP 연결 안에서 요청/응답을 **스트림**으로 나누고 프레임을 인터리빙 → 논리적으로 병렬. 헤더는 HPACK으로 압축.
- **HTTP/2에 남은 것 (TCP 레벨 HOL)**: 여러 스트림이 **하나의 TCP 연결**을 공유하는데, TCP는 바이트 순서를 보장하므로 **한 패킷이 유실되면 그 뒤 모든 스트림의 데이터가 커널 버퍼에서 대기**한다. 애플리케이션은 병렬인데 전송 계층이 막는 것.
- **HTTP/3가 해결한 것**: **QUIC(UDP 기반)** 이 스트림 다중화를 전송 계층 안으로 내려, 스트림마다 독립적으로 순서를 관리 → 한 스트림의 손실이 다른 스트림을 막지 않음. TLS 1.3을 내장해 **연결+암호화 핸드셰이크를 1-RTT(재접속 시 0-RTT)** 로 단축하고, IP가 바뀌어도 Connection ID로 세션이 유지됨(**connection migration**).

## 3. 비교표
| 버전 | 전송 | 다중화 | 남은 HOL | 헤더 | 대표 특징 |
|------|------|--------|----------|------|-----------|
| 1.1 | TCP | ✕ (연결당 순차) | 앱+TCP 둘 다 | 평문 | keep-alive, 브라우저가 연결 6개로 회피 |
| 2 | TCP | ✓ 스트림 | **TCP 레벨 잔존** | HPACK 압축 | 멀티플렉싱, 서버 푸시(현재 비권장) |
| 3 | **QUIC/UDP** | ✓ (전송 계층 내장) | 없음 | QPACK | 1-RTT/0-RTT, connection migration |

## 4. 핵심 포인트 (자주 하는 실수)
- 🔴 "HTTP/2면 HOL 블로킹이 완전히 사라진다" — ❌ **애플리케이션 레벨만** 해결. 여러 스트림이 단일 TCP 연결을 공유하므로 패킷 손실 시 **TCP 레벨 HOL**이 남고, 손실률 높은 망(모바일)에선 HTTP/1.1(연결 여러 개)보다 오히려 느릴 수 있다. 이걸 없앤 게 HTTP/3.
- 🟡 HTTP/3는 TCP가 아니라 **UDP 위 QUIC**이라 방화벽/미들박스가 UDP 443을 막으면 동작 못 해 TCP(HTTP/2)로 폴백한다.
- 🟡 서버 푸시는 캐시와 잘 안 맞고 이득이 적어 주요 브라우저가 지원을 접었다 — "HTTP/2 장점=서버 푸시"라고만 답하면 낡은 답변.
- 🟡 HTTP/2 멀티플렉싱 덕에 도메인 샤딩·이미지 스프라이트 같은 HTTP/1.1 최적화 기법은 오히려 불필요·역효과가 된다.

## 5. 예상 면접 질문

**Q1. "HTTP/1.1, 2, 3의 차이를 HOL 블로킹 중심으로 설명해주세요."**
> 1.1은 연결 하나에서 응답이 순서대로 와야 해 애플리케이션 레벨 HOL이 있고, 브라우저는 연결을 여러 개 열어 우회했습니다. 2는 한 연결 안에서 요청을 스트림으로 멀티플렉싱해 애플리케이션 레벨 HOL을 없앴지만, 스트림들이 단일 TCP를 공유해 패킷 손실 시 TCP 레벨 HOL이 남습니다. 3은 UDP 기반 QUIC로 스트림 다중화를 전송 계층에 내려 스트림별 독립 전송을 하므로 TCP 레벨 HOL까지 제거하고, 핸드셰이크도 1-RTT로 줄였습니다.

**Q2. (시나리오·"왜") "HTTP/2로 올렸는데 특정 사용자군에서 여전히 느립니다. 왜 그럴까요?"**
> HTTP/2는 애플리케이션 레벨 HOL만 해결하고 TCP 레벨 HOL은 남습니다. 손실률이 높은 모바일·불안정 망에서는 여러 스트림이 하나의 TCP 연결을 공유하기 때문에 패킷 하나만 유실돼도 그 뒤 모든 스트림이 커널에서 대기해, 연결을 여러 개 벌려 쓰던 HTTP/1.1보다 체감이 나쁠 수 있습니다. 이 경우 HTTP/3(QUIC) 적용을 검토하거나, 손실·RTT 지표를 먼저 측정해 병목이 전송 계층인지 서버 처리인지 구분합니다.

**Q3. (트레이드오프) "그럼 무조건 HTTP/3가 정답인가요?"**
> 아닙니다. QUIC은 UDP 443을 쓰는데 일부 기업 방화벽·미들박스가 이를 차단해 TCP로 폴백해야 하고, UDP 처리가 커널 최적화가 덜 돼 CPU 비용이 더 들 수 있습니다. 손실이 적은 안정적 유선망·내부 통신에서는 HTTP/2로 충분하고, 손실·이동성이 큰 공용 인터넷 클라이언트 트래픽에서 HTTP/3의 이득이 큽니다. 실제로는 HTTP/3를 제공하되 미지원 클라이언트는 HTTP/2로 자연스럽게 폴백하도록 함께 운영합니다.

**Q4. "HTTP/3가 연결 수립을 어떻게 빠르게 하나요?"**
> QUIC은 TLS 1.3을 프로토콜에 내장해 전송 연결과 암호화 협상을 합쳐 1-RTT에 끝냅니다. TCP+TLS라면 3-way 핸드셰이크 뒤 TLS 핸드셰이크가 별도로 붙어 최소 2~3 RTT가 드는 것과 대비됩니다. 한 번 붙었던 서버에는 세션 정보를 캐시해 0-RTT로 곧바로 요청을 실어 보낼 수도 있고, Connection ID로 식별하므로 Wi-Fi↔LTE 전환처럼 IP가 바뀌어도 연결을 새로 맺지 않습니다.

# 핵심 질문 (Quiz)

> 답변을 먼저 떠올린 뒤 펼쳐서 확인하세요.

<details>
<summary>Q1. OSI 계층에서 TCP와 IP는 각각 몇 계층이고, 왜 계층을 나누나요?</summary>

- IP: 3계층(네트워크) — 논리 주소·라우팅 담당
- TCP: 4계층(전송) — 포트 기반 종단 간 신뢰성 전달 담당
- 역할을 계층으로 분리 → 각 계층이 바로 위/아래하고만 인터페이스 → **계층 독립성**(한 계층 변경이 나머지에 영향 최소)
- 꼬리 질문 대비: OSI는 통신 이해·설계용 **참조 모델(이론)**, 실제 인터넷은 **TCP/IP 4계층 스택**으로 동작. Presentation·Session 역할은 실무에서 TLS·HTTP 같은 Application 계층 프로토콜이 흡수

</details>

<details>
<summary>Q2. TCP와 UDP의 차이는 무엇인가요?</summary>

| 구분 | TCP | UDP |
|------|-----|-----|
| 연결 | 연결형(handshake) | 비연결형 |
| 신뢰성 | 보장(ACK·재전송·순서) | 없음 |
| 속도/오버헤드 | 느림·큼 | 빠름·작음 |
| 흐름/혼잡 제어 | 있음 | 없음 |
| 용도 | HTTP, 파일, 메일 | 스트리밍, 게임, DNS, VoIP |

- 신뢰성이 필요하면 TCP, 실시간성이 중요하면 UDP (재전송 지연이 오히려 해로운 경우 UDP가 적합)

</details>

<details>
<summary>Q3. 3-way handshake와 4-way handshake를 설명해주세요. 2-way면 왜 안 되나요?</summary>

- **3-way (연결 수립)**: SYN(연결 요청, ISN 통지) → SYN+ACK(수락+역요청) → ACK(확인) — 양방향 송수신 능력과 초기 시퀀스 번호(ISN)를 상호 확인. 2-way면 서버가 클라의 수신 능력을 검증받지 못함
- **4-way (연결 종료)**: FIN → ACK → FIN → ACK — TCP는 전이중이라 방향별로 독립적으로 닫음. 한쪽이 FIN을 보내도 반대 방향은 열려 있어(half-close) 남은 데이터를 마저 보낼 수 있어 ACK와 FIN이 분리
- 종료 후 능동 종료측만 **TIME_WAIT**(2×MSL 대기) — 마지막 ACK 유실 대비 + 이전 연결 패킷이 새 연결에 섞이는 것 방지

</details>

<details>
<summary>Q4. 흐름 제어와 혼잡 제어의 차이는 무엇인가요?</summary>

| 구분 | 흐름 제어 | 혼잡 제어 |
|------|-----------|-----------|
| 목적 | 수신자 버퍼 오버플로 방지 | 네트워크(경로) 혼잡 방지 |
| 기준 변수 | rwnd (수신 윈도우) | cwnd (혼잡 윈도우) |
| 판단 근거 | 상대방이 알려주는 버퍼 여유 | 손실·타임아웃으로 추정 |
| 증가 방식 | 상대가 알려준 만큼 즉시 반영 | slow start(지수) → AIMD(선형) |
| 감소 트리거 | rwnd=0 | timeout(급감, cwnd→1) / 3중복ACK(반으로) |

- 실제 전송량은 `min(rwnd, cwnd)`
- 혼잡 제어는 수신자가 알려주는 게 아니라 **송신자가 손실 신호로 스스로 추정**

</details>

<details>
<summary>Q5. HTTP/1.1, HTTP/2, HTTP/3의 차이는 무엇인가요?</summary>

- **1.1**: keep-alive로 연결 재사용은 되지만 **HOL(Head-of-Line) 블로킹** 잔존
- **2**: 한 연결에서 **멀티플렉싱**(병렬 스트림) + 헤더 압축 + 서버 푸시로 HTTP 레벨 HOL 블로킹 해결
- **3**: **QUIC(UDP 기반)**로 TCP 레벨 HOL 블로킹까지 제거, 연결 수립도 빠름

</details>

<details>
<summary>Q6. 쿠키, 세션, JWT의 차이와 각각의 트레이드오프는?</summary>

| 구분 | 쿠키 | 세션 | JWT |
|------|------|------|-----|
| 저장 위치 | 클라이언트 | 서버(클라엔 세션 ID만) | 클라이언트(서버는 저장 안 함) |
| 서버 확장성 | 상태 없음 | 여러 서버 간 세션 공유 필요(Redis 등) | 무상태 → 수평 확장 용이 |
| 로그아웃(무효화) | 서버가 강제로 막기 어려움 | 서버에서 즉시 무효화 가능 | 만료 전까지 즉시 무효화 어려움 |

- JWT는 확장성은 좋지만 탈취 시 만료 전까지 막기 어려움 → **짧은 만료 + Refresh Token** 전략, 필요시 블랙리스트 병행

</details>

<details>
<summary>Q7. HTTPS에서 대칭키와 비대칭키를 왜 둘 다 쓰나요?</summary>

- 비대칭키(RSA/ECC): 안전하지만 **느림** → 키 교환·서버 인증(CA 서명 인증서로 신원 확인)에만 사용
- 대칭키(AES): **빠름** → 핸드셰이크로 교환된 세션키로 실제 데이터 암호화에 사용
- 즉 "비대칭으로 키 교환·인증, 대칭으로 실제 통신" — 속도와 안전성을 모두 취함
- 인증서(CA 서명)가 없으면 중간자(MITM)가 자기 공개키를 서버인 척 건네도 알아챌 수 없음

</details>

<details>
<summary>Q8. 브라우저에 URL을 입력하면 어떤 일이 일어나나요?</summary>

1. URL 파싱
2. **DNS** 조회로 도메인 → IP (Application, N5)
3. **TCP 3-way handshake**로 서버와 연결 (Transport, N2)
4. HTTPS면 **TLS 핸드셰이크**로 대칭키 교환 (N4)
5. **HTTP 요청** 전송(로드밸런서가 서버로 분산) (Application, N3)
6. 서버 응답(HTML/JSON) + 상태코드
7. 브라우저 렌더링(HTML 파싱 → DOM → 표시)
8. 연결 종료(4-way) 또는 keep-alive 재사용

- 재방문 시 생략되는 것: DNS(캐시 TTL 내면 생략), HTTP 요청 자체(강한 캐시 유효 시), TCP/TLS 핸드셰이크(keep-alive 연결 살아있으면 재사용)

</details>

<details>
<summary>Q9. (시나리오) 서버에 TIME_WAIT 소켓이 수만 개 쌓였습니다. 무슨 문제이고 어떻게 대응하나요?</summary>

- TIME_WAIT은 **능동 종료측**(먼저 FIN 보낸 쪽)에 생김 → 서버에 쌓였다면 **서버가 매 요청마다 커넥션을 먼저 끊고 있다**는 신호(keep-alive 미사용 등)
- 각 소켓이 로컬 포트·메모리를 2×MSL 점유 → **ephemeral 포트 고갈**, 신규 연결 실패
- 근본 대응: **커넥션 재사용(keep-alive·커넥션 풀)** 으로 종료 빈도 자체를 줄이고, 필요하면 능동 종료를 클라/프록시 쪽으로 이동
- 보조: `tcp_tw_reuse`(아웃바운드 재사용), `ip_local_port_range` 확대. (`tcp_tw_recycle`은 NAT 오작동으로 제거됨)
- 대비: **CLOSE_WAIT**이 쌓이면 이건 커널이 아니라 **앱이 소켓을 close() 안 하는 리소스 누수 버그** — 시간이 지나도 안 사라지는 게 단서

</details>

<details>
<summary>Q10. 느린 시작(Slow Start)과 AIMD를 설명하고, 손실 종류별로 cwnd를 왜 다르게 줄이나요?</summary>

- **Slow Start**: cwnd 1 MSS에서 시작해 RTT마다 ×2(지수). 이름은 "시작 값이 작다"는 뜻이지 증가가 느린 게 아님
- **AIMD (Congestion Avoidance)**: ssthresh 이후 RTT마다 +1 MSS(선형 증가), 손실 시 곱셈 감소 → cwnd 톱니 그래프
- **Timeout**(패킷 완전 유실, 심각) → cwnd=1로 리셋+slow start 재시작(강한 처벌)
- **3중복 ACK**(뒤 패킷은 도착 중 = 네트워크 살아있음) → cwnd 절반+congestion avoidance 재개(Fast Recovery, 가벼운 처벌)
- 흐름 제어(rwnd, 수신자 통지)와 달리 혼잡 제어(cwnd)는 **송신자가 손실로 추정**, 전송량 = min(rwnd, cwnd)

</details>

<details>
<summary>Q11. (시나리오·"왜") HTTP/2를 쓰는데 특정 사용자군에서 여전히 느립니다. 왜일까요?</summary>

- HTTP/2는 **애플리케이션 레벨 HOL만** 해결(스트림 멀티플렉싱)
- 여러 스트림이 **단일 TCP 연결 공유** → 패킷 하나만 유실돼도 TCP 순서 보장 때문에 뒤 모든 스트림이 커널에서 대기 = **TCP 레벨 HOL 잔존**
- 손실률 높은 모바일·불안정 망에선 연결 여러 개 벌리던 **HTTP/1.1보다 오히려 느릴 수 있음**
- 대응: HTTP/3(QUIC) 검토 — 스트림 다중화를 전송 계층에 내려 스트림별 독립 전송으로 TCP 레벨 HOL 제거. 단, 병목이 전송 계층인지 서버 처리인지 손실·RTT부터 측정

</details>

<details>
<summary>Q12. HTTP/3(QUIC)는 무엇을 더 해결했고, 무조건 정답인가요?</summary>

- **UDP 기반 QUIC**: 스트림 다중화를 전송 계층에 내장 → 스트림별 독립 순서 관리로 **TCP 레벨 HOL 제거**
- **TLS 1.3 내장** → 연결+암호화 핸드셰이크를 **1-RTT**(재접속 시 0-RTT). TCP+TLS의 2~3 RTT 대비 빠름
- **Connection Migration**: Connection ID로 식별해 Wi-Fi↔LTE로 IP가 바뀌어도 연결 유지
- 트레이드오프: UDP 443을 막는 방화벽·미들박스에선 **TCP로 폴백** 필요, UDP 처리 CPU 비용↑. 안정적 유선·내부망은 HTTP/2로 충분, 손실·이동성 큰 공용 트래픽에서 HTTP/3 이득
- 낡은 답 주의: "HTTP/2 = 서버 푸시"는 이제 비권장(브라우저 지원 중단)

</details>
