# V0. 헷갈리지 말 것 — RDB vs 그래프 DB vs 벡터 DB

**학습 목표**: *"Neo4j랑 벡터 DB 차이가 뭐예요?"* 에 5분간 답할 수 있다.

## 1. 비유 — 무엇으로 "찾느냐"
- **RDB**: 엑셀 표. 조건이 정확히 맞는 행을 찾음 (`WHERE id = 5`).
- **그래프 DB**: 인맥 지도. **연결(관계)을 따라간다** — "친구의 친구의 친구".
- **벡터 DB**: 의미 좌표 지도. **"뜻이 가까운 것"** 을 거리로 찾는다.

## 2. 결정적 오해
> **Neo4j는 그래프 DB이지 벡터 DB가 아니다.** 둘은 푸는 문제가 다르다.
> (요즘 Neo4j·pgvector 등이 벡터 검색 기능을 곁들이긴 하지만, 본질은 별개다.)

| 구분 | RDB | 그래프 DB | 벡터 DB |
|------|-----|----------|---------|
| 저장 단위 | 행(row) | 노드+엣지 | 벡터(임베딩) |
| 핵심 질의 | 정확 일치·집계 | 관계 경로 탐색 | 의미 유사도 top-k |
| 검색 방식 | 인덱스(B-Tree) | 그래프 순회 | ANN(근사 최근접) |
| 대표 | PostgreSQL, MySQL | **Neo4j** | Pinecone, Weaviate, **pgvector** |
| 대표 용례 | 트랜잭션·CRUD | 추천·소셜·사기탐지 | 시맨틱 검색·RAG |

# V1. 임베딩(Embedding) — 뜻을 좌표로

## 1. 비유 — 지도 위의 점
서울·부산·도쿄를 위도/경도 2개 숫자로 지도에 찍으면 "서울-부산이 서울-도쿄보다 가깝다"를 거리로 안다.
임베딩은 **텍스트를 수백 차원(예: 768) 숫자 벡터로 찍는 것**. 이러면 "고양이"와 "강아지"가 "고양이"와 "주식"보다 가깝게 배치된다.

## 2. 개념
> **임베딩 모델**(예: `nomic-embed-text`, `bge-m3`, OpenAI `text-embedding-3`)이 텍스트를 받아 **고정 길이 실수 벡터**를 내놓는다. 의미가 비슷하면 벡터도 가깝다.

```
"TCP와 UDP 차이?"        → [0.021, -0.11, 0.33, ... ] (768개)
"전송 계층 프로토콜 비교"  → [0.019, -0.10, 0.31, ... ] ← 위와 가까움 (단어 안 겹쳐도!)
"React useEffect 정리"    → [0.44,  0.02, -0.20, ... ] ← 멀리 떨어짐
```

핵심: **단어 일치(keyword)** 가 아니라 **의미(semantic)** 로 가까워진다. 이게 전통 `LIKE '%..%'` 검색과의 근본 차이.

# V2. 유사도 — 코사인은 왜 거리 아닌 각도인가

두 벡터가 얼마나 비슷한지 = **코사인 유사도(cosine similarity)**:

```
cos(a, b) = (a · b) / (‖a‖ · ‖b‖)      // 내적 / (크기 곱)
          = 두 벡터가 이루는 "각도"의 코사인
값 범위: -1(정반대) ~ 0(무관) ~ 1(동일 방향)
```

**왜 유클리드 거리가 아니라 각도?**
- 텍스트 임베딩은 **길이(문서가 길고 짧음)** 보다 **방향(의미의 방향)** 이 중요하다.
- 긴 문서/짧은 문서라도 주제가 같으면 방향이 비슷 → 코사인이 높게 나온다.
- (많은 임베딩 모델은 벡터를 정규화해 내보내므로 코사인 ≈ 내적이 되어 계산도 싸다.)

```js
function cosine(a, b) {
  let dot = 0, na = 0, nb = 0
  for (let i = 0; i < a.length; i++) { dot += a[i]*b[i]; na += a[i]*a[i]; nb += b[i]*b[i] }
  return dot / (Math.sqrt(na) * Math.sqrt(nb))
}
```

# V3. 벡터 DB가 하는 일 (ANN)

카드가 400개면 "쿼리 벡터 vs 400개 전부 코사인 계산"(브루트포스)이 충분히 빠르다. 하지만 **수백만~수억 개**가 되면 전부 비교는 느리다.

> **벡터 DB의 존재 이유 = ANN(Approximate Nearest Neighbor)**: 정확한 최근접 대신 **거의 최근접을 훨씬 빠르게** 찾는 색인(HNSW, IVF 등)을 제공한다. 약간의 정확도를 내주고 속도를 얻는 트레이드오프.

즉 벡터 DB = **"임베딩 저장 + ANN 색인 + top-k 유사도 질의"** 를 묶어주는 것. 소규모면 사실 그냥 배열 + 코사인으로 충분하다(이 저장소 실습이 그 경우).

# V4. RAG — 회사 지식베이스의 정체

"회사가 자기 문서로 챗봇을 만든다"의 표준 패턴 = **RAG(Retrieval-Augmented Generation)**.

```
[사전 준비 · 오프라인]
 사내 문서 수천 개 → 청크(조각)로 자르기 → 각 청크 임베딩 → 벡터 DB에 저장

[질의 · 온라인]
 사용자 질문 ──임베딩──▶ 벡터 DB에서 의미상 가까운 청크 top-k 검색
                                   │
                                   ▼
        검색된 청크 + 질문을 프롬프트에 함께 넣어 ──▶ LLM ──▶ 답변
```

**왜 이렇게?**
- LLM은 사내 문서를 학습한 적이 없다(환각 위험). "관련 문서를 찾아 프롬프트에 넣어주는" 검색을 벡터 DB가 담당.
- 파인튜닝보다 싸고, 문서가 바뀌면 **재임베딩만** 하면 됨(모델 재학습 불필요).
- 출처(어느 문서에서 왔는지)를 같이 줄 수 있어 신뢰성↑.

# V5. pgvector 실전 (PostgreSQL 내장)

별도 벡터 DB 없이 **PostgreSQL 확장 `pgvector`** 로 다 된다. (Supabase는 내장)

```sql
-- 1. 확장 켜기 (Supabase는 대시보드/SQL로 한 번)
create extension if not exists vector;

-- 2. 임베딩 컬럼 (nomic-embed-text = 768차원)
alter table cards add column embedding vector(768);

-- 3. ANN 색인 (HNSW: 정확·빠름 / ivfflat: 가볍고 빌드 빠름)
create index on cards using hnsw (embedding vector_cosine_ops);

-- 4. 유사도 검색: '<=>' = 코사인 거리(작을수록 유사). '<->'=L2, '<#>'=내적
select id, question, 1 - (embedding <=> $1) as similarity
from cards
order by embedding <=> $1   -- $1 = 쿼리 임베딩
limit 5;
```

**핵심**: 거리 연산자(`<=>`)로 `ORDER BY ... LIMIT k` 하면 그게 곧 top-k 유사 검색. 색인이 있으면 ANN으로, 없으면 전수 스캔으로 동작.

# V6. 그래프 DB(Neo4j) vs 벡터 DB — 언제 뭘

| 질문 형태 | 맞는 도구 |
|-----------|----------|
| "A와 **연결된** 노드? 최단 경로? 2-hop 이웃?" | **그래프 DB** |
| "이 문장과 **의미가 비슷한** 문서 top-k?" | **벡터 DB** |
| "정확히 이 조건인 행? 합계/집계?" | **RDB** |
| "관련 문서 찾아(벡터) + 그 문서들의 관계 따라가기(그래프)" | 둘 다 (GraphRAG) |

> 실무 팁: 규모가 작으면 새 DB 도입 말고 **PostgreSQL 하나로** 버텨라. 관계는 조인/재귀 CTE, 유사도는 pgvector. Neo4j·전용 벡터 DB는 규모·성능이 진짜 문제될 때 도입.

# V7. 이 프로젝트에서의 자리 + 실측

**채점 기능엔 벡터 DB가 필요 없다** (답변 1개 vs 모범답안 1개 비교 = 검색 아님). 벡터가 진짜 맞는 자리는:
1. **노트 시맨틱 검색** — 단어 아닌 뜻으로 노트 찾기.
2. **비슷한 카드 / 중복 탐지** — 400여 카드 중 겹치는 질문 찾기.
3. **연관 카드 추천** — interview-map 개념 연결 보강.

## 실습 결과 (로컬 `nomic-embed-text` + 코사인, 인프라 0)

이 저장소 `notes/**/*.md`에서 카드 질문 **524개**(main Q + 꼬리 Q)를 뽑아 임베딩 → 메모리 코사인으로 "비슷한 카드 top-5"를 돌렸다.

- 모델 `nomic-embed-text`(768차원), 배치 임베딩 **22ms/개**, 524개 총 11.6초. 벡터 DB 없이 **배열 + 코사인만으로** 충분 (이 규모에선 ANN 불필요).

**잘 된 예 (in-domain 정확):**
```
쿼리 [03-network] "OSI 계층에서 TCP와 IP는 각각 몇 계층?"
  0.835  [network] TCP는 스트림이라 메시지 경계가 없다는데 어떻게 구분?
  0.820  [network] TCP 서버 소켓 호출 순서
  0.815  [network] TCP 소켓과 UDP 소켓의 코드상 차이
→ 5개 전부 진짜 네트워크 질문. 좋은 결과.
```
좋은 **교차 도메인** 링크도 나옴: `HashMap 왜 TreeMap?`(jvm) → `Java HashMap 내부 구조`(**dsa**) = 폴더는 달라도 같은 개념. 이게 시맨틱 검색의 진짜 값어치.

**노이즈 (정직한 약점):**
```
쿼리 [02-os] "프로세스와 스레드 차이?"
  0.900  [database] 파티셔닝과 샤딩의 차이는?   ← 1위인데 주제 무관!
  0.862  [javascript] 프로토타입 체인이 뭔가요?  ← 노이즈
```
→ 모델이 기술 명사(프로세스/스레드)가 아니라 **의문문 "템플릿"**("~의 차이는?")에 과하게 반응. `왜 ~한가요?`, `실무에서 ~` 같은 한국어 질문 골격이 의미를 눌러버림. `인덱스는 왜 빠른가` 쿼리도 1위가 엉뚱하게 `데드락 만난 적?`이 나옴.

**교훈 (실무 직결):**
1. **작은 임베딩 모델은 표면형(phrasing) 매칭 경향** — 특히 짧은 한국어 질문. #1 결과를 맹신 금지.
2. **가장 싼 개선 = 도메인 필터** — 같은 도메인 안에서만 유사도 검색하면 최악의 교차 도메인 충돌이 사라짐.
3. 더 나은 품질은 **다국어 대형 임베딩 모델**(`bge-m3` 등) 또는 **리랭커(reranker)** 로. (트레이드오프: 속도·비용)
4. **결론**: "이것도 복습해보세요" 추천 위젯엔 지금도 충분. 정밀 검색엔 위 개선 필요.

# 예상 면접 질문

**Q1. "벡터 DB가 뭐고, 언제 쓰나요?"**
> 텍스트 등을 임베딩(의미를 담은 고차원 벡터)으로 바꿔 저장하고, 쿼리와 **의미상 가까운** 항목을 ANN 색인으로 top-k 검색하는 DB입니다. 키워드 일치가 아니라 의미 유사도가 필요할 때 — 시맨틱 검색, 추천, 특히 RAG의 검색 단계에 씁니다.

**Q2. "RAG를 설명해보세요."**
> 문서를 청크로 잘라 임베딩해 벡터 DB에 넣어두고, 사용자 질문을 임베딩해 가까운 청크를 검색한 뒤, 그 청크를 프롬프트에 함께 넣어 LLM이 답하게 하는 구조입니다. LLM이 학습 안 한 사내 지식을 환각 없이 답하게 하고, 문서 변경 시 재임베딩만으로 갱신되며 파인튜닝보다 쌉니다.

**꼬리 Q2-1. "왜 그냥 파인튜닝하지 않고 RAG를 쓰나요?"**
> 파인튜닝은 데이터셋 구축·학습·재학습 비용이 크고 최신 정보 반영이 느립니다. RAG는 문서만 다시 임베딩하면 즉시 반영되고, 출처를 함께 제시해 신뢰성과 검증이 쉽습니다. 지식이 자주 바뀌는 QA/사내봇에는 RAG가, 말투·형식 고정에는 파인튜닝이 더 맞습니다.

**Q3. "벡터 유사도로 코사인을 쓰는 이유는?"**
> 텍스트 임베딩은 벡터의 길이보다 방향(의미의 방향)이 중요합니다. 코사인은 각도만 보므로 문서 길이에 덜 민감하고, 정규화된 임베딩에서는 내적과 같아 계산도 쌉니다.

**Q4. "Neo4j 같은 그래프 DB와 벡터 DB는 뭐가 다른가요?"**
> 그래프 DB는 노드·엣지로 **관계 경로 탐색**(친구의 친구, 최단 경로)이 주특기이고, 벡터 DB는 임베딩으로 **의미 유사도 검색**이 주특기입니다. 문제가 "연결"이면 그래프, "비슷함"이면 벡터입니다. 둘을 결합한 GraphRAG도 있습니다.
