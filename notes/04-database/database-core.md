# D1. Database Index

**학습 목표**: *"인덱스가 뭐고 왜 빠른가요? 단점은?"* 에 답할 수 있다.

## 1. 비유 — 책 색인
500쪽 책에서 "트랜잭션" 단어 찾기: 처음부터 다 읽으면(Full Scan) 느림. 책 뒤 **색인(index)**에서 페이지 번호 보고 바로 펴면 빠름. DB 인덱스가 바로 이 색인.

## 2. 개념 + 자료구조
> **인덱스** = 특정 컬럼 값을 **정렬된 자료구조(B-Tree)** 로 별도 저장해, 검색을 O(N)→O(log N)으로 줄이는 것.
- 대부분 RDB 인덱스는 **B-Tree(정확히는 B+Tree)**: 정렬 유지, 범위 검색·정렬에 강함.
- 해시 인덱스는 등가 검색(`=`)만 O(1), 범위 검색 불가.

## 3. 다이어그램 — B+Tree
```
              [ 30 | 60 ]              ← 루트(중간 노드는 탐색용)
             /     |     \
      [10|20] [30|40|50] [60|70]       ← 리프에 실제 키(정렬)
        │  리프끼리 링크드리스트로 연결 → 범위 검색 빠름
```

## 4. 클러스터드 vs 논클러스터드
| 구분 | 클러스터드 | 논클러스터드(보조) |
|------|-----------|-------------------|
| 정렬 | 데이터 자체가 인덱스 순 정렬 | 별도 인덱스 + 실제행 포인터 |
| 개수 | 테이블당 1개(보통 PK) | 여러 개 |
| 속도 | 범위검색 매우 빠름 | 한 단계 더(포인터 조회) |

## 5. 핵심 포인트 (자주 하는 실수) ⭐
- 🔴 "인덱스 많을수록 좋다" — ❌ **쓰기(INSERT/UPDATE/DELETE) 때마다 인덱스도 갱신** → 쓰기 느려지고 저장공간↑. 읽기-쓰기 균형이 관건.
- 🔴 인덱스 컬럼에 **함수/연산** 쓰면 인덱스 못 탐: `WHERE YEAR(created)=2024` ❌ → `WHERE created >= '2024-01-01'` ✅.
- 🟡 **복합 인덱스는 순서가 중요**: `(a, b)` 인덱스는 `a` 또는 `a,b` 검색엔 타지만 `b` 단독은 못 탐(왼쪽 접두사 규칙).
- 🟡 **커버링 인덱스**: 쿼리가 필요한 컬럼이 인덱스에 다 있으면 테이블 접근 없이 인덱스만으로 응답 → 매우 빠름.
- 🟡 카디널리티 낮은 컬럼(예: 성별)은 인덱스 효과 적음.

## 6. 실행 계획(EXPLAIN) 읽기 ⭐
쿼리가 인덱스를 타는지는 **추측이 아니라 확인**해야 함. `EXPLAIN SELECT ...` (MySQL 기준) 로 옵티마이저가 세운 계획을 본다.

- **type** 컬럼이 핵심 지표:
  | type | 의미 |
  |------|------|
  | `ALL` | Full Scan — 인덱스 없이 전체 행 스캔 (경고 신호) |
  | `index` | 인덱스 전체를 순서대로 스캔 (Full Scan보다는 낫지만 여전히 큼) |
  | `range` | 인덱스로 범위 검색 (`BETWEEN`, `>`, `IN` 등) |
  | `ref` / `eq_ref` | 인덱스로 동등 비교 — `eq_ref`는 조인에서 PK/UNIQUE로 1행만 매칭 (이상적) |
  | `const` | PK/UNIQUE로 단 1행 확정 (가장 빠름) |
- **key** 컬럼: 실제 사용된 인덱스 이름 (`NULL`이면 인덱스 미사용).
- **rows**: 옵티마이저가 예상하는 검사 행 수 — 실제 결과 행 수와 크게 다르면 통계 정보(테이블 통계, `ANALYZE TABLE`)가 낡았을 가능성.
- **Extra**: `Using index`(커버링 인덱스로 해결됨, 이상적) vs `Using filesort`/`Using temporary`(정렬·임시테이블 발생, 튜닝 필요 신호).

### 옵티마이저가 인덱스를 무시하는 경우
- **카디널리티가 낮음**: 인덱스를 타도 전체 행의 상당 부분을 걸러내지 못하면, 옵티마이저는 "인덱스로 찾아가서 다시 테이블 읽기"보다 그냥 Full Scan이 더 싸다고 판단.
- **결과 행 비율이 큼**: 통계상 조건에 맞는 행이 테이블의 10~20% 이상이면 인덱스 사용이 오히려 손해(랜덤 I/O 다발) → Full Scan 선택.
- **통계 정보가 오래됨**: 대량 insert/delete 후 `ANALYZE TABLE`을 안 돌리면 옵티마이저가 잘못된 rows 추정으로 엉뚱한 계획을 세움.
- **암묵적 형변환**: `WHERE varchar_col = 123` 처럼 타입이 안 맞으면 인덱스 컬럼에 변환 함수가 걸려 인덱스 무시.
- 해결 순서: `EXPLAIN`으로 원인 확인 → 통계 갱신/쿼리 재작성 → 그래도 안 되면 `FORCE INDEX`(임시방편, 남용 금지).

## 7. 인덱스 심화 — 복합 인덱스 설계 ⭐
### 선두 컬럼 선택 기준
복합 인덱스 `(a, b, c)`는 왼쪽 접두사 규칙 때문에 **어떤 컬럼을 앞에 두느냐**가 성능을 좌우한다.
1. **카디널리티가 높은 컬럼을 앞에**: 한 번에 많이 걸러낼수록 뒤 단계에서 훑을 행이 줄어듦. (예: `status`(값 3개) 보다 `user_id`(수만 개)를 앞에)
2. **조건 빈도가 높은 컬럼을 앞에**: 카디널리티가 비슷하다면, WHERE 절에 **자주 등장하는** 컬럼을 앞에 둬야 다양한 쿼리가 인덱스를 재사용.
3. **등호(`=`) 조건 컬럼을 범위(`BETWEEN`, `>`) 조건 컬럼보다 앞에**: 범위 조건이 나오는 순간 그 뒤 컬럼은 정렬 이점을 못 씀. 예: `(user_id, created_at)`에서 `WHERE user_id=1 AND created_at > ?` 는 두 조건 다 인덱스 활용, 반대로 `(created_at, user_id)`면 `created_at` 범위 이후 `user_id`는 인덱스 안에서 추가로 못 좁힘.

### 커버링 인덱스 실제 예
```sql
-- orders(id PK, user_id, status, created_at, amount ...)
CREATE INDEX idx_user_status ON orders(user_id, status, created_at);

-- 이 쿼리는 인덱스 컬럼만으로 응답 가능 → 테이블(클러스터드) 접근 없이 인덱스만 읽음
SELECT user_id, status, created_at
FROM orders
WHERE user_id = 42 AND status = 'PAID';
-- EXPLAIN Extra: "Using index" 로 확인
```
- 만약 `SELECT *`를 쓰면 인덱스에 없는 컬럼(`amount` 등)을 가져오려고 테이블을 다시 찾아가야 해서 커버링 인덱스 효과가 사라짐 → **필요한 컬럼만 SELECT** 하는 습관이 성능에 직결.

## 8. 예상 면접 질문
**Q1. "인덱스는 왜 빠르고, 단점은?"**
> 정렬된 B-Tree로 검색을 O(log N)으로 줄여 조회가 빠릅니다. 단 쓰기마다 인덱스를 갱신해야 해 쓰기 성능이 떨어지고 저장공간을 더 쓰므로, 조회가 잦은 컬럼에 선별적으로 걸어야 합니다.

**Q2. "인덱스를 걸었는데 안 타는 경우는?"**
> 인덱스 컬럼에 함수·연산을 적용하거나, 복합 인덱스에서 왼쪽 접두사를 건너뛰거나, 카디널리티가 너무 낮거나, 옵티마이저가 Full Scan이 더 싸다고 판단할 때입니다.

**Q3. "쿼리가 인덱스를 타는지 어떻게 확인하나요?"**
> `EXPLAIN`으로 실행 계획을 봅니다. type이 `ALL`이면 Full Scan, `ref`/`range`/`const`면 인덱스 사용 중이고, key 컬럼으로 실제 어떤 인덱스가 쓰였는지, Extra의 `Using index`로 커버링 인덱스 여부를 확인합니다.

**Q4. "복합 인덱스 컬럼 순서는 어떻게 정하나요?"**
> 카디널리티가 높고 조건에 자주 등장하는 컬럼을 앞에 두고, 등호 조건 컬럼을 범위 조건 컬럼보다 앞에 둡니다. 왼쪽 접두사 규칙 때문에 순서가 곧 어떤 쿼리 패턴을 지원할지를 결정합니다.

**Q5. (시나리오) "인덱스를 분명히 걸었는데 안 타요. 왜 그럴 수 있죠?"**
> 먼저 `EXPLAIN`으로 확인부터 합니다. 흔한 원인은 ① 인덱스 컬럼에 함수·연산·암묵적 형변환이 걸린 경우(`WHERE DATE(created)=...`, `WHERE phone=01012345678`처럼 varchar에 숫자 비교), ② 복합 인덱스 왼쪽 접두사를 건너뛴 경우, ③ 선행 와일드카드 `LIKE '%kakao'`, ④ 조건에 맞는 행 비율이 커서(대략 20% 이상) 옵티마이저가 랜덤 I/O보다 Full Scan을 싸다고 본 경우, ⑤ `ANALYZE TABLE`을 안 돌려 통계가 낡은 경우입니다. 원인별로 쿼리 재작성·통계 갱신으로 풀고, `FORCE INDEX`는 최후의 임시방편으로만 씁니다.

**Q6. (시나리오) "복합 인덱스 `(a, b)`가 있는데 `WHERE b = ?` 로만 조회하면 어떻게 되나요?"**
> 왼쪽 접두사 규칙 때문에 이 인덱스를 정상적으로 타지 못합니다. 인덱스는 `a`로 먼저 정렬되고 그 안에서 `b`가 정렬된 구조라, `a` 없이 `b`만 주면 특정 구간으로 좁힐 수가 없습니다. 다만 인덱스에 `b`가 포함되어 있으니 옵티마이저가 인덱스 풀스캔(`type=index`)으로 커버링 삼을 수는 있는데, 이는 테이블 풀스캔보다 조금 나을 뿐 `range`처럼 좁혀 읽는 게 아닙니다. `b` 단독 조회가 잦으면 `(b)` 또는 `(b, a)` 인덱스를 따로 만들어야 합니다.

**Q7. (트레이드오프) "그럼 인덱스는 조회 컬럼마다 다 걸면 되지 않나요?"**
> 아닙니다. 인덱스는 읽기를 위해 쓰기·저장공간을 희생하는 트레이드오프입니다. 인덱스가 많으면 INSERT/UPDATE/DELETE마다 관련 B+Tree를 전부 갱신해야 해서 쓰기 지연이 커지고, 버퍼 풀·디스크도 더 씁니다. 쓰기가 많은 테이블(로그, 이벤트)일수록 인덱스를 최소화하고, 실제 느린 쿼리를 슬로우 쿼리 로그로 뽑아 필요한 것만 선별해서 겁니다.

# D2. Transaction and ACID

**학습 목표**: *"트랜잭션 ACID, 격리수준과 이상현상?"* 에 답할 수 있다. (Spring @Transactional과 직결)

## 1. 비유 — 계좌 이체
A→B 5만원 이체 = "A에서 차감" + "B에 입금" **두 작업이 하나로 묶여** 전부 되거나 전부 안 되어야. 중간에 죽으면 롤백. 이 묶음이 **트랜잭션**.

## 2. ACID
| 속성 | 의미 |
|------|------|
| **Atomicity(원자성)** | 전부 성공 or 전부 실패(롤백) |
| **Consistency(일관성)** | 트랜잭션 전후 무결성 규칙 유지 |
| **Isolation(격리성)** | 동시 트랜잭션이 서로 간섭 안 함 |
| **Durability(지속성)** | 커밋되면 장애에도 보존(로그) |

## 3. 격리 수준과 이상현상 ⭐
격리를 낮출수록 성능↑·이상현상↑. 이상현상 3종:
- **Dirty Read**: 커밋 안 된 값을 읽음
- **Non-Repeatable Read**: 같은 행을 두 번 읽었는데 값이 달라짐(중간에 UPDATE 커밋)
- **Phantom Read**: 같은 조건 재조회 시 행 개수가 달라짐(중간에 INSERT)

| 격리수준 | Dirty | Non-Repeatable | Phantom |
|----------|-------|----------------|---------|
| READ UNCOMMITTED | 발생 | 발생 | 발생 |
| READ COMMITTED (많은 DB 기본) | 방지 | 발생 | 발생 |
| REPEATABLE READ (MySQL 기본) | 방지 | 방지 | 발생* |
| SERIALIZABLE | 방지 | 방지 | 방지 |
> *MySQL InnoDB는 MVCC+갭락으로 REPEATABLE READ에서 팬텀도 상당부분 방지.

## 4. 핵심 포인트
- 🟡 격리수준은 **동시성 성능 vs 정확성**의 트레이드오프. 무조건 SERIALIZABLE은 성능 최악.
- 🟡 Spring `@Transactional(isolation=...)`로 지정 가능하지만 보통 DB 기본값 사용.

## 5. 커넥션 풀(Connection Pool)과 N+1 ⭐ (↔ 애플리케이션 레이어)
### 왜 필요한가
- TCP 연결 수립 + DB 인증은 **비용이 큰 작업**(수십 ms 단위). 요청마다 새 커넥션을 열고 닫으면 트랜잭션 실행 시간보다 연결 비용이 더 클 수 있음.
- **커넥션 풀**은 미리 커넥션 여러 개를 만들어두고 재사용 → 연결 비용 제거, DB 서버의 동시 연결 수 제한도 보호(무한정 커넥션이 열리면 DB가 죽음).
- Spring Boot 기본 구현체: **HikariCP**.

### 크기 설정
- 무조건 크게 잡으면 좋을 것 같지만 ❌ — 각 커넥션은 DB 서버 쪽에서 스레드/메모리를 점유. 풀이 너무 크면 DB 컨텍스트 스위칭 비용↑, 오히려 처리량↓.
- 경험칙(HikariCP 공식 가이드): `connections = ((core_count * 2) + effective_spindle_count)` 근사치 — **CPU 코어 수 기반**으로 작게 시작해 모니터링하며 조정. "많을수록 좋다"가 아니라 **DB가 실제로 병렬 처리 가능한 수준**이 상한.
- 풀이 너무 작으면: 커넥션 대기 큐가 쌓여 응답 지연(`connection timeout`) 발생 → 애플리케이션 스레드 자체가 블로킹되어 전체 처리량 저하로 이어짐.

### N+1 문제는 애플리케이션 이슈
- DB 자체 문제가 아니라 **ORM(JPA/Hibernate) 사용 패턴**에서 발생: 목록 1번 조회(1) 후 연관 엔티티를 각 행마다 지연 로딩(N번 추가 쿼리) → 총 N+1번 쿼리.
```java
List<Order> orders = orderRepository.findAll(); // 1번
for (Order o : orders) {
    o.getUser().getName(); // 지연 로딩 → 주문 수만큼 N번 추가 쿼리
}
```
- 해결: **fetch join**(`JOIN FETCH`), `@EntityGraph`, 또는 배치 사이즈 설정(`hibernate.default_batch_fetch_size`)으로 N번을 1~수 번으로 축소.
- 커넥션 풀과의 연결고리: N+1이 심하면 쿼리 수가 폭증해 **커넥션을 오래 점유** → 풀 고갈 → 다른 요청이 커넥션을 못 받아 전체 응답 지연으로 번짐. 즉 N+1(애플리케이션 버그)이 커넥션 풀 부족(인프라 증상)처럼 보이는 경우가 실무에서 흔함.

## 6. 예상 면접 질문
**Q1. "ACID를 설명해주세요."**
> 원자성(전부 or 전무), 일관성(무결성 유지), 격리성(동시 트랜잭션 간섭 방지), 지속성(커밋 후 영속). 이체처럼 여러 작업을 하나로 묶어 안전하게 처리하기 위한 성질입니다.

**Q2. "격리수준과 이상현상 관계는?"**
> 낮은 격리일수록 Dirty/Non-Repeatable/Phantom Read가 나타나고 성능은 좋아집니다. READ COMMITTED는 dirty만 막고, REPEATABLE READ는 non-repeatable까지, SERIALIZABLE은 전부 막지만 가장 느립니다.

**Q3. "커넥션 풀은 왜 필요하고 크기는 어떻게 정하나요?"**
> 매 요청마다 DB 커넥션을 새로 맺으면 연결 비용이 커서 미리 만들어둔 커넥션을 재사용합니다. 크기는 무조건 크게가 아니라 DB가 실제로 병렬 처리 가능한 수준(CPU 코어 수 기반)에 맞춰 작게 시작해 모니터링하며 조정합니다.

**Q4. "N+1 문제가 뭐고 어떻게 해결하나요?"**
> 목록을 조회한 뒤 각 행의 연관 엔티티를 지연 로딩으로 하나씩 추가 조회해 쿼리가 N+1번 발생하는 문제입니다. fetch join이나 배치 사이즈 설정으로 쿼리를 묶어서 해결하며, 방치하면 커넥션 풀 고갈로도 이어질 수 있습니다.

**Q5. (왜) "격리수준을 왜 굳이 REPEATABLE READ로 두나요? READ COMMITTED가 더 흔하다던데요."**
> DB마다 기본값이 다르기 때문입니다. Oracle·PostgreSQL·SQL Server는 READ COMMITTED가 기본이고, MySQL InnoDB만 REPEATABLE READ가 기본입니다. InnoDB가 RR을 기본으로 둔 역사적 이유는 과거 statement 기반 복제(binlog)에서 마스터·슬레이브 데이터 일관성을 지키려면 트랜잭션 내내 스냅샷이 고정돼야 했기 때문입니다. 실무에서는 한 트랜잭션 안에서 같은 조회를 반복해도 값이 안 흔들리길 원하면 RR, 항상 최신 커밋값을 보고 싶고 롱 트랜잭션에서 언두가 쌓이는 걸 피하고 싶으면 RC를 택합니다. 핵심은 "기본값이라 그냥 쓴다"가 아니라 우리 서비스가 어느 쪽 이상현상을 감수할 수 있는지로 정한다는 점입니다.

**Q6. (시나리오) "재고를 조회해서 1 남았길래 주문을 넣었는데 초과 판매가 났습니다. 격리수준으로 막을 수 있나요?"**
> 순수 격리수준만으로는 부족합니다. 흔한 실수가 `SELECT`로 재고를 읽고(스냅샷) 애플리케이션에서 판단한 뒤 `UPDATE`하는 패턴인데, 두 트랜잭션이 동시에 "1개 남음"을 읽으면 둘 다 통과합니다(lost update). 해결은 ① `SELECT ... FOR UPDATE`로 읽는 순간 배타 락을 잡거나, ② `UPDATE stock SET qty=qty-1 WHERE id=? AND qty>0`처럼 조건부 UPDATE로 원자적 감소를 하거나, ③ `@Version` 낙관적 락으로 충돌 시 재시도하는 것입니다. 격리수준을 올리는 것보다 "읽고-쓰는 구간을 원자적으로 만드는 것"이 핵심입니다.

# D3. Lock and MVCC

**학습 목표**: *"공유/배타 락, 낙관 vs 비관, MVCC?"* 에 답할 수 있다. (OS 동기화와 연결)

## 1. 비유 — 문서 공동 편집
같은 문서를 둘이 고치면 충돌. 해결:
- **비관적 락**: 편집 시작 시 **문서 잠금**(남이 못 건드림). 충돌 잦을 때.
- **낙관적 락**: 일단 각자 편집, 저장 시 **버전 비교**해 충돌이면 재시도. 충돌 드물 때.

## 2. 락 종류
- **공유 락(S, Read)**: 여러 트랜잭션 동시 읽기 허용, 쓰기 차단.
- **배타 락(X, Write)**: 하나만. 읽기·쓰기 모두 차단.

## 3. 낙관 vs 비관
| | 비관적 락 | 낙관적 락 |
|--|----------|----------|
| 방식 | 미리 DB 락 획득 | version 컬럼으로 충돌 감지 |
| 적합 | 충돌 잦음 | 충돌 드묾(웹서비스 다수) |
| 비용 | 락 대기·데드락 위험 | 충돌 시 재시도 |
> JPA: 비관=`@Lock(PESSIMISTIC_WRITE)`, 낙관=`@Version`.

## 4. MVCC (Multi-Version Concurrency Control) ⭐
- 데이터를 **여러 버전**으로 유지 → 읽기는 **스냅샷**을 보고, 쓰기는 새 버전 생성.
- 효과: **읽기가 쓰기를 막지 않고, 쓰기가 읽기를 막지 않음**(락 경합↓). MySQL InnoDB·PostgreSQL의 핵심.

## 5. MVCC 구현 상세 — undo log와 버전 체인 ⭐ (MySQL InnoDB 기준)
### 버전 체인은 어떻게 만들어지나
- 각 행에는 숨겨진 컬럼이 있음: **트랜잭션 ID(생성 버전)**와 **undo log 포인터**(이전 버전 위치).
- `UPDATE`가 일어나면 InnoDB는 원본 행을 덮어쓰기 전에 **변경 전 이미지를 undo log에 기록**하고, 새 행이 그 undo log를 가리키게 함 → 여러 버전이 **링크드리스트(버전 체인)**로 연결됨.
```
최신 행(txn 105) → undo log(txn 102가 남긴 이전 값) → undo log(txn 98이 남긴 이전 값) → ...
```
- 읽기 트랜잭션은 시작 시점에 **자기 스냅샷 기준(read view)**을 정하고, 버전 체인을 따라가며 "내 스냅샷 시점에 커밋되어 있던" 버전을 찾아 반환 → 다른 트랜잭션이 그 사이 값을 바꿔도 내 스냅샷은 그대로(Non-Repeatable Read 방지).
- undo log는 롤백 용도로도 쓰임(Atomicity) — MVCC와 트랜잭션 롤백이 같은 메커니즘을 공유.
- 오래된 버전은 그 버전을 참조하는 트랜잭션이 하나도 없어지면 **purge 스레드**가 정리(너무 오래 열린 트랜잭션이 있으면 undo log가 계속 쌓여 성능 저하 → "장시간 트랜잭션 금지"의 실제 이유).

### REPEATABLE READ에서 갭락(Gap Lock)으로 팬텀 막기
- MVCC(스냅샷 읽기)만으로는 **일반 SELECT**의 팬텀은 막아지지만, **쓰기가 섞인 경우**(`SELECT ... FOR UPDATE`, `UPDATE`, `INSERT` 재확인)는 락이 필요.
- InnoDB는 REPEATABLE READ에서 인덱스 레코드뿐 아니라 **레코드 사이의 "간격(gap)"에도 락**을 건다.
```sql
-- id: 10, 20, 30 이 있는 테이블
SELECT * FROM t WHERE id BETWEEN 10 AND 30 FOR UPDATE;
-- Gap Lock이 (10,20), (20,30) 사이 "틈"까지 잠가서
-- 다른 트랜잭션이 id=15 를 INSERT 하는 것도 막음 → 팬텀 방지
```
- **Next-Key Lock** = 레코드 락 + 그 앞 갭 락의 조합. InnoDB REPEATABLE READ의 기본 잠금 방식.
- 즉 "읽기 전용 조회"는 MVCC 스냅샷으로, "쓰기가 걸린 범위 조회"는 갭락/넥스트키락으로 팬텀을 막는 **이중 구조**.

## 6. 데드락 실전 예시와 예방 ⭐
### 두 트랜잭션이 락을 반대 순서로 잡는 경우
```
트랜잭션 A                         트랜잭션 B
----------------------------------------------------------
UPDATE accounts SET ... WHERE id=1;   UPDATE accounts SET ... WHERE id=2;
  (id=1 행에 X락 획득)                  (id=2 행에 X락 획득)

UPDATE accounts SET ... WHERE id=2;   UPDATE accounts SET ... WHERE id=1;
  (id=2 락 대기 → B가 쥐고 있음)         (id=1 락 대기 → A가 쥐고 있음)

              ⇒ 서로 상대가 풀어주기만 기다림 = 데드락
```
- A는 `id=1 → id=2` 순서로, B는 `id=2 → id=1` 순서로 락을 잡아 **순환 대기(circular wait)**가 발생 — 계좌 이체 로직에서 흔한 실수(A→B 이체와 B→A 이체가 동시에 실행될 때).
- DB는 보통 데드락 감지 시 **둘 중 하나(비용이 적은 쪽)를 강제 롤백**하고 나머지는 진행(`Deadlock found when trying to get lock` 에러).

### 예방
- **락 획득 순서 통일**: 모든 트랜잭션이 항상 "낮은 PK → 높은 PK" 순서로 접근하도록 애플리케이션에서 강제(예: 이체 시 계좌 ID 오름차순으로 정렬 후 UPDATE).
- **트랜잭션을 짧게**: 락을 오래 쥐고 있을수록 충돌 확률↑. 트랜잭션 안에서 불필요한 외부 호출·긴 로직 금지.
- **인덱스로 락 범위 최소화**: 인덱스가 없으면 조건에 안 맞는 행까지 락이 걸려(풀스캔 중 락) 충돌 범위가 커짐.
- **동일한 격리수준·락 전략을 팀 전체가 일관되게** 사용 — 일부만 `FOR UPDATE`를 쓰고 일부는 안 쓰면 예측 불가능한 충돌 패턴 발생.

## 7. OS 동기화와의 연결 ⭐ (cross-link)
- DB 락(S/X) = OS의 뮤텍스/세마포어와 같은 **상호배제** 문제(OS O4).
- DB 데드락 4조건도 OS 데드락과 동일 — 락 획득 순서 통일로 예방.

## 8. 예상 면접 질문
**Q1. "낙관적 락과 비관적 락 차이?"**
> 비관적 락은 미리 DB 락을 걸어 충돌을 원천 차단(충돌 잦을 때), 낙관적 락은 version으로 저장 시점에 충돌을 감지해 재시도(충돌 드물 때). 웹서비스는 보통 낙관적 락이 유리합니다.

**Q2. "MVCC가 뭔가요?"**
> 데이터를 다중 버전으로 관리해 읽기는 스냅샷을, 쓰기는 새 버전을 만드는 방식입니다. 읽기와 쓰기가 서로 막지 않아 동시성이 크게 향상됩니다.

**Q3. "MVCC는 실제로 어떻게 구현되나요?"**
> InnoDB는 UPDATE 전 원본 값을 undo log에 남기고 새 행이 그 로그를 가리키게 해 버전 체인을 만듭니다. 읽기는 자신의 스냅샷(read view) 시점 기준으로 체인을 따라가며 적절한 버전을 찾고, 오래된 버전은 참조하는 트랜잭션이 없어지면 purge됩니다.

**Q4. "REPEATABLE READ인데 어떻게 팬텀을 막나요?"**
> 일반 SELECT는 MVCC 스냅샷으로 막히지만, `FOR UPDATE`처럼 쓰기가 섞인 범위 조회는 레코드 사이 간격까지 잠그는 갭락(Next-Key Lock)으로 다른 트랜잭션의 INSERT를 막아 팬텀을 방지합니다.

**Q5. "데드락 예시와 예방법은?"**
> 두 트랜잭션이 각각 A→B, B→A 순서로 같은 두 행에 락을 걸면 서로 상대 락을 기다리며 순환 대기에 빠집니다. 예방은 모든 트랜잭션이 동일한 순서(예: PK 오름차순)로 락을 획득하게 하고, 트랜잭션을 짧게 유지하며, 인덱스로 락 범위를 최소화하는 것입니다.

**Q6. (시나리오) "낙관적 락을 걸었더니 트래픽이 몰릴 때 재시도가 폭증하고 응답이 느려졌습니다. 왜죠?"**
> 낙관적 락은 "충돌이 드물다"는 가정 위에서만 효율적이기 때문입니다. 인기 상품 재고 차감처럼 같은 행에 요청이 집중되면 버전 충돌이 계속 나서 재시도가 눈덩이처럼 불어나고(라이브락에 가까운 상태), 재시도 자체가 부하를 키웁니다. 이런 핫 로우(hot row)는 오히려 비관적 락(`FOR UPDATE`)으로 줄 세우거나, 아예 조건부 UPDATE(`WHERE qty>0`) 한 방으로 원자화하거나, Redis 원자 연산·큐로 직렬화하는 게 낫습니다. 즉 락 전략은 "충돌 빈도"라는 데이터 특성으로 선택하는 트레이드오프입니다.

**Q7. (왜) "일반 SELECT는 락을 안 잡는데 어떻게 REPEATABLE READ가 보장되나요?"**
> InnoDB의 일반 `SELECT`는 락을 잡지 않는 스냅샷 읽기(consistent read)입니다. 트랜잭션 첫 조회 시점에 read view를 고정하고, 그 이후 다른 트랜잭션이 커밋한 변경은 언두 로그의 이전 버전을 따라가 안 보이게 합니다. 그래서 락 경합 없이도 같은 행을 반복 조회할 때 값이 안 바뀝니다. 락은 쓰기(`UPDATE`)나 `FOR UPDATE`/`LOCK IN SHARE MODE`같은 잠금 읽기(current read)에서만 등장하고, 이때는 최신 커밋본을 보고 넥스트키락까지 겁니다. 이 "스냅샷 읽기 vs 잠금 읽기" 구분이 D9의 핵심입니다.

# D4. Normalization

**학습 목표**: *"정규화가 뭐고 왜/언제 반정규화하나요?"* 에 답할 수 있다.

## 1. 비유 — 중복 제거
주문 테이블마다 고객 이름·주소를 매번 적으면 고객이 이사할 때 모든 행을 고쳐야(이상현상). 고객 정보를 **별도 테이블로 분리**하고 ID로 참조 = 정규화.

## 2. 정규화 단계
| 단계 | 규칙 |
|------|------|
| **1NF** | 원자값(한 칸에 하나), 반복 그룹 제거 |
| **2NF** | 부분 함수 종속 제거(복합키 일부에만 종속되는 컬럼 분리) |
| **3NF** | 이행 종속 제거(키가 아닌 컬럼에 종속되는 컬럼 분리) |

## 3. 정규화 vs 반정규화
| | 정규화 | 반정규화 |
|--|--------|----------|
| 목적 | 중복 제거·무결성 | 조회 성능(조인↓) |
| 장점 | 갱신 이상 방지, 저장 효율 | 읽기 빠름 |
| 단점 | 조인 많아짐 | 중복·갱신 비용↑ |
> 실무: 정규화 기본, **읽기 병목이 검증되면** 선택적 반정규화(캐시성 컬럼 추가 등).

## 4. 핵심 포인트
- 🟡 반정규화는 "성급한 최적화" 주의 — 측정 후 적용.
- 🟡 조인이 많아 느리다고 무조건 반정규화 X, 인덱스·쿼리 튜닝이 먼저.

## 5. 예상 면접 질문
**Q1. "정규화의 목적과 반정규화하는 이유는?"**
> 정규화는 중복을 없애 갱신 이상을 막고 무결성을 지키는 것입니다. 다만 조인이 많아져 읽기가 느려질 수 있어, 읽기 병목이 확인되면 성능을 위해 선택적으로 반정규화(중복 허용)합니다.

**Q2. "정규화 단계를 하나의 예로 설명해보세요."**
> 주문 테이블에 고객 이름·주소를 매번 같이 저장하면(반복), 고객이 이사할 때 모든 주문 행을 수정해야 하는 갱신 이상이 생깁니다. 고객 테이블을 분리하고 FK로 참조하면 2NF/3NF를 만족하며 이 문제가 사라집니다.

# D5. SQL vs NoSQL and Redis

**학습 목표**: *"RDB와 NoSQL 언제 쓰나요? Redis는?"* 에 답할 수 있다.

## 1. 비유 — 엑셀 vs 메모앱
- **RDB** = 엑셀 시트(정해진 열, 관계, 규칙). 정합성 강함.
- **NoSQL** = 자유 형식 메모(스키마 유연, 대량·분산에 강함).

## 2. 비교표
| | RDB (SQL) | NoSQL |
|--|-----------|-------|
| 스키마 | 고정, 정규화 | 유연(문서/키값/컬럼/그래프) |
| 정합성 | 강함(ACID) | 유연(BASE, 결과적 일관성) |
| 확장 | 수직(scale-up) 위주 | 수평(scale-out) 강함 |
| 적합 | 트랜잭션·관계 복잡 | 대용량·유연 스키마·고속 |

## 3. Redis (In-Memory)
- 메모리 기반 key-value 저장소 → **초고속**. 용도: **캐시**, 세션 저장, 랭킹(Sorted Set), 분산 락, 큐.
- **캐시 전략**: Look-Aside(캐시 먼저 조회, 없으면 DB→캐시 저장), Write-Through 등.
- 캐시 이슈: **캐시 무효화**, TTL, Cache Stampede.

## 4. 핵심 포인트
- 🔴 "NoSQL이 항상 빠르고 좋다" — ❌ 트랜잭션·복잡 조인·강한 정합성엔 RDB가 유리. 용도 문제.
- 🟡 대부분 실무는 **RDB 주력 + Redis 캐시** 조합.
- 🟡 CAP: 분산 시스템은 일관성(C)·가용성(A)·분단내성(P) 중 P 하에 C와 A를 트레이드오프.

## 5. 파티셔닝 vs 샤딩 ⭐
"확장(수평/수직)"을 실제로 어떻게 하는지에 대한 두 가지 다른 접근.

### 파티셔닝 (Partitioning) — 한 DB 인스턴스 안에서 분할
- **수평 파티셔닝**: 같은 스키마의 테이블을 **행 기준**으로 여러 물리 조각(파티션)으로 나눔. 예: `orders` 테이블을 `created_at` 기준 월별 파티션으로 분리 → 오래된 데이터 조회/삭제가 해당 파티션만 건드려 빠름.
- **수직 파티셔닝**: 한 테이블을 **컬럼 기준**으로 나눔. 예: 자주 안 쓰는 대용량 컬럼(예: 본문 TEXT, 이미지 BLOB)을 별도 테이블로 분리해 자주 조회하는 핵심 컬럼만 있는 테이블을 가볍게 유지(정규화와 비슷한 효과지만 목적은 성능).
- 핵심: **여전히 하나의 DB 서버/인스턴스** 안에서 일어나는 분할. 애플리케이션 입장에선 대체로 투명(쿼리는 그대로, DB 내부적으로 파티션 프루닝).

### 샤딩 (Sharding) — 여러 DB 서버로 분산
- 데이터를 **여러 개의 독립된 DB 서버(물리적으로 분리)**로 나눠 저장. 각 샤드는 자기 데이터에 대해 완전한 DB 인스턴스(자체 CPU/메모리/디스크).
- 보통 특정 **샤드 키**(예: `user_id % N`, 해시 기반)로 어느 샤드에 저장할지 결정.
- 파티셔닝과 차이: 샤딩은 **여러 서버로 흩어지므로** 스케일 아웃 효과가 훨씬 크지만, 애플리케이션(또는 미들웨어)이 "어느 샤드로 갈지" 알아야 하고, **샤드 간 조인·트랜잭션이 어려움**(분산 트랜잭션 필요) → 설계 복잡도가 크게 증가.

| | 파티셔닝 | 샤딩 |
|--|---------|------|
| 범위 | 한 DB 인스턴스 내부 | 여러 개의 독립 DB 서버 |
| 애플리케이션 인지 | 대체로 투명 | 샤드 키 라우팅 로직 필요 |
| 조인/트랜잭션 | 그대로 가능 | 샤드 간 조인·트랜잭션 어려움 |
| 확장성 | 제한적(단일 서버 리소스 한계) | 이론상 무제한(서버 추가) |
| 대표 사례 | 월별 로그 테이블 파티션 | 대형 서비스의 user_id 기준 DB 분산 |

## 6. 예상 면접 질문
**Q1. "RDB와 NoSQL을 어떻게 선택하나요?"**
> 관계가 복잡하고 강한 트랜잭션 정합성이 필요하면 RDB, 스키마가 유연하고 대용량 수평 확장이 중요하면 NoSQL을 씁니다. 실무에선 RDB를 주력으로 두고 Redis를 캐시로 함께 쓰는 경우가 많습니다.

**Q2. "Redis를 어디에 쓰나요?"**
> 메모리 기반이라 캐시가 대표적이고, 세션 저장, 실시간 랭킹(Sorted Set), 분산 락, 큐 등에 씁니다. 캐시 사용 시엔 무효화 전략과 TTL 설계가 중요합니다.

**Q3. "파티셔닝과 샤딩의 차이는?"**
> 파티셔닝은 한 DB 인스턴스 안에서 행(수평) 또는 컬럼(수직) 기준으로 테이블을 나누는 것이고, 샤딩은 데이터를 여러 개의 독립된 DB 서버로 분산하는 것입니다. 샤딩은 확장성은 훨씬 크지만 샤드 간 조인·트랜잭션이 어려워 설계 복잡도가 큽니다.

**Q4. (시나리오) "조회가 느려서 Redis 캐시를 붙였는데 가끔 옛날 데이터가 보입니다. 어떻게 접근하나요?"**
> 캐시 정합성 문제이므로 먼저 무효화 전략을 봅니다. Look-Aside에서 DB를 UPDATE한 뒤 캐시를 지우는데, "DB 쓰기 → 캐시 삭제" 사이에 다른 요청이 옛 값을 다시 캐시에 채우면 stale이 남습니다. 실무 대응은 TTL을 짧게 둬 최악의 노출 시간을 제한하고, 쓰기 시 캐시를 갱신이 아니라 삭제(delete-on-write)해 다음 조회가 DB에서 새로 채우게 하며, 강한 정합성이 필요하면 캐시를 안 쓰고 DB로 직접 가는 예외 경로를 둡니다. 그리고 캐시가 동시에 만료돼 DB로 요청이 몰리는 Cache Stampede는 뮤텍스/짧은 락이나 TTL 지터로 완화합니다. 핵심은 "캐시는 정합성을 조금 포기하고 성능을 얻는 트레이드오프"임을 전제로 허용 가능한 stale 범위를 먼저 정하는 것입니다.

# D6. B-Tree / B+Tree 인덱스 구조

**학습 목표**: *"인덱스는 왜 하필 B+Tree인가요? 해시나 이진트리는 왜 안 쓰나요?"* 에 5분간 답할 수 있다.

## 1. 비유 — 도서관 서가 + 맨 아래칸 연결통로
책 색인(D1)이 "인덱스가 왜 필요한가"였다면, B+Tree는 "그 색인을 어떤 모양으로 쌓는가"이다. 도서관을 떠올리면: 위층 안내판(상위 노드)은 "어느 서가로 갈지"만 알려주고 **실제 책은 맨 아래칸(리프)에만** 꽂혀 있다. 게다가 맨 아래칸끼리 **옆으로 통로가 뚫려 있어서**(리프 링크드리스트), 한 번 시작점을 찾으면 옆으로 쭉 걸어가며 범위를 훑을 수 있다.

## 2. 개념 정의
> **B+Tree** = 균형 잡힌 다진 트리(한 노드가 자식을 수백 개 가짐)로, **내부 노드는 탐색 키(라우팅)만**, **리프 노드에만 실제 데이터(또는 데이터 포인터)**를 두고, **리프끼리 정렬된 순서로 링크드리스트 연결**한 자료구조.

- **B-Tree vs B+Tree**: 순수 B-Tree는 내부 노드에도 데이터를 저장한다. B+Tree는 데이터를 리프로 몰아넣어 ① 내부 노드가 더 많은 키를 담아 트리가 낮아지고(디스크 I/O↓), ② 리프 연결 덕분에 범위·정렬 스캔이 빠르다. 그래서 RDB 인덱스는 사실상 전부 B+Tree다.
- **왜 트리 높이가 낮은 게 중요한가**: 인덱스는 디스크(또는 버퍼 풀)에 페이지 단위로 저장되고, 노드 하나를 읽는 게 곧 한 번의 I/O에 가깝다. 한 노드에 수백 개 키가 들어가므로, 수백만~수천만 행도 높이 **3~4단**이면 리프에 닿는다 → 조회가 몇 번의 페이지 접근으로 끝난다.

## 3. 다이어그램 — 왜 이진트리가 아니라 다진트리인가
```
[이진 탐색 트리(BST)]              [B+Tree]
   높이 ≈ log2(N)                    높이 ≈ log_(수백)(N)
   1000만 행 → 높이 ~23               1000만 행 → 높이 3~4
   노드마다 I/O 1번 → 23번 I/O        노드마다 I/O 1번 → 3~4번 I/O

     내부: [ 30 | 60 | 90 ]        ← 라우팅 키만, 데이터 없음
           /     |    |    \
   리프: [10 20 30]→[40 50 60]→[70 80 90]→...
          └──────── 정렬된 링크드리스트로 연결 ────────┘
          범위검색: 시작 리프 찾고 옆으로 스캔
```
- 이진트리는 노드당 자식이 2개뿐이라 데이터가 많아지면 높이가 급격히 커진다 → 그만큼 디스크 접근이 많아 느리다. B+Tree는 **한 노드(=한 페이지)에 키를 최대한 채워** fan-out(가지 수)을 키워서 높이를 억제한다.

## 4. 왜 해시 인덱스가 아니라 B+Tree인가 (비교표)
| 기준 | B+Tree 인덱스 | 해시 인덱스 |
|------|--------------|------------|
| 등가 검색 `=` | O(log N), 충분히 빠름 | O(1), 이론상 더 빠름 |
| 범위 검색 `>`, `BETWEEN` | **가능**(리프 스캔) | **불가능**(해시는 순서 없음) |
| 정렬 `ORDER BY` | **인덱스 순서 그대로 활용** | 불가(재정렬 필요) |
| 왼쪽 접두사 `LIKE 'kakao%'` | 가능 | 불가 |
| 복합 인덱스 부분 활용 | 가능(선두 컬럼) | 불가(키 전체 해시) |

- 결론: 실무 쿼리는 등가 검색만 하는 게 아니라 **범위·정렬·부분일치**가 뒤섞인다. 해시는 `=` 하나만 잘하고 나머지를 다 못 하므로 범용 인덱스로 부적합하다. MySQL InnoDB는 아예 해시 인덱스를 직접 만들 수 없고(메모리 엔진·Adaptive Hash Index 예외), PostgreSQL은 `HASH` 인덱스를 제공하지만 등가 검색 전용이라 쓰임이 제한적이다.

## 5. 핵심 포인트 (자주 하는 실수) ⭐
- 🔴 "B-Tree와 B+Tree를 같은 것으로 뭉뚱그림" — 면접에서 "리프에만 데이터, 리프끼리 연결"이라는 B+Tree 특징을 말하면 한 단계 깊이 있다는 인상을 준다.
- 🔴 "해시가 항상 O(1)이라 더 빠르다" — 등가 검색 한정이고 범위·정렬·부분일치를 못 한다. 인덱스는 범용성이 중요.
- 🟡 리프가 링크드리스트로 연결돼 있다는 점이 **범위 검색·`ORDER BY`·페이징**이 인덱스로 빨라지는 근본 이유다.
- 🟡 트리가 낮은(3~4단) 이유는 노드=페이지에 키를 꽉 채워 fan-out을 키우기 때문 → "인덱스 컬럼이 너무 길면(긴 varchar) 노드당 키 수가 줄어 트리가 깊어지고 인덱스가 커진다"는 것도 여기서 나온다.

**Q1. "왜 인덱스 자료구조로 해시가 아니라 B+Tree를 쓰나요?"**
> 해시는 등가 검색은 O(1)이지만 순서가 없어 범위 검색·정렬·부분일치(`LIKE 'a%'`)를 전혀 못 합니다. 실무 쿼리는 범위와 정렬이 섞여 있어 이 모두를 지원하는 B+Tree가 범용적입니다. B+Tree는 리프에만 데이터를 두고 리프끼리 정렬된 링크드리스트로 연결해 범위 스캔까지 빠릅니다.

**Q2. "B-Tree와 B+Tree의 차이는 뭔가요?"**
> B-Tree는 내부 노드에도 데이터를 저장하고, B+Tree는 데이터를 리프에만 몰아넣고 리프끼리 연결합니다. 그래서 B+Tree는 내부 노드가 키를 더 많이 담아 트리가 낮아지고 디스크 I/O가 줄며, 리프 연결 덕에 범위·정렬 스캔이 빨라 RDB 인덱스는 사실상 전부 B+Tree입니다.

**Q3. (왜) "천만 행짜리 테이블인데 인덱스로 조회가 몇 번의 디스크 접근이면 끝나는 이유는?"**
> B+Tree는 한 노드가 한 페이지고 거기에 키를 수백 개씩 담아 fan-out이 큽니다. 그래서 천만 행이라도 트리 높이가 3~4단이면 리프에 닿고, 노드 하나 읽는 게 대략 I/O 한 번이라 몇 번의 페이지 접근으로 원하는 리프를 찾습니다. 이진트리라면 높이가 20단 이상이라 훨씬 느립니다.

# D7. 클러스터드 vs 논클러스터드 인덱스

**학습 목표**: *"클러스터드 인덱스가 뭐고, 보조 인덱스로 조회하면 실제로 무슨 일이 벌어지나요?"* 에 답할 수 있다.

## 1. 비유 — 사전 자체 vs 사전 뒤 색인
- **클러스터드 인덱스** = 국어사전. 단어(키)가 곧 본문의 물리적 배열 순서다. "가"로 시작하는 단어를 찾으면 그 자리에 뜻(=행 전체)이 바로 있다.
- **논클러스터드(보조) 인덱스** = 사전 뒤 "주제별 찾아보기". "동물이 나오는 단어" 목록에서 페이지 번호(=행 위치)를 보고, 그 페이지로 다시 넘어가야 본문을 본다. 한 번 더 찾아가는 과정이 있다.

## 2. 개념 정의
> **클러스터드 인덱스** = 테이블 데이터 자체가 인덱스 키 순서로 **물리적으로 정렬 저장**된 것. 리프 노드가 곧 데이터 행이다. InnoDB에서는 **PK가 클러스터드 인덱스**이며 테이블당 1개만 존재.
> **논클러스터드(보조/secondary) 인덱스** = 키와 "행을 찾아가는 포인터"만 담은 별도 구조. 테이블당 여러 개 가능.

- **InnoDB의 핵심 규칙**: 보조 인덱스의 리프는 물리 주소가 아니라 **PK 값**을 저장한다. 즉 보조 인덱스로 찾으면 PK를 얻고, 그 PK로 **클러스터드 인덱스를 한 번 더 타고 내려가** 실제 행을 읽는다. 이 2단계 조회를 **북마크 룩업(bookmark lookup)** 또는 흔히 **테이블 되찾기**라 부른다.

## 3. 다이어그램 — 보조 인덱스 조회의 2단계
```
SELECT * FROM users WHERE email = 'a@x.com';   -- email에 보조 인덱스

[1] 보조 인덱스(email)                [2] 클러스터드 인덱스(PK=id)
   ... 'a@x.com' → id=42  ──────────►   ... id=42 → (id, email, name, ...) 실제 행
        (키 + PK값)                          (리프 = 데이터 전체)
   = 인덱스에서 PK 획득               = PK로 다시 찾아가 행 전체 읽음  (되찾기)
```
- 만약 `SELECT id, email`처럼 **필요 컬럼이 보조 인덱스 안에 다 있으면** [2]단계를 생략(커버링 인덱스, `Using index`). `SELECT *`면 `name` 등 때문에 [2]단계가 반드시 발생.

## 4. 비교표
| 구분 | 클러스터드 인덱스 | 논클러스터드(보조) 인덱스 |
|------|------------------|--------------------------|
| 저장 방식 | 데이터 행 자체가 키 순 물리 정렬(리프=행) | 키 + PK값만 별도 저장 |
| 개수 | 테이블당 1개(InnoDB=PK) | 여러 개 가능 |
| 조회 경로 | 인덱스 = 곧 데이터, 1단계 | PK 얻고 클러스터드 재조회, 2단계(되찾기) |
| 범위 검색 | 매우 빠름(물리 인접) | 되찾기 랜덤 I/O로 상대적 느림 |
| PK 값이 크면 | — | 모든 보조 인덱스가 PK를 품어 전부 비대해짐 |

## 5. 핵심 포인트 (자주 하는 실수) ⭐
- 🔴 "보조 인덱스는 행의 물리 주소를 가리킨다" — InnoDB에선 **틀림**. PK 값을 가리킨다(그래서 되찾기가 생기고, PK 설계가 모든 보조 인덱스 크기에 영향). (SQL Server의 힙 테이블 등은 RID로 가리키므로 DB마다 다름.)
- 🔴 "PK를 UUID(문자열)나 큰 값으로 잡아도 상관없다" — ❌ InnoDB에선 **모든 보조 인덱스가 PK를 복사**해 담으므로, PK가 크면 인덱스 전체가 커지고 캐시 효율이 떨어진다. 그래서 **작고 단조증가하는 정수 PK(AUTO_INCREMENT)**가 권장된다.
- 🟡 클러스터드는 물리 정렬이라 **PK 범위 검색·정렬**이 매우 빠르다. 반대로 랜덤한 PK(UUIDv4)를 쓰면 INSERT 때 페이지 중간에 끼워넣느라 페이지 분할(split)이 잦아 쓰기 성능·단편화가 나빠진다.
- 🟡 보조 인덱스 조회는 되찾기(랜덤 I/O)가 따라오므로, 자주 함께 읽는 컬럼은 **커버링 인덱스**로 되찾기를 없애면 극적으로 빨라진다.

**Q1. "클러스터드 인덱스와 논클러스터드 인덱스의 차이는?"**
> 클러스터드는 데이터 행 자체가 인덱스 키 순서로 물리 정렬된 것이라 인덱스 리프가 곧 데이터고, 테이블당 하나(InnoDB는 PK)입니다. 논클러스터드는 키와 행을 찾아갈 포인터만 담은 별도 구조로 여러 개 만들 수 있고, InnoDB에선 그 포인터가 PK 값이라 조회 시 PK로 클러스터드를 다시 타는 되찾기가 발생합니다.

**Q2. (왜) "InnoDB에서 PK를 UUID 대신 AUTO_INCREMENT 정수로 쓰라는 이유는?"**
> InnoDB는 모든 보조 인덱스 리프에 PK 값을 복사해 담기 때문에 PK가 크면(UUID 36바이트 등) 보조 인덱스 전체가 비대해져 캐시 효율이 떨어집니다. 또 클러스터드는 PK 순 물리 정렬이라, 랜덤한 UUID는 INSERT마다 페이지 중간 삽입으로 페이지 분할·단편화를 유발합니다. 작고 단조증가하는 정수 PK는 인덱스도 작고 뒤에 순차 추가돼 이런 문제가 없습니다.

**Q3. (시나리오) "보조 인덱스를 탔는데도 조회가 기대만큼 안 빨라요. 왜죠?"**
> 되찾기(bookmark lookup) 때문일 수 있습니다. 보조 인덱스로 PK를 찾은 뒤 실제 행을 읽으려고 클러스터드 인덱스를 다시 타는데, 매칭 행이 많으면 이 되찾기가 랜덤 I/O로 다발하고, 결과 비율이 크면 옵티마이저가 아예 Full Scan을 택하기도 합니다. `SELECT`가 필요로 하는 컬럼을 인덱스에 포함시켜 커버링 인덱스(`Using index`)로 만들면 되찾기를 없앨 수 있고, `SELECT *`를 필요한 컬럼만으로 줄이는 것도 같은 효과입니다.

# D8. 트랜잭션 격리수준과 이상현상

**학습 목표**: *"격리수준 4단계와 각 단계가 막는 이상현상을 이유까지 설명"* 할 수 있다. (D2의 표를 근거·시나리오까지 심화)

## 1. 비유 — 공유 화이트보드를 보는 규칙
여러 사람이 같은 화이트보드를 동시에 보고 쓴다. "남이 **연필로 적다 만(커밋 안 된)** 글씨까지 읽어도 되나?", "내가 아까 읽은 줄을 **다시 봤을 때 값이 바뀌어도** 되나?", "다시 봤을 때 **줄 자체가 새로 생겨도** 되나?" — 이 세 질문에 얼마나 엄격히 "안 된다"고 답하느냐가 격리수준이다.

## 2. 개념 정의 — 3대 이상현상
> - **Dirty Read**: 아직 **커밋되지 않은** 다른 트랜잭션의 변경을 읽음. 그쪽이 롤백하면 나는 존재한 적 없는 값을 읽은 셈.
> - **Non-Repeatable Read**: 같은 **행**을 두 번 읽었는데 사이에 다른 트랜잭션이 **UPDATE 커밋**해 값이 달라짐.
> - **Phantom Read**: 같은 **조건**으로 두 번 조회했는데 사이에 다른 트랜잭션이 **INSERT/DELETE 커밋**해 **행 개수(집합)**가 달라짐.

- Non-Repeatable과 Phantom의 차이 포인트: 전자는 "**이미 있던 행의 값**"이 바뀌는 것, 후자는 "**조건에 맞는 행의 집합**"이 바뀌는 것. 그래서 팬텀은 행 단위 락만으로는 못 막고 범위(갭) 락이 필요하다.

## 3. 비교표 — 격리수준별 허용/차단
| 격리수준 | Dirty | Non-Repeatable | Phantom | 실현 방식(개념) |
|----------|-------|----------------|---------|----------------|
| READ UNCOMMITTED | 발생 | 발생 | 발생 | 사실상 락·스냅샷 없이 읽음 |
| READ COMMITTED | 방지 | 발생 | 발생 | 매 SELECT마다 최신 커밋 스냅샷 |
| REPEATABLE READ | 방지 | 방지 | 발생*(표준)/방지(InnoDB) | 트랜잭션 시작 스냅샷 고정 |
| SERIALIZABLE | 방지 | 방지 | 방지 | 사실상 직렬 실행(강한 락/범위 락) |
> *ANSI 표준상 REPEATABLE READ는 팬텀을 허용하지만, **MySQL InnoDB는 MVCC + 넥스트키락(갭락)**으로 REPEATABLE READ에서도 팬텀을 상당 부분 막는다. PostgreSQL의 REPEATABLE READ(=스냅샷 격리)도 일반 조회 기준 팬텀이 안 보인다. "표준과 실제 DB 동작이 다르다"가 면접 포인트.

## 4. 핵심 포인트 (자주 하는 실수) ⭐
- 🔴 "격리수준만 올리면 모든 동시성 버그가 사라진다" — ❌ Lost Update(둘 다 읽고 각자 덮어쓰기)는 RC/RR에서도 날 수 있어 `FOR UPDATE`나 원자적 UPDATE가 필요(D2 Q6). 격리수준은 "읽기 일관성"을 다루지, 애플리케이션의 read-modify-write 경합을 자동으로 막지 않는다.
- 🔴 "REPEATABLE READ가 MySQL 기본이니 다 그렇다" — ❌ Oracle·PostgreSQL·SQL Server 기본은 READ COMMITTED. DB마다 다르다.
- 🟡 SERIALIZABLE은 정확하지만 사실상 직렬 실행이라 처리량이 급감 → 대부분의 서비스는 RC 또는 RR로 두고, 정말 필요한 소수 트랜잭션만 국소적으로 `FOR UPDATE` 등을 쓴다.
- 🟡 격리수준을 낮출수록 성능↑·이상현상↑ 라는 **트레이드오프 축**을 먼저 말하고, 그 위에서 우리 서비스가 감수 가능한 이상현상으로 고른다고 답하면 좋다.

**Q1. "격리수준 4단계와 각각이 막는 이상현상을 설명해주세요."**
> READ UNCOMMITTED는 아무것도 안 막아 Dirty Read까지 납니다. READ COMMITTED는 커밋된 값만 읽어 Dirty를 막지만 매 조회가 최신 스냅샷이라 Non-Repeatable은 납니다. REPEATABLE READ는 트랜잭션 시작 스냅샷을 고정해 Non-Repeatable까지 막고(표준상 팬텀은 허용), SERIALIZABLE은 사실상 직렬 실행으로 팬텀까지 전부 막지만 가장 느립니다.

**Q2. (왜) "Non-Repeatable Read와 Phantom Read는 뭐가 다른가요?"**
> Non-Repeatable은 이미 있던 특정 행을 다시 읽었을 때 그 행의 값이 바뀐 것이고, Phantom은 같은 조건으로 다시 조회했을 때 조건에 맞는 행의 개수(집합)가 바뀐 것입니다. 전자는 행 하나에 락을 걸면 막히지만, 후자는 아직 존재하지 않는 행의 INSERT를 막아야 해서 행 락으로는 부족하고 범위를 잠그는 갭락이 필요합니다.

**Q3. (왜/트레이드오프) "그럼 항상 SERIALIZABLE로 두면 안전하지 않나요?"**
> 안전하지만 실무에선 안 그렇게 합니다. SERIALIZABLE은 사실상 트랜잭션을 직렬 실행시켜 동시성이 급감하고, 락 경합·대기·데드락이 늘어 처리량이 크게 떨어집니다. 격리수준은 정확성과 성능의 트레이드오프라, 대부분 RC나 RR을 기본으로 두고 정말 강한 일관성이 필요한 소수 구간만 `FOR UPDATE` 같은 국소적 잠금으로 올립니다.

**Q4. (지식 검증) "MySQL InnoDB는 REPEATABLE READ인데 왜 팬텀이 거의 안 나나요?"**
> 표준 REPEATABLE READ는 팬텀을 허용하지만, InnoDB는 MVCC로 일반 SELECT의 스냅샷을 고정하고, `FOR UPDATE`처럼 잠금 읽기가 걸린 범위 조회에는 레코드 사이 간격까지 잠그는 넥스트키락(갭락)을 걸어 다른 트랜잭션의 INSERT를 막습니다. 그래서 실제로는 팬텀이 상당 부분 방지됩니다. 표준 정의와 특정 DB의 실제 구현이 다르다는 점이 포인트입니다.

# D9. MVCC와 잠금

**학습 목표**: *"MVCC와 락은 어떤 관계인가요? 언제 스냅샷을 보고 언제 락을 잡나요?"* 에 답할 수 있다. (D3의 구현 상세를 "락과의 관계·DB 차이" 축으로 재정리)

## 1. 비유 — 사진 vs 현장 통제
- **MVCC(스냅샷 읽기)** = 트랜잭션 시작 순간 현장을 **사진 찍어두고** 그 사진만 본다. 남이 현장을 바꿔도 내 사진은 안 바뀐다 → 읽기끼리, 읽기-쓰기끼리 서로 안 막는다.
- **락(잠금 읽기/쓰기)** = 사진이 아니라 **현장을 직접 통제**한다. "이 구역 내가 쓸 거니 아무도 못 들어와" → 정확하지만 남을 기다리게 만든다.
- 핵심: MVCC는 "**읽기**를 락 없이 일관되게" 해주고, 실제 **쓰기·잠금 읽기**의 상호배제는 여전히 **락**이 담당한다. 둘은 대체재가 아니라 **역할 분담**이다.

## 2. 개념 정의
> **MVCC(다중 버전 동시성 제어)** = 한 행의 여러 버전을 유지해, 읽기는 자기 시점 스냅샷(버전)을 보고 쓰기는 새 버전을 만들게 함으로써 **읽기와 쓰기가 서로 블로킹하지 않게** 하는 기법.

- **스냅샷 읽기(consistent read)**: 일반 `SELECT`. 락을 안 잡고 언두 로그로 만든 read view의 버전을 읽음.
- **잠금 읽기(current read)**: `SELECT ... FOR UPDATE` / `LOCK IN SHARE MODE`, 그리고 `UPDATE`/`DELETE`. 최신 커밋본을 읽고 그 행(및 범위)에 락을 건다.
- undo log: 변경 전 이미지를 보관해 ① 다른 트랜잭션에 옛 버전을 보여주고(MVCC) ② 롤백에도 쓴다(Atomicity). 오래된 버전은 참조자가 없으면 purge된다 → **장시간 열린 트랜잭션은 언두가 계속 쌓여 성능 저하**를 부른다.

## 3. 다이어그램 — 스냅샷 읽기 vs 잠금 읽기
```
트랜잭션 T1 시작 (read view 고정)
   │
   │  SELECT price FROM item WHERE id=1;   → 스냅샷 읽기 (락 X)
   │        ┌───────── 버전 체인 ─────────┐
   │        최신(T9 커밋) → undo(T4) → undo(T1 시점 값)
   │        T1은 자기 시점에 커밋돼 있던 값을 따라가 읽음 → 반복해도 동일
   │
   │  SELECT ... FOR UPDATE;                → 잠금 읽기 (최신본 + X락/넥스트키락)
   │        다른 트랜잭션은 이 행/범위 쓰기를 기다려야 함
   ▼
```

## 4. 비교표 — MVCC와 락의 역할, DB 차이
| 항목 | MVCC(스냅샷 읽기) | 락(잠금 읽기/쓰기) |
|------|------------------|-------------------|
| 대상 | 일반 SELECT | `FOR UPDATE`, `UPDATE`, `DELETE` |
| 블로킹 | 읽기-쓰기 서로 안 막음 | 쓰기끼리·잠금읽기끼리 대기 |
| 보는 값 | 내 시점 스냅샷(과거 가능) | 최신 커밋본 |
| 막는 것 | Dirty·Non-Repeatable(일관 읽기) | Lost Update·팬텀(범위 락) |

| MVCC 구현 차이 | MySQL InnoDB | PostgreSQL |
|----------------|--------------|------------|
| 옛 버전 저장 위치 | **undo log**(테이블스페이스 분리) | **행 자체에 여러 버전(tuple) 누적** |
| 정리 방식 | purge 스레드 | **VACUUM**(죽은 튜플 회수) |
| 부작용 | 롱 트랜잭션 시 undo 팽창 | 죽은 튜플 누적 → 테이블 팽창(bloat), VACUUM 필요 |
| 기본 격리수준 | REPEATABLE READ | READ COMMITTED |

## 5. 핵심 포인트 (자주 하는 실수) ⭐
- 🔴 "MVCC가 있으니 락이 필요 없다" — ❌ MVCC는 **읽기 일관성**만 담당한다. 쓰기 충돌·Lost Update·팬텀 방지는 여전히 **락**의 몫. 둘은 역할 분담.
- 🔴 "일반 SELECT도 최신 값을 본다" — ❌ REPEATABLE READ에서 일반 SELECT는 트랜잭션 시작 시점 스냅샷을 본다. 최신 값이 꼭 필요하면 `FOR UPDATE`(잠금 읽기)를 써야 한다.
- 🟡 InnoDB는 옛 버전을 undo log에, PostgreSQL은 행에 직접 쌓는다 → PostgreSQL은 **VACUUM/오토배큠**으로 죽은 튜플을 정리해야 하고 안 하면 테이블이 팽창(bloat)한다. InnoDB는 롱 트랜잭션 시 **undo(히스토리 리스트) 팽창**이 문제.
- 🟡 그래서 두 DB 공통으로 **트랜잭션을 짧게** 유지하라는 조언이 나온다(오래 열면 옛 버전을 못 지워 성능 저하).

**Q1. "MVCC와 락은 어떤 관계인가요? MVCC가 있으면 락은 필요 없나요?"**
> MVCC는 읽기 일관성을 담당합니다. 각 트랜잭션이 자기 시점 스냅샷을 봐서 읽기와 쓰기가 서로 막지 않게 하죠. 하지만 쓰기 충돌, Lost Update, 팬텀 방지는 여전히 락이 담당합니다. 일반 SELECT는 스냅샷(락 없음), `FOR UPDATE`나 UPDATE는 최신본을 읽고 락을 거는 식으로 둘이 역할을 나눠 갖습니다.

**Q2. (지식 검증) "MySQL과 PostgreSQL의 MVCC 구현 차이를 아나요?"**
> InnoDB는 변경 전 이미지를 별도 undo log에 남겨 버전 체인을 만들고 purge 스레드가 정리하는데, 롱 트랜잭션이 있으면 undo가 계속 쌓입니다. PostgreSQL은 옛 버전을 행 자체에 여러 튜플로 누적하고 VACUUM(오토배큠)이 죽은 튜플을 회수하는데, 안 돌면 테이블이 팽창(bloat)합니다. 기본 격리수준도 InnoDB는 REPEATABLE READ, PostgreSQL은 READ COMMITTED로 다릅니다.

**Q3. (왜) "왜 장시간 열린 트랜잭션을 피하라고 하나요? MVCC랑 무슨 상관인가요?"**
> MVCC는 옛 버전을 참조하는 트랜잭션이 하나도 없어야 그 버전을 정리(InnoDB purge / PostgreSQL VACUUM)할 수 있습니다. 오래 열린 트랜잭션이 있으면 그 시점 스냅샷을 살려두느라 옛 버전을 못 지워, InnoDB는 undo가 팽창하고 PostgreSQL은 죽은 튜플이 쌓여 테이블이 커지고 조회가 느려집니다. 그래서 트랜잭션을 짧게 유지하는 게 MVCC 성능 유지의 핵심입니다.

**Q4. (시나리오) "방금 다른 세션이 UPDATE 후 커밋했는데 내 트랜잭션에선 여전히 옛 값이 보입니다. 버그인가요?"**
> 버그가 아니라 REPEATABLE READ + MVCC의 정상 동작입니다. 제 트랜잭션은 시작 시점 스냅샷을 고정해 보기 때문에, 그 사이 커밋된 변경은 일반 SELECT에 안 보입니다. 최신 값을 꼭 봐야 한다면 `SELECT ... FOR UPDATE`로 잠금 읽기(current read)를 하거나, 트랜잭션을 새로 시작하면 됩니다.

# 핵심 질문 (Quiz)

> 답변을 먼저 떠올린 뒤 펼쳐서 확인하세요.

<details>
<summary>Q1. 인덱스는 왜 빠르고, 단점은 무엇인가요?</summary>

- 정렬된 **B-Tree(B+Tree)** 자료구조로 저장 → 검색이 O(N) → O(log N)로 줄어듦
- 단점: **쓰기(INSERT/UPDATE/DELETE)마다 인덱스도 같이 갱신**되어야 해서 쓰기 성능↓, 저장공간↑
- 결론: 조회가 잦은 컬럼에 **선별적으로** 걸어야 함 (읽기-쓰기 균형)

</details>

<details>
<summary>Q2. 쿼리가 인덱스를 타는지 어떻게 확인하고, 안 타는 경우는 언제인가요?</summary>

- 확인: `EXPLAIN`으로 실행 계획 확인
  - `type`: `ALL`(Full Scan, 경고) vs `ref`/`range`/`const`(인덱스 사용)
  - `key`: 실제 사용된 인덱스 (NULL이면 미사용)
  - `Extra`: `Using index`(커버링 인덱스) vs `Using filesort`/`Using temporary`(튜닝 필요)
- 인덱스를 안 타는 경우
  - 인덱스 컬럼에 **함수/연산** 적용 (`WHERE YEAR(created)=2024`)
  - 복합 인덱스에서 **왼쪽 접두사 규칙**을 건너뜀
  - **카디널리티가 너무 낮음** (예: 성별)
  - 조건에 맞는 행 비율이 커서 옵티마이저가 Full Scan이 더 싸다고 판단
  - 통계 정보가 오래됨, 암묵적 형변환

</details>

<details>
<summary>Q3. ACID를 설명해주세요.</summary>

| 속성 | 의미 |
|------|------|
| Atomicity(원자성) | 전부 성공 or 전부 실패(롤백) |
| Consistency(일관성) | 트랜잭션 전후 무결성 규칙 유지 |
| Isolation(격리성) | 동시 트랜잭션이 서로 간섭 안 함 |
| Durability(지속성) | 커밋되면 장애에도 보존(로그) |

- 계좌 이체처럼 여러 작업을 하나로 묶어 안전하게 처리하기 위한 성질

</details>

<details>
<summary>Q4. 격리수준과 이상현상(Dirty/Non-Repeatable/Phantom Read)의 관계는?</summary>

- 격리수준을 **낮출수록 성능↑, 이상현상↑**
- Dirty Read: 커밋 안 된 값을 읽음
- Non-Repeatable Read: 같은 행 재조회 시 값이 바뀜
- Phantom Read: 같은 조건 재조회 시 행 개수가 바뀜

| 격리수준 | Dirty | Non-Repeatable | Phantom |
|----------|-------|----------------|---------|
| READ UNCOMMITTED | 발생 | 발생 | 발생 |
| READ COMMITTED | 방지 | 발생 | 발생 |
| REPEATABLE READ (MySQL 기본) | 방지 | 방지 | 발생* |
| SERIALIZABLE | 방지 | 방지 | 방지 |

- 무조건 SERIALIZABLE은 성능 최악 → 성능 vs 정확성 트레이드오프

</details>

<details>
<summary>Q5. 낙관적 락과 비관적 락의 차이는?</summary>

- **비관적 락**: 미리 DB 락을 걸어 충돌을 원천 차단 → 충돌이 잦을 때 적합, 데드락 위험
- **낙관적 락**: `version` 컬럼으로 저장 시점에 충돌 감지 후 재시도 → 충돌이 드문 웹서비스에 유리
- JPA: 비관 = `@Lock(PESSIMISTIC_WRITE)`, 낙관 = `@Version`

</details>

<details>
<summary>Q6. MVCC는 실제로 어떻게 구현되나요? (InnoDB 기준)</summary>

- `UPDATE` 시 원본 행을 덮어쓰기 전에 **변경 전 이미지를 undo log에 기록**하고 새 행이 그 로그를 가리키게 함 → 여러 버전이 **버전 체인**으로 연결
- 읽기 트랜잭션은 시작 시점의 **read view(스냅샷)** 기준으로 체인을 따라가며 자신의 스냅샷에 맞는 버전을 찾음 → Non-Repeatable Read 방지
- 오래된 버전은 참조하는 트랜잭션이 없어지면 **purge 스레드**가 정리 (장시간 트랜잭션 금지의 이유)
- 단, 쓰기가 섞인 범위 조회(`SELECT ... FOR UPDATE` 등)는 MVCC만으로 부족해 **갭락/Next-Key Lock**으로 팬텀을 추가 방지

</details>

<details>
<summary>Q7. 정규화의 목적과 반정규화하는 이유는?</summary>

- 정규화: 중복 제거 → **갱신 이상 방지**, 무결성 유지 (1NF 원자값 → 2NF 부분 함수 종속 제거 → 3NF 이행 종속 제거)
- 부작용: 조인이 많아져 읽기 성능↓
- 반정규화: **읽기 병목이 실측으로 확인된 경우에만** 선택적으로 중복 허용해 조인↓, 조회 성능↑
- 주의: 조인이 많다고 바로 반정규화 X — 인덱스/쿼리 튜닝이 먼저

</details>

<details>
<summary>Q8. 파티셔닝과 샤딩의 차이는?</summary>

| | 파티셔닝 | 샤딩 |
|--|---------|------|
| 범위 | 한 DB 인스턴스 내부 | 여러 개의 독립 DB 서버 |
| 애플리케이션 인지 | 대체로 투명 | 샤드 키 라우팅 로직 필요 |
| 조인/트랜잭션 | 그대로 가능 | 샤드 간 조인·트랜잭션 어려움 |
| 확장성 | 제한적(단일 서버 리소스 한계) | 이론상 무제한(서버 추가) |

- 파티셔닝: 행 기준(수평, 예: 월별) 또는 컬럼 기준(수직, 예: TEXT/BLOB 분리)
- 샤딩: 샤드 키(예: `user_id % N`)로 저장 서버 결정, 확장성은 크지만 설계 복잡도 크게 증가

</details>

<details>
<summary>Q9. 인덱스 자료구조로 왜 해시가 아니라 B+Tree를 쓰나요?</summary>

- 해시: 등가(`=`) 검색만 O(1), **순서가 없어** 범위·정렬·부분일치(`LIKE 'a%'`)를 못 함
- B+Tree: **리프에만 데이터**를 두고 **리프끼리 정렬된 링크드리스트로 연결** → 범위·정렬 스캔까지 빠름
- 실무 쿼리는 범위·정렬이 섞여 있어 범용성 있는 B+Tree가 적합
- B-Tree와 차이: B-Tree는 내부 노드에도 데이터 저장, B+Tree는 리프에 몰아넣어 트리가 낮아지고(I/O↓) 범위 스캔이 빠름

</details>

<details>
<summary>Q10. 천만 행 테이블도 인덱스 조회가 몇 번의 I/O로 끝나는 이유는?</summary>

- B+Tree는 **한 노드 = 한 페이지**에 키를 수백 개 담아 fan-out(가지 수)이 큼
- 그래서 천만 행이라도 트리 **높이가 3~4단** → 노드 하나 읽기 ≈ I/O 1번 → 몇 번의 페이지 접근으로 리프 도달
- 이진트리라면 높이가 20단 이상이라 훨씬 느림 → "다진트리로 높이를 억제"하는 게 핵심

</details>

<details>
<summary>Q11. InnoDB 클러스터드/논클러스터드 인덱스와, 보조 인덱스 조회의 되찾기(bookmark lookup)란?</summary>

- **클러스터드 인덱스** = 데이터 행 자체가 PK 순으로 물리 정렬(리프=행), 테이블당 1개(InnoDB=PK)
- **논클러스터드(보조)** = 키 + **PK 값**을 담은 별도 구조, 여러 개 가능
- 보조 인덱스로 조회 → PK 획득 → **PK로 클러스터드를 다시 타 실제 행 읽음(되찾기, 2단계·랜덤 I/O)**
- 필요한 컬럼이 인덱스에 다 있으면 되찾기 생략(**커버링 인덱스**, `Using index`)

</details>

<details>
<summary>Q12. InnoDB에서 PK를 UUID 대신 AUTO_INCREMENT 정수로 권장하는 이유는?</summary>

- **모든 보조 인덱스가 PK 값을 복사**해 담음 → PK가 크면(UUID 36B) 보조 인덱스 전체가 비대, 캐시 효율↓
- 클러스터드는 PK 순 물리 정렬 → 랜덤 UUID는 INSERT마다 **페이지 중간 삽입 → 페이지 분할·단편화**
- 작고 단조증가하는 정수 PK: 인덱스 작고 뒤에 순차 추가돼 분할이 거의 없음

</details>

<details>
<summary>Q13. Non-Repeatable Read와 Phantom Read의 차이는?</summary>

- **Non-Repeatable**: 이미 있던 **특정 행의 값**이 재조회 시 바뀜(다른 트랜잭션 UPDATE 커밋) → 행 락으로 방지 가능
- **Phantom**: 같은 **조건에 맞는 행의 집합(개수)**이 바뀜(INSERT/DELETE 커밋) → 아직 없는 행의 삽입을 막아야 하므로 **범위(갭) 락** 필요
- 격리수준: RC는 둘 다 허용, RR은 Non-Repeatable 방지(표준상 팬텀 허용, InnoDB는 갭락으로 대부분 방지), SERIALIZABLE은 전부 방지

</details>

<details>
<summary>Q14. 격리수준만 올리면 모든 동시성 버그가 사라지나요? (Lost Update)</summary>

- ❌ 아님. 격리수준은 **읽기 일관성**을 다루지, read-modify-write 경합을 자동으로 막지 않음
- 예: 두 트랜잭션이 재고 "1개 남음"을 각자 읽고 각자 차감 → **Lost Update**(초과 판매). RC/RR에서도 발생
- 해결: `SELECT ... FOR UPDATE`(비관), 조건부 UPDATE(`WHERE qty>0`, 원자적), `@Version`(낙관적 재시도)

</details>

<details>
<summary>Q15. MVCC와 락은 어떤 관계인가요? MVCC가 있으면 락은 필요 없나요?</summary>

- MVCC = **읽기 일관성** 담당: 각 트랜잭션이 자기 시점 스냅샷을 봐 읽기-쓰기가 서로 안 막음
- 락 = **쓰기 충돌·Lost Update·팬텀 방지** 담당 → 둘은 대체재가 아니라 **역할 분담**
- **스냅샷 읽기**(일반 SELECT, 락 X) vs **잠금 읽기**(`FOR UPDATE`/`UPDATE`, 최신본 + 락) 구분이 핵심

</details>

<details>
<summary>Q16. MySQL InnoDB와 PostgreSQL의 MVCC 구현 차이는?</summary>

| | InnoDB | PostgreSQL |
|--|--------|-----------|
| 옛 버전 저장 | **undo log**(분리 저장) | **행에 여러 튜플 누적** |
| 정리 | purge 스레드 | **VACUUM/오토배큠** |
| 부작용 | 롱 트랜잭션 시 undo 팽창 | 죽은 튜플 누적 → 테이블 bloat |
| 기본 격리수준 | REPEATABLE READ | READ COMMITTED |

- 공통 교훈: **트랜잭션을 짧게** — 오래 열면 옛 버전을 못 지워(참조 중) 성능 저하

</details>
