# D1. Database Index

**학습 목표**: *"인덱스가 뭐고 왜 빠른가요? 단점은?"* 에 답할 수 있다.

## 1. 비유 — 책 색인
500쪽 책에서 "트랜잭션" 단어 찾기: 처음부터 다 읽으면(Full Scan) 느림. 책 뒤 **색인(index)**에서 페이지 번호 보고 바로 펴면 빠름. DB 인덱스가 바로 이 색인.

## 2. 개념 + 자료구조
> **인덱스** = 특정 컬럼 값을 **정렬된 자료구조(B-Tree)** 로 별도 저장해, 검색을 O(N)→O(log N)으로 줄이는 것.
- 대부분 RDB 인덱스는 **B-Tree(정확히는 B+Tree)**: 정렬 유지, 범위 검색·정렬에 강함.
- 해시 인덱스는 등가 검색(`=`)만 O(1), 범위 검색 불가.

## 3. 다이어그램 — B+Tree
```
              [ 30 | 60 ]              ← 루트(중간 노드는 탐색용)
             /     |     \
      [10|20] [30|40|50] [60|70]       ← 리프에 실제 키(정렬)
        │  리프끼리 링크드리스트로 연결 → 범위 검색 빠름
```

## 4. 클러스터드 vs 논클러스터드
| 구분 | 클러스터드 | 논클러스터드(보조) |
|------|-----------|-------------------|
| 정렬 | 데이터 자체가 인덱스 순 정렬 | 별도 인덱스 + 실제행 포인터 |
| 개수 | 테이블당 1개(보통 PK) | 여러 개 |
| 속도 | 범위검색 매우 빠름 | 한 단계 더(포인터 조회) |

## 5. 핵심 포인트 (자주 하는 실수) ⭐
- 🔴 "인덱스 많을수록 좋다" — ❌ **쓰기(INSERT/UPDATE/DELETE) 때마다 인덱스도 갱신** → 쓰기 느려지고 저장공간↑. 읽기-쓰기 균형이 관건.
- 🔴 인덱스 컬럼에 **함수/연산** 쓰면 인덱스 못 탐: `WHERE YEAR(created)=2024` ❌ → `WHERE created >= '2024-01-01'` ✅.
- 🟡 **복합 인덱스는 순서가 중요**: `(a, b)` 인덱스는 `a` 또는 `a,b` 검색엔 타지만 `b` 단독은 못 탐(왼쪽 접두사 규칙).
- 🟡 **커버링 인덱스**: 쿼리가 필요한 컬럼이 인덱스에 다 있으면 테이블 접근 없이 인덱스만으로 응답 → 매우 빠름.
- 🟡 카디널리티 낮은 컬럼(예: 성별)은 인덱스 효과 적음.

## 6. 예상 면접 질문
**Q1. "인덱스는 왜 빠르고, 단점은?"**
> 정렬된 B-Tree로 검색을 O(log N)으로 줄여 조회가 빠릅니다. 단 쓰기마다 인덱스를 갱신해야 해 쓰기 성능이 떨어지고 저장공간을 더 쓰므로, 조회가 잦은 컬럼에 선별적으로 걸어야 합니다.

**Q2. "인덱스를 걸었는데 안 타는 경우는?"**
> 인덱스 컬럼에 함수·연산을 적용하거나, 복합 인덱스에서 왼쪽 접두사를 건너뛰거나, 카디널리티가 너무 낮거나, 옵티마이저가 Full Scan이 더 싸다고 판단할 때입니다.

# D2. Transaction and ACID

**학습 목표**: *"트랜잭션 ACID, 격리수준과 이상현상?"* 에 답할 수 있다. (Spring @Transactional과 직결)

## 1. 비유 — 계좌 이체
A→B 5만원 이체 = "A에서 차감" + "B에 입금" **두 작업이 하나로 묶여** 전부 되거나 전부 안 되어야. 중간에 죽으면 롤백. 이 묶음이 **트랜잭션**.

## 2. ACID
| 속성 | 의미 |
|------|------|
| **Atomicity(원자성)** | 전부 성공 or 전부 실패(롤백) |
| **Consistency(일관성)** | 트랜잭션 전후 무결성 규칙 유지 |
| **Isolation(격리성)** | 동시 트랜잭션이 서로 간섭 안 함 |
| **Durability(지속성)** | 커밋되면 장애에도 보존(로그) |

## 3. 격리 수준과 이상현상 ⭐
격리를 낮출수록 성능↑·이상현상↑. 이상현상 3종:
- **Dirty Read**: 커밋 안 된 값을 읽음
- **Non-Repeatable Read**: 같은 행을 두 번 읽었는데 값이 달라짐(중간에 UPDATE 커밋)
- **Phantom Read**: 같은 조건 재조회 시 행 개수가 달라짐(중간에 INSERT)

| 격리수준 | Dirty | Non-Repeatable | Phantom |
|----------|-------|----------------|---------|
| READ UNCOMMITTED | 발생 | 발생 | 발생 |
| READ COMMITTED (많은 DB 기본) | 방지 | 발생 | 발생 |
| REPEATABLE READ (MySQL 기본) | 방지 | 방지 | 발생* |
| SERIALIZABLE | 방지 | 방지 | 방지 |
> *MySQL InnoDB는 MVCC+갭락으로 REPEATABLE READ에서 팬텀도 상당부분 방지.

## 4. 핵심 포인트
- 🟡 격리수준은 **동시성 성능 vs 정확성**의 트레이드오프. 무조건 SERIALIZABLE은 성능 최악.
- 🟡 Spring `@Transactional(isolation=...)`로 지정 가능하지만 보통 DB 기본값 사용.

## 5. 예상 면접 질문
**Q1. "ACID를 설명해주세요."**
> 원자성(전부 or 전무), 일관성(무결성 유지), 격리성(동시 트랜잭션 간섭 방지), 지속성(커밋 후 영속). 이체처럼 여러 작업을 하나로 묶어 안전하게 처리하기 위한 성질입니다.

**Q2. "격리수준과 이상현상 관계는?"**
> 낮은 격리일수록 Dirty/Non-Repeatable/Phantom Read가 나타나고 성능은 좋아집니다. READ COMMITTED는 dirty만 막고, REPEATABLE READ는 non-repeatable까지, SERIALIZABLE은 전부 막지만 가장 느립니다.

# D3. Lock and MVCC

**학습 목표**: *"공유/배타 락, 낙관 vs 비관, MVCC?"* 에 답할 수 있다. (OS 동기화와 연결)

## 1. 비유 — 문서 공동 편집
같은 문서를 둘이 고치면 충돌. 해결:
- **비관적 락**: 편집 시작 시 **문서 잠금**(남이 못 건드림). 충돌 잦을 때.
- **낙관적 락**: 일단 각자 편집, 저장 시 **버전 비교**해 충돌이면 재시도. 충돌 드물 때.

## 2. 락 종류
- **공유 락(S, Read)**: 여러 트랜잭션 동시 읽기 허용, 쓰기 차단.
- **배타 락(X, Write)**: 하나만. 읽기·쓰기 모두 차단.

## 3. 낙관 vs 비관
| | 비관적 락 | 낙관적 락 |
|--|----------|----------|
| 방식 | 미리 DB 락 획득 | version 컬럼으로 충돌 감지 |
| 적합 | 충돌 잦음 | 충돌 드묾(웹서비스 다수) |
| 비용 | 락 대기·데드락 위험 | 충돌 시 재시도 |
> JPA: 비관=`@Lock(PESSIMISTIC_WRITE)`, 낙관=`@Version`.

## 4. MVCC (Multi-Version Concurrency Control) ⭐
- 데이터를 **여러 버전**으로 유지 → 읽기는 **스냅샷**을 보고, 쓰기는 새 버전 생성.
- 효과: **읽기가 쓰기를 막지 않고, 쓰기가 읽기를 막지 않음**(락 경합↓). MySQL InnoDB·PostgreSQL의 핵심.

## 5. OS 동기화와의 연결 ⭐ (cross-link)
- DB 락(S/X) = OS의 뮤텍스/세마포어와 같은 **상호배제** 문제(OS O4).
- DB 데드락 4조건도 OS 데드락과 동일 — 락 획득 순서 통일로 예방.

## 6. 예상 면접 질문
**Q1. "낙관적 락과 비관적 락 차이?"**
> 비관적 락은 미리 DB 락을 걸어 충돌을 원천 차단(충돌 잦을 때), 낙관적 락은 version으로 저장 시점에 충돌을 감지해 재시도(충돌 드물 때). 웹서비스는 보통 낙관적 락이 유리합니다.

**Q2. "MVCC가 뭔가요?"**
> 데이터를 다중 버전으로 관리해 읽기는 스냅샷을, 쓰기는 새 버전을 만드는 방식입니다. 읽기와 쓰기가 서로 막지 않아 동시성이 크게 향상됩니다.

# D4. Normalization

**학습 목표**: *"정규화가 뭐고 왜/언제 반정규화하나요?"* 에 답할 수 있다.

## 1. 비유 — 중복 제거
주문 테이블마다 고객 이름·주소를 매번 적으면 고객이 이사할 때 모든 행을 고쳐야(이상현상). 고객 정보를 **별도 테이블로 분리**하고 ID로 참조 = 정규화.

## 2. 정규화 단계
| 단계 | 규칙 |
|------|------|
| **1NF** | 원자값(한 칸에 하나), 반복 그룹 제거 |
| **2NF** | 부분 함수 종속 제거(복합키 일부에만 종속되는 컬럼 분리) |
| **3NF** | 이행 종속 제거(키가 아닌 컬럼에 종속되는 컬럼 분리) |

## 3. 정규화 vs 반정규화
| | 정규화 | 반정규화 |
|--|--------|----------|
| 목적 | 중복 제거·무결성 | 조회 성능(조인↓) |
| 장점 | 갱신 이상 방지, 저장 효율 | 읽기 빠름 |
| 단점 | 조인 많아짐 | 중복·갱신 비용↑ |
> 실무: 정규화 기본, **읽기 병목이 검증되면** 선택적 반정규화(캐시성 컬럼 추가 등).

## 4. 핵심 포인트
- 🟡 반정규화는 "성급한 최적화" 주의 — 측정 후 적용.
- 🟡 조인이 많아 느리다고 무조건 반정규화 X, 인덱스·쿼리 튜닝이 먼저.

## 5. 예상 면접 질문
**Q. "정규화의 목적과 반정규화하는 이유는?"**
> 정규화는 중복을 없애 갱신 이상을 막고 무결성을 지키는 것입니다. 다만 조인이 많아져 읽기가 느려질 수 있어, 읽기 병목이 확인되면 성능을 위해 선택적으로 반정규화(중복 허용)합니다.

# D5. SQL vs NoSQL and Redis

**학습 목표**: *"RDB와 NoSQL 언제 쓰나요? Redis는?"* 에 답할 수 있다.

## 1. 비유 — 엑셀 vs 메모앱
- **RDB** = 엑셀 시트(정해진 열, 관계, 규칙). 정합성 강함.
- **NoSQL** = 자유 형식 메모(스키마 유연, 대량·분산에 강함).

## 2. 비교표
| | RDB (SQL) | NoSQL |
|--|-----------|-------|
| 스키마 | 고정, 정규화 | 유연(문서/키값/컬럼/그래프) |
| 정합성 | 강함(ACID) | 유연(BASE, 결과적 일관성) |
| 확장 | 수직(scale-up) 위주 | 수평(scale-out) 강함 |
| 적합 | 트랜잭션·관계 복잡 | 대용량·유연 스키마·고속 |

## 3. Redis (In-Memory)
- 메모리 기반 key-value 저장소 → **초고속**. 용도: **캐시**, 세션 저장, 랭킹(Sorted Set), 분산 락, 큐.
- **캐시 전략**: Look-Aside(캐시 먼저 조회, 없으면 DB→캐시 저장), Write-Through 등.
- 캐시 이슈: **캐시 무효화**, TTL, Cache Stampede.

## 4. 핵심 포인트
- 🔴 "NoSQL이 항상 빠르고 좋다" — ❌ 트랜잭션·복잡 조인·강한 정합성엔 RDB가 유리. 용도 문제.
- 🟡 대부분 실무는 **RDB 주력 + Redis 캐시** 조합.
- 🟡 CAP: 분산 시스템은 일관성(C)·가용성(A)·분단내성(P) 중 P 하에 C와 A를 트레이드오프.

## 5. 예상 면접 질문
**Q1. "RDB와 NoSQL을 어떻게 선택하나요?"**
> 관계가 복잡하고 강한 트랜잭션 정합성이 필요하면 RDB, 스키마가 유연하고 대용량 수평 확장이 중요하면 NoSQL을 씁니다. 실무에선 RDB를 주력으로 두고 Redis를 캐시로 함께 쓰는 경우가 많습니다.

**Q2. "Redis를 어디에 쓰나요?"**
> 메모리 기반이라 캐시가 대표적이고, 세션 저장, 실시간 랭킹(Sorted Set), 분산 락, 큐 등에 씁니다. 캐시 사용 시엔 무효화 전략과 TTL 설계가 중요합니다.
